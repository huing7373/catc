## 2. 最成功的纪律（Epic 1+ 务必继续）

### 2.1 双 gate 漂移守门

**出处：** Story 0-6（error code registry）首次建立；Story 0-14（WS 消息注册表）完整化为"四步走"。

**机制：**
- **Gate 1 — CI 单元测试**：扫常量集合与 handler / dispatcher 注册集合，不一致则 fail
- **Gate 2 — 启动期 fail-fast**：`initialize.go` 调用 `validate*Consistency(...)`；漂移 `log.Fatal`

**为什么 Epic 1+ 要继续：** 每引入一套全局常量（error code / WS message type / feature flag / cron job name / Redis key prefix / Provider 名字 ……）漂移都是必然，靠人工 review 拦 100% 不可能。四步走纪律（加常量 → 在使用点注册 → 更新文档 → 跑 CI build）把"漏一步就挂"变成硬约束。

### 2.2 Empty* / Noop 占位 + 逐步填实

**出处：** Story 0-12（6 个 Empty Provider：User/Friends/CatState/Skins/Blindboxes/RoomSnapshot）+ Story 0-13（4 个 Noop / Empty：Pusher/TokenProvider/TokenCleaner/QuietHoursResolver）。

**核心约定：**
- Empty 返回**合法 JSON**（`null` / `[]`），**不**返回占位字符串或魔法值
- 消费侧接口在 `internal/*`；实现在 `pkg/*`（`internal/ws` 不能被 `pkg` 引用）
- Epic 1-8 每个 story 只需替换一个 Empty，**消费侧代码不改**

**为什么 Epic 1+ 要继续：** 允许整条消息路径 / 推送路径在骨架阶段就**端到端跑通**，business 代码不用等最末端的 Provider 就绪。Story 1.1 上线第一个真实 UserProvider 时，`session.resume` handler 零改动。

### 2.3 Fail-closed / Fail-open 场景化分类

**出处：** Story 0-11（WS 连接限流 / blacklist）/ Story 0-12（resume cache）/ Story 0-13（APNs Pusher）共同形成的模式。

**分类：**

| 场景 | 策略 | 故障时行为 | 出处 |
|---|---|---|---|
| 防护闸门（blacklist / ratelimit） | **fail-closed** | Redis 挂 → 拒连，防 J4 reconnect storm 重现 | 0-11 |
| 性能缓存（resume cache） | **fail-open** | Redis 挂 → 降级到 Provider，依然可服务 | 0-12 |
| 持久队列（APNs Enqueue） | **fail-closed** | XADD 失败 → 调用方报错，避免 push 丢失 | 0-13 |
| 辅助查询（token / quiet hours） | **fail-open** | 查询失败 → 走默认路径，push 仍发出 | 0-13 |

**关键约束**（与用户 memory `project_backup_fallback.md` 一致）：
- **不是**掩盖根因的 backup/fallback —— **是**按"故障的业务后果"分类的防护策略
- 每种选择都有**可观测点**：healthz / 审计日志 / DLQ，运维能看到"我正在降级"

**为什么 Epic 1+ 要继续：** 用户明确反对"backup/fallback 掩盖架构风险"。0-11/0-12/0-13 的场景化分类提供了判据 —— 这条降级**会不会暴露出去**（healthz dropping / DLQ 积压 / 审计日志）？能就是合法防护，不能就是遮掩。

## 3. 最值得反思（Epic 1+ 要改进）

### 3.1 Review 早启

**观察：** Epic 0 三条 P1 finding（0-14 r1 / 0-15 r1 / 0-15 r2）都是**语义正确性**问题——不是"bug 跑不通"，是"定义本身错"。Story 0-15 跑到 round 2 才锁清 reconnect 度量语义。

**问题根因：** review 发生在 implement 完成之后；语义失真在 AC 阶段其实已经可见，只是 AC 还没被"第二个视角"扫过。

**Epic 1+ 改进：** story 工具 / 指标 / 守门类的工作，AC draft 阶段请邀评核（可以是另一个 Claude session 做 "bmad-review-adversarial-general" 或 "bmad-review-edge-case-hunter"）。**不是"review implement"，是"review AC"**，成本低得多。

### 3.2 工具类 story 的成本/收益判据

**观察：** Epic 0 蹬出 3 个独立 CLI 工具：
- `tools/blacklist_user`（Story 0-11）
- `tools/dlq_dump`（Story 0-13 的 APNs DLQ 查看器）
- `tools/ws_loadgen`（Story 0-15）

**问题：** 每个都是 one-off，跟"监控 + 配置 + 文档" 三件套的打包时机不同。`blacklist_user` 没有对应的 Grafana dashboard；`ws_loadgen` 的 spike 报告 Phase A 只到"工具就绪"就收工；真机执行（Phase B）拆到了 Epic 9.1。

**Epic 1+ 改进：** 新增运维 CLI 时明确是否与"监控"/"runbook"/"alert" 同发布：
- **独立上线**：故障期真能用吗？人怎么找到它？
- **打包上线**：tooling 作为架构 milestone 的附属工件，不独立成 story

### 3.3 Spike / 真机类工作的单独管线

**观察：** Story 0-15（spike-op1 WS 稳定性测试）硬需要真机 + 人工执行，差点阻塞 Epic 0 收官。解决方案是把 Phase B 拆到新的 Epic 9，让 Epic 0 靠 "Phase A 工具就绪" 签字。

**教训：** Claude coding 瓶颈**不在代码**，在真机与人工。未来凡是涉及"物理 / 真机 / 人工 observability"的工作，一开始就划归 Epic 9（横向 spike epic），不要塞进业务 epic 的关键路径。Epic 10（客户端联调 MVP）做了类似的拆法 —— Epic 10 `internal/ws/room_mvp.go` MVP 上线，真机联调推给用户自己跑。

## 4. Epic 0 三条 P1 review finding 回顾

| Story | Round | 错误模式 | 教训 |
|---|---|---|---|
| 0-14 | r1 | 不变式守门在 debug 分支被 `if mode != "debug"` 整块跳过 —— 条件分支写反 | 所有"分模式的守门"都要在**双边**都跑测试，`TestX_DebugMode` + `TestX_ReleaseMode` 不能少 |
| 0-15 | r1 | `workerLongLived` 外层循环第 2 次一律 `reconnects.Add(1)`，把启动期 dial 失败误记为重连 | godoc 语义（"after the first successful connect"）与代码实现要**测试锁死**；只靠 godoc 描述不够 |
| 0-15 | r2 | `reconnectRatio` 分母 = 所有成功 dial（含每次重连），语义本该是 "% 会话被迫重连" | 比率 / 百分比度量必须附带**分子 + 分母的精确定义**；相对阈值（> 5%）严重依赖分母选择 |

**共性：** 都是**语义正确性**而非功能正确性。代码能跑、tests 能过；但定义（metric 语义 / 守门覆盖范围）本身失真，会让下游决策变成"数据支持的错误"。

**Epic 1+ 防御模式：** 加入 Review Checklist 问句："**如果这段代码运行出了错误结果但没 crash，会误导谁？** 把那个人请来 review。"
