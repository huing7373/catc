## Lesson 1: MySQL `VARCHAR(N)` 长度按"字符数"计，不是"字节数"

- **Severity**: P2
- **Category**: docs / contract / db-semantics
- **分诊**: fix
- **位置**: `docs/宠物互动App_V1接口设计.md:71`

### 症状（Symptom）

§2.5 全局编码约定原文：

> 所有字符串字段的最大长度以**字节**计（utf8mb4 编码下 1 个汉字占 3 字节），与 MySQL 表 `VARCHAR(N)` 一致。

如果 server / iOS 客户端按这条规则**用字节数**实施输入校验，会与 MySQL 真实存储语义不一致 —— MySQL 5.0+ 的 `VARCHAR(N)` 中 `N` 是**字符数**而非字节数，utf8mb4 下 1 个字符在底层存储可能占 1-4 字节，但 limit 仍按字符计数。错误规则会让"昵称 128 字"的合法中文输入被客户端按"384 字节超限"误拒。

### 根因（Root cause）

把 MySQL utf8mb4 的"**存储**字节数"和 `VARCHAR(N)` 的"**长度限制**单位"混为一谈：

- 存储层面：utf8mb4 1 字符占 1-4 字节（确实是按 byte 落盘）
- DDL 语义层面：`VARCHAR(N)` 中 `N` 是字符数 limit（这是 MySQL 5.0 之后的规范）

两个概念都和"字节"沾边，但作用层不同。文档把存储字节算到长度规则上，是常见的把"实现机制"当成"对外语义"的混淆。

### 修复（Fix）

把 §2.5 line 71 改为按字符数表述，并提示客户端 / server 用字符计数 API 实施校验：

```
- **字符串长度约束**：所有字符串字段的最大长度以**字符数**计（与 MySQL 表
  `VARCHAR(N)` 语义一致 —— `N` 是字符数 limit，**不是**字节数；utf8mb4 编码下
  1 个字符在底层存储占 1-4 字节，但长度校验按字符计数）。客户端 / server 实施
  长度校验时**必须**按 Unicode 字符数（如 Go `utf8.RuneCountInString` /
  Swift `String.count`）判断，**不**按字节数判断，否则会误拒合法的多字节输入。
```

### 预防规则（Rule for future Claude）⚡

> **一句话**：未来 Claude 在**写跨端接口契约文档涉及"字符串长度约束"**时，**必须**明确**单位是字符数还是字节数**并与持久层语义对齐。

> **展开**：
> - MySQL `VARCHAR(N)` / `CHAR(N)` 中 `N` 都是**字符数**（charset-independent length unit），不是字节数。即使是 utf8mb4，limit 也按字符计。
> - PostgreSQL `varchar(N)` 同理：N 是 character count。
> - 字节限制只在 `VARBINARY(N)` / `BINARY(N)` 这种二进制类型才成立。
> - 客户端实施校验**必须**用语言原生的字符计数 API（Go `utf8.RuneCountInString`、Swift `String.count`、JS `[...str].length` 而非 `str.length`）。**不要**用 `len(s)` (Go bytes) / `strlen` (C bytes) / `Buffer.byteLength` (Node) 这些字节级 API 做用户字符串长度校验。
> - **反例**：在 contract 文档里写"字符串字段最大 128 字节，因为 utf8mb4 下汉字 3 字节"是踩坑 —— 这会让任何严格按文档实施的客户端，把 128 个汉字（合法 MySQL `VARCHAR(128)` 输入）按"384 字节"拒掉。
> - **写完字符串长度规则**后，问自己一句："如果客户端工程师只读这一段，他会用字符数还是字节数实施校验？两种算法在多字节场景下产生的拒绝集合一致吗？"如果不一致，说明描述有歧义。

## Lesson 2: 改文档某处时必须 grep 全节，确保同节内部一致

- **Severity**: P2
- **Category**: docs / process
- **分诊**: fix
- **位置**: `docs/宠物互动App_V1接口设计.md:262`（顺带 line 274 / line 276）

### 症状（Symptom）

上轮 fix-review (round 2) 在 §4 `/me` 章节的 Future Fields 段（line 304）改了 `currentRoomId` 的语义：

> 本字段保留为 schema 占位但不计划在后续节点回填真实数据；服务端始终返回 null。

但同章节的 line 262（节点列）依然写：

> 节点 2 阶段 currentRoomId 必为 null；真实 currentRoomId 节点 4 由 Story 11.10 / 14.x 注入

两处描述**矛盾**。下游 iOS / server 实装看不同行会得出截然相反的结论："节点 4 后能拿到真实数据" vs "永远是 null"。round 3 codex 当场抓出。

### 根因（Root cause）

上轮改 Future Fields 时，**只改了引发 review finding 的那一行**，没有把"该字段的语义变化"在同章节其他位置（节点列、字段表说明、JSON 示例旁注）做一致性扫描。这是单点改动 + 没做"涉及字段全节扫描"的典型工艺漏洞。

底层原因：fix-review 的最小改动原则被误读为"只改 review 指到的那一行"。但**最小改动** ≠ **不做一致性扫描** —— 当一个字段的语义发生变化（不是仅纠错措辞），同字段在同章节的所有出现位置都属于"必须同步"的范围，否则一致性反而被破坏。

### 修复（Fix）

把 §4 `/me` 同章节其他描述位置全部对齐为"始终 null，无回填计划，真实房间归属用 `GET /home`"：

- line 262（节点列）：从 "节点 4 由 Story 11.10 注入" 改为 "始终返回 null —— 该字段保留为 schema 占位但不计划在后续节点回填真实数据；获取真实房间归属请改用 `GET /home`"
- line 274（字段表说明）：从 "节点 2 阶段强制返回 null（房间在节点 4 才落地）" 改为 "始终返回 null —— schema 占位字段，无后续节点回填计划"
- line 276（JSON 示例旁注）：从 "节点 4 之后真实数据见 Future Fields" 改为 "服务端始终返回 null；获取真实房间归属请改用 `GET /home`"

修完后跑 `grep -n currentRoomId docs/宠物互动App_V1接口设计.md` 复核，确认 §4 全节四处描述（line 262 / 274 / 276 / 304）口径一致。

### 预防规则（Rule for future Claude）⚡

> **一句话**：未来 Claude 在**改一份契约 / 设计文档某处描述某字段语义**时，**必须**先对该字段在**同章节 / 同接口段**做 grep，把所有相关位置（节点列、字段表、JSON 示例旁注、Future Fields、错误码表）一并同步，**不能**只改 review 指出的那一行。

> **展开**：
> - 触发条件：当改动是"语义变更"（如"会回填" → "永不回填"、"required" → "optional"、"允许 null" → "禁止 null"）时，必须做全章节扫描。仅纠错措辞 / 改格式不必。
> - 扫描范围：该字段在同章节内的所有 markdown 段落 / 表格行 / JSON 示例 / 引用块。跨章节的引用（其他接口里提到该字段时）也要列入候选清单，但跨章节同步可独立成新一轮 review，本节修复至少保证**自身章节内一致**。
> - 工具：`grep -n <fieldName> <file>` → 看哪些行在同章节 → 同步描述。
> - **反例**：上轮 round 2 fix-review 改 line 304 的 Future Fields 段，没扫同章节 line 262 / 274 / 276 → round 3 review 当场抓出"line 262 与 line 304 矛盾"。这种回归 finding 是工艺漏洞而非新坑。
> - **fix-review 工艺补丁**：对于"语义变更类"的 finding，修完那一处后，应**主动 grep 字段名**做最后一次同章节一致性 sweep，确保不引入新的自相矛盾。这步不属于 scope creep，是修复完整性的一部分。

## Meta: 本次 review 的宏观教训

两条 finding 看似独立（一条是 DB 语义错误、一条是同节自相矛盾），但它们共享同一个底层主题：**契约文档的内部一致性 / 与底层存储语义对齐**。

- Lesson 1 是**语义对齐**：契约规则必须与持久层语义（MySQL `VARCHAR` 的字符数语义）严格对齐，避免"看起来对的描述"误导下游实施。
- Lesson 2 是**章节自洽**：改契约文档时不能只改一处，必须保证同章节内所有同字段描述口径一致，否则下游会按不同行得出相反结论。

两者合起来给 fix-review 工艺一个补丁：**修文档类 finding 后，除了改 review 指到的那一行，还必须做两次扫描** —— 一次"对齐底层存储语义"，一次"同章节字段一致性 sweep"。这两次扫描成本极低（grep + 5 行复读），但能消除大量回归 finding。
