## Lesson 1: 已认证路由的 1005 限频维度必须按 user_id，不是按 IP

- **Severity**: medium (P2)
- **Category**: docs
- **分诊**: fix
- **位置**: `docs/宠物互动App_V1接口设计.md:592`

### 症状（Symptom）

§6.1 步数同步接口 (`POST /api/v1/steps/sync`，已认证) 错误码表格里 1005 写"rate_limit 中间件拦截（同 IP 每分钟 > 60 次）"。这与 Story 4.5 (`epics.md:1037-1041`) 的 "默认每用户每分钟 60 次" 矛盾 —— 已认证路由能从 token 拿到 user_id，应该按 user 限频；按 IP 限频会让大型 NAT 后所有用户共用同一计数桶，是**仅在没有 user 上下文时（如游客登录前）的 fallback 维度**。

### 根因（Root cause）

写 §6.1 时直接 copy-paste §4.1 (POST /auth/guest-login) 那行错误码描述，但 §4.1 是**未认证路由**（登录接口本身免 auth 中间件，按 IP 限频是正确选择，因为还没 user_id）；§6.1 是**已认证路由**，已经能从 Bearer token 解出 user_id，限频维度应翻转。文档维护者没意识到"限频维度由路由的认证状态决定"这条隐性规则。

### 修复（Fix）

§6.1 错误码表格 1005 行改为：

> `1005 | 操作过于频繁 | rate_limit 中间件拦截（**已认证路由**按 user_id 限频，每用户每分钟 > 60 次；按 Story 4.5 默认值，配置可调）`

### 预防规则（Rule for future Claude）⚡

> **一句话**：写 V1 接口设计文档的"可能的错误码"表格里 1005 描述时，**必须**先看接口元信息的"认证"行：**已认证路由按 user_id 限频，未认证路由按 IP 限频** —— 不能 copy-paste 跨认证状态的描述。
>
> **展开**：
> - 已认证路由（接口元信息表"认证"= **需要** Bearer token）：1005 描述模板"rate_limit 中间件拦截（已认证路由按 user_id 限频，每用户每分钟 > N 次）"
> - 未认证路由（接口元信息表"认证"= **不需要**，如 `/auth/guest-login`）：1005 描述模板"rate_limit 中间件拦截（同 IP 每分钟 > N 次）"
> - **基线锚点**：Story 4.5 AC `epics.md:1037-1041` 默认 user-per-min 60 次；IP-per-min 是**未认证 fallback 维度**，不是默认维度
> - **反例**：把 §4.1 (未认证) 的"同 IP 每分钟 > 60 次"原样复制到 §6.1/§6.2/§7.x/§8.x（已认证业务路由）的 1005 描述里 —— 这会推动下游 Story (7.3/8.5/...) 实装错误的限频维度，且对 NAT 用户产生误伤
> - **副作用**：含糊地写"rate_limit 中间件拦截"也优于错误地写"按 IP" —— 当不确定具体维度时，宁可省略也别写错维度

## Lesson 2: 错误码语义说明里"粘性错误码"声称必须与上方"服务端逻辑"严格一致，不能臆造跨调用持续状态

- **Severity**: medium (P2)
- **Category**: docs
- **分诊**: fix
- **位置**: `docs/宠物互动App_V1接口设计.md:598`

### 症状（Symptom）

§6.1 "关键约束" 第一段说："3001 当日剩余 sync 调用都会得到 3001，应静默接受 + 第二天再试"，把 3001 描述成**粘性错误码**（一次触发后当日所有 sync 都失败）。但同一节点 §6.1 "服务端逻辑" 步骤 3-4 明确说：

- 倒退场景（`clientTotalSteps < lastClientTotalSteps`）→ delta = 0，仍写日志，返回 `code = 0`
- 重复场景（`clientTotalSteps == lastClientTotalSteps`）→ delta = 0，正常入账日志
- 3001 仅在 `prevAccepted + curDelta > 50000` 时触发

文档自相矛盾。客户端如果信"关键约束"，会在首次 3001 后停止当日 sync —— 但其实当日仍可能产生有效的"差值=0"sync_log（审计需要），停止上报会丢审计数据。

### 根因（Root cause）

写"关键约束"时把"业务直觉"（封顶后 = 当日彻底关门）当成"协议契约"写了进去，没回头核对上方逻辑步骤。**"差值=0 + 仍写日志 + 返 code=0"** 这条规则让 3001 不可能成为粘性错误码 —— 一旦 prevAccepted 已 ≥ 50000，下一次 sync 不论 clientTotalSteps 多大都会触发 `prevAccepted + curDelta > 50000`，此时确实仍返 3001；但**只要客户端的 clientTotalSteps 不再增长（如设备静止）**，差值就 = 0，反而走 code = 0 路径。所以 3001 是"条件性触发"，不是"粘性"。

### 修复（Fix）

把第一条"关键约束"改写为：

> 错误码 3001 与 1002 的语义差异**必须**明确：1002 是**参数本身**不合法；3001 是**参数合法但触发服务端业务限制**（防作弊封顶，本次 delta 被强制 = 0）。**注意**：3001**不是粘性错误码** —— 当日只要 `prevAccepted + curDelta > 50000` 触发条件成立才返 3001；若客户端后续 sync 是倒退或重复，按"差值计算"步骤 delta 自然 = 0，**仍返 code = 0**。客户端**不应**假设当日首次 3001 后所有 sync 都失败而停止上报（仍需上报维持 sync_log 审计）。

### 预防规则（Rule for future Claude）⚡

> **一句话**：在"可能的错误码"或"关键约束"里描述某个错误码的**跨调用持续性**（粘性 / 一次性 / 条件性）时，**必须**用上方"服务端逻辑"步骤里**所有**会返该 code 的分支去验证；任何"一次触发后续都触发"的声称必须能从逻辑步骤推出。
>
> **展开**：
> - 区分三种错误码持续性：
>   - **一次性**：每次调用独立判定（如 1002 参数错误、1005 限频）
>   - **条件性**：只要触发条件再次成立就返（如 3001 的 `prevAccepted + curDelta > 50000`）
>   - **粘性**：一次触发后某段时间内强制重复返同一 code（极少见，需要 server 维护"已封禁"状态；MVP 没有此类设计）
> - **校验流程**：写完错误码描述后，反过来读"服务端逻辑"步骤，问 "**是否每条 delta 计算分支（首次 / 有记录 / 倒退 / 重复 / 跨日）都会返这个 code？**" 只要有一条不返 → 该 code 不是粘性
> - **客户端 retry 文案要点**：粘性 code 才能写"当日不要再调"；条件性 code 必须写"按业务规则继续按需调用"
> - **反例**：写出"3001 后当日剩余 sync 都返 3001" / "1009 服务繁忙后 5 分钟内不要重试，server 会持续返 1009" 这类**没有 server 状态机支撑**的粘性声称 —— 这会让客户端做出错误的 retry / UX 决策（如丢失审计数据 / 用户白等待）
