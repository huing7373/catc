## Lesson 1: 日封顶 3001 触发条件被写成"已达上限"会放过越界 sync

- **Severity**: high
- **Category**: docs（接口契约）
- **分诊**: fix
- **位置**: `docs/宠物互动App_V1接口设计.md:548`（服务端逻辑表）+ `:593`（错误码表 3001 行）

### 症状（Symptom）

§6.1 把 3001 触发条件写成「当日 `accepted_delta_steps` 累计 ≥ 50000 → 当次 delta 强制 = 0」。在 `prev=49000 + cur=4000`（截断后 cur=4000）这种场景，`prev < 50000` 不命中"已达上限"分支，于是 4000 正常入账到 53000，跨过 50000 红线 —— 等于日封顶被放过最后一次越界 sync。`epics.md` GAP K 原文是「单日累计 > 50000 → delta=0 + 3001」，跟"已达"语义直接矛盾。

### 根因（Root cause）

写"≥ 50000 触发"是把封顶判断的**方向**搞反了：上限保护应该是**前置 / 入账前判断本次会不会越界**，而不是**后置 / 已经越界才回头拒**。两种写法在边界恰好等于上限时等价，但只要 prev 严格小于上限、cur 又会跨过去，就会暴露差异。文档作者写时没在脑里走 `prev=49000, cur=4000` 这条具体数轴，于是表面"自然"的"累计达上限"措辞悄悄把语义偏成了后置判断。

### 修复（Fix）

把 §6.1 的 3001 触发条件改成"**本次入账后将超过**"判断：

- `:548`: `若同 sync_date 当日 accepted_delta_steps 历史累计 + 本次截断后 delta 将超过 50000（即 prevAccepted + curDelta > 50000） → 当次 delta 强制 = 0 + log warning + 返回 3001`，并补一句"判断条件是'本次入账后是否越界'，不是'已达上限'，否则会放过最后一次跨界 sync，例如 prev=49000 + cur=4000 应被拒"
- `:593`: 错误码表 3001 行同步改成"本次同步入账后当日 `accepted_delta_steps` 累计**将超过** 50000 上限（即 `prevAccepted + curDelta > 50000`）"

### 预防规则（Rule for future Claude）

> **一句话**：未来 Claude 在写**任何"上限 / 配额 / 封顶"类规则**时，**必须**用 **`prevState + delta op limit`** 形态的入账前判断，**禁止**用"累计已达 / 累计 ≥"形态的入账后判断。
>
> **展开**：
> - 上限判断永远写成 `if (current + incoming) > limit then reject`，不写 `if accumulated >= limit then reject`，前者守得住边界、后者必然放过最后一次越界请求
> - 写完任何阈值规则，**必须**在脑里走"prev 严格小于 limit、cur 会跨过去"这条具体数轴用例，能否被规则正确拒；走不通就是规则方向写反了
> - 文档表达上推荐直接写出公式 `prevAccepted + curDelta > 50000`（含变量名），而不是只写中文措辞 —— 公式没有"歧义解读空间"，措辞有
> - **反例**：`若当日 accepted 累计 ≥ 50000 → delta=0 + 3001`、`若 quota 已用完 → 拒绝`、`若 rate 已达上限 → 限频`（这三句都是后置判断，全都会放过最后一次跨界请求）

## Lesson 2: `clientTotalSteps ≤ 200000` 硬上限与参数校验/3001 防作弊三方语义不一致

- **Severity**: medium
- **Category**: docs（接口契约）
- **分诊**: fix
- **位置**: `docs/宠物互动App_V1接口设计.md:515`

### 症状（Symptom）

请求体字段表写 `clientTotalSteps: 0 ≤ value ≤ 200000（实际硬上限，超出按防作弊处理）`，但是：

1. 同节服务端逻辑步骤 2「参数校验」只列了 `clientTotalSteps ≥ 0`，**不**校验 ≤ 200000
2. 错误码 1002 触发条件只列「缺失或为负数」，**不**包含「> 200000」
3. 防作弊规则（§6.1 步骤 4 + GAP K）只约束 **`delta`** 的截断 / 封顶，**不**约束 `clientTotalSteps` 总值本身

三处表述彼此矛盾 → 同一输入 `clientTotalSteps=250000` 在 7.3 落地时既可能走 1002（如果开发者读字段表）、也可能不报错走防作弊路径（如果开发者读校验步骤），还可能完全不命中（如果开发者只读 GAP K）。`199900 → 200100` 这种正常推进也会因表面有 200000 上限而陷入"该不该 1002"的争议。

### 根因（Root cause）

字段表的"范围/约束"列被当成了"实现期望"而非"协议契约"，作者把"客户端实际不太可能上报超过 200000 步"这种先验假设直接写成了"硬上限"。但**硬上限**这个词在协议层有强语义 —— 必须有对应的校验失败路径（哪个错误码、哪条规则）。当字段表写了硬上限但下面的校验列表没接住时，三方就开始打架。

更深层根因：步数防作弊整套规则（GAP K）的认知模型是「**只对 delta 反作弊，server 是 delta 权威；clientTotalSteps 只是客户端原始输入，不做总值反作弊**」。这个模型决定了 clientTotalSteps 不该有业务硬上限。字段表"自然"加上 200000 是把不存在的契约写进去了。

### 修复（Fix）

把 §6.1 §6.1 请求体字段表 `:515` 行改成：

```
| `clientTotalSteps` | number (int) | 必填 | `value ≥ 0`（int32 自然上限即可，不设业务硬上限；server 不对总值本身做防作弊判断，所有反作弊只针对 delta，见下方"防作弊阈值"） | ... |
```

不需要再改 1002 触发条件（原本就只列 `< 0`，与新约束一致），不需要再改 GAP K（原本就只约束 delta）。一处改动让三方语义重新对齐。

### 预防规则（Rule for future Claude）

> **一句话**：未来 Claude 在**接口字段表里写"硬上限 / 硬下限 / enum"类约束**时，**必须**同时确认三处都把这条约束接住：(a) 服务端逻辑里的参数校验列表、(b) 错误码触发条件表、(c) 业务规则（防作弊 / 配额）描述。
>
> **展开**：
> - 字段表里的"范围/约束"列不是"实现期望提示"，是**协议层强契约**；每条约束必须有对应的失败路径（哪个 code、哪条规则）兜住，否则 server 实装时三方各执一词
> - 写"硬上限"前先问自己：超出硬上限走 1002（参数错误）还是走业务码（如 3001）？如果两种都不走，那就**不**是硬上限，是先验假设 —— 改写成"实际数据范围预期"放在说明栏，**别**放在范围列
> - 反作弊规则的对象（"管 delta 还是管 total"）一旦定下来，所有相关字段的范围列都要跟这个模型对齐 —— 模型说"只管 delta"，那 total 字段就**不**该写硬上限
> - **反例**：字段表写 `clientTotalSteps ≤ 200000`，但参数校验只查 `≥ 0`，错误码 1002 不提 200000，防作弊规则只管 delta —— 这就是三方各写各的、没一处兜得住"超 200000"的失败路径
> - **反例**：字段表写 `appVersion 长度 ≤ 32 字符`，但服务端校验只查非空 —— 同类型陷阱（约束写了但没校验路径）

## Meta: 本次 review 的宏观教训

两条 finding 都属于**接口契约文档内部一致性**问题：契约文档不是单一段落能成立的，它是「字段表 ↔ 服务端逻辑步骤 ↔ 错误码表 ↔ 业务规则段落」四方互相约束的网状结构。任何一处写一条规则，**必须**走一遍其它三处看是否兜得住，否则就会留下"看起来都写了但落地三方打架"的隐患。这个网状结构的检查在文档审阅时是**第一优先级**，比"措辞是否优美 / 例子是否完整"重要得多。

- **写完任何接口规则段，必走的 4 处一致性 checklist**：(1) 字段表的范围/约束列、(2) 服务端逻辑步骤里的参数校验、(3) 错误码表的触发条件、(4) 业务规则（防作弊/配额/事务）描述。四处任意两处对同一字段说不同的话 = 契约写错。
- **公式优先于措辞**：阈值类规则（封顶 / 限频 / 配额）能用公式（如 `prev + delta > limit`）就别用中文措辞（"已达上限 / 超过 / 满了"）—— 中文歧义会偷偷改方向。
