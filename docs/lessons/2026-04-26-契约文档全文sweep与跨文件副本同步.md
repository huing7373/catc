## Lesson 1: 修改契约 / 设计文档某处规则后，**必须**全文 grep 该规则关键词，包括所有副本

- **Severity**: P2 / medium
- **Category**: docs
- **分诊**: fix（3 条 finding 是同一根因的 3 个表现）
- **位置**: `docs/宠物互动App_V1接口设计.md` + `_bmad-output/implementation-artifacts/4-1-接口契约最终化.md`

### 症状（Symptom）

review round 2 / 3 改 V1 接口文档时反复出现"改一处漏一处"的自相矛盾：

- §4.3 表格 + §4.3 Future Fields 引用块都改成"`/me.currentRoomId` 始终 null + 无 planned story"
- 但 §1 顶部"Freeze Policy"bullet 仍把 `user.currentRoomId` 列在 Future Fields 列表里
- §2.5 改成"长度按字符数计"，但 §4.1 请求体字段表 + §3 错误码表仍写"1 ≤ length ≤ 128 字节" / "长度超过 128 字节"
- 同一份 schema 在 story 文件（dev 阶段实施说明产物）里仍是 pre-fix 状态 —— 后续 reviewer 看 story 文件会"复活"已删除的语义

### 根因（Root cause）

**当 Claude 修改文档某条规则时，倾向于只改"主要现场"（最相关的章节），不主动假设规则的"副本"分布**。具体三层副本盲区：

1. **同文档的总览 / freeze policy / 索引节**：契约文档常在顶部 §1 / §2 写一个"全局规则"段，又在 §4 / §5 各接口章节展开。改了具体接口的语义，**很容易忘记**回去同步顶部总览段
2. **错误码表 / 触发条件列**：长度约束改了字段表，但"参数错误"的触发条件描述也含具体数字 / 单位，是另一处"语义副本"
3. **story 文件 / artifacts 副本**：dev-story 阶段产生的"实施说明"产物里抄了一遍 schema 作为 AC 描述。改了 V1 文档**就必须**同步改 story 文件，否则下游 reviewer 看 story 文件会按 pre-fix 语义判定 "你没按 AC 实施"

### 修复（Fix）

**V1 文档**：

- 删除 line 22 Freeze Policy bullet 里的 `user.currentRoomId`，并补一句注释"`user.currentRoomId`（仅出现在 `GET /me`）**不**属于 Future Fields —— 该字段是**永久 schema 占位**（始终 `null`，无后续节点回填计划），见 §4.3 Future Fields 引用块"
- line 148 / 151 / 152 / 217："字节" → "字符"

**story 文件**：

- line 61：§2.5 副本同步成"字符数"长版描述（与 V1 文档 line 71 一致）
- line 94 / 97 / 98 / 164：guestUid / appVersion / deviceModel 长度约束 + 错误码触发条件，"字节" → "字符"
- line 186 / 197 / 199 / 211：§4.3 schema 副本同步成"始终 null + schema 永久占位 + 无 planned story"，Future Fields 引用块改为 permanent placeholder 语义
- line 330：错误码触发条件示例文本"超过 128 字节" → "超过 128 字符"
- line 345：§1 Freeze policy summary 删除 `user.currentRoomId`
- line 359 / 413：commit message 模板 + Task 2.2 描述里的"utf8mb4 字节长度" → "utf8mb4 字符长度"
- line 360 / 421 / 564：AC4 自检条 + Task 4.2 描述同步"始终 null / 永久占位"语义

### 预防规则（Rule for future Claude）⚡

> **一句话**：未来 Claude 在 **修改契约 / 设计 / 协议文档某条规则后**，**必须**在 commit 前对该规则的关键词跑**两轮** grep —— ① 当前文档**全文**；② **所有引用 / 复制了同份 schema 的其他文件**（story / artifacts / 别的 docs / 测试 fixture）—— 把每一个 hit 逐行核对是否与新规则一致；不一致的全部一次性改完。
>
> **展开**：
>
> - **第 1 轮 grep（同文档全文）**：以新规则的关键词（如改 `字节` 为 `字符`，关键词就是 `字节`；改 `节点 4 由 Story 11.x` 为 `始终 null`，关键词就是 `currentRoomId`）跑 `Grep` tool。把所有 hit 行号列出来，**逐行**判断该行语义是不是属于新规则范围 —— 是则改，不是则**显式**说明为何不改（写在 lesson 或 commit message 里）
> - **第 2 轮 grep（跨文件副本）**：在改契约 / 设计文档时，**默认**有这些副本会存在：
>   - `_bmad-output/implementation-artifacts/<story-file>.md`（dev 阶段实施说明产物，AC 描述里抄了 schema）
>   - 其他相关 `docs/宠物互动App_*.md`（如改了 V1 文档的字段长度，可能 `数据库设计.md` 也有 VARCHAR 定义）
>   - `_bmad-output/planning-artifacts/epics.md`（epic / story AC 里可能引用了字段名 / 类型）
>   - 服务端 / 客户端代码里的 DTO / Codable struct（节点 2 阶段还没写，节点 3+ 必须 grep）
>   - 测试 fixture（`*_test.go` / `*Tests.swift` 里的 expected response JSON）
> - **判断 grep hit 是否要改的启发式**：
>   - 同义改写还是引用？hit 是"该规则的另一种表述"则改；hit 是"对该规则的引用 / 元描述"则保留（如 §2.5 自身的解释里说"1 个字符占 1-4 字节"，这是讲原理，不改）
>   - "/me 上下文"vs"/home 上下文"vs"domain / DB 列上下文"：同一字段名（如 `currentRoomId`）在不同上下文有不同语义，逐行判断上下文，**不**机械全替换
> - **完成后跑第 3 轮验证 grep**：原关键词应**只剩**说明性 / 元描述性的 hit，**不应**还有"按旧规则"的实际语义 hit。验证 grep 必须出现在 commit 前
> - **反例（一定不要这么做）**：
>   - 改了某接口章节的字段语义，**只**在该章节内 sweep 就 commit
>   - 改了规则但**不**去 story 文件 / planning artifacts 里同步副本 —— 下游 dev / reviewer 会"复活"旧语义
>   - 用 `replace_all` 全文一刀切替换关键词 —— 容易把元描述（如"1 个字符占 1-4 字节"里的"字节"）也换掉，制造新的不一致

## Meta: 本次 review 的宏观教训

这是 round 3 lesson "改文档某处必须 grep 全节防自相矛盾" 的**进一步加强版**。round 3 lesson 解决的是"同章节内自相矛盾"，本轮揭示的是更深层的两个延伸：

1. **跨章节 / 文档顶部总览段**：契约文档天然有"freeze policy / 全局规则段 / 各章节展开"的层次结构，改了细节**必须**回顶部总览段同步
2. **跨文件副本（story / artifacts / DTO / 测试 fixture）**：契约文档不是孤立的，它的语义会被复制到 dev 阶段的实施说明产物 + 代码 DTO + 测试 fixture 里。改了源头**必须**同步所有副本

实操上这意味着：**任何**对契约 / 设计文档某条规则的修改，commit 前必须做 3 轮 grep（同文档全文 + 跨文件副本 + 验证 grep），并把"我搜了哪些关键词、扫了哪些文件、判断哪些 hit 不改的理由"写进 commit message 或 lesson —— 这是把"完整 sweep"从隐式自律变成显式审计，避免下一轮 review 又抓出新副本。
