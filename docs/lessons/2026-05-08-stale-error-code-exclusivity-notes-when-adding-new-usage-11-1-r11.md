## Lesson 1: 给已有错误码扩展使用接口时，必须反向 grep 整份契约文档里所有"该错误码仅 X"/"该错误码不在 Y 出现"类陈述并同步对齐

- **Severity**: P2
- **Category**: docs
- **分诊**: fix
- **位置**: `docs/宠物互动App_V1接口设计.md` §10.1 创建房间（行 1140）+ §10.4 加入房间（行 1494）

### 症状（Symptom）

r7 给 §10.3 GET /rooms/{roomId} 新增 6004（ACL fail）后，文档里出现两条相互矛盾的契约陈述：

- §10.3（新）：6004 = caller 不是该房间成员 → 拒绝查看
- §10.1 行 1140（旧）：「6004（用户不在房间中）仅在 `POST /rooms/{roomId}/leave` 出现」
- §10.4 行 1494（旧）：「6004 仅用于 leave 接口」

§3 全局错误码表行 138（`6004 用户不在房间中`）描述足够泛化，能覆盖 leave + GET ACL fail 两种语义，**没有**问题；问题集中在每个具体接口里 reviewer 当年加的"反向澄清注"——这些注当时是在帮助实装者快速排除"我这里要不要返 6004"的疑问，但默认了"6004 只有一个使用接口"这个写死的前提，新增使用时没人扫到反向陈述。

实装影响：客户端 / 服务端工程师按 §10.4 join 接口或 §10.1 create 接口的注释做"我这里能不能触发 6004"的反向推理时，会得到"6004 仅 leave 用 → GET /rooms/{roomId} 不应该返 6004"的错误结论；遇到 §10.3 ACL fail 实际下发的 6004 → 走 fallback path → 可能直接当 server bug 上报或者把用户卡在 stale state。

### 根因（Root cause）

契约 freeze 类 story 在每轮 review 给某个错误码扩展使用接口时，只往新接口章节加触发条件 + 错误码表行 + 注释，**没有反向扫整份文档里"该错误码仅在 X 出现 / 不在 Y 出现"类反向陈述并同步对齐**。这种"只加正向 forward path、不扫反向 reverse path"的修法在以下场景特别容易踩坑：

- 同一份文档内多个接口章节互相 cross-reference（§10.1 ~ §10.5 都在谈房间错误码）
- 文档里习惯加"本接口不会触发 X / Y / Z 因为..."类反向澄清注（出于减负目的，但这种注里写的"仅在 W 出现"是写死的硬陈述）
- 错误码本身具有"扩展性"（语义足够泛 → 多个接口都可能复用），但首版文档里因为只有一个接口用 → 反向陈述写得很绝对（"仅"/"only"/"exclusive to"）

更深层：**反向陈述和正向陈述应该对称维护**——给某错误码新增使用 = 同时更新所有正向陈述（错误码表 + 触发条件）+ 所有反向陈述（"不在 X 出现"/"仅在 Y 出现"）。前 10 轮 review/fix 都没在"反向陈述对齐"维度做横扫，到 r11 才被 codex 抓到，说明该维度是漏洞。

### 修复（Fix）

- **§10.1 行 1140（创建房间注）**：
  - **before**：`> **注**：本接口**不**会触发 6001 / 6002 / 6004 / 6005 —— 房间在事务内才被创建...；6004（用户不在房间中）仅在 POST /rooms/{roomId}/leave 出现。`
  - **after**：`> **注**：本接口**不**会触发 6001 / 6002 / 6004 / 6005 —— 房间在事务内才被创建...；6004（用户不在房间中）仅出现在 GET /rooms/{roomId}（§10.3，ACL fail：caller 不是该房间成员）和 POST /rooms/{roomId}/leave（§10.5，重复 leave / 不在该房间）两条接口。`
- **§10.4 行 1494（加入房间注）**：
  - **before**：`> **注**：本接口**不**触发 6004（用户不在房间中）—— 6004 仅用于 leave 接口。`
  - **after**：`> **注**：本接口**不**触发 6004（用户不在房间中）—— 6004 用于 GET /rooms/{roomId}（§10.3，ACL fail：caller 不是该房间成员）和 POST /rooms/{roomId}/leave（§10.5，重复 leave / 不在该房间），与本接口（join）的语义场景正交。`
- **§3 全局错误码表（行 138）**：保持原样不动（描述"用户不在房间中"已足够泛化覆盖两种用法；§3 没有"出现接口"列需要对齐 —— 已 grep 确认）
- **§10.3 / §10.5 的 6004 触发条件描述**：保持原样不动（§10.3 r7 已经写好；§10.5 leave 接口本来就是 6004 originating 接口，注释无矛盾）

### 预防规则（Rule for future Claude）⚡

> **一句话**：未来 Claude 在 **给某个 error code / status code / event type 新增一个使用接口（或新增一处使用场景）** 时，**必须**在改完正向条目（接口章节内的触发条件 + 错误码表行）**之前**先 `grep` 整份契约文档里 **所有反向陈述模式**（"仅在 X 出现"/"only used in Y"/"exclusive to Z"/"不会触发 X"等），逐条同步对齐到"完整使用接口列表"，**禁止**留下任何"仅 X 用"类陈述与新增用法矛盾。
>
> **展开**：
> - **触发条件**：契约 freeze 类 story（V1 接口设计 / 数据库设计 / 时序图等"全文档体系内多接口共享同一概念"的文档）做 review/fix 给某错误码扩展使用接口时
> - **必做检查**：
>   - `grep -n "<错误码> .{0,30}仅"` —— 中文"仅"反向陈述
>   - `grep -n "<错误码>.{0,30}\(only\|exclusive\)"` —— 英文反向陈述
>   - `grep -n "不.{0,5}触发.{0,5}<错误码>\|<错误码>.{0,30}不.{0,5}出现"` —— 中文否定反向陈述
>   - `grep -n "本接口.{0,5}不"` 整体扫一下"本接口不..."类减负注，看其中是否绑了某错误码的写死陈述
> - **必做改写**：把所有"仅 X"陈述改成"用于 X + Y + Z（完整列表）"或"用于 X / Y（按章节链接）"，并在每个接口的反向澄清注里指明"与本接口的语义场景正交"而不是"仅在某接口用"
> - **必做扫描原则**：错误码 / 事件类型 / 状态码这类**有 cardinality 增长潜力的 token**，每次扩展使用都要做反向扫描；不能只扫"哪些接口已经用了它"（forward），还要扫"哪些接口注里写了它的反向 exclusivity claim"（reverse）
> - **反例**：r7 给 §10.3 加了 6004 ACL fail，正向条目（§10.3 错误码表 + 触发条件 + ACL 注）都更新了，**但漏扫了 §10.1 / §10.4 注释里的"6004 仅用于 leave"反向陈述**，到 r11 才被 codex 抓到 —— 留下 4 轮的"潜伏期 contract bug"。这种"forward path 改完就声明 done、不做 reverse sweep"是踩坑模式
> - **反例 2**：通常人会觉得"反向澄清注是减负工具不是契约硬约束，可以容忍小程度的 stale"——**这是错的**，反向陈述里写"仅 X / 不 Y"就是在用"绝对语气"做硬契约（实装者会反向推理 → 写死 if-else），stale 反向陈述等价于 stale 正向陈述，必须同步对齐
> - **反例 3**：仅看 r7 patch 的局部 diff（§10.3 范围）容易判断 "OK 一切对齐"；必须把整份文档当 single source 扫，因为 V1 接口设计文档就是 contract single source（其他文档/客户端/服务端都按它写）
