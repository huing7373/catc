# Review Lessons — 2026-04-26 — 契约 schema 字段可空性必须显式声明 + JSON 示例标签必须与示例内容一致

## 背景

Story 4-1（接口契约最终化）round 5 review。codex 在 V1 接口设计 §5.1（GET /home）发现两处契约缺陷：

## Lesson 1: 契约 schema 字段表必须显式声明每一层容器的可空性，不能光靠子字段不写当默认非空

- **Severity**: P2 (Medium)
- **Category**: docs
- **分诊**: fix
- **位置**: `docs/宠物互动App_V1接口设计.md` §5.1 line 335-340

### 症状（Symptom）

V1 §5.1 GET /home schema 字段表原本只列 `data.pet.id` / `data.pet.petType` / `data.pet.name` / `data.pet.currentState` / `data.pet.equips` 五个子字段，**没有任何一行说明 `data.pet` 容器对象本身可能为 null**。

但 epics.md Story 4.8 AC 明确写："edge: 用户无 pet（理论不该）→ pet 字段返回 null"。这意味着 server 实装时 `/home` 在用户无 pet 时必须回 `{"data": {"pet": null, ...}}`；下游 iOS / Go DTO 按当前 schema 表会把 pet 解析成必有 object —— 跑 Story 4.8 edge case 测试时直接 decode 失败。

### 根因（Root cause）

字段表用"列子字段"的方式描述对象 schema 时，**子字段的存在隐含了"父字段必有"的语义**，但没有任何机制提醒文档作者去显式声明"父字段可空"。这是一个由"子字段优先"思路引发的盲区：

- 列字段时只关心叶子字段（`pet.id` / `pet.petType`），父级 `pet` 被当作"显然存在的容器"
- 真实业务里 `pet` 可能不存在（无默认 pet 的边缘场景），但因为 happy case 写得很顺，作者忘记翻 epics.md / 数据库设计去 cross-check 容器层的可空性
- iOS Codable / Go JSON unmarshal 都按 schema 字段表生成 DTO；如果 schema 没标可空，DTO 就是非 optional

类似的盲区也出现在 `room` 容器：`data.room.currentRoomId` 标了 `string | null`，但 `data.room` 本身没标可空 —— 只是这次 review 没逮到。

### 修复（Fix）

V1 文档 §5.1 字段表新增一行 `data.pet | object | null`：

```diff
 | `data.user.avatarUrl` | string | `""` | 头像 URL |
+| `data.pet` | object \| null | 真实数据 | 默认猫信息容器；用户**无默认 pet（理论不该发生，但 Story 4.8 edge case 强制覆盖）时返回 `null`**。客户端必须按可空对象解析（iOS 端 `Optional<PetDTO>` / Go 端 `*PetDTO`），不得假设 pet 永远非空。下方 `data.pet.*` 子字段**仅当 `data.pet ≠ null` 时存在** |
 | `data.pet.id` | string | 真实数据 | 默认猫主键（仅当 `data.pet ≠ null`） |
 | `data.pet.petType` | number | `1` | 宠物类型枚举（仅当 `data.pet ≠ null`） |
 ...
```

并在每个 `data.pet.*` 子字段说明里追加"（仅当 `data.pet ≠ null`）"作为二次提醒，让下游开发者无论从哪一行入手都能看到约束。

故事文件副本（`_bmad-output/implementation-artifacts/4-1-接口契约最终化.md` §AC5 schema 块）同步同样的字段表。

### 预防规则（Rule for future Claude）

> **一句话**：未来 Claude 在写"对象层级 schema 字段表"时，**必须**给每一层容器对象（不光是叶子字段）显式声明类型 + 可空性，并 cross-check 该接口对应的 epics.md AC（尤其是 edge case AC）确认每个容器的真实可空语义。
>
> **展开**：
> - 写 schema 字段表时，先列**每个嵌套层级的容器**，再列子字段。例如 `data.pet | object \| null` 是父行，`data.pet.id | string` 是子行。父行先确定 nullability，子行再继承父行的"仅当父 ≠ null 时存在"语义
> - cross-check 来源：epics.md 对应 story 的 AC（特别是 edge / unhappy path）、数据库设计的 NOT NULL 约束、时序图里"无数据"分支
> - 子字段说明里追加"（仅当 父 ≠ null）"作为兜底提醒，避免下游开发者只看子字段忽略容器可空性
> - **反例**：
>   - schema 表只列 `data.pet.id` / `data.pet.name`，未列 `data.pet` 容器 → DTO 把 pet 当成非 optional → edge case 解码失败
>   - schema 表 `data.room.currentRoomId | string | null` 列了，但没列 `data.room` 容器是否可空 → 同样的容器层盲区

## Lesson 2: JSON 示例的标签语义必须严格匹配示例内容，不能给"未来才会有的字段"提前贴标签

- **Severity**: P3 (Low)
- **Category**: docs
- **分诊**: fix
- **位置**: `docs/宠物互动App_V1接口设计.md` line 389-413

### 症状（Symptom）

JSON 示例上方标签写：

> 节点 9 / 节点 10 之后的真实数据，pet.equips 由 Story 26.6 填充，pet.equips[].renderConfig 由 Story 29.6 填充

但实际 sample payload 里 `pet.equips[0]` 只有 `slot` / `userCosmeticItemId` / `cosmeticItemId` / `name` / `rarity` / `assetUrl`，**没有 `renderConfig`**。

读者按节点 10 语义 copy 此示例时，会得到一个"标签说有 renderConfig，sample 没有 renderConfig"的自相矛盾载体。下游 iOS / Go 实装容易因此疑惑：到底这个示例是对的还是错的？

### 根因（Root cause）

写 JSON 示例时，作者想用**一个示例**覆盖**两个节点**（节点 9 + 节点 10），所以在标签上同时声明"两节点之后"+"由 Story 26.6 / 29.6 填充"。但 sample 实际只展示了节点 9 的字段（pet.equips 有数据），**节点 10 才落地的 renderConfig 字段没补**（因为 Story 29.6 的具体 renderConfig schema 还没冻结）。

这种"标签声明了未来字段，但 sample 只覆盖当前已冻结字段"的状态，是文档作者**用一个示例承担多个节点的展示职责**导致的内在不一致。正确做法：要么"标签语义匹配 sample 实际内容"（仅声明节点 9），要么"sample 内容补全所有标签声明的字段"（节点 10 也展示 renderConfig）—— 二选一，不能让标签跑在 sample 前面。

### 修复（Fix）

选简单修法 a：把标签限定到节点 9，节点 10 + renderConfig 留待 Story 29.6 落地时单独补示例。

```diff
-JSON 示例（节点 9 / 节点 10 之后的真实数据，pet.equips 由 Story 26.6 填充，pet.equips[].renderConfig 由 Story 29.6 填充）：
+JSON 示例（节点 9 之后的真实数据，pet.equips 由 Story 26.6 填充；节点 10 起 pet.equips[] 还会带 renderConfig 子对象，本示例**不**展示 —— 待 Story 29.6 落地时单独补充节点 10 版本示例）：
```

同步：

- V1 文档 §5.1 Future Fields 引用块里"见上方'节点 9 / 节点 10 之后的真实数据'示例" → 改为"见上方'节点 9 之后的真实数据'示例"
- 故事文件副本 `_bmad-output/implementation-artifacts/4-1-接口契约最终化.md` line 297 的相同标签同步

### 预防规则（Rule for future Claude）

> **一句话**：未来 Claude 在写 JSON 示例时，**示例标签的语义必须严格等于 sample payload 的实际字段集**：标签声明的每个字段必须在 sample 里出现，sample 里不展示的字段不能在标签里声明"由 Story X 填充"。
>
> **展开**：
> - 标签描述未来字段时，明示该字段"本示例**不**展示，待 Story X 落地时单独补示例"，把缺失变成显式声明而不是暗坑
> - 一份 sample 不能同时承担多节点的展示职责。如果想覆盖节点 9 + 节点 10，就写两份 sample（节点 9 版 + 节点 10 版），各自标签清晰
> - 修标签或修 sample 时，必须 sweep 所有引用该 sample 的文档块（如 Future Fields 引用），同步标签文字 —— 标签一变，引用必变
> - **反例**：
>   - 标签写"节点 9 / 节点 10 之后的数据，含 renderConfig" + sample 没 renderConfig → 读者陷入二选一的疑惑
>   - 改了 sample 里某个字段名，没改标签里对该字段的引用 → 标签 + sample 同节自相矛盾
>   - 标签里贴"由 Story X 填充"但 sample 不演示该字段 → 给未来开发者一个"自己脑补 Story X schema"的空白题，多人脑补必产生分歧

## Meta: 本次 review 的宏观教训

两条 lesson 共同指向一个元问题：**契约文档的"完整性"不能用"happy case 看起来对"来自我安全感**。

具体表现：

- Lesson 1：happy case 字段表"看起来很完整"（user / pet / step / chest / room 全列了），但作者没翻 edge case AC 去 cross-check 每层容器的可空性 → 容器层语义被"happy case 顺手写"埋掉
- Lesson 2：JSON 示例"看起来覆盖了未来"（标签写了节点 9 + 节点 10 + Story 26.6 + Story 29.6），但 sample 实际只展示了 happy case 字段集 → 标签的"未来语义"成了空头支票

**元规则**：写契约文档（schema 表 / JSON 示例 / Future Fields 块）的最后一步**必须**做下面三个 cross-check：

1. **AC sweep**：对每个被写入文档的接口，grep epics.md 该接口对应 story 的所有 AC（尤其 edge / unhappy），看每条 AC 的字段语义在文档里是否被显式覆盖
2. **示例 vs 标签 sweep**：对每个 JSON 示例，对照标签里声明的所有字段名，逐个在 sample payload 里查找；不存在 → 修标签（限定语义）或修 sample（补字段），二选一
3. **跨副本 sweep**：契约文档改了某节，必须 grep 该节内容在故事文件副本（`_bmad-output/implementation-artifacts/<story>.md`）里的复制，同步修改 —— 否则 round X+1 review 会再次发现副本不同步（参见 round 4 lesson `2026-04-26-契约文档全文sweep与跨文件副本同步.md`）

本轮其实就是 round 4 元规则的延续应用：round 4 的元教训是"改契约文档必须 grep 全文 + 所有副本"，本轮发现的两处缺陷就是 round 4 sweep 不够全留下的。证明元规则有效但执行强度还可以再提一档：**sweep 时不仅要按文字 grep，还要按 AC 语义 cross-check**（容器可空性 / 示例字段集等"语义层面"的同步问题，单凭文字 grep 抓不到）。
