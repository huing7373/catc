## Lesson 1: 冻结契约不得把"哪个错误码"留给实装层二选一——会让被钦定的单测 case 永不可达

- **Severity**: medium (P2)
- **Category**: docs（契约设计）
- **分诊**: fix
- **位置**: `docs/宠物互动App_V1接口设计.md:1492`（服务端逻辑步骤 4）+ `:1542`（错误码表 5001 行）

### 症状（Symptom）

§8.3 equip step 4 原文："行不存在 → 5001（含'实例 id 不存在'与'实例存在但不属于当前用户'两种 —— 实装层可合并为 `WHERE id=? AND user_id=?` 查不到即 5001；若需区分返 5002，由实装层决定，契约层接受两种映射，但 5002 必须可达）。"

同一段落同时声明"契约接受 5001/5002 两种映射"**和**"5002 必须可达"——两句话互斥：若实装选了被允许的"合并查询返 5001"路径，他人实例场景永远返 5001，5002 在该 case 下不可达，client 永远见不到文档承诺的 5002。且 epics.md §Story 26.3 AC 第 3561 行把 `edge: 实例不属于当前用户 → 5002` 列为**必须覆盖的单测 case**——选 5001 实装时该单测无法通过。

### 根因（Root cause）

写冻结契约时为"给实装留灵活度"把**可观测的对外行为**（哪个 case 返哪个错误码）降级成实装自由项。错误码映射是 client 依赖的契约面（iOS 按 5001/5002 走不同 UI 文案，见 epics.md §Story 27.x AC `5001 → "道具已不存在"` / `5002 → "道具不属于你"`），不是实装内部细节。把它写成"二选一"等于让契约对同一输入有两个合法输出，下游 iOS 和 server 各按一种实装就会观测到不一致。"实装弹性"只能给**不可观测的内部实现**（用一条 SQL 还是两条、加不加索引），绝不能给**对外可观测的错误码/字段/状态**。叠加坑：同段落自己又写了"5002 必须可达"的反向约束，等于契约文档内部自相矛盾还没被作者自检发现。

### 修复（Fix）

把 step 4 改成单一确定映射，消除实装自由度：

- step 4 改为：`SELECT ... FROM user_cosmetic_items WHERE id = ?`（**仅按 id 查，禁止本步加 `AND user_id = ?` 过滤**）→ 行完全不存在 → 5001；行存在但 `user_id != 当前用户` → **强制 5002**；行存在且属当前用户 → 继续。附"5002 必须可达（不变量）"说明，明确"实装无自由度"。
- 错误码表 5001 行收紧为"实例 id **完全无 row**（**仅**此一种；'实例存在但属他人'恒走 5002，**不**得合并进 5001）"。
- 自检：§8.3 全段 5001/5002/5003/5008 语义已自洽；§3 错误码表（codes 未删）、§1 冻结边界（仍列 `equip：5001/5002/5003/5008`，只是 case→code 映射去歧义）、DB 设计 §8.4（高层"校验实例归属"不规定具体码，不冲突）、epics.md §26.3 AC（`实例不属于当前用户 → 5002` 现可达）全部一致。

### 预防规则（Rule for future Claude）⚡

> **一句话**：未来 Claude 在写/改**冻结契约**（尤其多端共享、被 epics AC 单测钦定的）时，**禁止**把任何**对外可观测行为**（错误码映射 / 字段值 / 状态流转）写成"实装层可二选一 / 由实装决定 / 契约接受多种映射"——可观测行为必须是输入→输出的**单一确定函数**。
>
> **展开**：
> - "实装弹性"只能授予**不可观测的内部实现细节**（一条 SQL vs 两条、索引选择、缓存与否）；判据：换实装时 client 能否观测到差异？能 → 必须钉死，不能 → 可留弹性。
> - 写完一段契约立刻做**自洽自检**：本段是否同时存在"允许 X"和"必须非 X"的互斥句？被 epics.md AC 列为单测 case 的行为，契约里对应的码/字段是否被某条"实装可选"分支架空？grep 该错误码/字段在本接口全文出现处，逐处确认语义不打架。
> - 错误码是 client 的契约面：iOS 按不同码走不同 UI（5001→"已不存在" vs 5002→"不属于你"），所以"返哪个码"恒为契约硬约束，绝非实装细节。
> - **反例**：契约写"行不存在 → 5001（实装层可合并查询即查不到都返 5001，或区分返 5002，二选一），但 5002 必须可达"——一段话里"二选一"与"必须可达"互斥，且 epics 钦定了 `→5002` 单测 case，选 5001 实装直接让该单测无法通过，iOS/server 各按一种实装即跨端行为分裂。这种"为灵活而灵活"的契约措辞是冻结契约头号反模式。

## Meta: 本次 review 的宏观教训

冻结契约的危险措辞家族："实装层可…由实装决定…契约接受多种…"。这类措辞每出现一次，都要追问："这个被留弹性的点，client / 另一端能观测到差异吗？被任何 epics AC 单测钦定了吗？"任一为是 → 该点必须钉成单一确定行为。契约的价值正是消除歧义；给可观测行为留二选一等于把歧义写进权威源。
