## Lesson 1: 接口文档"全局类型规则"必须显式标出例外字段，不能靠"all 后面默认收敛"

- **Severity**: P2
- **Category**: docs
- **分诊**: fix
- **位置**: `docs/宠物互动App_V1接口设计.md:68` (§2.5 字段类型与编码约定)

### 症状

§2.5 写"**所有**时间字段统一使用 ISO 8601 UTC 格式"，但同一文档 §6.2 `POST /steps/sync` 请求体定义 `clientTimestamp: 1776920345000`（epoch milliseconds 整数）。下游实现按 §2.5 的全局规则会把 `clientTimestamp` 序列化成 ISO 字符串，与 step-sync story 既有形态冲突。

### 根因

写"全局类型约定"章节时，作者只想着 response 里的 `unlockAt` / `createdAt` 这类 datetime 字段，没盘点全文档其他章节的 request 字段。`clientTimestamp` 是 request 端的客户端心跳时间戳，定义在另一个章节，作者写 §2.5 时没回到 §6.2 校验是否一致。"all 时间字段" 的 quantifier 表述在 LLM / 人类读者眼里都是绝对全称命题，不会自动派生例外。

### 修复

把 §2.5 时间字段规则**收紧到 response datetime 类型**，并显式列出 request epoch ms 字段的例外：

```diff
- - **时间字段**：所有时间字段统一使用 ISO 8601 UTC 格式（如 `"unlockAt": "2026-04-23T10:20:00Z"`），客户端按本地时区显示。
+ - **时间字段**：**response 中的 datetime 类型字段**统一使用 ISO 8601 UTC 字符串（如 `"unlockAt": "2026-04-23T10:20:00Z"`），客户端按本地时区显示；**request 中的 epoch 时间戳类字段**（如 `POST /steps/sync` 的 `clientTimestamp`）保留 number 类型（毫秒整数），不做 ISO 字符串化 —— 该类字段在各接口章节单独标注。
```

### 预防规则（Rule for future Claude）

> **一句话**：在接口契约文档写"全局类型 / 编码 / 格式"约定时，**禁止**用"所有 X 字段都 Y"这种绝对全称命题，**必须**先 grep 全文确认字段范围 + 显式列出例外字段及其原因。
>
> **展开**：
> - 写 §2.5 / §2.x 这类 cross-cutting 约定时，先 `Grep "时间|timestamp|clientTimestamp|unlockAt|createdAt"` 等候选字段名扫一遍全文档，盘点出现位置和当前类型。
> - 当某个 cross-cutting 规则要加进来时，按 **request 字段 / response 字段 / WS payload 字段**三类分开陈述，不要写"所有"。
> - **反例**：`所有时间字段统一使用 ISO 8601 UTC` —— 不分 request/response，且 grep 结果会立刻找到 `clientTimestamp` 是 epoch ms 整数。
> - **正例**：`response 中的 datetime 类型字段用 ISO 8601 UTC 字符串；request 中的 epoch 时间戳类字段（如 clientTimestamp）保留 number 类型`，明示分类 + 显式例外。

## Lesson 2: Future Fields 承诺必须对应 planning artifacts 中已规划的 story，否则改成死信明示

- **Severity**: P2
- **Category**: docs
- **分诊**: fix
- **位置**: `docs/宠物互动App_V1接口设计.md:303-305` (§4.3 GET /me Future Fields)

### 症状

§4.3 GET /me Future Fields 注释写 `data.user.currentRoomId` "节点 4 房间链路落地后，由 Story 11.x 在加入 / 退出房间事务里维护"，引导客户端按可空解析、未来会有真实数据。但 `_bmad-output/planning-artifacts/epics.md` 中**只有 Story 11.10** 给 `GET /home` 的 `room.currentRoomId` 加了真实化（line 2005-2024），**没有** /me 的对应 story。下游若按这条注释期待 `/me.user.currentRoomId` 未来会有值，会永远等不到。

### 根因

写 future fields 注释时，作者复用了"节点 4 起 currentRoomId 会有真实值"的模式（这一条对 /home 是真的，对 /me 不是），没有在 planning artifacts 里逐字段确认 backfill story 是否真的存在。两个接口都返回 currentRoomId，但只有一个被规划成"未来由 Story 11.10 写入真实数据"，另一个静默废弃 —— 文档没有跟上这个二分。

### 修复

把 /me 的 future field 改成"始终为 null + 明示无 planned story + 引导客户端使用 /home"：

```diff
- > **Future Fields**（节点 4 起激活）：
- > - `data.user.currentRoomId`：节点 4 房间链路落地后，由 Story 11.x 在加入 / 退出房间事务里维护；节点 2 阶段服务端**强制**返回 `null`，客户端**必须**按可空解析。
+ > **关于 `data.user.currentRoomId`**：本字段**保留为 schema 占位**但**不计划在后续节点回填真实数据**（planning artifacts 中无对应 story）。服务端**始终**返回 `null`，客户端**必须**按可空解析；获取真实房间归属请改用 `GET /home` 的 `data.room.currentRoomId`（节点 4 由 Story 11.10 回填真实数据）。
```

### 预防规则（Rule for future Claude）

> **一句话**：在接口契约文档写 "Future Fields / 节点 X 起激活" 类承诺时，**必须**先去 `_bmad-output/planning-artifacts/epics.md` 确认对应 story 真实存在；找不到 → **必须**改成"始终 null / 不计划回填" + 引导客户端用替代字段，**禁止**留空头支票。
>
> **展开**：
> - 每写一条 `> Future Fields ...` 注释，配套 `Grep` 一遍 planning artifacts，确认承诺的 "Story X.Y" 真在 epic / story 列表里且 AC 提到该字段。
> - 同名字段在不同接口（如 `/me` 和 `/home` 都有 `currentRoomId`）不要假设 backfill story 都覆盖 —— 必须分别确认。
> - 多接口都返回相同语义字段时，**优先选一个 canonical 接口**（如 /home 是首页聚合）作为 backfill 目标，其他接口里该字段标"始终 null + 引导用 canonical 接口"，避免重复维护。
> - **反例**：`data.user.currentRoomId：节点 4 房间链路落地后，由 Story 11.x 在加入 / 退出房间事务里维护` —— Story 编号占位 `11.x`、未具体到 11.10、且 planning artifacts 里没有 /me 字段相关 story，是无锚点的承诺。
> - **正例**：明示"始终为 null + 无 planned story" + 引导客户端用 `/home` 的同名字段；如果未来真要 /me 也回填，必须先在 epics.md 加 story 再回头改这条注释。

## Meta: 本次 review 的宏观教训

两条 finding 指向同一个 docs 思维漏洞：**契约文档的全局陈述 / 远期承诺 与文档其他位置或外部锚点（planning artifacts）必须双向自洽**。

具体表现：
- "全局类型规则"vs"具体字段类型"：一边定 cross-cutting 约定，一边在具体 endpoint 章节定字段，写 cross-cutting 时不回看具体章节 → 全称命题踩例外。
- "Future Fields 承诺"vs"planning artifacts story 列表"：一边写"未来由某 story 实装"，一边的 story 列表实际只覆盖部分字段 → 承诺脱锚。

future Claude 写 / 改 / review 这类契约文档时，**写完 cross-cutting 章节立刻回扫 endpoint 字段**、**写完 Future Fields 立刻回扫 planning artifacts**，把一致性检查内联到当前编辑会话里，不要积累到 review round 才被 codex 抓出来。
