# 宠物互动App MVP节点规划与里程碑

## 1. 文档目标

本文档用于沉淀当前 MVP 的研发推进顺序、每个节点的目标、范围、验收标准与阶段产出。

本文档主要回答 4 个问题：

- 当前版本先做什么，后做什么
- 每个节点的边界在哪里
- 做到什么程度算完成
- 哪些能力明确后置，不在当前节点内扩张

---

## 2. 当前 MVP 节点规划原则

当前项目的节点设计遵循以下原则：

1. **先打通工程与链路，再叠加玩法**
   - 先证明 iOS、Go、MySQL、Redis、HTTP、WebSocket 能串起来
   - 再逐步补登录、猫动作、房间、同步、表情、宝箱、仓库、穿戴、渲染、合成、分享

2. **先完成房间骨架，再补房间内实时内容**
   - 先做房间创建、加入、退出、成员进入退出通知
   - 再做猫动作同步
   - 再做表情广播

3. **先验证宝箱链路，再接入资产归仓**
   - 先完成倒计时、开箱、奖励弹窗与奖励配置
   - 再把奖励真正写入仓库

4. **先完成仓库，再做穿戴；先完成穿戴，再叠加渲染配置**
   - 仓库提供查看实例的能力
   - 穿戴完成"道具能挂在猫身上"的事务模型；视觉缺失时以文字代替，保证可独立验收
   - 渲染配置补齐美术：让每件装扮在猫身上的位置 / 大小 / 朝向有可分辨差异

5. **合成依赖仓库，但不依赖穿戴 / 渲染**
   - 合成只读取实例数据 + 写新实例，不关心是否穿戴或可视化
   - 节点 11 可与节点 9 / 10 并行推进

6. **分享能力后置**
   - 房间主链路与链接深链能力分开处理

7. **基础设施按需引入，不一次堆完**
   - 节点 1 仅做 `cmd/server/main.go` + 配置 + Gin + ping + 重做 `scripts/build.sh` + **Server / iOS 测试基础设施**
   - **MySQL 推迟到节点 2**（首次需要持久化时引入）
   - **Redis + WebSocket 网关推迟到节点 4**（首次需要 presence / 实时态时引入）
   - `auth` / `rate_limit` 中间件推迟到节点 2
   - 这样避免节点 1 范围爆炸，也保证每个基础设施都"刚好在被需要时引入"，符合"先打通再叠加玩法"

8. **测试基础设施必须在节点 1 内建立**
   - server 和 iOS 各自需在节点 1 完成测试库选型 / mock 策略 / 本地跑法 / CI 配置
   - 不允许等到首次需要测试时才临时补

## 3. 节点总览

| 节点 | 名称 | 核心目标 |
|---|---|---|
| 节点1 | App 与 Server 可运行 | 打通最小工程与 ping 联调 |
| 节点2 | 默认游客登录 | 实现无登录页自动游客登录 |
| 节点3 | 小猫动作与步数上报 | 本地猫动作表现 + 服务端步数接收 |
| 节点4 | 基础房间功能 | 创建、加入、退出、成员进出通知 |
| 节点5 | 房间内猫动作同步 | 房间成员可看到其他人的猫动作状态 |
| 节点6 | 房间内表情发送 | 点击自己的猫发送表情并广播给他人 |
| 节点7 | 宝箱倒计时与奖励弹窗 | 倒计时、1000 步开箱、奖励展示、道具配置 |
| 节点8 | 仓库功能 | 奖励入仓并可查看拥有的道具（仅展示） |
| **节点9** | **穿戴 / 卸下** | 玩家从仓库选择实例穿戴 / 卸下；缺图时以文字代替图像 |
| **节点10** | **装扮渲染配置** | server 按 cosmetic_items.render_config 提供渲染参数；client 按参数渲染 |
| 节点11 | 合成功能 | 手动选择 10 个同品质实例合成更高品质道具 |
| 节点12 | 分享房间链接 | 生成房间链接并支持通过链接加入 |

## 4. 节点详细说明

### 4.1 节点1：App 和 Server 能运行

#### 节点目标

完成最小工程骨架，证明客户端与服务端都可以本地稳定启动，并完成最小联调。**严格收紧范围，避免基础设施一次堆完导致节点 1 范围爆炸**。

#### 节点范围

服务端（**仅这些**）：
- `cmd/server/main.go` 入口
- 配置加载（YAML，多环境占位）
- Gin 路由 + `GET /ping` 接口
- 中间件：`request_id` / `recover` / `logging`
- 重做 `scripts/build.sh`（对齐新结构）
- **Server 测试基础设施**：mock 库选型 / fixture / 本地 `go test` + race + cover 跑法 / CI 配置

客户端（**仅这些**）：
- 决定 `ios/` 目录改造方案（复用 Cat.xcodeproj 还是新建）
- App 可以启动
- 能看到主界面壳子
- 能显示 App 名称
- `APIClient` 框架 + ping 调用
- **iOS 测试基础设施**：XCTest 配置 + Mock 框架选型 + CI 跑法

联调：
- App 可以请求 ping 接口
- 可以看到响应结果或至少在日志中确认请求成功

#### 当前不做（按需引入原则）

- **MySQL** —— 推迟到节点 2 接入（首次需要持久化）
- **Redis** —— 推迟到节点 4 接入（首次需要 presence）
- **WebSocket 网关** —— 推迟到节点 4 接入
- `auth` / `rate_limit` 中间件 —— 推迟到节点 2
- 登录态 / 宝箱 / 仓库 / 房间

#### 验收标准

- App 在真机或模拟器可稳定启动
- Server 在本地可稳定启动
- `GET /ping` 返回成功
- App 与 Server 之间至少存在一条真实联调链路
- **server 端可跑 `go test ./...` 通过（即使没有业务测试，测试基础设施已搭好）**
- **iOS 端可跑单元测试套件（哪怕只有一条 dummy 测试）**

#### 阶段产出

- iOS 工程初始化
- Go 服务初始化
- 基础环境配置
- ping 联调通过
- **测试基础设施可运行**

### 4.2 节点2：App 和 Server 完成登录功能

#### 节点目标

实现默认游客登录，不提供显式登录页，让用户进入 App 后直接拥有身份。**首次引入 MySQL 与 auth 中间件**。

#### 节点范围

服务端：
- **首次引入 MySQL**：GORM / sqlx 选型 + 连接池 + `migrations/` 框架建立
- migrations：`users` / `user_auth_bindings` / `pets` / `user_step_accounts` / `user_chests` 五张表
- 中间件：`auth`（Bearer token 校验）/ `rate_limit`（保护登录接口）
- 游客登录接口
- 初始化事务（user / binding / 默认 pet / step 账户 / 首个 chest）
- token 签发

客户端：
- App 启动后自动发起游客登录
- App 能持久化 token（Keychain）
- App 后续请求可以自动带 token

#### 当前不做

- 微信绑定
- 登录页 UI
- 昵称头像编辑

#### 验收标准

- 用户首次启动 App 可自动获得游客身份
- 再次启动后不会丢失身份
- Server 可识别同一个游客账号
- 首页后续接口可带登录态访问

#### 阶段产出

- 游客登录接口
- token 存储机制
- 客户端会话管理

### 4.3 节点3：App 和 Server 完成小猫功能

#### 节点目标

实现客户端小猫基础动作表现，并完成步数上报链路。

#### 节点范围

客户端：
- 能看到小猫 `walk / run / idle` 三种状态
- 状态先由客户端本地驱动

服务端：
- 能接收客户端上报的步数
- 能存储步数同步日志

#### 当前不做

- 房间内同步别人猫动作
- 宝箱开箱
- 仓库

#### 验收标准

- App 内小猫三种状态能正常展示
- 客户端可发起步数同步请求
- 服务端能记录并返回步数同步结果

#### 阶段产出

- 本地猫动作逻辑
- 步数同步接口
- 步数日志与账户基础能力

### 4.4 节点4：App 和 Server 完成房间功能

#### 节点目标

先完成房间骨架，只解决房间存在、成员进入退出与成员列表更新。**首次引入 Redis 与 WebSocket 网关**。

#### Server 端拆为两个 Epic（避免单 Epic 过胖）

**Epic A · WebSocket 网关基础设施**：
- **首次引入 Redis**
- WS 网关骨架（连接管理 / Session / Heartbeat / 广播 primitive `BroadcastToRoom(roomId, msg)`）
- Redis presence repo（`room:{id}:online_users` / `user:{id}:ws_session`）
- 心跳超时清理
- 房间快照下发框架（业务内容由 Epic B 填充）

**Epic B · 房间业务**：
- `rooms` / `room_members` 表 migration
- 创建 / 加入 / 退出事务
- 房间快照构造（基于成员关系 + 在线 presence）
- 成员加入 / 离开事件通过 Epic A 的 broadcast primitive 推送

#### 节点范围（业务层面）

- 创建房间
- 通过输入房间号加入房间
- 退出房间
- 房间内成员能看到别人进入和退出
- WebSocket 房间通道建立成功

#### 当前不做

- 看到别人小猫的动作
- 房间内发表情
- 分享房间链接

#### 验收标准

- 用户可创建房间并自动加入
- 其他用户可通过房间号加入
- 房间成员可收到加入/离开广播
- 用户退出后房间成员列表正确刷新

#### 阶段产出

- 房间接口
- 房间 WS 建连能力
- 成员进出广播

### 4.5 节点5：App 和 Server 完成房间内同步功能

#### 节点目标

在已有房间骨架上，增加房间成员的猫动作状态同步。

#### 节点范围

- 房间内可看到其他成员小猫当前状态
- 同步范围仅限 `walk / run / idle`
- 状态通过 WebSocket 广播或房间快照同步

#### 当前不做

- 表情
- 装扮同步
- 精确位置同步

#### 验收标准

- 房间内至少两名用户在线时，可看到对方的猫状态变化
- 状态变化延迟可接受
- 断线重连后可恢复最近状态

#### 阶段产出

- 猫状态同步协议
- 房间内状态广播机制

### 4.8 节点8：App 和 Server 完成仓库功能（仅展示）

#### 节点目标

把宝箱奖励正式写入仓库，并提供基础查看能力。

#### 节点范围

- 宝箱开出的道具自动进入仓库
- 可以打开仓库页面查看自己拥有的道具
- 仓库展示基础信息：名称、品质、部位、数量/实例

#### 当前不做

- 穿戴（拆出独立节点 9；**仓库页可预留穿戴入口按钮位置，行为留空，节点 9 实装真正的事务调用**）
- 仓库复杂筛选与排序
- 道具收藏 / 锁定

#### 验收标准

- 开箱奖励可稳定进入仓库
- 仓库页能看到已有道具
- 仓库数据与后端一致

#### 阶段产出

- 背包接口（catalog + inventory，**不含 equip / unequip**）
- 仓库页面（仅展示）
- 奖励入仓链路

### 4.9 节点9：App 和 Server 完成穿戴 / 卸下功能

#### 节点目标

让玩家可以从仓库选择装扮实例穿戴到猫身上对应槽位，或卸下；保证穿戴关系与实例状态一致；视觉缺失时以文字代替，保证节点 9 可独立验收，**不依赖节点 10**。

#### 节点范围

- 从仓库点选具体实例
- 穿戴到对应槽位（事务）
- 卸下当前装备（事务）
- 同槽位换装：先卸旧再穿新，单事务
- **视觉降级（关键约束）**：当装扮缺少有效 `asset_url` 或未配置 `render_config` 时，客户端使用文字代替（如显示道具名 + 槽位标签），不阻断穿戴流程

#### 当前不做

- 装扮可视化精细渲染（offset / scale / rotation）—— 由节点 10 提供
- 穿戴 / 卸下的精美动画过渡

#### 验收标准

- 玩家可对自己拥有的实例穿戴 / 卸下，状态以 server 为准
- 同槽位换装时新旧实例状态正确切换
- 缺图实例可在猫身上以文字标签展示，不阻断穿戴流程
- `UNIQUE(pet_id, slot)` 与 `UNIQUE(user_cosmetic_item_id)` 约束在事务中得到维护

#### 阶段产出

- 穿戴 / 卸下接口（事务）
- iOS 穿戴交互页
- 缺图文字降级 UI

---

### 4.10 节点10：App 和 Server 完成装扮渲染配置

#### 节点目标

把节点 9 的"文字降级"升级为正式视觉渲染，让每件装扮在猫身上呈现可分辨的位置 / 大小 / 朝向。

#### 节点范围

服务端：
- `cosmetic_items` 表新增 `render_config JSON` 字段（**A 方案**：直接加列）
- 字段结构示例：`{"offset_x":..., "offset_y":..., "scale":..., "rotation":..., "z_layer":...}`
- 通过 migration / seed 写入初始配置
- `catalog` / `inventory` 接口返回 `render_config`

客户端：
- 渲染层按 `render_config` 在猫身上正确显示装扮图像
- 配置缺失时仍能降级为文字（节点 9 行为）

#### 当前不做

- **admin 写入接口**（MVP 通过 migration / seed 维护，未来需要再加）
- 按宠物状态（rest / walk / run）的偏移变体（结构预留扩展位，MVP 不强制实装）
- 装扮间的 z_layer 复杂排序（MVP 阶段简单按字段值排序即可）

#### 验收标准

- 同一槽位的不同装扮在猫身上可视化差异明显（hat A 和 hat B 一眼能区分）
- `render_config` 缺失时客户端仍能降级为文字（节点 9 行为不被破坏）
- 猫动作切换（rest / walk / run）时装扮位置不错位（MVP 可接受 sprite 锚点固定，无 per-state 偏移）
- 通过修改 server 端 seed 即可调整任意装扮的渲染参数，无需 App 重发版

#### 阶段产出

- `cosmetic_items.render_config` 字段 + migration + seed
- catalog / inventory 接口返回 render_config
- 客户端渲染层（替换节点 9 的文字降级为图像渲染，文字降级路径保留作为 fallback）

### 4.11 节点11：App 和 Server 完成合成道具功能

#### 节点目标

完成合成功能，让玩家手动选择材料并获得更高品质道具。

#### 节点范围

- 玩家手动选择 **10 个同品质道具实例**
- 发起合成
- 产出一个更高品质的随机道具实例
- 合成结果自动进入仓库

#### 当前规则

- 不要求同部位
- 不要求相同配置 id
- 只要求品质一致
- 每个材料必须是具体实例 id

#### 当前不做

- 多配方系统
- 一键自动选材
- 合成保底或特殊活动规则

#### 验收标准

- 玩家可以手动选满 10 个材料
- 服务端能校验品质与归属
- 合成成功后材料被消耗
- 新道具进入仓库
- **demo 依据**：合成产出实例进入仓库可见即算通过，**不依赖节点 9（穿戴）或节点 10（渲染配置）的视觉效果**

#### 阶段产出

- 合成接口
- 合成页面与材料选择逻辑
- 合成日志与材料日志

---

### 4.12 节点12：App 和 Server 完成分享房间链接功能

#### 节点目标

让用户可以更方便地邀请他人进入房间。

#### 节点范围

- 生成房间分享链接
- 分享给外部用户
- 通过链接拉起 App
- 通过链接中的房间号加入房间

#### 当前不做

- 复杂分享卡片样式
- 多平台深链适配扩展
- 房间权限体系增强

#### 服务端

- 复用 `/api/v1/rooms/{roomId}/join`，**无新接口**
- 仅需确认 URL 协议规约（route 命名 / scheme / Universal Link 域名等约定可写在 V1接口设计 文档）

#### 验收标准

- 可生成有效房间链接
- 打开链接可解析房间号
- 已安装 App 时可拉起并尝试加入房间

#### 阶段产出

- 房间分享参数设计
- 客户端深链解析
- 通过链接加入房间的跳转链路

## 5. 节点依赖关系

推荐依赖顺序如下：

## 7. 当前版本最值得注意的边界

### 7.1 节点 4 与 节点 5 不要合并过早

房间骨架与猫动作同步是两件事。先完成房间生命周期，再补实时状态，会更稳。

### 7.2 节点 7 与 节点 8 的拆分是合理的

先验证宝箱倒计时、1000 步开箱与奖励抽取，再把奖励真正接进仓库，可以降低返工风险。

### 7.3 节点 9 在节点 10 之前可独立验收

文字降级机制保证节点 9（穿戴）即使在没有 render_config 的情况下也能完整验收：玩家可以"穿戴"任何实例到对应槽位，UI 会以"道具名 + 槽位"文字形式展示在猫身上。这避免了节点 10 成为节点 9 的阻塞前置。

### 7.4 节点 10 是节点 9 的视觉升级

节点 10 不引入新的业务能力，只是把节点 9 的"文字降级"升级为图像渲染。它本身不阻塞任何后续节点。

### 7.5 节点 11（合成）必须建立在"实例化道具"基础上

由于玩家需要手动选择道具材料，且每个道具实例都有唯一 id，所以合成不能建立在"纯数量型库存"之上。节点 11 只依赖节点 8，**可与节点 9 / 10 并行推进**。

### 7.6 节点 12 可以严格后置

房间功能本身优先级高于分享链接。分享能力适合在房间链路稳定后再加。节点 12 server 端无新接口，仅复用 `/rooms/{id}/join`，工作量集中在 iOS Universal Link 接入。

## 8. 当前确认的关键规则

当前项目已经确认的关键规则如下：

- 默认游客登录，后续可绑定微信
- 步数读取 iPhone 系统统计步数
- 跑步与步行收益一致
- 宝箱同时只能存在 1 个
- 宝箱倒计时为 10 分钟
- 开启宝箱消耗 **1000 步**
- 开箱奖励先展示，再进入仓库
- 每个道具是实例化数据，每件实例都有唯一 id
- **装扮渲染参数（render_config）按 cosmetic_item 配置，缺失时客户端文字降级**
- **render_config MVP 阶段通过 migration / seed 维护，不做 admin 写入接口**
- **基础设施按需引入**：节点 1 不接 MySQL / Redis / WS，仅 cmd/server + Gin + ping + 测试基础设施
- **测试基础设施必须在节点 1 内建立**（server 端 + iOS 端各自）
- **`cosmetic_items` 预置数据数量约束**：common ≥ 8 / rare ≥ 4 / epic ≥ 2 / legendary ≥ 1，覆盖 ≥ 4 个槽位（保证开箱 / 合成 demo 不会反复出同一件）
- 合成时玩家手动选择 **10 个同品质道具实例**
- 合成不要求同部位，也不要求相同配置 id
- 合成结果是一个更高品质的随机装扮实例
- 房间最多 4 人
- 用户同时只能加入 1 个房间
- 房间内可发送固定系统表情
- 房主不能踢人
