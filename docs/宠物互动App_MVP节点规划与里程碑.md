# 宠物互动App MVP节点规划与里程碑

## 1. 文档目标

本文档用于沉淀当前 MVP 的研发推进顺序、每个节点的目标、范围、验收标准与阶段产出。

本文档主要回答 4 个问题：

- 当前版本先做什么，后做什么
- 每个节点的边界在哪里
- 做到什么程度算完成
- 哪些能力明确后置，不在当前节点内扩张

---

## 2. 当前 MVP 节点规划原则

当前项目的节点设计遵循以下原则：

1. **先打通工程与链路，再叠加玩法**
   - 先证明 iOS、Go、MySQL、Redis、HTTP、WebSocket 能串起来
   - 再逐步补登录、猫动作、房间、同步、表情、宝箱、仓库、合成、分享

2. **先完成房间骨架，再补房间内实时内容**
   - 先做房间创建、加入、退出、成员进入退出通知
   - 再做猫动作同步
   - 再做表情广播

3. **先验证宝箱链路，再接入资产归仓**
   - 先完成倒计时、开箱、奖励弹窗与奖励配置
   - 再把奖励真正写入仓库

4. **先完成仓库，再做合成**
   - 合成依赖道具实例化、仓库存储与选择材料能力

5. **分享能力后置**
   - 房间主链路与链接深链能力分开处理

---

## 3. 节点总览

| 节点 | 名称 | 核心目标 |
|---|---|---|
| 节点1 | App 与 Server 可运行 | 打通最小工程与 ping 联调 |
| 节点2 | 默认游客登录 | 实现无登录页自动游客登录 |
| 节点3 | 小猫动作与步数上报 | 本地猫动作表现 + 服务端步数接收 |
| 节点4 | 基础房间功能 | 创建、加入、退出、成员进出通知 |
| 节点5 | 房间内猫动作同步 | 房间成员可看到其他人的猫动作状态 |
| 节点6 | 房间内表情发送 | 点击自己的猫发送表情并广播给他人 |
| 节点7 | 宝箱倒计时与奖励弹窗 | 倒计时、1000 步开箱、奖励展示、道具配置 |
| 节点8 | 仓库功能 | 奖励入仓并可查看拥有的道具 |
| 节点9 | 合成功能 | 手动选择 10 个同品质实例合成更高品质道具 |
| 节点10 | 分享房间链接 | 生成房间链接并支持通过链接加入 |

## 4. 节点详细说明

### 4.1 节点1：App 和 Server 能运行

#### 节点目标

完成最小工程骨架，证明客户端与服务端都可以本地稳定启动，并完成最小联调。

#### 节点范围

客户端：
- App 可以启动
- 能看到主界面壳子
- 能显示 App 名称

服务端：
- Go Server 可以启动
- 提供 ping 接口
- 对 ping 请求能返回成功响应

联调：
- App 可以请求 ping 接口
- 可以看到响应结果或至少在日志中确认请求成功

#### 当前不做

- 登录态
- 宝箱
- 仓库
- 房间
- WebSocket

#### 验收标准

- App 在真机或模拟器可稳定启动
- Server 在本地可稳定启动
- `GET /ping` 或等价接口返回成功
- App 与 Server 之间至少存在一条真实联调链路

#### 阶段产出

- iOS 工程初始化
- Go 服务初始化
- 基础环境配置
- ping 联调通过

---

### 4.2 节点2：App 和 Server 完成登录功能

#### 节点目标

实现默认游客登录，不提供显式登录页，让用户进入 App 后直接拥有身份。

#### 节点范围

- App 启动后自动发起游客登录
- Server 能创建或复用游客账号
- Server 返回 token
- App 能持久化 token
- App 后续请求可以自动带 token

#### 当前不做

- 微信绑定
- 登录页 UI
- 昵称头像编辑

#### 验收标准

- 用户首次启动 App 可自动获得游客身份
- 再次启动后不会丢失身份
- Server 可识别同一个游客账号
- 首页后续接口可带登录态访问

#### 阶段产出

- 游客登录接口
- token 存储机制
- 客户端会话管理

---

### 4.3 节点3：App 和 Server 完成小猫功能

#### 节点目标

实现客户端小猫基础动作表现，并完成步数上报链路。

#### 节点范围

客户端：
- 能看到小猫 `walk / run / idle` 三种状态
- 状态先由客户端本地驱动

服务端：
- 能接收客户端上报的步数
- 能存储步数同步日志

#### 当前不做

- 房间内同步别人猫动作
- 宝箱开箱
- 仓库

#### 验收标准

- App 内小猫三种状态能正常展示
- 客户端可发起步数同步请求
- 服务端能记录并返回步数同步结果

#### 阶段产出

- 本地猫动作逻辑
- 步数同步接口
- 步数日志与账户基础能力

---

### 4.4 节点4：App 和 Server 完成房间功能

#### 节点目标

先完成房间骨架，只解决房间存在、成员进入退出与成员列表更新。

#### 节点范围

- 创建房间
- 通过输入房间号加入房间
- 退出房间
- 房间内成员能看到别人进入和退出
- WebSocket 房间通道建立成功

#### 当前不做

- 看到别人小猫的动作
- 房间内发表情
- 分享房间链接

#### 验收标准

- 用户可创建房间并自动加入
- 其他用户可通过房间号加入
- 房间成员可收到加入/离开广播
- 用户退出后房间成员列表正确刷新

#### 阶段产出

- 房间接口
- 房间 WS 建连能力
- 成员进出广播

---

### 4.5 节点5：App 和 Server 完成房间内同步功能

#### 节点目标

在已有房间骨架上，增加房间成员的猫动作状态同步。

#### 节点范围

- 房间内可看到其他成员小猫当前状态
- 同步范围仅限 `walk / run / idle`
- 状态通过 WebSocket 广播或房间快照同步

#### 当前不做

- 表情
- 装扮同步
- 精确位置同步

#### 验收标准

- 房间内至少两名用户在线时，可看到对方的猫状态变化
- 状态变化延迟可接受
- 断线重连后可恢复最近状态

#### 阶段产出

- 猫状态同步协议
- 房间内状态广播机制

---

### 4.6 节点6：App 和 Server 完成在房间内发送表情的功能

#### 节点目标

在房间实时链路上增加表情交互能力。

#### 节点范围

- 点击自己的小猫
- 弹出交互面板
- 选择固定表情
- 发送到房间内其他成员

#### 当前不做

- 自定义表情
- 表情收藏
- 表情对宠物产生额外效果

#### 验收标准

- 点击自己的猫能弹出表情面板
- 发送后服务端广播成功
- 其他成员可看到对应表情提示

#### 阶段产出

- 表情配置
- emoji WS 协议
- 房间表情广播链路

---

### 4.7 节点7：App 和 Server 完成宝箱倒计时功能

#### 节点目标

先打通宝箱玩法主链路：倒计时、1000 步开箱、奖励配置、奖励展示。

#### 节点范围

- 宝箱倒计时 10 分钟
- 倒计时结束后宝箱可开启
- 开启宝箱需要消耗 **1000 步**
- 显示获取了什么道具
- 完成基础道具配置能力

#### 当前不做

- 奖励真正写入仓库
- 装扮穿戴
- 合成

#### 验收标准

- 首页能展示宝箱剩余倒计时
- 倒计时结束后宝箱状态正确变更
- 用户可在步数足够时开启宝箱
- 开启后能看到奖励弹窗

#### 阶段产出

- 宝箱状态流转逻辑
- 奖励抽取逻辑
- 道具配置表/配置能力

---

### 4.8 节点8：App 和 Server 完成仓库功能

#### 节点目标

把宝箱奖励正式写入仓库，并提供基础查看能力。

#### 节点范围

- 宝箱开出的道具自动进入仓库
- 可以打开仓库页面查看自己拥有的道具
- 仓库展示基础信息：名称、品质、部位、数量/实例

#### 当前不做

- 穿戴
- 仓库复杂筛选与排序
- 道具收藏/锁定

#### 验收标准

- 开箱奖励可稳定进入仓库
- 仓库页能看到已有道具
- 仓库数据与后端一致

#### 阶段产出

- 背包接口
- 仓库页面
- 奖励入仓链路

---

### 4.9 节点9：App 和 Server 完成合成道具功能

#### 节点目标

完成合成功能，让玩家手动选择材料并获得更高品质道具。

#### 节点范围

- 玩家手动选择 **10 个同品质道具实例**
- 发起合成
- 产出一个更高品质的随机道具实例
- 合成结果自动进入仓库

#### 当前规则

- 不要求同部位
- 不要求相同配置 id
- 只要求品质一致
- 每个材料必须是具体实例 id

#### 当前不做

- 多配方系统
- 一键自动选材
- 合成保底或特殊活动规则

#### 验收标准

- 玩家可以手动选满 10 个材料
- 服务端能校验品质与归属
- 合成成功后材料被消耗
- 新道具进入仓库

#### 阶段产出

- 合成接口
- 合成页面与材料选择逻辑
- 合成日志与材料日志

---

### 4.10 节点10：App 和 Server 完成分享房间链接功能

#### 节点目标

让用户可以更方便地邀请他人进入房间。

#### 节点范围

- 生成房间分享链接
- 分享给外部用户
- 通过链接拉起 App
- 通过链接中的房间号加入房间

#### 当前不做

- 复杂分享卡片样式
- 多平台深链适配扩展
- 房间权限体系增强

#### 验收标准

- 可生成有效房间链接
- 打开链接可解析房间号
- 已安装 App 时可拉起并尝试加入房间

#### 阶段产出

- 房间分享参数设计
- 客户端深链解析
- 通过链接加入房间的跳转链路

## 7. 当前版本最值得注意的边界

### 7.1 节点 4 与 节点 5 不要合并过早

房间骨架与猫动作同步是两件事。先完成房间生命周期，再补实时状态，会更稳。

### 7.2 节点 7 与 节点 8 的拆分是合理的

先验证宝箱倒计时、1000 步开箱与奖励抽取，再把奖励真正接进仓库，可以降低返工风险。

### 7.3 节点 9 必须建立在“实例化道具”基础上

由于玩家需要手动选择道具材料，且每个道具实例都有唯一 id，所以合成不能建立在“纯数量型库存”之上。

### 7.4 节点 10 可以严格后置

房间功能本身优先级高于分享链接。分享能力适合在房间链路稳定后再加。

## 8. 当前确认的关键规则

当前项目已经确认的关键规则如下：

- 默认游客登录，后续可绑定微信
- 步数读取 iPhone 系统统计步数
- 跑步与步行收益一致
- 宝箱同时只能存在 1 个
- 宝箱倒计时为 10 分钟
- 开启宝箱消耗 **1000 步**
- 开箱奖励先展示，再进入仓库
- 每个道具是实例化数据，每件实例都有唯一 id
- 合成时玩家手动选择 **10 个同品质道具实例**
- 合成不要求同部位，也不要求相同配置 id
- 合成结果是一个更高品质的随机装扮实例
- 房间最多 4 人
- 用户同时只能加入 1 个房间
- 房间内可发送固定系统表情
- 房主不能踢人
