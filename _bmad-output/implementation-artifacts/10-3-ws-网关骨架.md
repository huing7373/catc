## Acceptance Criteria

**AC1 — Go module 依赖与 WS 库选型 ADR**

`server/go.mod` 必须添加 WebSocket 库依赖：

- **库选型**：`github.com/gorilla/websocket`（gorilla/websocket，是 Go 生态最广泛使用的 WS 库；与 V1 §12.2 / §12.3 信封字段类型 100% 兼容；与 stdlib net/http 直接兼容、Gin 通过 gin.WrapH / 拦截 hijack 协议升级即可挂载）
- **版本 pin**：`github.com/gorilla/websocket v1.5.4`（或当前最新稳定 v1.5.x；以 `go get github.com/gorilla/websocket@v1.5.4` 拿到的版本为准；**不**用 latest）
- 添加后跑 `go mod tidy` 让 go.sum / 间接依赖正确锁定
- 不引入其他 WS client（如 nhooyr/websocket / coder/websocket），避免选型不一致

**关键决策（写 ADR-0011 文档）**：

新建 `_bmad-output/implementation-artifacts/decisions/0011-ws-stack.md`，含以下内容：

- §1 上下文：节点 4 引入 WS，需选 server 端 Go WS 库
- §2 候选方案：
  - **gorilla/websocket**（v1.5.4）：✅ Go 生态事实标准；✅ 字段层 ABI 稳定（v1.x API 多年未破坏性改动）；✅ 与 Gin 通过 hijack 直接兼容；⚠️ 包级 maintenance 模式（仍接受 PR 但活跃度不高）
  - **nhooyr.io/websocket**（v1.8.x）：✅ context-aware API 更现代；⚠️ 与 Gin 集成需要更多 boilerplate（不直接支持 hijack 模式）；⚠️ 与 miniredis-style in-process server 的测试 fixture 兼容性未验证
  - **coder/websocket**（v1.8.x）：⚠️ 较新（fork from nhooyr），生态成熟度低于 gorilla
- §3 决策：选 gorilla/websocket
- §4 理由：
  - 项目 ADR-0001 §3 钦定"先求一致再求性能"——gorilla 是事实标准，未来人员上手成本低
  - V1 §12.x 协议骨架已通过 Story 10.1 冻结，gorilla 字段层语义匹配；切其他库不带来契约层收益
  - 与 Gin（已集成）兼容性最好（c.Writer.Hijack 模式是 gorilla 推荐路径）
  - 测试栈 miniredis / sqlmock / dockertest 均与 gorilla 兼容
- §5 反例：不选 nhooyr：API 更现代但 Gin 集成复杂度提升 + 团队学习曲线
- §6 future：节点 9+ 多实例部署 + 性能压测时若发现 gorilla 性能瓶颈，再评估切 nhooyr / coder

**AC3 — `Session` 对象（session.go）**

`server/internal/app/ws/session.go` 必须定义 `Session` struct + 完整 godoc 注释：

```go
// Session 表示单个用户的单个 WS 连接。
//
// Session 内部状态字段对照 Go 项目结构 §9.1 钦定（userID / roomID / conn /
// sendChan / lastHeartbeatAt 五字段），其中：
//   - lastHeartbeatAt 在本 story（10.3）阶段**仅写入**（每次收到 client 消息
//     更新），但**不读**（10.4 心跳超时扫描才会读这个字段做超时判定）；本 story
//     预留字段是为了让 10.4 接管时不需要修改 Session 结构
//
// Session 生命周期：
//
//	1. gateway.go upgrade 完成 → newSession 构造（userID / roomID / conn / sendChan / lastHeartbeatAt=now）
//	2. SessionManager.Register(s)（触发 OnSessionRegister 钩子）
//	3. 同步段写 placeholder room.snapshot（**不**走 sendChan，直接 conn.WriteMessage）
//	4. go s.readLoop() 启动读 goroutine
//	5. go s.writeLoop() 启动写 goroutine（消费 sendChan）
//	6. （生命周期内）readLoop 读到 ping → 推 pong 占位 message 到 sendChan
//	7. （正常断开）client close / readLoop 出错 → s.Close() → SessionManager.Unregister(s)（触发 OnSessionUnregister 钩子）→ writeLoop 退出
//
// 并发安全：
//   - sendChan 是 buffered chan（容量 32，详见本注释末"sendChan 容量决策"）；
//     Send 调用方（多 goroutine）可并发推消息，writeLoop 单 goroutine 消费；
//     不需要额外 mutex
//   - lastHeartbeatAt 用 atomic.Int64（unix ms epoch）；readLoop 写 + 10.4 心跳
//     扫描 goroutine 读，无锁原子访问
//   - closed 字段用 atomic.Bool；多次调用 Session.Close() 只关一次 chan / 触发
//     一次 Unregister；Close 必须幂等（与 *sql.DB / RedisClient.Close 一致）
//
// sendChan 容量决策（buffered=32）：
//   - 0（unbuffered）：Send 必须同步等待 writeLoop 消费 → BroadcastToRoom（10.5）
//     对一个 room 1000 个 Session 串行 Send 会变成线性时间，违反"广播是 fast path"
//   - 1024（过大）：内存浪费 + 客户端慢 / 阻塞时积压消息让用户感受到的是大段
//     stale 消息批量到达，UX 反而差
//   - 32：足够吸收节点 4 阶段（最多 4 user / room）的瞬时 burst（如 4 个 ping 同时
//     到达 server，server 同时回 4 个 pong；同房间 broadcast 4 条消息），同时保持
//     内存占用合理（每 Session 32 * pointer-sized = ~256B）
//
// **关键反例（不要做）**：
//   - 不要把 sendChan 容量设成 1：会让一个客户端的慢消费阻塞 BroadcastToRoom（10.5）
//   - 不要在 Session 上加 mutex 保护 sendChan 的"是否已关"状态判断（用 atomic.Bool
//     标志 + select-with-default 模式更简洁，详见 Send 实装）
type Session struct {
    userID          uint64                  // 已认证 user ID（来自 Story 4.4 token util Claims.UserID）
    roomID          uint64                  // 路径参数 roomId（已通过 §12.1 校验顺序第 5 步：room_members.user_id = userID AND room_id = roomID）
    conn            *websocket.Conn         // gorilla/websocket conn；只在 Session 内部持有，**不**导出
    sendChan        chan []byte             // server → client 消息队列（buffered=32，详见顶部注释）
    lastHeartbeatAt atomic.Int64            // unix ms epoch；本 story 阶段仅由 readLoop 写入；10.4 加扫描 goroutine 读取做超时判定
    closed          atomic.Bool             // 防止多次 Close 重复关闭 chan / 重复 Unregister
    closeOnce       sync.Once               // 防止 sendChan close 多次 panic（Go close-already-closed 会 panic）
    logger          *slog.Logger            // 携带 sessionID / userID / roomID 的 contextual logger（每条 Session 日志都自带这三个字段）
}
```

**Session 接口方法**：

```go
// Send 把消息字节流入队 sendChan。
//
// 选 fire-and-forget 语义而非阻塞 Send：
//   - BroadcastToRoom（10.5）需要"一个慢 client 不阻塞其他 client"，超时丢弃比阻塞合理
//   - sendChan 容量 32 足够吸收瞬时 burst；超过即视为 client 严重滞后，丢弃比积压合理
//   - 如 future 业务消息需要"必送达"语义（如表情广播必须每个客户端都收到），
//     由调用方上层（Service 层）做重传策略，**不**在 Session 层加阻塞 Send
//
// 错误返回：
//   - Session 已 Close → ErrSessionClosed sentinel（非 panic；让调用方安全循环）
//   - sendChan 满 → ErrSessionSendBufferFull sentinel；调用方根据语义决定是否重试
//   - 写 chan 成功 → nil（**不**保证消息已经被 client 收到，仅保证已入队）
//
// **关键约束**：调用方**不应**调用 Send 后假设消息已发送；如需"必送达"语义，
// 必须在 Service 层用 ack message + retry 实装。
func (s *Session) Send(msg []byte) error

// Close 关闭 Session：
//   - 标记 closed
//   - 关闭 sendChan（用 sync.Once 防 close-twice panic）
//   - 关闭底层 *websocket.Conn（gorilla 自身幂等）
//   - 触发 SessionManager.Unregister（间接触发 OnSessionUnregister 钩子）
//
// 必须幂等：多次调用不 panic / 不返 error（与 *sql.DB.Close / RedisClient.Close
// 一致）；与 readLoop / writeLoop 同时调用安全（用 atomic.Bool 防止重入）。
//
// 选不接 ctx：Close 是同步纯内存操作（< 1µs），无 IO 等待；与 *sql.DB.Close 一致。
func (s *Session) Close() error

// readLoop 是读 goroutine 主循环（私有方法）。
//
// 主流程：
//   1. for { msg, err := s.conn.ReadMessage() }
//   2. err != nil → break loop（client 主动 close / 网络错 / read deadline 超时）
//   3. 收到消息 → 更新 s.lastHeartbeatAt = time.Now().UnixMilli()
//   4. 根据 envelope.type 路由：
//      - "ping" → 推 pong 占位 message 到 sendChan（writeLoop 消费）
//      - 其他 type → log warn + 安全忽略（与 V1 §12.3 末尾"安全忽略 + log warn"钦定一致）
//   5. loop 退出 → s.Close()
//
// 解析 envelope：用 json.Unmarshal 到 ClientEnvelope struct（仅含 type + requestId
// + payload 三字段），失败 → log warn + 安全忽略（**不** close 连接，与 V1 §12.3
// 末尾钦定一致）；不严格校验 type 字段长度 / 字符集（10.4 / 业务 epic 阶段才补）
//
// **不在本 story 范围**：
//   - 60s 心跳超时扫描（10.4 才做）
//   - 业务消息（emoji.send 等）路由到 service 层（Epic 11 / 17 才做）
//   - close code 4006 / message too large（10.4 阶段补完）
//   - 真正校验 ping payload 字段表（10.4 阶段补完精确字段表）
func (s *Session) readLoop()

// writeLoop 是写 goroutine 主循环（私有方法）。
//
// 主流程：
//   1. for msg := range s.sendChan { s.conn.WriteMessage(websocket.TextMessage, msg) }
//   2. sendChan 关闭 → loop 退出
//   3. 写错 → log warn + s.Close()（不传播错误给上层；wire 错通常意味着 conn 已死）
//
// **关键约束**：
//   - 必须用 TextMessage（V1 §12.2 关键约束钦定）
//   - 必须设 WriteDeadline（避免慢 client 卡住 server 写）—— 节点 4 阶段 5s 足够
//     （ConfigKey: ws.write_timeout_sec，YAML 默认 5；prod 不可覆盖，dev 可调）
//   - 写错 → 触发 s.Close() 而非简单 log；wire-level 写失败通常表示 conn 已死，
//     继续 writeLoop 也会持续失败
func (s *Session) writeLoop()
```

**关键约束**（设计层面，dev-story 阶段必须遵守）：

- Session 字段全部小写（不导出）；外部访问通过方法
- 不导出 *websocket.Conn（外部不能直接 Read/Write）
- Send 是 fire-and-forget（详见上文方法注释）
- closeOnce 用 sync.Once 防止 close chan 二次 panic（Go close-already-closed 是 panic 不是 no-op）
- readLoop 和 writeLoop 在 gateway.go 同步段写完 snapshot **之后**才启动（**严格按 V1 §12.1 钦定的 5 步握手成功顺序**：1. 创建 Session 2. Register 3. 同步段写 snapshot 4. 启动读/写 goroutine 5. （10.6 阶段）Redis presence）
- Session 必须挂 contextual logger：每条日志带 sessionID（用 google/uuid.New() 生成 short）+ userID + roomID 三字段，方便 grep 排障

**AC4 — `SessionManager`（session_manager.go）**

`server/internal/app/ws/session_manager.go` 必须定义 `SessionManager` 接口 + 进程内 in-memory 实装：

```go
// SessionManager 是进程内全局 Session 注册中心。
//
// 设计目的：
//   - 让 BroadcastToRoom（Story 10.5）通过 ListSessionsByRoomID 拿到目标 Session 列表
//   - 让 Story 10.6 Redis presence repo 挂在 OnSessionRegister / OnSessionUnregister 钩子上
//   - 让 graceful shutdown（Epic 36+）批量遍历所有 Session 主动 close 1001
//
// 范围边界：
//   - 进程内 in-memory（map[sessionID]*Session + map[roomID]map[sessionID]*Session 双索引）
//   - **不**做跨实例 / Pub/Sub（节点 9+ 多实例部署才考虑）
//   - **不**直接消费 Redis client（钩子模式让 Story 10.6 注入，避免直接耦合）
//
// 钩子注入：用 functional option 模式让 bootstrap 期注入：
//
//	mgr := ws.NewSessionManager(
//	    ws.WithRegisterHook(func(s *Session) { presenceRepo.AddOnline(...) }),  // 10.6 加
//	    ws.WithUnregisterHook(func(s *Session) { presenceRepo.RemoveOnline(...) }),
//	)
//
// 本 story 阶段两个钩子默认 no-op（10.6 落地时才注入具体实现）。
type SessionManager interface {
    // Register 注册 Session 到 manager；触发 OnRegister 钩子（如挂的话）。
    // sessionID 由 manager 内部生成（uuid 短串），返回供 Session 持有作为 logger
    // 的关联字段。
    //
    // 同一 user 重复 Register（如重连） → 旧 Session 被强制 Close（用 sentinel
    // ErrSessionReplaced 标识；调用方可在 close hook 里区分日志）。
    // 节点 4 阶段单 user 单 session 假设由本约束兜底；多 session 抢占语义留给
    // Epic 36+。
    Register(ctx context.Context, s *Session) (sessionID string, err error)

    // Unregister 注销 Session；触发 OnUnregister 钩子。
    // sessionID 不存在 → 返 nil（idempotent；与 Session.Close 多次调用一致）。
    Unregister(ctx context.Context, sessionID string) error

    // ListSessionsByRoomID 返回 roomID 对应的所有 active Session（按 sessionID
    // 字典序，让 BroadcastToRoom 调用时遍历顺序确定，便于排障）。
    //
    // roomID 没有任何 Session → 返 ([], nil)。
    // 调用方**不应**修改返回的切片（manager 返回 snapshot，但内部 *Session 共享 ——
    // 修改字段需通过 Session 自身的方法）。
    ListSessionsByRoomID(ctx context.Context, roomID uint64) []*Session

    // Close 关闭 manager：批量 close 所有 Session（不主动推 close 1001 frame，
    // graceful shutdown 业务由 Epic 36 接管）。
    // 必须幂等（与 *sql.DB.Close 一致）。
    Close() error
}

// SessionManagerOption 是 functional option（让 bootstrap 期注入钩子）。
type SessionManagerOption func(*sessionManager)

// WithRegisterHook 注入 Session 注册时的回调钩子（Story 10.6 用来加 Redis presence）。
func WithRegisterHook(fn func(s *Session)) SessionManagerOption

// WithUnregisterHook 注入 Session 销毁时的回调钩子。
func WithUnregisterHook(fn func(s *Session)) SessionManagerOption
```

**实装关键约束**：

- 用 `sync.RWMutex` 保护双索引（不用 channel-based actor 模式，因为读多写少 + sessionID 唯一索引查询是热路径）
- Register 内部生成 sessionID（用 google/uuid.New().String() 取前 8 chars）
- 同 user 重连：触发 ErrSessionReplaced 路径（旧 Session.Close + 新 Session 替换索引）—— 用 user → sessionID 单向映射快速查重
- ListSessionsByRoomID 返回值是 snapshot 切片（read-lock copy），调用方拿到后 manager 改动不影响（避免遍历过程中 manager 改 map 的并发问题）
- Close 必须幂等 + 用 sync.Once 防止 close chan 二次 panic

## Dev Notes

### 实装关键决策

#### 1. WS 库选型：gorilla/websocket vs nhooyr.io/websocket

**决策**：选 gorilla/websocket v1.5.4。

理由：
- Go 生态事实标准，未来人员上手成本低
- V1 §12.x 协议骨架已通过 Story 10.1 冻结，gorilla 字段层语义匹配
- 与 Gin（已集成）兼容性最好（c.Writer.Hijack 模式是 gorilla 推荐路径）
- 测试栈 miniredis / sqlmock / dockertest 均与 gorilla 兼容

**反例**（不要做）：
- 不要选 nhooyr：API 更现代但 Gin 集成复杂度提升
- 不要让两个 WS 库共存（增加测试矩阵 + 维护负担）

详见 ADR-0011（本 story Task 1.4 创建）。

#### 2. WS 路由不挂 RateLimit / Auth 中间件

V1 §12.1 行 1275 钦定 "WS **不**走 HTTP rate_limit"；本 story 实装时**禁止**误把 RateLimit 挂在 WS 路由：

- 鉴权在 Gateway.Handle 内部按 V1 §12.1 校验顺序实装（解析 query → token → roomId → room → membership），失败发 close frame 而非 HTTP 401（与 HTTP auth 中间件路径不同）
- Rate limit 由 Story 10.4 心跳超时框架 + 后续业务 epic 各自的消息频率限制处理（不复用 HTTP RateLimit）

**反例**（不要做）：
- 不要把 WS 路由挂在 `api := r.Group("/api/v1", middleware.Auth, middleware.RateLimit)` 子组下 —— path 都不一样（`/ws/rooms/{roomId}` 不带 `/api/v1` 前缀）
- 不要在 Gateway.Handle 顶部调 middleware.Auth(deps.Signer)(c) —— 校验失败要发 close frame，HTTP 中间件路径写 envelope 是错的

#### 3. 同步段 placeholder snapshot 写入路径（10.7 接管前的临时实装）

V1 §12.1.3 钦定 "room.snapshot 是握手成功后**必发**的第一条消息"；本 story 阶段 SnapshotBuilder 接口尚未引入（10.7 才做），placeholder snapshot 必须 inline 在 gateway.go 同步段写入：

```go
// gateway.go Handle() 第 5 步同步段：
members, err := g.roomMember.ListMembers(ctx, roomID)
if err != nil {
    g.closeWithCode(conn, 1011, "snapshot build failed")
    g.logger.Error("snapshot build failed", slog.Any("error", err), slog.Uint64("roomID", roomID))
    return  // **不**启动读 / 写 goroutine
}
snapshotBytes, err := g.buildPlaceholderSnapshot(ctx, roomID, members)
if err != nil {
    g.closeWithCode(conn, 1011, "snapshot build failed")
    return
}
if err := conn.WriteMessage(websocket.TextMessage, snapshotBytes); err != nil {
    g.closeWithCode(conn, 1011, "snapshot build failed")
    return
}
// **现在**才启动 goroutine（snapshot 已经在 wire 上）
go session.readLoop()
go session.writeLoop()
```

10.7 接管时的替换路径：

- 引入 `SnapshotBuilder` interface（`BuildSnapshot(ctx, roomID) (Snapshot, error)`）
- gateway.go 用 `g.snapshotBuilder.BuildSnapshot(ctx, roomID)` 替换 inline `buildPlaceholderSnapshot`
- inline placeholder 实装移到 `internal/app/ws/snapshot.go` 作为 `placeholderSnapshotBuilder` 实装

**反例**（不要做）：
- 不要在 readLoop 启动后再写 snapshot —— V1 §12.1.3 钦定 first-must-be-snapshot；启动 goroutine 后 server 会先回 client 的 ping → pong 在 snapshot 前到达
- 不要走 sendChan 推 snapshot —— writeLoop 此时还没启动，sendChan 满了无人消费会立刻阻塞 / 失败

#### 6. Session.Send 的 fire-and-forget 语义

Send 必须是 fire-and-forget（详见 AC3 内的方法注释）：

```go
func (s *Session) Send(msg []byte) error {
    if s.closed.Load() {
        return ErrSessionClosed
    }
    select {
    case s.sendChan <- msg:
        return nil
    default:
        return ErrSessionSendBufferFull
    }
}
```

理由：
- BroadcastToRoom（10.5）需要"一个慢 client 不阻塞其他 client"
- sendChan 容量 32 足够吸收节点 4 阶段的瞬时 burst
- "必送达"语义由调用方 service 层用 ack message + retry 实装，**不**在 Session 层加阻塞 Send

#### 7. SessionManager 同 user 重连：抢占语义

节点 4 阶段单 user 单 session 假设：

- 同 user 调用 Register → 旧 Session 被强制 Close（用 sentinel ErrSessionReplaced）
- userToSessionID 单向索引快速查重
- 不实装多 session 抢占（如 web + mobile 双开）—— Epic 36 prod launch 阶段才考虑

**反例**（不要做）：
- 不要让多 session 共存 —— Epic 12 客户端实装 reconnect 时假设单 session；多 session 会让客户端不知道哪个是 active 的
- 不要在 BroadcastToRoom 阶段加"多 session 同 user 都收到广播"语义 —— 单 session 假设让 BroadcastToRoom 简化为遍历 ListSessionsByRoomID

#### 8. close code 必须先 Upgrade 才能 emit

V1 §12.1 钦定校验失败必须发 close frame；**关键约束**：close frame 是 WS 协议层的，必须先 Upgrade 成功（HTTP 101 Switching Protocols）才能 emit：

```go
// gateway.go Handle()：
// 1. 校验失败前的代码路径都要先 Upgrade
conn, err := g.upgrader.Upgrade(c.Writer, c.Request, nil)
if err != nil {
    // Upgrade 自身失败（如 client 不发 Upgrade header） → 走 HTTP 400 路径
    // gorilla.Upgrade 内部已经写 HTTP 400，不需要再处理
    g.logger.Warn("ws upgrade failed", slog.Any("error", err))
    return
}
defer conn.Close()

// 2. 校验失败 → 写 close frame（已经 Upgrade 成功，wire 上是 WS frame）
// 用 conn.WriteControl(websocket.CloseMessage, FormatCloseMessage(code, reason), deadline)
g.closeWithCode(conn, 4001, "missing token")
```

**反例**（不要做）：
- 不要在 Upgrade 之前写 HTTP 401 / 403 —— V1 §12.1 钦定校验失败发 close frame，**不**发 HTTP 错误
- 不要忘记 defer conn.Close() —— close frame 写完后必须 Close 底层 conn

### Anti-patterns to AVOID

1. **不要**直接暴露 `*websocket.Conn`：所有调用方走 Session.Send / Close 接口；只 `internal/app/ws/` 包内部用具体 conn 实例
2. **不要**在 readLoop / writeLoop 内部裸用 `context.Background()` 或 `context.TODO()`：必须用 Session 持有的 ctx（ADR-0007）
3. **不要**写"自定义 in-memory WS mock"：用 httptest.NewServer + gorilla.Dialer 起真实 wire 路径；自定义 mock 容易遗漏 frame 边界 / control message
4. **不要**给 WS 三字段加 env override（`CAT_WS_*`）：保持 env 表面积小，YAML 配置足够；与 Redis DB / PoolSize 同模式
5. **不要**让 redis.Nil-style error 透传给调用方：RoomMemberRepo 三方法明确返 `(bool, nil)` / `([], nil)` 而非透传底层 sql.ErrNoRows（与 redis 抽象层一致）
6. **不要**忘记 main.go defer sessionMgr.Close()：避免 SessionManager 内部 Session map 泄漏（与 mysql / redis 同模式）
7. **不要**在本 story 里实装心跳超时扫描 / BroadcastToRoom / Redis presence / SnapshotBuilder：超出范围，下游 story 各自实装
8. **不要**改 V1 接口设计文档：V1 §12 已在 Story 10.1 冻结，本 story 是字段层契约**实装**，**不**反向修改文档
9. **不要**在 SessionManager 接口里加非"注册 / 注销 / 列表 / 关闭"四类方法：保持单一抽象边界；如 future 真需 broadcast / heartbeat 扫描，**新增**接口方法或独立模块（10.5 broadcast 在新文件 broadcast.go / 10.4 heartbeat 在 session.go 扩 readLoop 钩子）
10. **不要**把 WS 路由挂在 `/api/v1` 子组下：V1 §12.1 钦定 path 是 `/ws/rooms/{roomId}`；挂在 `/api/v1/ws/rooms/...` 会让 client 拼 URL 错（也违反 V1 协议契约）
11. **不要**把 RateLimit / Auth HTTP 中间件挂在 WS 路由：V1 §12.1 行 1275 钦定不走 HTTP rate_limit；鉴权在 Gateway.Handle 内部按 V1 §12.1 校验顺序实装
12. **不要**在 Upgrade 之前发 close frame：close frame 是 WS 协议层的，必须先 Upgrade 成功才能 emit；Upgrade 之前 emit 是 HTTP 路径（错的）
13. **不要**让 readLoop 在写 snapshot 之前启动：V1 §12.1.3 钦定 first-must-be-snapshot；启动 goroutine 后窗口期 client 可能发 ping → server 已有读 goroutine → server 可能先回 pong 让 snapshot 不再是第一条
14. **不要**让 SessionManager 直接消费 RedisClient：用 functional option 模式注入钩子，让 10.6 落地时再挂具体 presence handler（保持单一抽象边界）
15. **不要**预先实装 rooms / room_members migration（Epic 11.2 才做）：本 story 集成测试用 setup 阶段临时建表，Epic 11.2 落地后再切到 official path
16. **不要**忘记 Session.Close 幂等：用 atomic.Bool + sync.Once 防重入；否则 readLoop / writeLoop / SessionManager.Unregister 三处可能并发调 Close 导致 close-already-closed panic

   `closeWithCode` 末尾 `conn.Close()`；happy path conn 生命周期由 `Session.Close`
   接管。这条坑值得归档为 lesson（review fix 阶段如果 codex 命中 → 再补 lesson md）。
3. **conn.Write 路径必须显式 SetWriteDeadline**：handshake 同步段写 snapshot 时
   显式 `conn.SetWriteDeadline(now + writeTimeout)`；writeLoop 每条消息也设；
   `closeWithCode` 走 WriteControl(deadline) —— 三条写路径都带本地 timeout，
   防 lesson `2026-04-26-startup-blocking-io-needs-deadline.md` 钦定的"IO 必带
   deadline"被破。
4. **Same-user reconnect 替换路径**：`Register` 内同 user 重连时，旧 Session
   的 Close **必须**在锁外调（避免 Close → notifyClosed → Unregister 重入死锁）；
   实装上锁内只做 `removeFromIndicesLocked`，把 replaced *Session 留到锁外
   `replaced.Close()`。这是 review-checkable 的并发关键点，review 时务必看
   `session_manager.go Register` 的 unlock 顺序。
5. **测试可观察性**：`ws_test.go` 全部用 httptest.NewServer + gorilla.DefaultDialer.Dial
   起真实 wire 路径（与 redis_test.go / mysql_test.go 用 sqlmock / miniredis 不同模式）。
   理由 V1 §12.x 协议字段层契约太密（close code 数值 + reason 字符串 + envelope
   JSON 字段），mock conn 容易遗漏 frame 边界 / control message。集成测试
