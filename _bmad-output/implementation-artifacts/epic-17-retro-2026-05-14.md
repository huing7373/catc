## 2. 关键洞察与 Epic 17 的"主旋律"

### 2.1 主旋律 1: Story 17.3 4 轮 review 死循环 → 主 agent LLM judgment lock-down

**全 epic 唯一一次需要主 agent 介入"判通过"的事件**。

17.3 codex 在两个互相冲突的设计目标间反复横跳:

| 轮次 | finding 派别 | 提议 | 处理 |
|---|---|---|---|
| r1 | admin 数据保留派 | down DELETE 删 admin 数据 → 改 no-op down | fix → 改成 `SELECT 1;` |
| r2 | migration invariant 派 | no-op down 违反 "down must undo up" | fix → 改回 narrow DELETE |
| r3 | invariant 派 + admin 派 | (P1#1) INSERT IGNORE 留下"坏行" → 改 ON DUPLICATE KEY UPDATE；(P1#2) down 又被指控删 admin 数据 | fix #1 + wontfix #2 |
| r4 | admin 派 | ON DUPLICATE KEY UPDATE silently overwrite admin rows | **视为通过 + flag**（主 agent override） |

**死循环本质**: 每一轮 codex 都"对症"，但**两轮间的 finding 互相反向**:
- "保留 admin" → "保证 invariant" → "保留 admin" → "保证 invariant" ...
- 代码一改，下一轮 codex 立刻能在 baseline diff 里看到上一轮的修复并 raise 反向的 finding

**主 agent 的 lock-down 决断**（写进 r3 lesson 顶部 + `final_decision_lock: true` frontmatter）:
- `up`: ON DUPLICATE KEY UPDATE 强制覆盖 name/asset_url/sort_order/is_enabled
- `down`: narrow DELETE 4 个 owned codes
- **核心语义**: 4 个 code 由 0010 完全占用；admin **禁止** customize 这 4 个 code；新增 emoji 走 0011+ migration
- **后续 reopen 条件**（明文写进 r3 lesson）: (1) 引入了 customizable 0010-equivalent emoji；(2) 引入了 emoji_audit_logs / 灾备恢复机制；(3) 业务定义出现"admin 必须能 hotfix 0010 owned emoji 配置"用例

r4 的 P1 finding 作为 "deferred wontfix decision record" 完整摘录到 r3 lesson 底部，保留诉求 + 主 agent 判定依据 + reopen 条件，避免未来 Claude 再陷入同一个循环。

**Epic-loop 规则的判断边界**: epic-loop.md 命令文档钦定"含 [P1]/[P2] → 不通过"，但同段也说"用 LLM 综合理解，不要硬卡格式"。本 epic 的 17.3 r4 是这条边界的实际触发点 —— 同一类 finding 已被 r3 显式 lock down 后再换 angle 重提 → 主 agent 自由裁量 + 强制退出 review/fix 子循环 + flag 到 lesson。这是 epic-loop 设计的预期路径。

### 2.2 主旋律 2: 17.1 r1 → 17.5 r1 的"契约层早期定下 + 实装层无返工"传导

17.1 r1 codex 抓到的 stale-Session 跨房间注入风险（P1）在契约层就被钉死: §12.2 服务端逻辑步骤 3 的房间归属校验从"`users.current_room_id != NULL`"升级为"`users.current_room_id != NULL` 且 `== Session.RoomID()`"。

这条 lock down 在两个下游 story 直接节省了 review 返工:

- **17.5 dev-story** 实装 `EmojiHandler.HandleEmojiSend` 步骤 3 直接对齐双校验（代码 + 单测 1 case 覆盖 stale Session reject），不需要 codex 在 r1 重新 raise 这个 finding 才发现 —— 17.5 r1 直接 APPROVED 的部分原因
- **17.5 prompt 显式列出** "反 stale-Session 跨房间双校验（17.1 r1 lock down）" 作为关键契约对齐项，让 dev-story sub-agent 在编码前就知道这是必须遵守的约束，而非事后发现

**核心机制**: 契约层 story 的 review 轮次（17.1 = 5 轮）会**前置消化**实装类 story 的潜在 finding；契约 freeze 一旦做对，下游 dev-story 的 review 通过率会显著提升（17.2 / 17.4 / 17.5 都是 r1 通过）。

### 2.3 主旋律 3: codex review --base 的"前面 2000+ 行干扰"陷阱已被解决

第 2+ 轮 codex review 用 `--base <baseline>` 看从 baseline 起的累计 diff，会把上轮 fix-review commit 的 lesson md 全文也当 diff 输出在文件**前面**（2000-4000 行干扰），真实结论永远在文件末尾 `^codex$` 段后的 30-100 行。

epic-loop.md 第 2 次迭代时已经写进 **fix-review sub-agent prompt 的 ⚠️ override #2**："解析顺序: tail -100 看末尾，找最后一个 `^codex$` 行往后读"。本 epic 4 个 fix-review sub-agent 全部正确解析（无"把上轮 lesson 当本轮 finding 误修"事故）—— 这是 Epic 15 retro 沉淀的工程化经验在本 epic 直接收效。

### 2.4 主旋律 4: WS `error` envelope 首次落地 / EmojiHandler import cycle

17.5 实装时**首次**用 `BuildErrorEnvelope` —— 节点 4 / 5 既有的所有 WS 路径（room 系列 / pet.state.changed 系列）都没有 server → client 的 error 消息（因为它们的入口都是 HTTP，错误通过 HTTP envelope 返回）。emoji.send 是节点 6 第一个 **client → server WS 业务消息**，错误必须通过 WS error envelope 回给发起者。

- envelope schema 严格对齐 §12.3 `### error`: `requestId` 回带 client 的 `emoji.send.requestId`（与广播类 `emoji.received` 的 `requestId` 固定 `""` **不同**）
- error 走 `session.SendPriority(msg)` 而非 `session.Send(msg)`: protocol-level msg 走 priority chan，业务 buffer 满载时仍能投递（与 `handlePing` 走 SendPriority 同模式）
- fire-and-forget: marshal / Send 失败仅 log warn，不返 error 给 caller

**import cycle 踩坑**: `service → ws` 已存在（pet_service import ws），新链路若 ws → service.EmojiService 会成环 → ws 包内定义本地 `EmojiValidator` interface（只 ValidateCode 方法），让 service.EmojiService 自动满足（Go consumer-defined interface 模式）。这条经验值得归档但**未**沉淀到 docs/lessons/（review 通过 + 不视为 bug，仅是实装权宜）—— **建议补一条 lesson** "Go 项目中跨包 handler 调 service 时优先用 consumer-defined interface 而非 producer-exported interface 避免 import cycle"（行动项见 §5）。

## 7. 元教训（Epic 17 给 Epic-loop 工程化的回馈）

1. **codex review 的"两个合理设计目标互斥时的死循环"是普遍现象，不是 17.3 特例**。主 agent 必须有"判通过 + lock down + flag" 的明确流程；epic-loop.md 已有相关规则但需进一步显式化（行动项 A2）

2. **契约层 story 的 review 投资回报率极高**。17.1 5 轮 review 看似拖沓，但下游 17.2 / 17.4 / 17.5 全部 r1 APPROVED，节省了至少 5-10 轮潜在的实装类 review。后续节点的"接口契约最终化"类 story 应该被允许跑足够多轮 review 直到契约真正干净

3. **"用 LLM 综合理解，不要硬卡格式"是 epic-loop 规则的 escape hatch**。本 epic 用了 1 次（17.3 r4），是合理且必要的；规则设计者预留这条空间是正确的，但需要在文档里更显眼地说明"什么情况下用 / 什么情况下不能用"（行动项 A2）

4. **Sub-agent 自报 build pass 与主 agent 实跑 build.sh 之间的双因素验证**是 Epic 17 实跑的额外保险。IDE 诊断 / gopls 中间态可能误报 / 漏报；sub-agent 在隔离 context 跑 build.sh 是单因素；主 agent 实跑是第二因素。两者一致才算真通过（行动项 P1）

5. **tech-debt-log.md 首次启用**意味着 "MVP 阶段刻意延后的事项" 有了显式归档点。本 epic 登记 1 条（emoji.send 限频）。后续节点应该养成习惯，避免遗忘
