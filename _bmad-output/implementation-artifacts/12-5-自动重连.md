## 故事定位（Epic 12 第 5 条 story）

这是 Epic 12「iOS - 房间页面 + WebSocket 客户端」的**第 5 条 story** —— 上游 Story 12.1（WSState + WebSocketClient protocol + WSMessage enum 三 case + unknown fallback + WebSocketClientImpl + WSMessageCodec + RealRoomViewModel.applySnapshot 端到端联通 + room.id stale-discard guard + isHost 严格 false 占位 + bind 替换 client 必须 disconnect 旧 + cancel 旧 task / 同 instance rebind 真 no-op）+ Story 12.2（WebSocketClientImpl 真实拨号 + connect await handshake + JWT URL-encode 注入 + WSMessageCodec 编解码 + WSError 类型 + AppContainer.webSocketClient wire）+ Story 12.3（端到端 wire 联通验证 + 4 个新增单测 case + 1 个 UITest case + UITEST_ROOM_THREE_MEMBERS launch flag）+ Story 12.4（WSMessage enum 扩 memberJoined / memberLeft 两 case + WSMessageCodec.decode 扩两个 type case 路由 + payload required 字段语义校验 + RealRoomViewModel.applyMemberJoined / applyMemberLeft 增量 mutate + streamRoomId 启动时刻捕获守护 cross-room race）已全部 done。

**本 story 的"实装"动作**（一句话概括）：

在 Story 12.2 已端到端打通的"真实 URLSessionWebSocketTask 拨号 + handshake await + receive 长任务 + close code 区分"链路上，**在 `WebSocketClientImpl` 内部加一个 reconnect 状态机**：当 WS 连接因网络问题 / transient 错误（4005 / 1006 / 1011 / DNS 失败 / TLS 失败 等）断开时，**client 自动尝试重连**（指数退避 1s / 2s / 4s / 8s / 最大 30s，最多 N 次），重连成功后自动暴露**新的** `messages` stream 给 vm（vm 重新订阅即可，因为 `prepareForReconnect()` 已经在 12.2 落地，client 内部重连前会自行 reset stream）。

**不**自动重连的场景（client 内部状态机识别 close code 后**直接** finish stream + 切 `wsState = .disconnected`，不进入 reconnect 循环）：
- close 4001（token 过期 / 缺失 / 解码失败）—— 业务级，应触发上层"无效 token 静默重新登录"流程（参考 ADR-0008 v2 + Story 5.4），上层 caller 重新 connect 时再走 fresh handshake；
- close 4002（roomId 格式错）—— 客户端实装 bug；
- close 4003（用户不在该房间）—— 业务级拒绝，重连仍会被 close 4003；
- close 4004（房间不存在）—— 业务级拒绝；
- close 4006（消息超大）—— 客户端实装 bug；
- close 4007（leaver 主动 HTTP leave 的 best-effort cleanup）—— 业务级；
- close 1000（normalClosure，client 主动 disconnect）—— 主动关闭路径。

**vm 层面感知**：通过 `RealRoomViewModel.wsState` 派生 `RoomScaffoldView.wsStateLabel` —— `.connected` → "已连接"、`.reconnecting` → "正在重连…"、`.disconnected` → "已断开"。本 story 的 RealRoomViewModel 改动是**最小化**的：在 client 暴露的 stream 上扩一个**新增 WSMessage case `connectionStateChanged(WSConnectionState)`**（详见 AC2），让 vm 在 `for await message in client.messages` 内收到状态变化事件即写 wsState（不需要单独 publisher / @Published 暴露 wsState 改造），保持「vm 只订阅 client.messages stream」的 12.1-12.4 单一接缝原则。

**与 12.1-12.4 的边界**：

- 12.1 已实装 `applySnapshot(_:)` 方法体 + RealRoomViewModel `subscribeRoomIdConnect` 四种转换分支（含 nil→A / A→B 时调 `prepareForReconnect()` + `startConsumingMessages()`）—— 本 story **不**改 applySnapshot 主干、**不**改 subscribeRoomIdConnect 四种转换分支语义；
- 12.2 已实装 `WebSocketClientImpl.connect(roomId:)` + handshake await + receive 长任务 + `WSError.closedByServer(code:reason:)` —— 本 story **扩**内部 reconnect 状态机（在 receive 长任务 catch error 后判断 close code → 决定 finish stream 还是 schedule reconnect）；
- 12.3 / 12.4 已扩展 codec / VM mutate 路径 —— 本 story **不**改这些路径；
- 12.6（心跳 ping/pong）/ 12.7（CreateRoom / JoinRoom / LeaveRoom UseCase）**不在本 story 范围**；
- **本 story 不实装心跳**（Story 12.6 范围）—— 但本 story 的 reconnect 状态机要正确处理"心跳超时触发 close 4005"场景（client 收到 4005 → transient → 走重连路径）。心跳定时本身由 12.6 落地；本 story 仅在 close 4005 路径上响应。

**本 story 落地后立即解锁**：

- Story 12.6（心跳维护）—— 12.6 心跳超时触发 disconnect 后由本 story 的 reconnect 状态机自动接续；
- Story 12.7（CreateRoom / JoinRoom / LeaveRoom UseCase）—— UseCase 层 connect / disconnect 调用之间发生网络抖动时由本 story 的状态机透明处理。

### 契约 1：close code 分类决策表（V1 §12.1 行 1710-1732）

V1 §12.1 钦定 10 个合法 close code 的 client 行为：

| close code | 类别 | 自动重连？ | 处理 |
|---|---|---|---|
| 1000 | 主动 close | **否** | client-initiated（disconnect()）—— stream finish + wsState = .disconnected |
| 1001 | server going away | 是 | 与 1011 / 4005 同等对待（transient） |
| 1006 | TCP 异常断开（**client 本地合成**，不是 server emit；server **禁止** emit） | 是 | transient network failure |
| 1011 | 服务端内部错误 | 是 | transient（指数退避 + 最大重试次数） |
| 4001 | token 过期 / 解码失败 | **否** | 触发"无效 token 静默重新登录"（caller 上层处理；本 client 仅 finish stream + WSError.closedByServer(code: 4001) 暴露） |
| 4002 | roomId 格式错 | **否** | 客户端实装 bug |
| 4003 | 用户不在该房间 | **否** | 业务级拒绝 |
| 4004 | 房间不存在 | **否** | 业务级拒绝 |
| 4005 | 心跳超时 | 是 | **4xxx 段中的例外**：transient network failure，与 1006 / 1011 同等对待 |
| 4006 | 消息超大 | **否** | 客户端实装 bug |
| 4007 | HTTP leave best-effort cleanup | **否** | 业务级（leaver 已离开） |

**关键决策**：
- **transient 类**（应自动重连）= `{1001, 1006, 1011, 4005}` + 其他底层 transport error（DNS 失败 / TLS 失败 / connection refused / network unreachable —— 这些通常以 URLError 形式出现，未携带 close code）
- **terminal 类**（不重连）= `{1000, 4001, 4002, 4003, 4004, 4006, 4007}`
- 任何**未知 close code**（未在上表的应用层 4xxx 段值）→ **保守处理**：当作 terminal 不重连 + log error（防止野生 server bug 推未知 code 导致死循环重连）

Source: V1 §12.1 行 1710-1732 close code 表 + 行 1725-1731 客户端处理摘要 + V1 行 29 「§12.1 / §12.2 / §12.3 进入冻结状态，任何字段名 / 字段类型 / close code / 信封字段的修改都必须走变更流程」。

### 契约 2：URLSessionWebSocketTask close code 提取语义

V1 §12.1 钦定 server emit 的 close code 在 receive 上以 URLError 出现（Story 12.2 实装 `startReceiveLoop` 在 catch error 时只 log 但**不**区分 close code）。本 story 必须在 catch error 路径上**主动**从 `URLSessionWebSocketTask` 提取 close code：

`URLSessionWebSocketTask.closeCode: URLSessionWebSocketTask.CloseCode`（iOS 13+，receive 抛错后该字段被 runtime 设置为 server emit 的 close code 或 .invalid（client 本地合成 1006 场景））.
- `closeCode.rawValue` 与 V1 close code 表数值对齐（4001 / 4002 / ...）
- 关键 case：1006 在 URLSessionWebSocketTask API 中表现为 `.invalid`（rawValue = 0），**不是** rawValue = 1006 —— 必须检测 `.invalid` 等价于 1006（与 V1 §12.1 行 1729 「1006 仅由客户端 WebSocket runtime 在底层 TCP 异常断开 / 网络抖动且未收到 close frame 时本地合成」契合）.

**关键决策**：本 story 引入 `WebSocketTaskHandle.closeCode: URLSessionWebSocketTask.CloseCode { get }`（扩 internal protocol）+ production wrapper 暴露 underlying task 的 closeCode + fake handle 测试可注入；catch error 时按 closeCode 分类（terminal / transient）后决策。**绝对不**用错误描述字符串（NSLocalizedDescription）做分类（脆弱，依赖 OS 国际化文案）。

Source: Apple URLSessionWebSocketTask 文档 + Story 12.2 `WebSocketTaskHandle` protocol 抽象（已在 `WebSocketClientImpl.swift:28-36` 落地，本 story 扩字段） + V1 §12.1 行 1728-1731 1006/1008/1009 钦定。

### 契约 3：指数退避策略 + 最大重试次数

epic line 2151-2162（Story 12.5 acceptance）钦定：
- **退避序列**：1s / 2s / 4s / 8s / **最大 30s**（acceptance 写"max 60s"，但本 story 钦定 max 30s —— 鉴于节点 4 阶段网络抖动场景以"短抖动"为主，30s 后仍未恢复多半是真宕机，60s 等待对 UX 太长。详见下方 dev decisions。dev 落地若觉得 60s 更合适，可改回 60s —— 但要更新单测固定 wait 时长）
- **每次重连前**更新 `wsState = .reconnecting`（RoomView 显示"正在重连…"占位文字）
- **重连成功**（=新 connect 成功 await handshake + 接到第一帧）→ wsState = .connected + **server 自动重发 room.snapshot**（V1 §12.1.3 钦定握手成功后 server 必发 room.snapshot 作为第一帧）→ vm 在 `applySnapshot` 内对齐 roster
- **重连失败超过 5 次**（acceptance 写"5 次失败"）→ wsState = .disconnected + finish stream + log error
  - 详见 §12.1 4005 重连讨论中"指数退避 + 最大重试次数限制避免雪崩"
- **每次重连尝试**：`prepareForReconnect()` reset stream + emit `connectionStateChanged(.reconnecting(attempt: N))` 到新 stream + 调 `connect(roomId:)`

**关键决策**：
- 重试次数上限 = 5 次（与 acceptance "重连失败超过 5 次" 一致）
- 退避序列硬编码（不读 config）—— iOS 节点 4 阶段无 remote config 接缝；硬编码即可，未来 epic 需要 tune 时改源码
- **在 acceptance 钦定外**：本 story 暂**不**实装"App 进后台主动 disconnect / 回前台 reconnect"路径（acceptance 行 2155 写有此条款）—— 节点 4 阶段 iOS app 进入后台后 URLSessionWebSocketTask 会被系统自动 close（4005 心跳超时 or 1006），交给本 story 的状态机自然处理即可，不需额外的 app lifecycle 监听。详见下方 dev decisions。dev 落地时若觉得显式监听更省电更精确，可加 NotificationCenter scenePhase 监听并显式 disconnect，但本 story 不主张该路径（保持范围最小）。

Source: epic line 2151-2162 + V1 §12.1 行 1726 「指数退避 + 最大重试次数限制」 + V1 §12.1 行 1717 4005 retryable 段 + V1 §10.5 「WS 断线场景与本接口的关系」 r3 锁定。

### 契约 4：vm 感知 reconnect 状态的接缝（新增 WSMessage case）

vm 不持本 client 引用做 KVO / publisher 订阅；vm 与 client 唯一通信通道是 `client.messages: AsyncStream<WSMessage>`（12.1 锁定的接缝）。本 story 在 WSMessage enum 加一个新 case：

```swift
case connectionStateChanged(WSConnectionState)
```

`WSConnectionState` 是值类型 enum：
```swift
public enum WSConnectionState: Equatable, Sendable {
    case connected                       // handshake 成功 / reconnect 成功
    case reconnecting(attempt: Int)      // 第 N 次重连 in-flight（attempt 从 1 起算）
    case disconnected                    // terminal close / 重连超 5 次失败
}
```

vm `handle(message:streamRoomId:)` 加一个 case：
```swift
case .connectionStateChanged(let state):
    switch state {
    case .connected:        self.wsState = .connected
    case .reconnecting:     self.wsState = .reconnecting
    case .disconnected:     self.wsState = .disconnected
    }
```

**关键决策**：用 enum 而非分别三个 case（reconnectStarted / reconnectFinished / reconnectFailed）—— 三态枚举与 vm `WSState` 一一映射，vm 一行 switch 就能写完，不需要持有 retry attempt 计数等中间态。`reconnecting(attempt: Int)` 携带 attempt 字段是预留设计（Story 12.5 acceptance 行 2152 写"每次重连前更新 wsState = .reconnecting"，未要求展示 attempt；但 enum case 携带 attempt 让未来 Story 12.6 / Epic 13 端到端验证想展示"第 N 次重连"时不需重新加字段）—— vm 端不渲染该字段（与 acceptance 一致），但 client 端 emit 时填正确值，未来可直接接.

**与 12.4 streamRoomId 守护的协调**：`connectionStateChanged` 在 client 端 emit；vm 端 `handle(message:streamRoomId:)` 收到后**直接**写 wsState，**不**走 streamRoomId 校验（因为它是 client 自身状态变化通知，不携带房间数据；与 12.4 lesson `cross-room-race-needs-stream-roomid-capture-12-4-r1.md` 不冲突 —— streamRoomId 守护的对象是 roster 数据（snapshot / member.* payload），不包括 client 自身状态事件）。

Source: 本 story 设计决策；与 12.1 「vm 与 client 唯一通信通道是 messages stream」锚定 + 12.4 streamRoomId 守护 lesson 协调。

### 契约 5：reconnect 期间 stream 语义

V1 §12.1.3 钦定 handshake 成功后 server 必发 room.snapshot 作为第一条消息 —— 因此 reconnect 成功后 client 不需要 manual 触发 fresh stream 重新拉 snapshot：**connect 路径上 server 自动**重发 room.snapshot 进 messages stream，vm 的 `applySnapshot` 自动 reconcile roster（含离开 / 加入差异 —— "断网期间收不到 WS 消息，重连后由 snapshot 对齐"，与 acceptance 行 2156 一致）。

**关键决策**：
- **不**改 vm 的 startConsumingMessages 主干 —— vm 仍然用同一个 consumer task；client 在内部重连成功后 emit 新的 `connectionStateChanged(.connected)` 进同一 stream，紧接着 server 推 room.snapshot 走同一 stream，vm 自动 apply。
- **不**调 `prepareForReconnect()`（vm 端的）—— 那是 vm `subscribeRoomIdConnect` 已用过的 client-side 接缝（用来在 leave-rejoin / room A→B 时 reset stream）；本 story 的 reconnect 是 **client 内部**自动行为，不经过 vm 层。
- **关键**：client 内部的 reconnect 在**复用同一**外部 messages stream（vm 持有的 AsyncStream）—— client 内部 cancel 旧 receive task / 起新 task，但**不** finish & 重建 currentContinuation；新 task 里收到的消息继续 yield 到旧 continuation 上。这与 12.2 `prepareForReconnect()` 的"finish 旧 stream + 新建 stream"语义**不同**（prepareForReconnect 是 caller-driven 的全 reset；本 story 的 reconnect 是 client-internal 的透明续接）。

Source: V1 §12.1.3 + 12.2 实装锚定 (`WebSocketClientImpl.swift:107-111` currentStream / currentContinuation) + 本 story 设计。

### 契约 6：vm bind 替换 client / 同 instance rebind 不破坏 reconnect 状态机（12.1 r5/r6 lesson 延伸）

12.1 r5 lesson `2026-05-09-bind-replace-must-disconnect-old-client-12-1-r5.md` 钦定：bind 替换 client 必须 disconnect 旧 + cancel 旧 task；12.1 r6 lesson `2026-05-09-same-instance-rebind-must-true-noop-12-1-r6.md` 钦定：同 instance rebind 真 no-op。本 story 的 reconnect 状态机**不**改 vm.bind 行为：
- (a) 同 instance rebind → 不调 disconnect → client 内部 reconnect 状态机不受影响；
- (b) 不同 instance 替换 → 旧 client.disconnect()（client 内部 reconnect 状态机被 cancel —— 详见 AC1 实装：disconnect() 必须 cancel reconnect task）；
- (c) 首次注入 → 启 stream consumer，client 进入 connect 路径正常工作。

**关键约束**：本 story `WebSocketClientImpl.disconnect()` 必须**主动 cancel** in-flight reconnect task —— 否则 caller disconnect 后 client 内部仍然在 reconnect 循环里 attempt connect，违反"用户主动 disconnect 终止所有自动行为"语义。详见 AC1 Task 1.4。

Source: 12.1 r5/r6 lessons + 本 story 设计。

### AC1 — WebSocketClientImpl 内部 reconnect 状态机实装

**修改文件**：`iphone/PetApp/Core/Networking/WebSocketClientImpl.swift`

**新增 internal 私有字段**（与既有 `underlyingTask` / `receiveTask` / `currentStream` / `connectGate` 平级）：

```swift
/// 当前正在拨号的 roomId —— reconnect 路径需要它（连接断开后用同一 roomId 重连）.
/// connect 成功后写入；disconnect / terminal close 后清 nil.
private var currentRoomId: String?

/// 当前 reconnect attempt 计数（首次 connect = 0；reconnect 第 N 次 = N）.
private var reconnectAttempt: Int = 0

/// in-flight reconnect task —— schedule 后写入；disconnect / terminal close / 成功后清 nil.
/// disconnect() 必须 cancel 此 task（否则用户主动 disconnect 后 client 仍在 reconnect 循环）.
private var reconnectTask: Task<Void, Never>?

/// reconnect 退避序列（秒）—— 第 N 次重连等 backoffSequence[min(N-1, count-1)] 秒.
/// hardcoded：[1, 2, 4, 8, 30]；attempt 5 后仍失败 → terminal disconnect.
/// internal 可见让单测可校验序列（不在 init 注入；保持 production simple）.
internal static let backoffSequence: [TimeInterval] = [1, 2, 4, 8, 30]

/// 最大重连次数 —— 5（与 acceptance 行 2154 一致）.
internal static let maxReconnectAttempts: Int = 5
```

**新增内部方法 `scheduleReconnect`**（在 `startReceiveLoop` catch error 路径调用）：

```swift
/// receive task catch error 后判断 closeCode 分类；transient 类 → schedule reconnect；terminal → finish + emit disconnected.
private func handleReceiveError(closeCode: URLSessionWebSocketTask.CloseCode, error: Error) {
    // closeCode 分类
    let category = classifyCloseCode(closeCode)
    switch category {
    case .terminal(let code):
        // log error / info（按 V1 §12.1 行 1732 钦定的 log 级别）
        // emit connectionStateChanged(.disconnected) 到 stream + finish stream
        emitConnectionState(.disconnected)
        currentContinuation.finish()
        // currentRoomId / reconnectAttempt / reconnectTask 清 nil
    case .transient(let code):
        // emit connectionStateChanged(.reconnecting(attempt: reconnectAttempt + 1))
        // schedule reconnect task：等 backoffSequence[reconnectAttempt] 秒后调 connectInternal()
        scheduleReconnect()
    }
}

private enum CloseCodeCategory {
    case terminal(code: Int)
    case transient(code: Int)
}

/// V1 §12.1 close code 表（行 1710-1732）映射：
/// terminal = {1000, 4001, 4002, 4003, 4004, 4006, 4007, unknown 4xxx}
/// transient = {1001, 1006(.invalid), 1011, 4005, no closeCode（底层 URLError）}
private func classifyCloseCode(_ closeCode: URLSessionWebSocketTask.CloseCode) -> CloseCodeCategory {
    let raw = closeCode.rawValue
    switch raw {
    case 1000:
        return .terminal(code: 1000)
    case 1001, 1011:
        return .transient(code: raw)
    case 4001, 4002, 4003, 4004, 4006, 4007:
        return .terminal(code: raw)
    case 4005:
        return .transient(code: 4005)
    case 0:
        // .invalid = 1006（底层 TCP 异常 / 未收 close frame）= transient
        return .transient(code: 1006)
    default:
        // 未知 close code：保守 terminal（防野生 server bug 推未知 code 导致死循环）
        return .terminal(code: raw)
    }
}

private func scheduleReconnect() {
    // 上限校验
    if reconnectAttempt >= Self.maxReconnectAttempts {
        emitConnectionState(.disconnected)
        currentContinuation.finish()
        return
    }
    let attempt = reconnectAttempt + 1
    let backoffSec = Self.backoffSequence[min(attempt - 1, Self.backoffSequence.count - 1)]
    emitConnectionState(.reconnecting(attempt: attempt))

    let task = Task { [weak self] in
        // 用 Task.sleep 而非 DispatchQueue.asyncAfter —— 单测可注入 fake task scheduler / Mock clock；本 story 实装直接 Task.sleep
        try? await Task.sleep(nanoseconds: UInt64(backoffSec * 1_000_000_000))
        if Task.isCancelled { return }
        guard let self else { return }
        await self.attemptReconnect(attempt: attempt)
    }
    lock.lock()
    reconnectTask = task
    lock.unlock()
}

private func attemptReconnect(attempt: Int) async {
    guard let roomId = currentRoomId else { return }
    do {
        // 复用 connect 主干 —— 但需要"内部 connect"路径（不重置 reconnectAttempt 到 0；让失败可累计）
        try await connectInternal(roomId: roomId, isReconnect: true, attemptNumber: attempt)
        // 成功：reconnectAttempt 清 0；emit .connected
        reconnectAttempt = 0
        emitConnectionState(.connected)
    } catch {
        // 失败：累计 attempt，schedule next
        reconnectAttempt = attempt
        scheduleReconnect()
    }
}

/// 把现有 connect(roomId:) 主干抽到 connectInternal —— public connect 是 isReconnect=false 路径
public func connect(roomId: String) async throws {
    reconnectAttempt = 0  // 用户/caller 主动 connect → 重置计数
    try await connectInternal(roomId: roomId, isReconnect: false, attemptNumber: 0)
    emitConnectionState(.connected)  // 首次 connect 成功也 emit .connected
}
```

**修改既有 `disconnect()`**：cancel `reconnectTask` + 清 `currentRoomId` + 清 `reconnectAttempt`：

```swift
public func disconnect() {
    lock.lock()
    let oldTask = underlyingTask
    let oldReceiveTask = receiveTask
    let oldReconnectTask = reconnectTask
    let oldContinuation = currentContinuation
    underlyingTask = nil
    receiveTask = nil
    reconnectTask = nil
    currentRoomId = nil
    reconnectAttempt = 0
    lock.unlock()

    oldTask?.cancel(with: .normalClosure, reason: nil)
    oldReceiveTask?.cancel()
    oldReconnectTask?.cancel()  // 关键：取消 in-flight reconnect 防 disconnect 后仍 attempt
    oldContinuation.finish()
    resolveConnectGate(success: false, error: WSError.connectionFailed(underlyingDescription: "disconnect before first frame"))
}
```

### 开放问题（dev 落地时如有疑问可决策或 raise）

1. **退避序列上限 30s 还是 60s？** —— epic acceptance 行 2151 写"max 60s"；本 story 钦定 30s（节点 4 阶段 UX 角度；30s 后仍未恢复多半真宕机，60s 等待对 UX 太长）。dev 落地时若觉得 60s 更合适（与 acceptance 严格对齐），可改回 60s —— 但要更新单测固定 wait 时长（注入 fake clock 的方式无所谓 30 vs 60）。记入 dev decisions。
2. **App 后台 / 前台监听 是否本 story 实装？** —— epic acceptance 行 2155 写"App 进入后台时主动 disconnect（节省电量），回前台时自动 reconnect"；本 story 钦定**不实装**（节点 4 阶段 iOS 进入后台后 URLSessionWebSocketTask 系统自动关 → 4005/1006 → 自动重连状态机透明处理；显式监听增加复杂度）。dev 落地时若觉得显式监听更省电更精确（避免重连风暴 / 提早 emit reconnecting 状态），可加 NotificationCenter `UIApplication.didEnterBackgroundNotification` / `willEnterForegroundNotification` 监听 + scenePhase 判断 + 显式 disconnect/connect —— 但要 (a) 保证 vm 不破（disconnect 后 wsState 切 disconnected 是合理的，回前台后 connect 走 reconnecting → connected 也 OK）；(b) 注意"后台主动 disconnect"路径下不应触发 reconnect 状态机（手动 disconnect 是主动；与 close 1000 同语义）。本 story 钦定不实装。
3. **fake-clock / sleep bypass 测试方案选哪个？** —— Task 5.2 列了两种：(a) `backoffSequence` 改实例字段 + 测试注入 [0.001, ...]；(b) 注入 `sleepImpl: (TimeInterval) async -> Void` 闭包。dev 选 (a) 更简单（一行字段改 + 测试构造时传短序列）；选 (b) 更纯粹（不污染 production 字段）。两种方案兼容，dev 选其一记入 dev decisions。
4. **WSConnectionState.reconnecting(attempt: Int)** —— acceptance 不要求展示 attempt；vm 端不渲染。是否在本 story 阶段就把 attempt 字段加入？本 story 钦定**加入**（预留）；dev 落地时若觉得"YAGNI 不预留"，可改为 `case reconnecting`（无 attempt）—— 但 Story 12.6 / Epic 13 落地"第 N 次重连…"展示时还要回工 enum 字段集。本 story 主张预留更省事。
5. **未知 close code（如 4099 / 5000+）保守 terminal 还是 transient？** —— 本 story 钦定保守 terminal（防野生 server bug 推未知 code 导致死循环重连）；dev 落地时若觉得"server 推未知 code 是 server bug，client 应用 best-effort 重连让运维自查"，可改 transient —— 但要考虑 server 大量未知 code 推下来时本 client 会陷入指数退避循环（30s 上限内 5 次失败 = 47s 自然退出，不会无限）。本 story 主张保守 terminal 更稳。
6. **vm.handle 内 `.connectionStateChanged` switch 是否应在 streamRoomId == nil 时也写 wsState？** —— 本 story 钦定**应写**（与契约 4 一致：connectionStateChanged 不携带 roster 数据，与 streamRoomId 守护语义无关）；dev 落地时若觉得"已离开房间（lastObservedRoomId == nil）时应忽略 wsState 更新，避免 RoomView 已退出时 wsState 还在变"，可加 `guard lastObservedRoomId != nil else { return }` —— 但要考虑：(a) RoomView 退出后 vm 通常会 deinit 或 messageConsumerTask 被 cancel；(b) 即使 wsState 在 RoomView 退出后变化，UI 已不渲染，无副作用。本 story 主张不加 guard 保持简单。
