## 2. 关键洞察与 Epic 15 的"主旋律"

**Attempt 1 的失败模式（13 轮 codex finding 总览）**：

| Round | Finding 类型 | 被引入的复杂度 |

**Attempt 1 的根本问题**：每一轮 codex 找到的 finding 都"合理"，每一轮 fix 都"对症"，但**fix 互相冲突 / 引入新边界**。13 轮过后 service 累积了：`lastSentState` / `lastSentAt` / `lastSentAtByState` / `pendingState` / `pendingCommitTimestamp` / `inFlightTask` / `inFlightCommit` / `spawnGeneration` / `lastKnownRoomId` / `hasStartedOnce` —— 10 个状态字段构成的复杂状态机，且字段间的不变量极易破坏。

**Attempt 2 的成功模式**：把 attempt 1 的"反模式清单"显式列入 story 文件红线段（`⚠️ 实装边界（Attempt 1 教训：必读）`），明确**禁止** 6 个过度设计：
1. ❌ per-state `[MotionState: Date]` throttle dict
2. ❌ coalesce-to-latest pending-state queue
3. ❌ Room-edge active sync（让 15.5 处理）
4. ❌ Stop-time in-flight Task throttle 回滚
5. ❌ Publisher subscribe-replay vs real-transition 区分
6. ❌ Service-internal in-flight gate / serialize / Task chain

加上**显式行数预算**（service ≤200 / test ≤350）+ **sibling template 引用**（Story 8.5 `StepSyncTriggerService`，249 行）。Attempt 2 dev-story sub-agent 严格遵守，落地 167 行 service + 213 行 test，codex r1 直接 APPROVED。

### 2.2 主旋律 2: Story 15.2 generation gate —— 4 轮迭代实质设计演化

15.2 是 epic 15 中**正常**走完 review/fix 流程的 story，4 轮 review 每轮都落地有效设计改进而非互相打架（vs 15.4 attempt 1 的"打架"模式）：

- **r1 → r2**：从"defer 同房间 rejoin race（lesson 写明）"反转为"实际加 streamGeneration gate 覆盖 4 case"。决策反转的诚实记录留 lesson `2026-05-12-pet-state-changed-stream-roomid-guard-residual-risk-same-room-rejoin-15-2-r1.md`（保留不删）+ `2026-05-12-per-stream-generation-guard-fixes-same-room-rejoin-race-15-2-r2.md`（反转记录）。
- **r2 → r3**：r2 把 `.roomSnapshot` exempt 是疏忽 —— snapshot 整体覆盖比单字段 merge 更危险，必须纳入 gate。
- **r3 → r4**：r3 完成 gate 覆盖后 codex 找到 gate 上游的 race —— consumer task 启动时 `client.streamGeneration` 与 `client.messages` 不原子捕获。**新增 protocol API** `currentStreamSnapshot: (AsyncStream<WSMessage>, Int)` 在 lock 内原子读取。
- **r4 → r5**：APPROVED。

15.2 4 轮 lesson **被未来 stories 复用**：Story 15.5 的 `triggerManualResync` 复用 15.2 r4 的 `currentStreamSnapshot` 原子 API；Story 15.4 attempt 2 实装明确**避免**复刻 generation gate（service 不维护连接状态机）。这是 lesson 沉淀的复利模式。

### 2.3 主旋律 3: 简化设计的"复利" —— 15.3 / 15.5 一轮过的同类机制

15.3 一轮过：Story 8.4 r2 已落地 `PetSpriteView` 的 SwiftUI content-swap idiom（`.id() + .transition + .animation(value:)`），15.3 实质工作只是 transition 类型 + duration tweak（5 行改动）+ 测试锁定 +ios-simulator 录屏归档。**收益来源**：节点 3 阶段的 lesson 投入（8.4 r2 `swiftui-content-swap-needs-id-and-transition.md`）让节点 5 的动画 story 一次到位。

15.5 一轮过：复用 15.1 `applySnapshot` + 15.2 `currentStreamSnapshot` 原子 API + Story 8.5 `StepSyncTriggerService` lifecycle pattern + 15.4 attempt 2 的"实装边界红线"模式（story 文件预先列禁止反模式）。**service 增量 48 行 / vm 增量 21 行 / RootView 增量 7 行** —— 全部紧贴预算上限内。

**总结**：15.3 / 15.4 attempt 2 / 15.5 三个一轮过的 story，共同特征是 **dev-story sub-agent 在 story 文件中已经看到了"什么不能做"和"sibling 长啥样"**。这与 15.4 attempt 1 的"sub-agent 自由发挥 → codex 反复打磨"形成强烈对比。

### 4.3 Epic 16 实装风险点（预测）

1. **节点 5 跨端 e2e 是节点 1-5 的第一次完整跨端联调**：节点 1 ping / 节点 2 自动登录 / 节点 3 步数-猫动作 / 节点 4 房间 lifecycle 都跑过 demo（epic 3 / 6 / 9 / 13 demo epics），但节点 5 把"步数 → 猫动作 → 房间内同步"这条链路打通需要看实际表现。**预期 e2e 会暴露**：
   - 模拟器之间的网络环境（localhost vs 局域网）
   - 真实 motion 事件的不可预测性（CoreMotion 在模拟器不会真发；只有 Story 8.4 的"模拟器 fake motion"路径或者真机才能验证完整链路）
   - 自动重连 + snapshot 重对齐的实际时序（vs 单元测试的快进时钟）
2. **15.4 attempt 2 极简设计的 production 验证**：attempt 2 的极简 service 在单元测试 + ios-simulator launch test 都通过，但**完整跨端 e2e 才是验收标准**。如发现"motion 变化没及时上报到房间"或"重连后没立即对齐"等问题，回溯路径在 15.5 triggerManualResync。
3. **14.3 / 14.4 / 15.x 全链路时序**：节点 5 是节点 1-5 首次同时**全部上线**的 demo。任何一个环节的 bug 都会在 e2e 表现为"看不到猫动了"。建议 16.1 e2e 测试用 dev-tools 端点（`/dev/grant-steps` / `/pets/current/state-sync` 强制触发）作 fallback，避免完全依赖 motion 真实事件。
4. **Epic 16 不能进 /epic-loop**：demo 黑名单 epic（含 16）的硬约束在 epic-loop.md 钦定 —— 自动循环跑不动 demo 验收。**需要手动**跑两端模拟器 + 真机联调。

## 9. Key Takeaways（关键收获）

1. **简化设计 > 完美设计**：15.4 attempt 1 的 13 轮 review 演化是"完美主义陷阱"的真实案例 —— 每一轮 finding 都"对"，每一次 fix 都"合理"，但累积构成不可维护的复杂度。Attempt 2 的"行数预算 + 反模式红线 + sibling template"是反过度设计的有效工具。
2. **Story 文件红线段是新工具**：把"什么不能做 + 为什么不能做"写入 story 文件本身，比留 lesson 在 `docs/lessons/` 等待 sub-agent 主动查阅，**强制性**和**及时性**都更高。15.4 attempt 2 / 15.5 都验证有效。
3. **decision reversal 是健康的工程模式**：15.2 r1 defer + r2 反转 fix 是诚实的工程决策模式 —— 第一轮 trade-off 评估错了，第二轮 codex 给出 strong signal 时就要反转。保留 r1 defer lesson + 新增 r2 反转 lesson + frontmatter 反链是反转的 audit trail。
4. **codex review 对 Swift / iOS 代码覆盖度与 Go 同等**：Epic 14 retro 4.3 段对 codex Swift 覆盖度的预测被验证。Attempt 1 的 13 轮 finding **全部命中真实问题**，没有误报。
5. **/epic-loop 主 agent 的 review 子循环熔断需要做更主动**：Epic 15 攻略前 3 story 顺利后，15.4 attempt 1 的 13 轮才意识到不收敛是反复 / 浪费。下次需要在 r5 主动 HALT 给 user 决策权。
6. **lesson 沉淀的复利可见**：15.3 一轮过得益于 8.4 r2 lesson；15.5 一轮过得益于 15.2 r4 lesson + 15.4 attempt 2 lesson + 8.5 sibling template。每条 lesson 不只是"避免再犯同错"，更是"未来 story 的简化路径预设"。
7. **工具依赖的脆弱性需要监测**：codex AMFI 隔离事件揭示 /epic-loop 关键工具不在自己掌控下；下次 epic 启动前自动检测是简单但有效的 safeguard。
8. **节点 5 iOS 实装与 server 实装节奏完全不同**：Epic 14 server 端是"契约 11 轮 + 实装 1 轮过 × 3"（前置 11 轮换后续 0 轮）；Epic 15 iOS 端是"实装 1-5 轮分布 + 1 个 reset 重做"。节点 5 的真实 e2e 会在 Epic 16 demo 验收揭示哪种节奏的实装质量更可靠。
