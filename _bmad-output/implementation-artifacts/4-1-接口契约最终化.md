**AC2 — 全局契约约定（开篇章节补全）**

在 V1 文档 §2.4（通用响应结构）后追加一段 **§2.5 字段类型与编码约定**（新增小节，但属 §2 现有"统一约定"大章节范围内，不破坏 §3 ~ §11 编号）：

```markdown
## 2.5 字段类型与编码约定

- **主键 / 外键 ID**：所有 BIGINT 主键 / 外键在 JSON 里以**字符串**形式下发（如 `"id": "1001"`），避免 JavaScript 端 `Number.MAX_SAFE_INTEGER` 精度丢失；客户端解析为 `String`，如需算术运算自行转换为 `Int64` 或等价类型。
- **时间字段**：所有时间字段统一使用 ISO 8601 UTC 格式（如 `"unlockAt": "2026-04-23T10:20:00Z"`），客户端按本地时区显示。
- **可空字段**：可空字段在 JSON 中显式以 `null` 表示（如 `"currentRoomId": null`），不省略 key；客户端解析为 `Optional<T>` / `T?`。
- **占位字段（节点 2 阶段）**：节点 2 阶段尚未实装的字段（如 `pet.equips` / `room.currentRoomId` / `chest.unlockable` 等动态状态）按"future fields"标注，文档中会在每个接口章节末尾以 `> Future Fields (节点 X 落地)` 引用块列出。
- **字符串长度约束**：所有字符串字段的最大长度以**字符数**计（与 MySQL 表 `VARCHAR(N)` 语义一致 —— `N` 是字符数 limit，不是字节数；utf8mb4 编码下 1 个字符在底层存储占 1-4 字节，但长度校验按字符计数）。客户端 / server 实施长度校验时**必须**按 Unicode 字符数（如 Go `utf8.RuneCountInString` / Swift `String.count`）判断，**不**按字节数判断，否则会误拒合法的多字节输入。
- **枚举字段**：枚举字段以 `TINYINT` 数值下发（如 `petType: 1`），不下发字符串字面量；具体取值见各接口章节。
```

**关键约束**：

- 此 §2.5 是**全局**约定，所有后续接口章节（§4 / §5 / §6 / §7 / §8 / §9 / §10 / §11）共享，不在每个接口里重复写
- 这条规则在节点 2 第一次显式写下；节点 1 阶段（Epic 1 / 2）的 `/ping` / `/version` 不涉及主键，未触发；本 story 把契约**冻结**点定在节点 2 起，明确"主键以字符串下发"是 V1 全局规则

## Dev Notes

### 关键设计原则

1. **契约先行（contract first）**：本 story 是节点 2 第一个 epic 的开局，**禁止**先写 Go 代码再回头补文档。理由：iOS Epic 5 必须基于稳定契约并行开工，先实装后补文档会导致 iOS 端 Codable struct 与 server response 字段对不上 → 节点 2 demo 时双向返工。
2. **数据库 → API 单向映射**：API 字段语义由 `docs/宠物互动App_数据库设计.md` §5.1 ~ §5.6 + `docs/宠物互动App_总体架构设计.md` §3.1 推导，**不**反向修改数据库。如发现 API 需求与数据库 schema 不符（如某个字段长度不够），优先改 API 设计或后置到下一 epic 处理，**不**直接改 DB。
3. **占位字段必须显式 null / 空数组**：节点 2 阶段未实装的字段（pet.equips / room.currentRoomId 等）服务端必须**显式**返回 `null` / `[]`，**禁止**省略 key。理由：客户端 Codable 解析需要明确知道字段一定存在但可能为占位；省略 key 会让 iOS 端 `keyDecodingStrategy` 出错。
4. **主键统一字符串化**：所有 BIGINT 主键序列化为 JSON 字符串。理由：JavaScript Number.MAX_SAFE_INTEGER = 2^53 - 1 ≈ 9e15，BIGINT 上限 2^63 - 1 ≈ 9e18，跨端最大公约数 = string；本约定 V1 全局生效，client 必须按 String 解析。
5. **Future Fields 不算契约变更**：节点 2 文档侧已**预留**节点 9 / 10 / 4 之后才激活的字段位置（pet.equips / pet.equips[].renderConfig / room.currentRoomId），后续节点激活这些字段**不**视为契约破坏，对应节点 epic 自然填充。

### 架构对齐

**领域模型层**（`docs/宠物互动App_总体架构设计.md` §3.1）：
- 账号系统：游客 UID 在 Keychain 持久化，server 只认 UID 标识同一账号
- 自动登录：客户端默认走游客身份，无显式登录页
- 这两条规则决定 §4.1 必须**幂等**：同一 guestUid 重复调用 → 同一 user_id

**数据库层**（`docs/宠物互动App_数据库设计.md`）：
- §5.1 users.guest_uid VARCHAR(128) → §4.1 guestUid 长度上限 128
- §5.2 user_auth_bindings 唯一约束 `UNIQUE(auth_type, auth_identifier)` → 同一 guestUid 二次登录走复用分支
- §5.6 user_chests.unlock_at + status → §5.1 home 接口的 chest 字段直接映射

- handler 层 DTO 严格按本契约：handler 不做业务计算，只做参数校验 + 调 service + 转 response
