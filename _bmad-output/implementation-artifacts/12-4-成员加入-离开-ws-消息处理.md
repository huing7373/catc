**本 story 的"实装"动作**（一句话概括）：

在 Story 12.3 已端到端打通的"server `room.snapshot` → `WebSocketClientImpl` → `WSMessageCodec.decode` → `WebSocketClient.messages` AsyncStream → `RealRoomViewModel.applySnapshot`"链路上，**扩展支持 `member.joined` / `member.left` 增量 WS 消息**：

1. 扩展 `WSMessage` enum 增加 `case memberJoined(MemberJoinedPayload)` + `case memberLeft(MemberLeftPayload)` 两 case + 新增 `MemberJoinedPayload` / `MemberLeftPayload` 值类型；

3. 在 `RealRoomViewModel.handle(message:)` 内新增对应 enum case 分支调用 `applyMemberJoined(_:)` / `applyMemberLeft(_:)`；这两个方法实装"增量 mutate `members` 数组"语义（**不是** full snapshot replacement）：

5. 单元测试 ≥5 case 覆盖 epic line 2133-2138 钦定的全部边界。

## 核心契约依据（必读）

> 本 story 严格遵循 V1 §12.3 `### 成员加入` / `### 成员离开` 字段表 + 关键约束 + Story 11.8 server-side 实装锚定，所有实装 / 测试不允许偏离。

### 契约 1：`member.joined` payload 完整自包含成员展示字段

V1 §12.3 行 2003-2013 字段表钦定 payload **必须**含 `userId` + `nickname` + `avatarUrl` + `pet`（nullable）：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `payload.userId` | string | 必填 | 加入的成员 user 主键（BIGINT 字符串化） |
| `payload.nickname` | string | 必填，**必非空字符串** | 来自 `users.nickname`；与 `room.snapshot` placeholder 阶段允许空字符串的语义**不同** —— member.joined 在加入事务**成功提交后**触发，server 必有真实 nickname |
| `payload.avatarUrl` | string | 必填 | 来自 `users.avatar_url`；可空字符串 `""`（节点 4 阶段头像未实装），**不**为 null |
| `payload.pet` | object \| null | 必填（nullable） | pet-less 账号下发 `null`；否则下发 `{petId, currentState}` 完整 object |
| `payload.pet.petId` | string | 必填（仅当 `pet ≠ null`） | BIGINT 字符串化 |
| `payload.pet.currentState` | int | 必填（仅当 `pet ≠ null`） | 节点 4 阶段固定 `1`（rest） |

**关键决策**：本 story `MemberJoinedPayload` 值类型必须**完整**含上述 5 个字段（`userId` + `nickname` + `avatarUrl` + `pet`，其中 `pet` 是 `MemberJoinedPet?`）—— **禁止**简化为仅 `userId + nickname`（避坑：V1 §12.3 行 2059 明确"client 收到 member.joined 后**可直接** append 一条完整的 roster entry，**不**需要走 `GET /api/v1/rooms/{roomId}` 二次拉取"）。

Source: V1 §12.3 行 1994-2063 + Story 11.8 实装 `MemberJoinedPayload` struct（server/internal/app/ws/snapshot.go:416-421）。

### 契约 2：`member.left` payload 精简为仅 `userId`

V1 §12.3 行 2073-2080 字段表钦定 payload **仅** `userId`：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `payload.userId` | string | 必填 | 离开的成员 user 主键（BIGINT 字符串化） |

**关键决策**：本 story `MemberLeftPayload` 值类型仅一个字段 `userId: String` —— V1 §12.3 行 2097 明确"离开事件 client UX 不需要显示昵称（client 从已有 roster 查 nickname；UX 文案降级为'有人离开'也可接受）"。

Source: V1 §12.3 行 2065-2101 + Story 11.8 实装 `MemberLeftPayload` struct（server/internal/app/ws/snapshot.go:428-430）。

### 契约 3：增量 mutate 而非 full replacement + dedup by userId + 不存在时 ignore

V1 §12.3 行 2061-2062 / 2099-2101 钦定 client 增量 mutate 语义：

- **`member.joined` 处理**：
  - (a) roster 中已存在该 `userId` entry → 字段级 enrich（按 client merge contract"非空覆盖、空字符串保留 client 已有值、null 直接覆盖（authoritative）"规则；本 story `avatarUrl` 可能下发空串则保留 client 已有真实值；`pet = null` 是 authoritative pet-less 信号则覆盖）；
  - (b) roster 中不存在该 `userId` entry → 新增完整 entry（含本字段表所有字段，`pet = null` 时 entry 中 pet 字段置空，渲染降级"无宠物"占位）；
  - **关键约束**：dedup by userId —— 同一 userId 重复 emit `member.joined`（server bug / 网络重发）**不应**重复添加成员（防"4 人房间显示 5 个成员"）。
- **`member.left` 处理**：
  - 从 roster 中**移除** `payload.userId` 对应 entry；
  - **关键约束**：roster 中不存在该 `userId` 时 → ignore + log warning；**禁止**抛错 / **禁止**因找不到 entry 把 vm.members 整体清空（防御性"诡异"server state 下不至于把整个房间清空）。
- **关键约束**：收到广播时**不重新拉 snapshot**（信任增量；epic line 2132 钦定"收到广播时**不重新拉 snapshot**（信任增量），但每次 reconnect 后重新拉 snapshot 对齐"——reconnect 路径属于 Story 12.5 范围，本 story 不实装）。

Source: V1 §12.3 行 2061-2062 / 2099-2101 + Story 11.8 r3 lesson `2026-05-09-async-must-preserve-sync-observable-invariants-11-8-r3.md`（async 化必须保留 caller 同步可观察 invariants —— 本 story 客户端层守护 emitter 顺序导致的 client view 不变量）。

### 契约 4：room.id 校验语义（fix-review r3 lesson 12.1 延续）

`member.joined` / `member.left` payload **不带** `room.id` 字段（V1 §12.3 钦定 payload 精简），server 端按 fanout 范围保证只投递到该房间的 sessions。但 client 端 `lastObservedRoomId == nil`（已离开房间）时收到的任何 member.* 消息属于"stale stream 上排队的 late message" → **必须**丢弃 + log debug，与 12.1 r3 lesson `2026-05-09-stale-snapshot-discard-by-room-id-12-1-r3.md` 同精神：

- `lastObservedRoomId == nil` + 收到 `member.joined` / `member.left` → 丢弃 + log debug
- `lastObservedRoomId != nil` + 收到 `member.joined` / `member.left` → 处理（信任 server fanout 范围）
- **不**做 payload 层 `room.id` 校验（payload 不带该字段）

Source: 12.1 r3 lesson `2026-05-09-stale-snapshot-discard-by-room-id-12-1-r3.md`（room.id 校验扩展到 member.* 消息层）+ V1 §12.3 行 2070 / 2098 钦定 server fanout 仅到达该房间在线成员。

## Story

As an iPhone 用户,
I want 当其他用户进入或离开房间时，我的成员列表实时刷新（不重新拉 snapshot），
So that 我能感知房间动态而不需要手动刷新，并为后续 12.5（自动重连）/ 12.7（创建/加入/退出闭环）提供视觉真实态.

private struct MemberJoinedEnvelope: Decodable {
    let payload: MemberJoinedPayloadDTO

    struct MemberJoinedPayloadDTO: Decodable {
        let userId: String
        let nickname: String
        let avatarUrl: String
        let pet: PetDTO?  // V1 §12.3：null = pet-less authoritative 信号

        struct PetDTO: Decodable {
            let petId: String
            let currentState: Int

        func toDomain() -> MemberJoinedPayload {
            MemberJoinedPayload(
                userId: userId,
                nickname: nickname,
                avatarUrl: avatarUrl,

### AC3 — RealRoomViewModel.handle 扩 `memberJoined` / `memberLeft` 分支 + applyMemberJoined / applyMemberLeft 增量 mutate

**修改文件**：`iphone/PetApp/Features/Room/ViewModels/RealRoomViewModel.swift`

**修改 `handle(message:)` switch 增加两个 case**（在既有 `.roomSnapshot` / `.pong` / `.error` / `.unknown` 之间）：

```swift
private func handle(message: WSMessage) {
    switch message {
    case .roomSnapshot(let payload):
        // ... 既有 stale-discard guard + applySnapshot ...
    case .memberJoined(let payload):
        // **room.id 校验**：lastObservedRoomId == nil（已离开房间）→ 丢弃 stale stream 上 late message.
        // payload 不带 room.id 字段（V1 §12.3 钦定 payload 精简）；server 端按 fanout 范围保证只投递
        // 到该房间的 sessions —— client 端不做 payload 层 room.id 校验，仅做 lastObservedRoomId nil 守护.
        guard lastObservedRoomId != nil else {
            os_log(.debug,
                   "RealRoomViewModel: discard stale member.joined (userId=%{public}@, no current room)",
                   payload.userId)
            return
        }
        applyMemberJoined(payload)
    case .memberLeft(let payload):
        guard lastObservedRoomId != nil else {
            os_log(.debug,
                   "RealRoomViewModel: discard stale member.left (userId=%{public}@, no current room)",
                   payload.userId)
            return
        }
        applyMemberLeft(payload)
    case .pong:
        break
    case .error(let code, let message, _):
        os_log(.error, "RealRoomViewModel WS error: code=%{public}d, msg=%{public}@", code, message)
    case .unknown(let rawType):
        os_log(.error, "RealRoomViewModel WS unknown message type: %{public}@", rawType)
    }
}
```

**新增方法 `applyMemberJoined(_:)`**：

```swift
/// V1 §12.3 client merge contract `member.joined` 处理路径（行 2061）.
///
/// 增量 mutate 语义（**不是** full snapshot replacement）：
///   - (a) roster 中已存在该 userId entry → 字段级 enrich（nickname 非空覆盖、avatarUrl 空字符串保留
///         existing、pet null 直接覆盖（authoritative pet-less 信号））
///   - (b) roster 中不存在该 userId entry → 新增完整 entry
///
/// **dedup by userId 关键约束**：同一 userId 重复 emit `member.joined`（server bug / 网络重发）
/// **不应**重复添加成员 —— 走分支 (a) enrich 而非追加（防"4 人房间显示 5 个成员"）.
///
/// **节点 4 阶段实装策略**（与 applySnapshot 主干保持一致）：
///   - `level / status / isHost` 沿用 applySnapshot 的占位（默认 8 / "在玩耍" / false）；
///     `isHost` 严格 false（fix-review r4 lesson 同精神：不做位置启发式）
///   - 节点 4 阶段 server `pet.currentState` 固定 1 → **不写入** `memberPetStates`（与 applySnapshot 一致；
///     待 Epic 14 真实驱动）
///   - `nickname` 空字符串场景理论不会发生（V1 §12.3 行 2008 钦定 member.joined nickname 必非空），
///     但防御性写：empty string fallback "成员"（与 applySnapshot 同精神）
///   - `avatarUrl` 字段当前 RoomMember struct 无对应 field —— 节点 4 阶段不挂入 RoomMember；
///     server payload 已有 avatarUrl 但 client 还未渲染头像图（Story 37.13 a11y 表 + Story 30.x 真实
///     渲染时挂入），本 story 仅 codec / payload 层透传，应用层 RoomMember 仍只用 id / name / level /
///     status / isHost；avatarUrl 字段以"server payload 已传 + client RoomMember 待 Epic 接续"路径
///     落地（与 applySnapshot 同 source of truth：snapshot 也下发 avatarUrl，节点 4 阶段同样不挂入
///     RoomMember）.
private func applyMemberJoined(_ payload: MemberJoinedPayload) {
    // 分支 (a)：dedup by userId —— 已存在 entry → 字段级 enrich（不重复 append）
    if let existingIndex = members.firstIndex(where: { $0.id == payload.userId }) {
        let existing = members[existingIndex]
        // nickname 非空覆盖；空字符串保留 existing.name（防御性，理论不应发生）
        let mergedName: String = payload.nickname.isEmpty ? existing.name : payload.nickname
        // level / status / isHost 沿用 existing（applyMemberJoined 不应回退已有 level / status；isHost 严格 false）
        let merged = RoomMember(
            id: existing.id,
            name: mergedName,
            level: existing.level,
            status: existing.status,
            isHost: false
        )
        members[existingIndex] = merged
        os_log(.debug,
               "RealRoomViewModel: applyMemberJoined enriched existing entry (userId=%{public}@)",
               payload.userId)
        return
    }
    // 分支 (b)：新增完整 entry（含全部展示字段；占位字段降级）
    let newName: String = payload.nickname.isEmpty ? "成员" : payload.nickname
    let newMember = RoomMember(
        id: payload.userId,
        name: newName,
        level: 8,                  // 节点 4 阶段占位（与 applySnapshot 一致）
        status: "在玩耍",           // 节点 4 阶段占位（与 applySnapshot 一致）
        isHost: false              // 严格 false（fix-review r4 lesson 同精神）
    )
    members.append(newMember)
    os_log(.debug,
           "RealRoomViewModel: applyMemberJoined appended new entry (userId=%{public}@, members.count=%{public}d)",
           payload.userId, members.count)
}
```

**新增方法 `applyMemberLeft(_:)`**：

```swift
/// V1 §12.3 client merge contract `member.left` 处理路径（行 2099）.
///
/// 增量 mutate 语义：从 roster 中**移除** payload.userId 对应 entry.
///
/// **关键约束**：roster 中不存在该 userId 时 → ignore + log warning；
/// **禁止**抛错 / **禁止**因找不到 entry 把 vm.members 整体清空（防御性 server bug 路径下
/// 不至于把整个房间清空）.
private func applyMemberLeft(_ payload: MemberLeftPayload) {
    guard let index = members.firstIndex(where: { $0.id == payload.userId }) else {
        os_log(.error,
               "RealRoomViewModel: applyMemberLeft userId not in members (userId=%{public}@, members.count=%{public}d)",
               payload.userId, members.count)
        return
    }
    members.remove(at: index)
    // memberPetStates 节点 4 阶段保持空 map；本 story 不动 memberPetStates（与 applySnapshot 一致）.
    // 但若未来 Epic 14 落地时该 user 的 memberPetStates entry 已写入，**应**同步 remove —— 本 story
    // 提前预埋安全语义：从 memberPetStates 也 remove 该 userId（节点 4 阶段空 map 下 no-op，但
    // Epic 14 后真实驱动后该路径自动正确）.
    memberPetStates.removeValue(forKey: payload.userId)
    os_log(.debug,
           "RealRoomViewModel: applyMemberLeft removed entry (userId=%{public}@, members.count=%{public}d)",
           payload.userId, members.count)
}
```

**红线**（不允许偏离）：

- **不**改 `applySnapshot` 算法主干（Story 12.1 已锁定）
- **不**让 `applyMemberJoined` / `applyMemberLeft` 触发 `appState.setCurrentRoomId(_:)`（这两个方法仅 mutate vm.members；房间归属由 HTTP join / leave authoritative 改变 —— 那是 Story 12.7 范围）
- **不**让 `applyMemberLeft` 在找不到 userId 时抛错 / 清空 members（防御性 ignore + log warning）
- **不**让 `applyMemberJoined` 在 dedup 路径上重复 append（必须走 enrich 分支）
- **不**做"`member.left` 之后 vm 自动断开 WS"的副作用 —— payload.userId 不一定是当前 user（可能是别的成员），盲目断开 WS 会破坏链路；当前 user leave 路径在 Story 12.7 LeaveRoomUseCase 由 HTTP 200 响应触发 `appState.setCurrentRoomId(nil)`，进而由 `subscribeRoomIdConnect` A→nil 分支主动 disconnect
- **不**实装"reconnect 后重新拉 snapshot"路径（epic line 2132 后半句"每次 reconnect 后重新拉 snapshot 对齐"属于 **Story 12.5 范围**；本 story 仅守护 `lastObservedRoomId == nil` 时 ignore 增量消息）
- **不**在 `applyMemberLeft` 内复用 `members.removeAll(where:)` —— 必须先 firstIndex 校验存在性 + log warning，否则吞掉 server bug 信号
- **不**对 `member.joined` payload 的 `userId == 当前 user.id` 走 self-fanout 防御过滤（V1 §12.3 行 2063 钦定 server 已从 fanout 排除加入者自己 —— 防御性 noop 即可，不主动过滤）：但**也不**让 self-fanout 误进 enrich/append 路径污染 vm.members；当前 user 没有"自己加自己"语义，server 端实装层 fanout 已排除 —— 本 story 不主动加防御过滤逻辑（与 Story 11.8 server 端 r3 lesson `2026-05-09-async-must-preserve-sync-observable-invariants-11-8-r3.md` 相互信任）

2. **`avatarUrl` 字段是否需要挂入 `RoomMember`？** —— 当前 acceptance 钦定**不**挂入（节点 4 阶段 RoomMember 字段集与 Story 37.8 落地一致 = id / name / level / status / isHost；avatarUrl 仅在 codec / payload 层透传，应用层不渲染头像图）；待 Story 37.13 a11y 表 + Story 30.x 真实 sprite 渲染时再挂入。如 dev 觉得本 story 就该挂入（防止后续 Epic 落地时 RoomMember 类型迁移成本），可在 RoomMember 加 `avatarUrl: String?` 可选字段（默认 nil）—— 但需要同步更新 Story 37.8 落地的 mock 路径（RoomScaffoldDefaults / MockRoomViewModel）；本 story 不主张该路径（保持 RoomMember 字段集稳定）。
3. **`applyMemberJoined` 在 enrich 分支（dedup 路径）是否要保留 existing.level / existing.status？** —— 当前 acceptance 钦定保留 existing（避免在节点 4 阶段无真实 level / status 数据时把"在玩耍" placeholder 退化为新进 entry 的占位）；如 dev 觉得 enrich 分支应同步覆盖 level / status（与 applySnapshot 字段级 merge 同精神），可改为 `level: 8` / `status: "在玩耍"` 重新派生 —— 但要考虑 Epic 14 后真实驱动时 level / status 由 memberPetStates 层派生，applyMemberJoined enrich 路径不应回退已有 level / status；本 story 主张保留 existing 更安全。
