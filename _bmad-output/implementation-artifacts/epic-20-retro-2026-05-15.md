## 2. 关键洞察与 Epic 20 的"主旋律"

### 2.1 主旋律 1: 契约 finalize 类 story 的深度迭代是 server-side 唯一无 dev-time CI 兜底的设计 work

20.1 跑满 15 轮 codex review。**这不是浪费**：r1~r11 每一轮都识别了真问题（不可实现的 rate_limit guarantee / 非原子 Redis idempotency / time-derived 字段缓存 stale / MVCC pending 不可见 / 等），其中 r4-r5 完成了一次根因翻转（Redis 非事务 → MySQL 同事务），后续 r6-r11 把这个根因方案的所有边界 case 全部锁定。r12-r15 是文档跨段一致性（V1 文档 + 数据库设计 + story 文件 + epics.md 之间的 cross-doc anchor 同步）。

**关键观察**：20.6 (业务核心实装) 之所以 review r1 即过、零返工，是因为 20.1 把所有幂等 / 事务 / 边界 case 都在契约层迭代清楚了。dev-time CI / build / test 抓不住"契约模糊"或"契约相互矛盾" — 必须靠 LLM review 反复挑刺 + 切换视角（架构层 → 字面层 → 跨文档层）逐渐收敛。这与 docs 类 story 的本质不同：代码类 story 有编译器 + test runner 兜底；契约类 story 唯一兜底是 LLM 视角的 adversarial review。

对应 lesson：`2026-05-14-contract-finalization-cross-doc-consistency-and-sql-pk.md` + `2026-05-14-stale-snapshot-delete-vs-warn-in-iterating-design-doc-20-1-r14.md` + `2026-05-14-story-file-must-track-canonical-contract-drift-20-1-r12.md` + 其他 r1~r15 各阶段 lesson 共约 15 条。

**对未来 server-side 契约 finalize story 的指导**（Epic 23.1 / 26.1 / 29.1 / 32.1 等）：
- 不要因为"已经迭代 N 轮"就早停 —— 每轮 LLM finding 都可能是新设计层面的盲点
- 注意 over-correction chain 信号：当 r_n+1 推翻 r_n 改动时，立即跳出表层微调，去找根因（如 20.1 r4 Redis → MySQL 根因翻转）
- 同步成本是真实成本：契约改了一处，story 文件 AC / epics.md / 数据库设计 / 错误码表 都得同步 — fix-review sub-agent 必须负责跨文件 sweep（r12 / r14 都是因为跨段 stale 引入新 finding）

### 2.2 主旋律 2: over-correction chain 是 review 第 3 轮起的常见反模式 — 终结条件是"上移一层抽象"

Epic 20 三次完整的 over-correction chain：

**20.7 race condition chain（5 轮）**：
- r1: race after FOR UPDATE（unlock 跑到 next chest）
- r2: client 传 chestID 但 RowsAffected 不能等同 NOT FOUND
- r3: 加回 RowsAffected check 但同毫秒重复误判
- r4: **事务包 SELECT FOR UPDATE + UPDATE 根因解决**（跳出 RowsAffected 表层）
- r5: contract regression（chestId 从 optional 变 required）→ fix-with-doc-only

**20.8 stub endpoint chain（4 轮）**：
- r1: silent success false-positive → 改返失败
- r2: ErrServiceBusy → HTTP 500 + ERROR log
- r3: 新错误码 ErrNotImplemented → 501 + WARN，但 5xx counter 仍污染
- r4: **metrics 层 skip /dev/* 根因解决**（跳出错误码语义微调）

**20.9 testing 责任分离 chain（7 轮）**：
- r1: 期望 3002 实际 4002（业务步骤顺序错判 — 真 bug）
- r2: rarity 分布断言 flaky + boundary 不精确
- r3-r7: 在"防 flaky" vs "防 bypass detection" 之间反复横跳
- r8: **承认 testing 层 fundamental tension 不可同时满足，flag bypass + 责任分离落 lesson**（跳出测试断言微调）

**模式标定**：当 review 进入第 3+ 轮、每轮 finding 都来自上一轮修复（不是新缺陷）→ 必须**上移一层抽象**：
- driver 层模糊性（RowsAffected）→ 用事务约束消除
- 错误码语义微调（500 vs 503 vs 501）→ 改 metrics 层路径过滤
- 测试断言强弱 trade-off → 用责任分离（单测覆盖算法 / 集成测试覆盖 wiring / observability 覆盖 race）

对应 lesson：`2026-05-15-domain-aware-rowsaffected-and-over-correction-chain-20-7-r3.md` + `2026-05-15-transaction-eliminates-rowsaffected-ambiguity-20-7-r4.md` + `2026-05-15-integration-vs-unit-test-responsibility-split-weighted-picker-20-9-r3.md` + `2026-05-15-test-reliability-over-completeness-20-9-r6.md`。

**对未来任何 review 进入 ≥ 3 轮 same-dimension 微调的 story 的指导**：立即在元层面提问 "我是否在追求不可达的完美 / 应该把哪些职责丢给其他层" —— 强行往下修会持续引入新边界 case。epic-loop.md 的 r15 hard cap 是兜底，但更好的做法是 r3-r4 时主动跳出。

### 2.3 主旋律 3: 实装类 story 的"review r1 即过"是契约层迭代的回报

20.2 / 20.3 / 20.4 / 20.5 / 20.6 五条 story review r1 即过 — 占 Epic 20 一半以上。它们的共同特点：
- 实装产物紧贴一份已经迭代成熟的契约（20.1 锁定的 V1 §7.1 / §7.2 / §13 + 数据库设计 §5.8 / §5.16）
- 不涉及新设计决策 — 只是把契约 mapped 到 Go / SQL
- 测试覆盖随实装一并落地（dockertest + sqlmock unit）

**对未来 story 拆分的指导**：当一个 epic 的核心是"server-side 业务实装" 时，先用一条 contract-finalize story（如 X.1）把所有契约钦定到无 ambiguity，后续实装 story 自然就能一次过。这是 Epic 11 (room) / Epic 17 (emoji) / Epic 20 (chest) 三次反复验证的 pattern。

## 5. 关键 Takeaways（Top 5）

1. **契约 finalize 类 story 不要因"已迭代 N 轮"早停**：20.1 r1~r15 每轮都识别真问题，根因翻转发生在 r4-r5（Redis → MySQL）。这是 server-side 设计 work 唯一无 dev-time CI 兜底的 step，必须靠 LLM adversarial review 反复挑刺收敛。
2. **over-correction chain 第 3-4 轮起必须上移一层抽象**：driver 层模糊（RowsAffected）→ 事务约束；错误码语义微调（500/503/501）→ metrics 路径过滤；测试断言强弱 trade-off → 责任分离。三个 chain (20.7 / 20.8 / 20.9) 都验证了"持续往下修引入新边界 case"的反模式。
3. **实装类 story 一次过的前提是契约层已迭代到无 ambiguity**：20.2-20.6 五条 review r1 即过的 story 都基于 20.1 已锁定的契约。Epic 11 / 17 / 20 三次反复验证此 pattern。
4. **sub-agent 中断恢复 = audit + resume，不要回滚**：Epic 20 期间 2 次中断（quota + socket）通过 audit 现有产物 + 补缺失部分恢复，无数据破坏。epic-loop.md 步骤 1.5 的恢复点诊断机制有效。
5. **time-derived 字段在缓存场景下必须穷举处理**：20.1 r9 / r11 / r13 / r14 反复在不同章节修复 nextChest.status / remainingSeconds / requestId 的缓存处理 — 任何新字段加入 idempotency response_json 缓存前，先问"是不是 time-derived / per-request trace？"。
