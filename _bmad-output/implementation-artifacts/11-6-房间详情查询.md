- **V1 §10.3 GET /rooms/{roomId} 服务端逻辑钦定**（V1接口设计.md 行 1222-1407）—— **本 story 实装路径严格按 V1 §10.3 + 数据库设计 §8.8**：

  事务边界：开 MySQL 事务（**REPEATABLE READ** 隔离级别，InnoDB 默认；**不**需要显式调 `SET TRANSACTION ISOLATION LEVEL`，GORM `db.Transaction(...)` 默认走 InnoDB 默认级别即 REPEATABLE READ）包以下全部 4 个步骤。**snapshot 隔离 + FOR SHARE 行锁两个机制是互补的，缺一不可**。

  按顺序执行（**全部在同一 REPEATABLE READ 事务内**）：

  1. **步骤 1a**：查 `users.current_room_id`（事务内 `userRepo.FindByID(txCtx, callerID)`，与请求的 `roomId` 不一致（含 `current_room_id` 为 null）→ 立即返回 **6004 用户不在房间中**（caller 不是该房间成员，禁止查看其他成员隐私字段）
  2. **步骤 1b**：通过后追加 `SELECT 1 FROM room_members WHERE room_id = ? AND user_id = caller FOR SHARE LIMIT 1`（**取共享锁**锁定 caller 自己的成员行，确保本读事务持续期间该行不会被并发 leave 的 DELETE 提交 —— DELETE 需要排他锁，与 FOR SHARE 互斥，必须等本读事务 commit 后才能继续）；命中 0 行 → 视同步骤 1a ACL 失败，返回 **6004**（兜底语义；正常情况下步骤 1a 通过即意味着该行存在，本步骤是 race 兜底）
  3. **步骤 2**：查 `rooms WHERE id = ?`（事务内 `roomRepo.FindByID(txCtx, roomID)`），**找不到** → 返回 **6001 房间不存在**（理论上步骤 1a 已通过意味着 caller 在该房间，rooms 行必存在；本步骤是兜底）
  4. **步骤 3**：查 `room_members WHERE room_id = ?` + INNER JOIN `users` + **LEFT JOIN `pets`** 聚合（按 `room_members.joined_at ASC` 稳定排序）—— **必须用 LEFT JOIN `pets`**（不能用 INNER JOIN）：pet-less 账号（用户无活跃 pet 行）若用 INNER JOIN 会被静默丢行 → 违反 `memberCount === members.length` 不变量；LEFT JOIN 时 `pets.*` 列为 NULL，service 实装层据此把 `data.members[].pet` 整体下发为 `null`
  5. **步骤 4**：提交事务并返回 `data.{room, members}`

  **关键约束**（V1 §10.3 + 数据库设计 §8.8 钦定）：
  - **6004 双路径必须等价**：步骤 1a 预检 + 步骤 1b FOR SHARE 兜底两条路径都返 `apperror.New(ErrUserNotInRoom, ...)`，handler 端响应 envelope 完全一致 —— client **不**应区分这两个子场景（含"用户不在任何房间" + "用户在其他房间" + race 下成员行已被并发 leave 删除）
  - **commit-after-response 残留窗口**（r12 锁定）：FOR SHARE 锁在事务 commit 时释放，handler 通常的执行顺序是 `commit tx → 序列化 JSON → ctx.WriteJSON / flush response body`，因此存在一个 **commit → flush** 之间的窗口（μs 量级）：在该窗口内并发 leave 可能 commit 其 DELETE，从而 caller 的 roster 字节流是在 caller 已离开后才到达对端 —— 这是**协议层面**承认的 best-effort 残留 race，**不**强制要求 server 在 commit 前先序列化 + WriteJSON 后再 commit（实装代价大，需重构 handler 流程）；端到端实质安全由 **client 防御性 discard** 兜底（client 已 leave 后收到的 stale roster 直接丢弃），不是 server 端协议保证
  - **deadlock 风险**：caller 锁的是 `(room_id, user_id=caller)` 这一**特定行**；并发 leave 也只锁 `user_id=caller` 这一行（取排他锁后 DELETE）；其他成员 / 其他房间走的是不同 `(room_id, user_id)` 锁路径，不存在锁顺序循环 → 无 deadlock 风险。最坏情况是并发 leave 等本读事务 commit 后再提交，等待时长 = 本读事务全步骤时长（μs ~ 数 ms 量级），可忽略
  - **错误码触发顺序**：步骤 1a → 6004（预检 ACL）；步骤 1b → 6004（FOR SHARE 0 行兜底）；步骤 2 → 6001（rooms 兜底）；任何 DB 异常 → 1009。**不**对外暴露 6002 / 6003 / 6005（V1 §10.3 行 1347 钦定纯查询不涉及 join 路径错误码；6005 在节点 4 阶段**不**用于本接口，即使 closed 房间也允许查询，但前提是 caller 仍是该房间成员才能走到这步）
  - **roster 不变量**：`data.room.memberCount` **必须严格等于** `data.members[]` 数组长度；`data.members[]` 必须包含**全部** `room_members WHERE room_id = ?` 行；service 实装层**禁止**做"在线态过滤"（与 §12.3 `room.snapshot` 不变量一致：snapshot / GET /rooms 都是 full roster view，不是 online-only view）
  - **status 与 members 关系**：`status = 2 closed` 时 `members[]` 必为 `[]`（最后一人离开同时删除最后一行 + status 变 2）；`status = 1 active` 时 `members[]` 必非空（房间至少 1 个成员）

**AC1 — `RoomMemberRepo.ExistsForShareByRoomAndUser` 方法新增**

修改 `server/internal/repo/mysql/room_member_repo.go`：

- `RoomMemberRepo` interface 新增 `ExistsForShareByRoomAndUser(ctx context.Context, roomID, userID uint64) (bool, error)` 方法签名（紧接既有 `DeleteByRoomAndUser` 方法之后）：
  - doc comment 说明：取共享锁（FOR SHARE）锁定 (room_id, user_id) 双键定位的单行 room_members（V1 §10.3 步骤 1b 钦定）；命中 1 行 → (true, nil)；0 行 → (false, nil)；query 失败 → (false, raw error 透传给 service 层（service 包成 1009）。
  - **必须在事务内调用**（与同事务 `userRepo.FindByID`（步骤 1a） + `roomRepo.FindByID`（步骤 2） + `RoomMemberRepo.ListRosterByRoomID`（步骤 3）一起执行）；事务外调用 FOR SHARE 锁立即释放，违反 V1 §10.3 + 数据库设计 §8.8 钦定的串行化路径 —— 跨事务并发 leave 的 DELETE 不再被阻塞，post-leave 数据泄漏 race 重新出现
- `roomMemberRepo.ExistsForShareByRoomAndUser` impl（紧接既有 `DeleteByRoomAndUser` impl 之后）：

  ```go
  // ExistsForShareByRoomAndUser 取 FOR SHARE 共享锁锁定 (room_id, user_id) 双键定位
  // 的单行 room_members（Story 11.6 引入；用于 V1 §10.3 步骤 1b ACL 共享锁）。
  //
  // **必须在事务内调用**（与同事务 userRepo.FindByID 步骤 1a + roomRepo.FindByID
  // 步骤 2 + ListRosterByRoomID 步骤 3 一起执行）；事务外调用 FOR SHARE 锁立即
  // 释放（autocommit 模式下任何 SELECT 完成即 commit），违反 V1 §10.3 + 数据库
  // 设计 §8.8 钦定的串行化路径 —— 跨事务并发 leave 的 DELETE 不再被阻塞，
  // post-leave 数据泄漏 race 重新出现。
  //
  // FOR SHARE vs FOR UPDATE：
  //   - FOR SHARE 取共享锁（S 锁），多个读事务可同时持锁，但与排他锁（X 锁，
  //     leave 事务的 DELETE 取）互斥；本 story 多个 GET /rooms/{roomId} 同时进
  //     行可彼此并行，仅与并发 leave 事务串行化
  //   - FOR UPDATE 取排他锁（X 锁），与所有锁互斥，性能更差；本 story 是只读
  //     接口，用 FOR SHARE 已足够（snapshot 隔离 + FOR SHARE 锁双机制提供
  //     ACL 边界保护，详见 V1 §10.3 rationale + 数据库设计 §8.8）
  //
  // 用 GORM Raw("SELECT 1 FROM room_members WHERE room_id = ? AND user_id = ? FOR SHARE LIMIT 1") +
  // Scan(&dummy) 路径生成 SELECT ... FOR SHARE SQL；走 idx_room_id 索引 +
  // uk_room_user 唯一约束高效定位（最多 1 行匹配）。
  //
  // 返：
  //   - 命中 1 行 → (true, nil)
  //   - 0 行 → (false, nil) —— 与 RoomExists / IsUserInRoom 同模式，Raw + Scan
  //     在 0 行时不返 ErrRecordNotFound 而是保持 dummy zero-value
  //   - query 失败 → (false, raw error 透传给 service（service 包成 1009）
  func (r *roomMemberRepo) ExistsForShareByRoomAndUser(ctx context.Context, roomID, userID uint64) (bool, error) {
      db := tx.FromContext(ctx, r.db)
      var dummy int
      err := db.WithContext(ctx).
          Raw("SELECT 1 FROM room_members WHERE room_id = ? AND user_id = ? FOR SHARE LIMIT 1", roomID, userID).
          Scan(&dummy).Error
      if err != nil {
          if stderrors.Is(err, gorm.ErrRecordNotFound) {
              return false, nil
          }
          return false, err
      }
      return dummy == 1, nil
  }
  ```

  - **不**新增哨兵 error（与 11.4 `CountByRoomID` / 11.5 `DeleteByRoomAndUser` 同模式；rowsAffected == 0 路径不是异常路径，是合法业务场景，由 service 层判定）

  ```go
  //   - DB error → (nil, raw error 透传给 service（service 包成 1009）
  //
  // 用 GORM Raw 路径而非 Joins / Preload —— Raw 让 SQL 显式可读 + Scan 到自定
  // 义 RosterRow struct 避免 GORM Preload 多查（preload 路径会触发额外 1 query
  // 加载 pets，违反"读快照事务步骤 3 单 SQL"钦定）。
  func (r *roomMemberRepo) ListRosterByRoomID(ctx context.Context, roomID uint64) ([]RosterRow, error) {
      db := tx.FromContext(ctx, r.db)
      var rows []RosterRow
      err := db.WithContext(ctx).
          Raw(`SELECT room_members.user_id AS user_id, users.nickname AS nickname, users.avatar_url AS avatar_url, pets.id AS pet_id
               FROM room_members
               INNER JOIN users ON room_members.user_id = users.id
               LEFT JOIN pets ON pets.user_id = room_members.user_id AND pets.is_default = 1
               WHERE room_members.room_id = ?
               ORDER BY room_members.joined_at ASC`, roomID).
          Scan(&rows).Error
      if err != nil {
          return nil, err
      }
      if rows == nil {
          // GORM Scan 在 0 行时可能保持 nil；统一返 [] 让调用方不需要 nil-check
          return []RosterRow{}, nil
      }
      return rows, nil
  }
  ```

  - **不**新增哨兵 error（0 行不是异常路径，是合法业务场景）
  - **不**用 GORM Joins / Preload 路径（避免 N+1 query / 多 SQL；本 story 严格要求步骤 3 单 SQL）

  ```go
  // 4 步事务（Story 11.6 引入）：
  //
  //	步骤 1（事务内 4 步）：
  //	  1a. userRepo.FindByID(txCtx, UserID) → CurrentRoomID 校验 → 不一致 → 6004

  //	  3.  roomMemberRepo.ListRosterByRoomID(txCtx, RoomID) → roster
  //	步骤 2（事务后）：commit / rollback；commit 成功 → 拼装 GetRoomDetailOutput
  //	  含 BIGINT 字段 + LEFT JOIN pet-less nullable + 节点 4 固定字段
  //
  // **错误码触发顺序锁定**（V1 §10.3 钦定）：步骤 1a → 6004（预检）；1b → 6004（兜底）；
  // 步骤 2 → 6001；任何 DB 异常 → 1009。**不**对外暴露 6002 / 6003 / 6005。
  //
  // **6004 双路径必须等价**：步骤 1a + 步骤 1b 都返 apperror.New(ErrUserNotInRoom, ...)，
  // handler 端响应 envelope 完全一致。
  //
  // **ctx 用法**：fn 内全部 4 个 repo 调用用 txCtx；用错 ctx 让 FOR SHARE 锁立即释放，
  // post-leave 数据泄漏 race 重新出现。
  //
  // **节点 4 硬编码字段**：MemberPetOutput.CurrentState 固定 1 (rest)；Equips 固定 [];
  // 节点 5 / 9 / 10 由 Epic 14 / 26 / 29 真实驱动时改为 query 结果。
  //
  // **LEFT JOIN pets 行为**：RosterRow.PetID 为 *uint64；nil 表示 pet-less（LEFT JOIN
  // 行 NULL）→ MemberOutput.Pet == nil；非 nil → MemberOutput.Pet = &MemberPetOutput{
  // PetID: *RosterRow.PetID, ...}（节点 4 硬编码 CurrentState / Equips）。
  func (s *roomServiceImpl) GetRoomDetail(ctx context.Context, in GetRoomDetailInput) (*GetRoomDetailOutput, error) {
      var out *GetRoomDetailOutput

      err := s.txMgr.WithTx(ctx, func(txCtx context.Context) error {
          // 步骤 1a: ACL 预检（事务内查 users.current_room_id）
          user, err := s.userRepo.FindByID(txCtx, in.UserID)
          if err != nil {
              return err // 含 ErrUserNotFound 哨兵 / DB raw error；外层包成 1009

          if user.CurrentRoomID == nil || *user.CurrentRoomID != in.RoomID {
              return apperror.New(apperror.ErrUserNotInRoom, apperror.DefaultMessages[apperror.ErrUserNotInRoom])
          }

          // 步骤 1b: ACL FOR SHARE 兜底（race 防御）
          inRoom, err := s.roomMemberRepo.ExistsForShareByRoomAndUser(txCtx, in.RoomID, in.UserID)
          if err != nil {
              return err // DB raw error；外层包成 1009
          }
          if !inRoom {
              // 步骤 1a 通过但步骤 1b 0 行 → 并发 leave 已删该行 → 6004 兜底
              return apperror.New(apperror.ErrUserNotInRoom, apperror.DefaultMessages[apperror.ErrUserNotInRoom])
          }

          // 步骤 2: 查 rooms（不带锁普通查询）
          room, err := s.roomRepo.FindByID(txCtx, in.RoomID)
          if err != nil {
              if stderrors.Is(err, mysql.ErrRoomNotFound) {
                  return apperror.New(apperror.ErrRoomNotFound, apperror.DefaultMessages[apperror.ErrRoomNotFound])
              }
              return err
          }

          // 步骤 3: 查 roster（INNER JOIN users + LEFT JOIN pets，ORDER BY joined_at ASC）
          roster, err := s.roomMemberRepo.ListRosterByRoomID(txCtx, in.RoomID)
          if err != nil {
              return err
          }

          // 拼装 output
          members := make([]MemberOutput, 0, len(roster))
          for _, r := range roster {
              m := MemberOutput{
                  UserID:    r.UserID,
                  Nickname:  r.Nickname,
                  AvatarURL: r.AvatarURL,
              }
              if r.PetID != nil {
                  // 非 pet-less：节点 4 阶段硬编码 CurrentState=1 / Equips=[]
                  m.Pet = &MemberPetOutput{
                      PetID:        *r.PetID,
                      CurrentState: 1, // V1 §10.3 节点 4 阶段固定 1 (rest)
                      Equips:       []EquipOutput{}, // 节点 4 阶段固定 []
                  }
              }
              members = append(members, m)
          }

          out = &GetRoomDetailOutput{
              RoomID:        room.ID,
              CreatorUserID: room.CreatorUserID,
              MaxMembers:    room.MaxMembers,
              Status:        room.Status,
              MemberCount:   len(members), // 不变量：== len(Members)
              Members:       members,
          }
          return nil
      })

      if err != nil {
          // 业务码分流：apperror 已包装的直接透传（步骤 1a / 1b 6004 + 步骤 2 6001）
          var ae *apperror.AppError

              return nil, err
          }

          // / ExistsForShareByRoomAndUser error）走 generic 1009 兜底
          return nil, apperror.Wrap(err, apperror.ErrServiceBusy, apperror.DefaultMessages[apperror.ErrServiceBusy])
      }

      return out, nil
  }
  ```

### 与 Story 4.6 / 7.3 / 11.3 / 11.4 / 11.5 业务事务模式的对照

本 story 实装路径**与既有写事务 story 模式有 5 个关键差异**：

| 维度 | 11.3 / 11.4 / 11.5（写事务） | 11.6 GetRoomDetail（读快照事务） |
|---|---|---|
| 锁类型 | FOR UPDATE 排他锁（rooms 行） | FOR SHARE 共享锁（room_members 行） |
| 锁对象 | rooms 行（与 join / leave 互斥） | caller 自己的 room_members 行（与 leave DELETE 互斥） |
| 锁释放时机 | commit | commit（同写事务） |
| 业务规则 | 多 step 写 + 状态变更 | 多 step 读 + ACL 校验 |
| 错误码集 | 6001/6002/6003/6004/6005/1009（写路径业务码） | 6001/6004/1009（读路径仅 ACL + rooms 兜底） |
| 事务隔离 | InnoDB 默认 REPEATABLE READ（写路径默认） | InnoDB 默认 REPEATABLE READ（读快照隔离 + FOR SHARE 互补） |
| post-commit 钩子 | 11.4 1 个（broadcast member.joined） / 11.5 2 个（close 4007 + broadcast member.left） | **无**（纯查询无副作用） |

**反模式（已避免）**：
- handler 直接调多 repo 拼事务（违反 §5.1 "service 是事务边界归属"）
- service 用 `*gorm.DB` 直接调 `db.Transaction(...)`（绕过 tx.Manager interface）
- fn 内 repo 用外层 `ctx` 而非 `txCtx`（绕过 tx 走 db pool 新连接）—— 对 FOR SHARE 路径**致命**，锁立即释放
- 6004 预检 + 兜底两条路径返不同 message（client 难以一致处理）
- 步骤 2 SELECT FOR UPDATE rooms 而非 SELECT 普通查询（**违反 V1 §10.3 钦定 read 路径用 FindByID**；与 11.4 join 接口的 6001 模式不同 —— 本 story 步骤 2 不锁 rooms，仅靠 snapshot 隔离 + 步骤 1b 的 FOR SHARE 锁 caller 自己 room_members 行）
- LEFT JOIN pets 但 INNER JOIN 同模式（**违反 §10.3 钦定 LEFT JOIN，让 pet-less 账号 silently drop**）
- service 层不硬编码 CurrentState=1 / Equips=[]（让节点 4 阶段假装真实驱动）—— V1 §10.3 字段表节点 4 列**显式钦定**这两个字段为固定值，service 层硬编码避免 query 层无效查询
- handler 用 string 字段而非 *string 字段下发 nullable roomId（让 nil 序列化为空字符串而非 JSON null）
- handler 用 []Equip 不带 omitempty（避免空数组下发为 JSON null —— 节点 4 阶段必须 `[]`）
