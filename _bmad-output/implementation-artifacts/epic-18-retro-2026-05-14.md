## 2. 关键洞察与 Epic 18 的"主旋律"

### 2.1 主旋律 1: codex review 在 iOS 端集中暴露"async window + transient state lifecycle"两大短板

Epic 18 共 5 个 [P2] finding，**4 个**落在两个明确范畴：

**async window 范畴**（2 条）：
| Story | Finding | 修复方向 |
|---|---|---|
| 18.1 r1 | `LoadEmojisUseCase` actor 内 `if cache == nil { cache = await repo.list() }` 在 await 处释放 actor 隔离，两个并发 caller 都看到 `cache == nil` → 各自发 GET /emojis 破坏 single-load 契约 | 加 `inflightTask: Task<[EmojiConfig], Error>?` 字段，miss 路径先看 inflight，无则建并存，多 caller 共享同 Task；done 后清空 inflight |
| 18.3 r2 | `onEmojiSelected` Step B（catalog 校验 await）期间用户切房间 → Step C（useCase.execute）调当前 WS 连接（指向 room B）发 emoji.send（无 roomId 字段无法 server 端按 room 分流）→ room A 选的表情飞到 room B | 入口 capture `snapshotRoomId = appState.currentRoomId`；await 之后比对 `currentRoomId == snapshotRoomId`，不等则 skip WS send（本地动效不回滚） |

**两条 finding 的共性**：Swift actor / async/await 提供的隔离**仅在单个不跨 await 的 critical section 内有效**；任何跨 await 的 read-modify 流程都必须显式 capture-then-recheck 或 single-flight。这是 iOS Epic 18+ 全期都需要警惕的 anti-pattern，对应 lesson 已落 2 条（`2026-05-14-actor-reentrancy-needs-inflight-task-for-single-flight.md` + `2026-05-14-emoji-send-cross-room-race-needs-snapshot-roomid-guard-18-3-r2.md`），未来 Epic 21 / 23 / 27 等任何 client 端 use case 在 async 里有"读 → await → 写"的路径都应被 cite。

**transient state lifecycle 范畴**（2 条）：
| Story | Finding | 修复方向 |
|---|---|---|
| 18.3 r1 | `activeEmojis` 新增 room-scoped 字段没在 `subscribeRoomIdConnect` 三个 reset 分支（nil→A / A→B / A→nil）清空；旧房间表情会残留到新房间 | 三个 reset 分支都加 `self.activeEmojis = []`，与既有 `members = []` / `memberPetStates = [:]` 同位置同模式 |
| 18.4 r1 | `EmojiAnimationLayer` 的 1.5s 自动 expire 只覆盖 remote `applyEmojiReceived` 路径；local `onEmojiSelected` 入队路径永不被清理 → ForEach 累积 invisible cells + a11y 节点污染 | 把 1.5s 过期 Task 提到入队点（onEmojiSelected Step A 之后），两路径同 owner 同 lifecycle |
| 18.4 r1 | `memberAnchors` PreferenceKey reduce 用 merge 模式（旧+新）保留离开成员的 stale anchor → emoji.received for 离开 user 应该走 centerAnchor fallback 但拿到 stale anchor → 破坏 V1 §12.3 (c) | reduce 改为完全覆盖（每帧整 dict 替换式赋值，离场子视图自然消失） |

**三条 finding 的共性**：在 SwiftUI / ViewModel 里新增任何 `@Published` 集合 / `@State` 派生数据时，必须同时回答两个问题：(1) 谁负责清空？(2) 上游事件结束时（room 切换 / member leave / animation 完成）资源是否回收？这是 ViewModel 层"数据 owner"思维的体现。对应 lesson 已落 3 条（`2026-05-14-room-transient-state-must-reset-on-room-transition.md` + `2026-05-14-swiftui-preferencekey-merge-vs-replace-on-roster-change.md` + 后者同条 lesson 复合覆盖 transient queue owner-side expire），未来 Epic 21 / 24 / 27 / 33 新增任何 room-scoped 或 view-scoped state 都应被 cite。

**剩 1 条 [P2]（18.1 r2）属于第三个范畴 - "feature ViewModel 自建 errorMapper 必须镜像 AppErrorMapper 二分"**：`.localStoreFailure`（transient retryable）被错归到 `.missingCredentials`（terminal）的 "登录已失效" 文案。这条 lesson（`2026-05-14-viewmodel-error-mapping-must-mirror-apperrormapper-transient-vs-terminal-18-1-r2.md`）是 Epic 27 / 33 等会写 ViewModel friendlyMessage 的 story 应当回看的。

### 2.2 主旋律 2: 一次 quota HALT + resume 模式跑通 → /epic-loop 的"in-progress 状态恢复"路径事实可行

Epic 18 中段最特殊事件：18.4 dev_story sub-agent 跑到 Task 7（RoomScaffoldView + EmojiAnimationLayer wire）中段时 Anthropic API 配额耗尽（"You've hit your limit · resets 12:30pm Asia/Shanghai"），sub-agent 截断退出，sprint-status 中 18-4 状态已被推到 `in-progress` 但 12 个 Task checkbox 全部 unchecked。

按 /epic-loop.md 现有规则，`case 'in-progress'` 应当 HALT 让用户手动决定 resume / 放弃。本 epic 实际走了一条**未在 /epic-loop 文档明文钦定**的路径：

1. 主 agent HALT 给出三个方案（A 重跑、B 续做、C 手动）
2. 用户回 "继续"
3. 主 agent grep story 文件确认 Task checkbox 全 unchecked + 列出工作区残留改动文件
4. 派**新的 dev_story sub-agent**，prompt 中显式告知"上一个 sub-agent 部分实装留存在工作区，请先 git diff 审计哪些 AC 已被部分/完整实装，避免回归 lesson；不要盲目重写"
5. 新 sub-agent 审计后判定上次完成了 Task 1-6 + Task 7 大部分（构造参数 / @State / GeometryReader 外层 / EmojiAnimationLayer 调用 / coordinateSpace 已落地）；缺失 Task 7.5（PetSpriteView 包 GeometryReader anchor 报告）/ 7.7（MemberAnchorPreferenceKey + CGRect.midPoint helper）/ 7.8（删除 placeholder）+ Task 8-12 整体未启动
6. 续做补完所有缺失 → 状态推到 review → 跑后续 review/fix-review/story-done 链路正常收敛

**结论**：resume 模式事实可行，前提是上次 sub-agent 没造成"半成品损坏既有代码"（本次只 add / 简单 modify，没破坏既有 wire）。这条经验应被显式写进 /epic-loop.md 的 in-progress 分支或一份独立 lesson（行动项 A2）。

### 2.3 主旋律 3: 18.2 r1 一次过证明"前置 story 把 ViewModel 基类规范写对"的复利

18.2 是 epic-18 唯一一次过（0 fix-review）的 story。直接原因是它**复用了 Epic 12（room ViewModel 基类）和 17 个既有 lesson 形成的模式**：

- `RoomViewModel` 基类的 abstract method override 模式（与 `onLeaveTap` / `applyRoomLeft` 等既有 abstract method 同模式）
- `currentUserId` 字段的 fail-safe nil 语义（与既有 `currentRoomId` 同语义）
- View 层 `member.id == state.currentUserId` 自己/他人分支（直接 mirror 既有 host vs guest 分支判定模式）
- `.sheet` 挂载 EmojiPanelView（直接 mirror Epic 11.7 JoinRoomModal 既有 .sheet 模式）
- UITest 走 `--uitest-emoji-panel-host` launch arg + AppContainer mock 路径（直接 mirror 18.1 已建立的 UITest hook 模式）

codex r1 review 唯一找到的"问题"是 Swift 6 EnvironmentKey 数据竞争警告（当前 Swift 5 build 下不是错误），明确判 not actionable。

**结论**：当 story 边界小 + 复用既有模式时，codex 一次过的概率高。对比 18.1 / 18.3 / 18.4 都需要做新机制（actor 缓存 / async window guard / preference key 收集 / 动画 layer）—— 都触发了 [P2]。这条规律对未来 epic 拆 story 有指导意义：**优先把"使用既有 ViewModel/View 模式的 wire-up 类工作"切成独立 small story，让它一次过**；把"引入新机制"的工作集中到独立 story 接受多轮 review。

### 2.4 主旋律 4: 环境 quirk 反复干扰 → sub-agent 学会绕过路径

本机当前 Xcode 26.5 SDK 已安装但 iOS 26.5 simulator runtime 未安装（只有 iOS 26.4 runtime），导致 `bash iphone/scripts/build.sh` 在 xcodegen regen 后 `xcodebuild -showdestinations` 返回的 Available 段没有 concrete simulator → 脚本的 destination 解析 fallback 失效 → build 失败。

这条 quirk 在 Epic 18 期间**至少 5 次反复出现**（每个 story 的 dev_story 阶段都遇到，fix-review 阶段也遇到一次 CoreSimulatorService 崩溃）。绕过路径：直接调 `xcodebuild build -project iphone/PetApp.xcodeproj -scheme PetApp -destination "platform=iOS Simulator,id=<具体 UDID>" -derivedDataPath iphone/build/DerivedData`，**不**跑 xcodegen regen → 复用既有 pbxproj → BUILD SUCCEEDED。

第一次（18.1 dev_story）由 sub-agent 自己摸索出来；后续每个 dev_story / fix_review sub-agent 的 prompt 都把这条绕过路径写进"环境陷阱"段，避免每个 sub-agent 都重新踩坑。这是 /epic-loop 主 agent 在 sub-agent 间传递"环境知识"的有效模式。

**结论**：环境 quirk 不算 epic 工作产物，但应作为环境维护 tech-debt 登记（行动项 A1），未来其他 iPhone story 都会受影响。

## 7. 元教训（Epic 18 给 Epic-loop 工程化的回馈）

1. **codex review 在 iOS Swift 端的 [P2] 抓取能力同样可靠**。Epic 17 server Go 端 codex 抓的是 stale-Session 跨房间注入（17.1 r1）、INSERT IGNORE 留坏行（17.3 r3）、INSERT 留 admin 数据（17.3 r4 重提）等 Go-specific 模式；Epic 18 iOS Swift 端 codex 抓的是 actor reentrancy、async window race、room transition reset 漏清、PreferenceKey merge 漏覆盖、ViewModel errorMapper 不对齐 AppErrorMapper —— 全部是 Swift/SwiftUI/MVVM 圈典型陷阱。证明 codex 不偏科，跨语言跨架构都能 raise relevant finding

2. **Resume 模式的工程化价值**：单次 sub-agent 跑大 story（如 18.4 12 个 Task）确实有 quota 风险；但只要工作区残留有效（add / 简单 modify、不破坏既有 wire）、story 文件 Task checkbox 状态可读，resume 可以在不丢工作的前提下续完。这条经验降低了未来对"超大 story 必拆"的强制要求 —— 拆 story 仍是好实践，但即使没拆，事故场景下也有恢复路径

3. **环境 quirk 在跨 sub-agent 传递的重要性**：单个 sub-agent context 隔离的代价是"每个 sub-agent 都从零开始"。如果主 agent 不主动在 prompt 里传递"全局生效但非 story-specific"知识（如 build 脚本绕过路径），每个 sub-agent 都要重新踩坑 5-10 分钟。本 epic 主 agent 在 5 次 sub-agent 派单 prompt 里都加了"环境陷阱"段，事实证明是高 ROI 的 prompt 投资

4. **iOS Epic 的 review/fix 轮次密度低于 server Epic 的部分原因**：iOS 端 ViewModel/View 改动通常是 "use case + view 拼装"，而 server 端契约 + migration + handler + service 链路有更多"跨层一致性"约束（17.1 5 轮、17.3 4 轮就是这种链路的产物）。Epic 18 的 9 轮（最大 3）相比 Epic 17 的 12 轮（最大 5）显著少 —— 这部分是 iOS 端特性，部分是 Epic 17 5 条 contract / migration 链路 lesson 的下游消化效果

5. **本 epic 没触发 17.3 那种"两个互斥设计目标死循环"的 escape hatch**。5 个 [P2] 都是"看一眼方向就明确 + fix 走单向"的常规 finding。这印证 Epic 17 retro 的判断："17.3 r4 lock-down 是预期路径，但确实是低频事件" —— Epic 18 4 个 story 都没触发，可见这种情况是稀有的，不是 epic-loop 流程的主旋律

## 8. 总结
