# Story 2.6: 基础错误 UI 框架（Toast / AlertOverlay / RetryView + ErrorPresenter）

Status: done

<!-- Validation 可选。建议运行 validate-create-story 在 dev-story 前做一次质检。 -->

## Story

As an iOS 开发,
I want 一套统一的错误 UI 组件（Toast / AlertOverlay / RetryView）+ 与 `APIError` 自动联动的 `ErrorPresenter` + 把组件挂载到 root 的 ViewModifier,
so that 后续业务页（Epic 4 GuestLogin / Epic 5 自动登录 / Epic 7+ 业务接口）能直接 `errorPresenter.present(error)` 触发统一 UX，不必各自实装错误展示，**且本 story 不动任何已有业务 flow（HomeViewModel / Ping flow / RootView 挂载点除外的部分）**。

## Acceptance Criteria

**AC1 — `AppErrorMapper`：APIError → 用户文案映射**

新建 `iphone/PetApp/Shared/ErrorHandling/AppErrorMapper.swift`：

```swift
import Foundation

/// `AppErrorMapper`：把 APIError 映射到面向用户的文案 + 选定呈现样式（toast / alert / retry）。
///
/// 设计目标：让 ViewModel **只调** `errorPresenter.present(error)`，不必关心错误类型 → UI 样式 →
/// 文案的三段映射。新增错误码时只在本文件单点扩展。
///
/// 映射规则（与 epics.md Story 2.6 AC + 总体架构 §V1错误码规范 + iOS 架构 §8.3 对齐）：
///
/// - `.business(code, message, _)` → AlertOverlay；文案优先用本地 codeMessage 表（按 V1 错误码字典精挑短句），
///   未命中时退回 server 返回的 `message`。
/// - `.unauthorized` → Toast；文案 "登录已过期，正在重新登录..."（实际重登逻辑归 Epic 5 Story 5.4，
///   本 story 只负责 UI 提示）。
/// - `.network(_)` → RetryView；文案 "网络异常，请检查后重试"。
/// - `.decoding(_)` → AlertOverlay；文案 "数据异常，请稍后重试"。
///
/// 错误码字典（中文短句，全部 hardcode；MVP 阶段不接 i18n）：
/// 见 `localizedMessage(forBusinessCode:fallback:)` 内 switch（V1 §3 全 32 码，含 0=success 占位虽不应触达此 mapper）。
public enum AppErrorMapper {
    /// 把 APIError 映射成 ErrorPresentation（呈现样式 + 文案）。
    /// 入参 error 必须是 APIError；其它 Error 类型走 fallback `.alert("操作失败", "请稍后重试")`。
    public static func presentation(for error: Error) -> ErrorPresentation {
        guard let apiError = error as? APIError else {
            return ErrorPresentation.alert(title: "操作失败", message: "请稍后重试")
        }
        switch apiError {
        case let .business(code, message, _):
            let userMessage = localizedMessage(forBusinessCode: code, fallback: message)
            return ErrorPresentation.alert(title: "提示", message: userMessage)

        case .unauthorized:
            return ErrorPresentation.toast(message: "登录已过期，正在重新登录...")

        case .network:
            return ErrorPresentation.retry(message: "网络异常，请检查后重试")

        case .decoding:
            return ErrorPresentation.alert(title: "提示", message: "数据异常，请稍后重试")
        }
    }

    /// 错误码 → 用户文案。覆盖 V1 §3 全部 32 码。未命中（业务方传未知 code）退回 server 返回的 message。
    /// 表里短句**一律不超过 12 字**，避免 alert 排版换行；超过 12 字的（如"操作过于频繁，请稍后再试"）单独 review。
    public static func localizedMessage(forBusinessCode code: Int, fallback: String) -> String {
        switch code {
        // 1xxx 通用错误（V1 §3）
        case 1001: return "登录已过期，请重新登录"
        case 1002: return "请求参数错误"
        case 1003: return "资源不存在"
        case 1004: return "权限不足"
        case 1005: return "操作过于频繁，请稍后再试"
        case 1006: return "当前状态不支持此操作"
        case 1007: return "数据冲突，请重试"
        case 1008: return "操作重复，请稍后再试"
        case 1009: return "服务繁忙，请稍后重试"

        // 2xxx 账号
        case 2001: return "账号不存在"
        case 2002: return "微信已绑定其他账号"
        case 2003: return "当前账号已绑定微信"

        // 3xxx 步数
        case 3001: return "步数同步异常"
        case 3002: return "步数不足，再走走吧"

        // 4xxx 宝箱
        case 4001: return "宝箱不存在"
        case 4002: return "宝箱尚未解锁"
        case 4003: return "暂时不能开启宝箱"

        // 5xxx 道具 / 装扮 / 合成
        case 5001: return "道具不存在"
        case 5002: return "道具不属于你"
        case 5003: return "道具状态不可用"
        case 5004: return "装备槽位不匹配"
        case 5005: return "合成材料数量错误"
        case 5006: return "合成材料品质不一致"
        case 5007: return "合成目标品质不合法"
        case 5008: return "装扮已装备"

        // 6xxx 房间
        case 6001: return "房间不存在"
        case 6002: return "房间已满"
        case 6003: return "你已在房间中"
        case 6004: return "你不在房间中"
        case 6005: return "房间状态异常"

        // 7xxx 表情 / WS
        case 7001: return "表情不存在"
        case 7002: return "实时连接未就绪"

        // 未知 code：退回 server 原文（如 server message 也是空字符串则给通用文案）
        default:
            return fallback.isEmpty ? "操作失败，请稍后重试" : fallback
        }
    }
}
```

**关键约束**：

- 用 `enum AppErrorMapper { ... }` + `static func` 而非 class / struct：本类是无状态查表器，enum + static 最贴切（与 `AccessibilityID` 风格一致）。
- 第一个公共 API `presentation(for:)` 接收 `Error` 而非 `APIError`：让 ViewModel 在拿到任意 `Error`（如 catch block 不一定 narrow 到 APIError）时也能直接调；非 APIError 走 generic fallback。
- 错误码字典**一站式覆盖 V1 §3 全部 32 码**：让后续业务 stories（Epic 4-36）在新增业务时**不必动 mapper**——除非业务码本身扩了。
- `localizedMessage(forBusinessCode:fallback:)` 标 `public`：测试可直接断言（无需经过 `presentation(for:)` 包装）。
- `default` 分支：`fallback.isEmpty ? "操作失败，请稍后重试" : fallback` —— server 返回空 message 时给兜底；非空时尊重 server 原文（让 server 仍能透传错误描述给 dev / 测试场景）。
- **不**在 mapper 里调 logger：错误日志归 ViewModel 的 catch block（→ Story 2.7 logger 框架接入后再加）。
- **不**做 i18n（Localizable.strings）：MVP 阶段全中文 hardcode；如未来要 i18n，把字典 key 改成 `String(localized:)` 即可，对 caller 透明。
- 顶部仅 `import Foundation`（`APIError` 在 PetApp module 内 public 可用）。

**AC2 — `ErrorPresentation`：呈现样式枚举**

新建 `iphone/PetApp/Shared/ErrorHandling/ErrorPresentation.swift`：

```swift
import Foundation

/// `ErrorPresentation`：错误呈现样式 + 关联文案。
///
/// 三态对应 epics.md Story 2.6 的三个 UI 组件：
/// - `.toast(message:)` → `ToastView`：顶部短暂浮现，2 秒自动消失。
/// - `.alert(title:message:)` → `AlertOverlayView`：全屏阻塞 alert，单 OK 按钮。
/// - `.retry(message:)` → `RetryView`：全屏 placeholder + 重试按钮，由调用方注入 `onRetry` 闭包。
///
/// `Equatable` 用于测试断言；`Identifiable` 用于 SwiftUI 的 `.fullScreenCover(item:)` / `.sheet(item:)`
/// 等需要 stable id 的 modifier（虽然本 story 用 `.overlay(...)` 而非 sheet，仍 conform 让未来切换轻便）。
public enum ErrorPresentation: Equatable, Identifiable {
    case toast(message: String)
    case alert(title: String, message: String)
    case retry(message: String)

    /// 用 String(describing:) 拼 id：保证不同 case 的 id 一定不同；
    /// 同 case 不同 message 的 id 也不同（避免 alert 队列里两个 "数据异常" 被 SwiftUI 当作同一项）。
    public var id: String {
        switch self {
        case let .toast(message):
            return "toast::\(message)"
        case let .alert(title, message):
            return "alert::\(title)::\(message)"
        case let .retry(message):
            return "retry::\(message)"
        }
    }
}
```

**关键约束**：

- 用 `enum` + 关联值：与 `APIError` / `PingResult` 的 value type 风格保持一致；`Equatable` 自动合成。
- `Identifiable` 协议 conform + `id: String`：未来若把 alert 改成 `.sheet(item:)` 弹出（替代 overlay）可零改动迁移。
- **不**带 `onRetry: () -> Void` 闭包字段：闭包不是 `Equatable` 的一等公民（会让 `Equatable` 合成失败）。`RetryView` 的 `onRetry` 闭包由 `ErrorPresenter.present(_:onRetry:)` 入口单独传，存在 presenter 内部的私有字段（见 AC4）。
- 顶部仅 `import Foundation`。

**AC3 — `ErrorPresenter`：呈现态管理器（含队列）**

新建 `iphone/PetApp/Shared/ErrorHandling/ErrorPresenter.swift`：

```swift
import Foundation
import Combine

/// `ErrorPresenter`：错误展示态的中心。
///
/// 职责：
/// - 接收 `present(error:onRetry:)` / `presentToast(_:)` 调用 → 转换成 `ErrorPresentation`
/// - 维护当前展示项 `current`（`@Published`）+ 等待队列 `queue`（FIFO）
/// - 为 toast 启动自动消失定时器（默认 2 秒；通过 `toastDuration` 注入测试快路径）
/// - 暴露 `dismiss()` 让 SwiftUI alert 的 OK 按钮 / RetryView 的 onRetry 闭包推进队列
///
/// 不职责：
/// - **不**渲染 UI（UI 由 SwiftUI ViewModifier `errorPresentationHost` 订阅 `current` 完成）
/// - **不**处理"重试"语义（`onRetry` 闭包透传给 RetryView，由调用方决定重试逻辑）
/// - **不**写日志（→ Story 2.7 logger 接入后再加 sink）
@MainActor
public final class ErrorPresenter: ObservableObject {

    /// 当前正在展示的呈现项；nil = 无。SwiftUI 视图订阅此字段渲染对应组件。
    @Published public private(set) var current: ErrorPresentation?

    /// 队列：current 消失后弹出下一个。
    /// 复合错误场景：caller 短时间内多次 present(...) 时，按 FIFO 排队；不丢弃也不合并。
    private var queue: [ErrorPresentation] = []

    /// retry 呈现态对应的 onRetry 闭包；与 `current` 的 `.retry(...)` 配对存活。
    /// 单独存放避免污染 `ErrorPresentation` 的 Equatable 合成。
    private var pendingOnRetry: (() -> Void)?

    /// toast 自动消失定时任务的句柄；`dismiss()` 时取消（防 toast 被手动 dismiss 后定时器误触）。
    private var toastDismissTask: Task<Void, Never>?

    /// toast 自动消失时长（秒）。生产默认 2.0；测试注入 0.05 加速。
    private let toastDuration: Double

    public init(toastDuration: Double = 2.0) {
        self.toastDuration = toastDuration
    }

    // MARK: - Public API

    /// 主入口：把任意 Error 映射到 ErrorPresentation 并展示。
    /// - Parameters:
    ///   - error: 任意错误；APIError 走 `AppErrorMapper`，其他类型走 generic fallback。
    ///   - onRetry: 仅在 mapper 选定 `.retry(...)` 时使用；其他呈现样式忽略本闭包。
    ///              传 nil 时 RetryView 仍能展示，但点"重试"按钮只 dismiss（无副作用）。
    public func present(_ error: Error, onRetry: (() -> Void)? = nil) {
        let presentation = AppErrorMapper.presentation(for: error)
        enqueue(presentation, onRetry: onRetry)
    }

    /// 直接展示一条 toast（用于成功提示 / 业务无错但需要轻量反馈，如"已同步"）。
    public func presentToast(_ message: String) {
        enqueue(.toast(message: message), onRetry: nil)
    }

    /// 直接展示一条 alert（用于本地校验失败、不来自 server 的业务规则提示）。
    public func presentAlert(title: String, message: String) {
        enqueue(.alert(title: title, message: message), onRetry: nil)
    }

    /// 关闭当前展示并推进队列。供 AlertOverlay OK 按钮、RetryView 重试按钮、Toast 定时器调用。
    /// `triggerOnRetry`：仅 `.retry` 时有意义——true 时调用 caller 注入的 onRetry 闭包后再 dismiss。
    public func dismiss(triggerOnRetry: Bool = false) {
        // 取消可能进行中的 toast 自动消失任务（手动 dismiss 时清理）
        toastDismissTask?.cancel()
        toastDismissTask = nil

        // 仅 .retry 且明确请求时调 onRetry
        if triggerOnRetry, case .retry = current {
            pendingOnRetry?()
        }
        pendingOnRetry = nil
        current = nil

        // 推进队列：弹下一个（如果有）
        if !queue.isEmpty {
            let next = queue.removeFirst()
            present(next, onRetry: nil)
        }
    }

    // MARK: - Private

    /// 入队 / 立即展示。current=nil → 直接 set；current 非 nil → push 到 queue 末尾。
    private func enqueue(_ presentation: ErrorPresentation, onRetry: (() -> Void)?) {
        if current == nil {
            present(presentation, onRetry: onRetry)
        } else {
            queue.append(presentation)
            // onRetry 入队时先丢失：当前实现不支持队列项携带 onRetry。
            // 设计权衡：retry 通常是"操作失败 → 重试"的强语义，并发出现概率低；MVP 不支持队列内 retry。
            // 若未来需要，把 queue 元素类型改为 (ErrorPresentation, (() -> Void)?) 即可。
        }
    }

    /// 真正展示一项（不入队）。toast 启动自动消失定时器；alert / retry 等手动 dismiss。
    private func present(_ presentation: ErrorPresentation, onRetry: (() -> Void)?) {
        current = presentation
        pendingOnRetry = onRetry

        if case .toast = presentation {
            toastDismissTask?.cancel()
            toastDismissTask = Task { [weak self, toastDuration] in
                try? await Task.sleep(nanoseconds: UInt64(toastDuration * 1_000_000_000))
                guard !Task.isCancelled else { return }
                await self?.handleToastTimeout()
            }
        }
    }

    /// toast 定时器触发：仅当当前仍是 toast 时 dismiss（防止用户在 2s 内手动 dismiss 后定时器还触发）。
    private func handleToastTimeout() {
        guard case .toast = current else { return }
        dismiss()
    }
}
```

**关键约束**：

- `@MainActor`：所有公共 API 都在主线程——SwiftUI 订阅 `@Published` 必须主线程。
- `final class`：不可被继承（presenter 是稳定单例式实例，子类化无价值）。
- `@Published var current: ErrorPresentation?` 是**唯一**对外可订阅状态字段；其他都 private。
- `ObservableObject`：让 SwiftUI ViewModifier 通过 `@ObservedObject` 订阅。
- **必须** `import Combine`（`ObservableObject` 来自 Combine，遵守 lesson `2026-04-25-swift-explicit-import-combine.md`）。
- **队列设计权衡**：当前实现入队的项**不**携带 onRetry —— retry 通常并发概率低，MVP 不复杂化。后续若发现并发 retry 频繁出现，把 queue 元素改成 `(ErrorPresentation, (() -> Void)?)` 即可（注释中已说明扩展路径）。
- `toastDismissTask` 用 `Task` + `Task.sleep`：替代 `Timer` / `DispatchQueue.main.asyncAfter`，与 Story 2.5 的 async/await 风格一致；`weak self` + `Task.isCancelled` check 防 leak / 防过期触发。
- `toastDuration: Double = 2.0`：通过 init 参数注入；测试用 0.05 秒加速。
- `dismiss(triggerOnRetry:)`：默认不触发 retry；仅 RetryView 的"重试"按钮显式传 `triggerOnRetry: true`；Alert OK 按钮 / Toast 自动消失走默认（不触发）。

import SwiftUI
import Combine

/// 把 `ErrorPresenter.current` 投影成对应 SwiftUI 组件的 ViewModifier。
///
/// 用法：在 RootView body 最外层 `.errorPresentationHost(presenter: container.errorPresenter)`。
///
/// 渲染策略（按 ErrorPresentation case）：
/// - `.toast(message)` → `ToastView` 顶部浮现，叠在主内容之上（ZStack overlay alignment .top）
/// - `.alert(title, message)` → `AlertOverlayView` 居中弹出，遮罩全屏阻塞
/// - `.retry(message)` → `RetryView` 全屏覆盖（直接替换主内容）
///
/// 动画（hardcoded MVP）：
/// - toast：`.transition(.move(edge: .top).combined(with: .opacity))`
/// - alert / retry：`.transition(.opacity)`
public struct ErrorPresentationHostModifier: ViewModifier {
    @ObservedObject var presenter: ErrorPresenter

    public func body(content: Content) -> some View {
        ZStack {
            // 主内容；retry 出现时被全屏覆盖
            content
                .accessibilityHidden(retryActive)
                .opacity(retryActive ? 0 : 1)

            // Retry 全屏覆盖
            if case let .retry(message) = presenter.current {
                RetryView(message: message, onRetry: { presenter.dismiss(triggerOnRetry: true) })
                    .transition(.opacity)
                    .zIndex(2)
            }

            // Alert 居中弹层
            if case let .alert(title, message) = presenter.current {
                AlertOverlayView(title: title, message: message, onDismiss: { presenter.dismiss() })
                    .transition(.opacity)
                    .zIndex(3)
            }

            // Toast 顶部浮现
            if case let .toast(message) = presenter.current {
                VStack {
                    ToastView(message: message)
                        .padding(.top, 12)
                        .transition(.move(edge: .top).combined(with: .opacity))
                    Spacer()
                }
                .frame(maxWidth: .infinity)
                .allowsHitTesting(false)   // toast 不吃点击：让用户继续操作下方内容
                .zIndex(4)
            }
        }
        .animation(.easeInOut(duration: 0.2), value: presenter.current)
    }

    private var retryActive: Bool {
        if case .retry = presenter.current { return true }
        return false
    }
}

extension View {
    /// 把 ErrorPresenter 挂到 view 上：根据 presenter.current 自动渲染对应错误组件。
    /// 应在 RootView body 最外层调用一次（**不**在每个子页重复挂）。
    public func errorPresentationHost(presenter: ErrorPresenter) -> some View {
        modifier(ErrorPresentationHostModifier(presenter: presenter))
    }
}

## Dev Notes

### 1. ErrorPresenter 不接入 HomeViewModel —— 仅搭框架

**为什么**：epics.md Story 2.6 AC 写的是"提供组件 + ErrorPresenter，让 ViewModel 通过 `errorPresenter.present(error)` 触发 UI"——**不**强制本 story 把任何具体 ViewModel 接入 errorPresenter。HomeViewModel 当前 ping flow 失败时显示 `serverInfo = "offline"`（Story 2.5 已 done），**不**走 ErrorPresenter——这是有意设计：

- 主界面版本号是装饰性元素，错误不应阻断渲染；让 ping 失败弹全屏 retry 会破坏首页可用性。
- HomeViewModel 接入 ErrorPresenter 涉及 init 签名扩展（注入 `errorPresenter`）+ Story 2.5 lesson 的"@StateObject + bind() 模式"复制（详见 `2026-04-26-stateobject-init-vs-bind-injection.md`）；改动面 + 测试覆盖面会爆涨。

**收益 vs 成本**：本 story 不接 HomeViewModel，确保**框架就位 + 0 业务流回归风险**；Epic 4 GuestLogin / Epic 5 LoadHome 接入时，那些 ViewModel **从一开始就**带 errorPresenter init 注入，不用重复 Story 2.5 的 bind() 双 init 体操。

### 2. ErrorPresenter 通过 AppContainer 暴露而非 EnvironmentObject

iOS 架构 §18.2 + ADR-0002 §3.3 锁定的 DI 模式是 `AppContainer` 持有所有共享依赖，让 RootView `@StateObject` 持有 container，再通过 init 注入往下传。本 story 沿用此模式：

```swift
public final class AppContainer: ObservableObject {
    public let apiClient: APIClientProtocol
    public let errorPresenter: ErrorPresenter   // ← 新增
    // ...
}

// RootView 用法
container.errorPresenter   // 拿同一个实例
```

**不**用 `@EnvironmentObject`：MVP 阶段未引入 SwiftUI environment scope，且 EnvironmentObject 测试时需要 `.environmentObject(...)` 包装，复杂度上升；与 Story 2.5 的"AppContainer 工厂方法"模式不一致。

未来 Epic 12 RoomView 接入 sheet-scoped 错误展示时，可以考虑：(a) 复用 root container.errorPresenter 让 alert/retry 在 root 层弹出（**当前默认**）；(b) 给 RoomView 自建 sheet-scoped `RoomErrorPresenter` 实例。届时按业务需求再决策。

### 3. SwiftUI ViewModifier 用 ZStack 而非 overlay —— 避开 lesson `2026-04-25-swiftui-zstack-overlay-bottom-cta.md`

`ErrorPresentationHostModifier` 用 `ZStack { content; alertView; retryView; toastView }` 而非 `content.overlay(...)` 链。理由：

- `overlay` 默认 alignment 是 `.center`，多层 overlay 嵌套时 z-order 不直观；用显式 `zIndex` + ZStack 让层级一目了然（toast=4, alert=3, retry=2）。
- Lesson `2026-04-25-swiftui-zstack-overlay-bottom-cta.md` 的踩坑场景是"装饰元素叠在交互元素之上"——本 story **故意**让 alert / retry 叠在 content 之上（这是错误展示的预期行为），但 alert / retry **本身**自带 OK / 重试按钮处理 dismiss，不存在"装饰元素吃了交互按钮的点击"的问题。
- Toast 加 `.allowsHitTesting(false)`：让 toast 浮在内容之上但**不吃**点击——用户可以在 toast 浮现时继续操作下方内容（语义对：toast 是非阻塞提示）。这正是 lesson 提到的"toast / FAB 显式让出 hit area"的最佳实践。
- Retry 用 `accessibilityHidden(retryActive)` + `.opacity(retryActive ? 0 : 1)` 让 content 在 retry 出现时**完全隐藏**：retry 是全屏覆盖语义，下方 content 不应被 VoiceOver 朗读 / 不应被点击。

### 4. Toast 定时器用 `Task` 而非 `Timer` —— 避开 lesson `2026-04-26-swiftui-task-modifier-reentrancy.md`

`ErrorPresenter` 的 toast 自动消失用 `Task { try? await Task.sleep(nanoseconds:); ... }`：

- `Task.sleep` 比 `Timer.scheduledTimer` 更适合 actor-isolated 上下文（presenter 是 `@MainActor`）。
- `Task.isCancelled` check + `weak self`：cover "presenter 已 deinit" 与 "手动 dismiss 后定时器不应触发" 两个边界。
- `dismiss()` 内 `toastDismissTask?.cancel()`：手动 dismiss 时立刻取消定时器（防过期触发覆盖后续展示项——AC9 case#10 专测此场景）。
- **不**用 `DispatchQueue.main.asyncAfter`：闭包不能取消，会污染队列。

Lesson `swiftui-task-modifier-reentrancy.md` 的"hasFetched flag 防 .task 重启"在本 story **不适用**：本 story 没有 `.task` modifier 调"一次性 side effect"；`.errorPresentationHost(presenter:)` modifier 内部不调任何 async 工作，纯渲染。

### 5. 不写 UI 测试（XCUITest）—— 是对 epics.md AC 的有意收窄

epics.md AC 第 4 条写："**集成测试覆盖**: UITest 触发各类错误（mock APIClient 返回不同 APIError）→ 验证对应 UI 组件出现"。

本 story **不**实装此 UITest，理由：

- HomeViewModel 不接入 ErrorPresenter（详见 Dev Note #1），UITest 没有"业务流触发错误"的入口可以走。LLDB `expr` 测可以验证组件渲染但不是 UITest。
- 给本 story 加一个"测试专用"的 dev 按钮触发各种 APIError（类似 Story 2.8 的 dev keychain 重置按钮）违反"按钮归 Story 2.8"的范围划分。
- Epic 4 GuestLoginViewModel / Epic 5 LoadHomeViewModel 接入 ErrorPresenter 后，UITest 自然有真实业务流入口可以走（如"mock APIClient 返回 1001 → UITest 断言 toast 出现"）；那时再补 UITest 比现在硬造测试入口更合适。

**对 epics.md AC 的偏离**：本 story 的 Dev Notes 显式记录此偏离 + 留 follow-up TODO 给 Epic 4 / 5 补 UITest。如未来 retro 认为这是 gap，单独立 story 补；不在本 story 范围。

### 6. 错误码字典覆盖范围 + 修改门槛

`AppErrorMapper.localizedMessage(forBusinessCode:fallback:)` 内 switch case **一站式覆盖 V1 §3 全部 32 码**（V1接口设计 §3 + 总体架构 §错误码规范）。修改原则：

- **新增业务码**（如 server 端节点 X 加新 code 8001）→ 在本 mapper 加新 case，**同时**把 epics.md / V1接口设计 同步登记到字典；不要让 case 与协议文档漂移。
- **修改既有文案**（如把"步数不足，再走走吧"改成"再多走走"）→ 单独 PR；改动是文案级的，无需 Story 升级。
- **改 case 含义**（如 4002 从"宝箱尚未解锁"变成别的语义）→ 这是 V1 协议变更，先动 V1接口设计.md，再同步 mapper + 各端实装。

mapper 不做"按 code 范围 (1xxx / 2xxx) 匹配"的兜底逻辑：宁可 default 走 fallback message，也不让"未知 1xxx 通用错误"被误归到通用文案——错过实际错误细节会让 dev 排查变难。

### 7. AppContainer 改动小到不需要新单测

本 story 给 `AppContainer` 加 `errorPresenter` 字段是 2 行改动（字段 + init）；既有 `AppContainerTests.swift`（Story 2.5 落地）会自动通过—— `init()` / `init(apiClient:)` 都还能 work，新加字段不影响既有 case。**不**为新字段单独加 case："`container.errorPresenter is ErrorPresenter` 永远 true" 这种空测试无价值。

### 8. ErrorPresentation enum 不带 onRetry 闭包 —— 设计权衡

`ErrorPresentation.retry(message:)` 的 case **不**携带 `onRetry: () -> Void` 字段。理由：

- 闭包不是 `Equatable` 一等公民，会让 `ErrorPresentation: Equatable` 自动合成失败（要手写 `==`，麻烦且容易出错）。
- 闭包不是 `Identifiable.id` 计算的稳定输入。
- `onRetry` 仅在 `.retry` 时有用；alert / toast 都没此概念，硬塞会让其他 case 也带空字段。

**替代方案**：把 `onRetry` 存在 `ErrorPresenter` 的私有字段 `pendingOnRetry: (() -> Void)?`，与 `current` 配对生命周期（dismiss 时清空）。这种"状态拆分"模式让 enum 保持纯数据 / Equatable 干净，闭包归 owner 自己管。

### 9. 关于 Story 2.7 的 logger 接入留位

ErrorPresenter `present(_:onRetry:)` 当前**不**调 logger。Story 2.7 落地 logger 框架后，建议在 mapper 入口加一行：

```swift
// 未来（Story 2.7 落地后）
public func present(_ error: Error, onRetry: (() -> Void)? = nil) {
    Logger.error.log("ErrorPresenter received: \(error)")  // ← 未来
    let presentation = AppErrorMapper.presentation(for: error)
    enqueue(presentation, onRetry: onRetry)
}
```

本 story **不**预埋此调用：避免引入未实现的 `Logger` 符号让 build 红。Story 2.7 完成后追加 PR 即可（mapper 是 ViewModel-friendly 入口，logger 接入是单点修改）。
