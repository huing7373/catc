### r1 P1 关键决策：SnapshotMember.Pet 字段从值类型改为 pointer 类型

**问题**：Story 10.7 落地时 SnapshotMember.Pet 是值类型 SnapshotPet（不是 pointer），原因是 placeholder 阶段所有 member 都下发 `pet ≠ null + petId: ""`（V1 §12.3 行 1978 r14 锁定的 Story 10.7 落地形态），不需要表达 pet-less → null 语义。但 Story 11.7 going-forward 契约要求 LEFT JOIN pets pet-less 时下发 `pet: null`（V1 §12.3 行 1881 钦定），值类型 SnapshotPet 无法在 JSON 序列化时下发 null。

**决策**：把 SnapshotMember.Pet 字段类型从值类型 SnapshotPet 改为 pointer 类型 `*SnapshotPet`，让 nil 序列化为 JSON null。

**兼容性影响**：
- placeholder 阶段（Story 10.7）所有 member 仍下发 `pet ≠ null + petId: ""`，即 wire 上 `"pet": {"petId": "", "currentState": 1}`；改 Pet 为 pointer 后 placeholder 实装层同步为 `m.Pet = &SnapshotPet{...}` pointer 构造，行为不变（Pet 始终非 nil + petId 空字符串）。
- 真实阶段（本 story 引入）pet-less 下发 `"pet": null`、否则下发 `"pet": {"petId": "<id>", "currentState": 1}`，与 V1 §12.3 going-forward 契约严格对齐。
- 客户端（Story 12.3 落地）按 `Optional<MemberPetDTO>` 处理（iOS Codable decodeIfPresent / Go json.Unmarshal *SnapshotPet 自然 nullable），无需改 decoder 即可同时正确处理两个阶段。

**review 阶段必查**：
- snapshot.go 内 `SnapshotPet{`（值类型字面值）在 ws 包内**仅** placeholderSnapshotBuilder 的 `&SnapshotPet{...}` 路径 + realSnapshotBuilder 的 `&SnapshotPet{...}` 路径 + 测试 fixture 内的 `&SnapshotPet{...}` 路径，没有任何"值类型 SnapshotPet{...}"赋给 pointer 字段的编译错误
- 既有 ws_test.go / snapshot_test.go / snapshot_integration_test.go / ws_integration_test.go 内所有 `SnapshotPet{...}`（值类型字面值赋给 SnapshotMember.Pet 字段）已同步改为 `&SnapshotPet{...}`
