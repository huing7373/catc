## 2. 关键洞察与 Epic 23 的"主旋律"

### 2.1 主旋律 1: 契约定稿 story 的多轮 review = 下游 N 条实装 story 的歧义一次性清算

| Story | 表面复杂度 | 锚定度 | review 轮次 |
|---|---|---|---|
| 23-1 契约定稿 | 低（纯文档）| **极低**：它本身在*定义*契约，无上游兜底 | **4 轮** |
| 23-2 migration | 中 | 高（23-1 刚冻结 schema + DB §5.9）| 1 轮过 |
| 23-3 catalog | 中 | **高**（23-1 §8.1 逐字段冻结 + emojis 模板）| 1 轮过 |
| 23-4 inventory | 高（三态矩阵）| **高**（23-1 r3 已把三态矩阵 + 两级全序写进冻结契约）| 1 轮过 |
| 23-5 开箱事务 + dev grant | 高 | 混合（开箱高锚定 / dev grant 低锚定）| **3 轮** |

`23-1 → 23-2/3/4` 的因果链是本 epic 最强证据：codex 在 23-1 死磕的三件事（全序排序 / 不静默丢已拥有数据 / 三态矩阵全枚举），正是 23-4 inventory 能一次过的原因——23-4 实装者拿到的是被对抗性 review 锤过 3 轮的契约。**契约定稿 story 的 review 轮次不是浪费，是把痛苦前置吸收。** 这是 Epic 20 主旋律 3 + Epic 21 主旋律 2 的又一次实证强化。

### 2.2 主旋律 2: 锚定度而非功能复杂度才是难度预测变量（Epic 21 主旋律 2 重演）

23-5 的痛苦不在高锚定的开箱事务（Epic 20 钦定 5a~5k + txCtx 铁律，codex r2/r3 一字未挑），而在低锚定的 **dev grant 激活**——`/dev/grant-cosmetic-batch` 是 Story 20.8 留的 501 stub，23-5 才第一次真实写库，`count` 语义从未被任何上游契约钉死过 → codex r1 P1（静默少发）+ r2 over-correction chain。meta-lesson 与 Epic 21 一致：复杂度的"锚定度"才是预测变量。

### 2.3 主旋律 3: AP1 固化的首次实战回报兑现

Epic 21 retro AP1（把 over-correction 防御固化进 `.claude/commands/epic-loop.md` fix_review override #3）已完成（commit `40b3823`）。23-5 r1→r2 是经典 over-correction：dev-story `ORDER BY RAND() LIMIT count` → codex r1 flag 静默少发 → fix r1「池<count 拒绝 1009」→ codex r2「这拒绝把合法主 demo（common 池 8 但要 grant 10 实例做合成测试）打死」。**关键：它没膨胀**——对比 Epic 20 race chain 5 轮 / testing chain 7 轮 / Epic 21 21-1 4 轮，23-5 因 override #3 强制「禁止表层微调 + 回根因」，r2 一轮锁定根因（`count` 是实例数不是 distinct 配置数；user_cosmetic_items 无 UNIQUE、合成 feature 本就要重复实例），改有放回 pick + 整批事务化 + 守门测试，r3 一次过。验证 Epic 21 Top Takeaway #5：**流程改进必须固化进模板，不能靠主 agent 临场记得**——机制有效。

### 2.4 主旋律 4: 中断恢复 + 主 agent 边界纪律

23-5 dev sub-agent 跑到一半账号 quota 耗尽强制打断，留下编译不过半成品。处理做对了：(1) 主 agent 出 HALT 报告不盲继续，给路径 A/B 选择；(2) 主 agent 全程**不自己 commit / 不改码**（硬约束），只诊断 + 调度；(3) 路径 A 续做 sub-agent 被明确告知「先 git diff 看半成品已对部分，别推翻重做」→ 生产代码全保留，零工作丢失。再次验证 Epic 21 主旋律 4。

### 2.5 固有噪音: gopls 陈旧中间态快照

23-3 / 23-4 / 23-5 每条有码 story，sub-agent 编辑后 harness 都抛 gopls `redeclared`/`itoa` 编译错误，但每次独立 `go vet` 四配置核验均确认是中间态陈旧快照（磁盘最终态干净）。若盲信 gopls 直接 HALT 会误杀 3 条 story。**机制**：每条有码 story 必须主 agent 独立 `go vet` 四配置核验，不盲信 harness 抛的编译错误。

> 当本 story 含时序 / 并发 / race / 事务 / 幂等 / 缓存透传 任一面时，**必须在 dev 开工前**显式划分下面三层测试责任，不要把责任混到一层后靠 review 轮次反复调断言强弱（Epic 20.9 testing chain 跑 7 轮的失败模式根因）：

## 5. 关键 Takeaways（Top 5）

1. **契约定稿 story 的多轮 review 不是浪费，是把下游 N 条实装 story 的歧义一次性清算**——23-1 死磕 4 轮换 23-2/3/4 三连一次过。
2. **锚定度是难度预测变量**：23-5 痛苦在低锚定的 dev grant 激活，不在高锚定的开箱事务。
3. **AP1 固化回报兑现**：override #3 让 23-5 over-correction 链子第 2 跳即断，未膨胀（vs Epic 20/21 的 5/7/4 轮）。
4. **中断恢复 + 主 agent 边界纪律**：quota 打断 → HALT 不盲继续 → 路径 A 续做不重做 → 零工作丢失；主 agent 全程不自己 commit/改码。
5. **gopls 陈旧中间态快照是固有噪音**：每条有码 story 需独立 `go vet` 四配置核验，不盲信 harness 抛的编译错误。
