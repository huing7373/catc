**§12.1.2 完整 URL 模板**（**保留**现有 `GET /ws/rooms/{roomId}?token=xxx` 示例，**前置**模板说明）：

```markdown
连接 URL 模板：

\`\`\`text
{ws_scheme}://{host}/ws/rooms/{roomId}?token={url_encoded_token}
\`\`\`

- `ws_scheme`：生产环境 **`wss`**（TLS）；dev / test 环境 **`ws`**（明文，仅本地）
- `host`：与 HTTP 接口同 host（不分离 WS 域名，简化 Info.plist / CORS 配置）
- `roomId`：路径参数，必须是用户当前所在房间的 ID（client 从 `GET /me.user.currentRoomId` 或 `GET /home.room.currentRoomId` 拿；当前节点 4 阶段也可以用 client 已知的 roomId，如刚 `POST /rooms/{roomId}/join` 成功后立即建连）
- `token`：query 参数，URL-encoded（必须 url-encode，避免 token 中 `+` `/` `=` 等字符影响 query 解析）；token 来自 §2.3 Bearer token 同源（`POST /auth/guest-login` 返回的 `token`）

服务端连接建立后需要校验：

- token 合法（解码 + 签名校验 + 未过期）
- 用户确实在该房间中（查 `room_members` 表，节点 4 阶段 Story 10.3 实装查询路径）
```

**§12.1.4 握手失败 — close code 表**（**新增** H4 块）：

```markdown
#### close code 表

握手 / 连接期间任一校验失败，服务端必须主动 close 连接，并使用以下 close code：

| close code | 触发条件 | 服务端是否带 reason | 客户端推荐处理 |
|---|---|---|---|
| 4001 | token 缺失 / 解码失败 / 签名不匹配 / token 已过期 | 是（reason = `"invalid token"` / `"token expired"`） | 触发"无效 token 静默重新登录"流程（参考 iOS Story 5.4）：清 keychain 旧 token → guest-login 拿新 token → 重连；**不**自动无限重试 |
| 4003 | token 合法但 user 不在该 roomId 的 `room_members` 表中 | 是（reason = `"user not in room"`） | **不**自动重连（业务级拒绝，重试无意义）；UX 提示用户"未加入该房间"，回退到主界面 |
| 4002 | roomId 路径参数格式错（非数字 / 缺失） | 是（reason = `"invalid roomId"`） | **不**自动重连；视为客户端实装 bug，记 log 后回退 |
| 4004 | room 不存在（`rooms` 表无该 ID 或已 archived） | 是（reason = `"room not found"`） | **不**自动重连；UX 提示"房间不存在"，回退到主界面 |
| 1000 | 服务端 / 客户端正常关闭 | 否 | 客户端主动 close 不重连；服务端主动 close（如 graceful shutdown）客户端可选重连 |
| 1001 | going away（服务端重启 / 客户端切后台） | 否 | 服务端重启 → 客户端**应**自动重连（指数退避，参考 iOS Story 12.5）；客户端切后台 → 客户端自身决定 |
| 1006 | 异常断开（无 close frame，TCP 中断 / 网络抖动） | 否（无 reason，因为没收到 close frame） | 客户端**应**自动重连（指数退避） |
| 1011 | 服务端内部错误（panic / 不可恢复异常） | 是（reason 应包含简短错误提示但**不**泄漏 stack trace） | 客户端**应**自动重连（指数退避，但限制最大重试次数避免雪崩） |

**关键约束**：

- 4001 / 4003 / 4002 / 4004 是**业务级**拒绝（4xxx 段是应用自定义；WebSocket RFC 6455 规定 4000-4999 为应用保留段），重试无意义，客户端**不**应自动重连；客户端应展示明确 UX 提示并回退
- 1000 / 1001 / 1006 / 1011 是**协议 / 网络**级断开（1xxx 段是 RFC 标准段），客户端**应**自动重连（除 1000 主动关闭）
- 4001 触发时 server **不**写 log error（这是常态，token 过期是正常业务），写 log info；4003 / 4004 写 log warn（疑似客户端实装 bug 或数据不一致）；1011 写 log error（必排查）
```

**§12.1.5 服务端校验顺序**（**新增** H4 块）：

```markdown
#### 服务端校验顺序

握手期间服务端必须按以下顺序校验，任一步失败立即 close 并使用对应 close code：

1. **解析 query**：缺 `token` 参数 → close 4001（reason = `"missing token"`）
2. **路径参数校验**：`roomId` 非数字 / 缺失 → close 4002
3. **token 校验**（不查 DB，仅本地校验签名 + 过期，复用 Story 4.4 token util）：失败 → close 4001
4. **room 存在性校验**（查 `rooms` WHERE `id = ?`）：失败 → close 4004
5. **用户房间归属校验**（查 `room_members` WHERE `user_id = ? AND room_id = ?`）：失败 → close 4003
6. **内部错误**（panic / DB 异常等）：close 1011

校验通过后：

- 创建 Session 对象（详见 Go 项目结构 §9.1）
- 注册到 SessionManager
- 启动读 / 写 goroutine
- 推送 `room.snapshot`（§12.3 schema）
- 在 Redis presence 记录在线（详见 Story 10.6）

**注意**：第 5 步的 DB 查询是热点路径（每次连接 / 重连都查一次），Story 10.3 / 10.6 实装时**应**将 `room_members` 在线态缓存到 Redis presence（避免每次连接都查 MySQL）；但**协议层面**第 5 步仍是契约一部分（client 不可见，但 server 必须保证语义一致 —— Redis 缓存与 MySQL 一致性由 Story 10.6 + Story 11.4 加入房间事务保证）。
```

```markdown
#### 通用消息信封（client → server）

所有客户端 → 服务端 WS 消息共享同一信封结构：

| 字段 | 类型 | 必填 | 范围/约束 | 说明 |
|---|---|---|---|---|
| `type` | string | 必填 | 1 ≤ length ≤ 64 字符；只允许 `[a-z0-9.]` | 消息类型，全小写点分（如 `"ping"` / `"emoji.send"`）；服务端按 `type` 路由到对应 handler |
| `requestId` | string | 选填 | 0 ≤ length ≤ 64 字符；客户端定义格式（推荐 `<short-prefix>_<timestamp_ms>` 或 UUID） | 客户端生成的请求 ID，用于追踪 / 日志关联；服务端**响应**当前请求时**应**带回同一 `requestId`（如 `pong` 回带 `ping` 的 `requestId`）；**广播消息**（如 `emoji.received`）的 `requestId` 由服务端置空字符串 `""` |
| `payload` | object | 必填 | 任意 JSON 对象；具体 schema 由 `type` 决定 | 业务负载；空 payload 也必须**显式**写 `"payload": {}`，**不**省略 key |

JSON 通用骨架：

\`\`\`json
{
  "type": "<message_type>",
  "requestId": "<optional_client_id>",
  "payload": { "...": "..." }
}
\`\`\`

**关键约束**：

- 信封**不**带 `ts` 字段（client → server 方向不需要 client 时间戳；如某业务消息需要 client 时间戳，由该业务消息 payload 自带，如 `emoji.send.payload.clientTs`，详见 Story 17.1）
- `requestId` **不**进 server 持久化日志的关键字段，仅用于本次连接 session 内的 request-response 配对（避免无意义的 cross-session 追踪）
- `type` 字段值大小写敏感：服务端 strict match；客户端**必须**按文档锚定的字面量发送（如 `"ping"` 不是 `"PING"` 或 `"Ping"`）
- 消息 frame 必须是 WebSocket text frame（opcode 0x1）+ UTF-8 JSON；**不**使用 binary frame（opcode 0x2）
- 单条消息 frame **大小上限 16 KB**（默认值；配置 `ws.max_message_size_bytes`，**prod 必须使用默认值不可覆盖**，dev/test 可覆盖）；超限服务端 close 1009（message too big）+ 不重连（客户端实装 bug）
```

1. **契约先行（contract first）**：本 story 是节点 4 第一个 epic 的开局，**禁止**先写 Go 代码再回头补文档。理由：iOS Epic 12 必须基于稳定 WS 协议骨架并行开工；server 业务 Epic 11 / 14 / 17 的 §X.1 story 也基于本骨架扩展；先实装后补文档会导致 client / server / 后续 epic 三方对不上 → 节点 4 demo 时三向返工。
2. **协议骨架 vs 业务消息分离**：本 story **只**锚定协议骨架（连接 / 信封 / 心跳 / 房间快照占位 / 错误），**不**锚定业务消息。理由：协议骨架由 Epic 10 一次性冻结，业务消息按 epic 顺序逐步冻结（11.1 / 14.1 / 17.1）；混锚会引入跨 epic 耦合 + 让本 story 范围爆炸。
3. **既有示例保留 vs 修订**：§12 已有 6 个 JSON 示例（`emoji.send` / `room.snapshot` / `member.joined` / `member.left` / `emoji.received` / `pong` / `error`），其中 3 个（`emoji.send` / `member.joined` / `member.left` / `emoji.received`）是业务消息草稿，2 个（`pong` / `error`）缺 `requestId` 字段违反信封规范。处理策略统一：**示例保留 git diff 最小**+**文档侧 blockquote 标注修订 / 延后锚定**。理由：示例 JSON 字面量被历史 lessons / 项目讨论引用过，改字面量会让历史引用语义漂移；契约权威是字段表（H4 H5 表格），不是示例 JSON。
4. **close code 选用决策**：4001（token 失败）/ 4003（用户不在房间）/ 4002（roomId 格式错）/ 4004（room 不存在）/ 1011（服务端内部错）。设计依据：
   - WebSocket RFC 6455 规定 4000-4999 为应用保留段，本 story 用 4001-4004
   - 1011（server error）是 RFC 标准段，避免应用重复占用
   - 1006（异常断开）/ 1000（正常关闭）/ 1001（going away）也是 RFC 标准段，本 story 列入文档但**不**由 server 主动选用（server 主动 close 用 4xxx + 1011；1006 / 1000 / 1001 是网络层 / 客户端触发）
   - 对应 epics.md AC："含 token 校验失败 / 用户不在该房间时的 close code"已覆盖（4001 / 4003 是核心；4002 / 4004 / 1011 是补全）
5. **配置 key 默认值属契约一部分**：`ws.heartbeat_timeout_sec`（60s，已在 Go 项目结构 §12.2 锚定）+ `ws.max_message_size_bytes`（16 KB，新增）。这些默认值在节点 4 冻结声明里**显式**列入"prod 必须使用默认值不可覆盖"白名单。理由（参照 Story 7.1 GAP K 同模式）：不同 prod 实例若使用不同心跳 / message 大小阈值会导致客户端实装层面"为什么 A 实例没踢人 B 实例踢人"的不一致 → 重新引入跨端契约漂移。dev / test 可覆盖（仅用于单测 / fixture，不影响 prod 体验）。
6. **client / server 信封不对称**：client→server 三字段（`type` / `requestId` / `payload`），server→client 四字段（多 `ts`）。理由：client 不需要带本机 ts（业务时间戳由 payload 自带，如 `emoji.send.payload.clientTs`）；server 需要 ts 用于 client 端日志 / UI 排序（避免多条 broadcast 乱序）。这条不对称必须在文档侧**显式**标注，避免 server 实装时画蛇添足让 client 也带 ts。
7. **未识别 type 安全忽略策略**：节点 4 阶段 client / server 都**不**对未识别 `type` close 连接，仅 log warn。理由：节点 4 后续 Epic 11 / 14 / 17 / 18 会陆续上线新 type；如果 client 灰度滞后（用户没更新 app）导致 server 推了新 type，client 不识别就 close 连接 = 强制用户升级；同理 server 灰度滞后也不能让客户端 crash。安全忽略 + log warn 是兼容性最佳实践（参考 Slack / Discord WS 协议设计）。

### 架构对齐

**领域模型层**（`docs/宠物互动App_总体架构设计.md` §3.10 / §6.10 / §7.9 / §8.2）：

- WebSocket 是房间互动专用通道，不承载普通业务请求（普通业务走 REST）
