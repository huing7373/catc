## Story

As a 服务端开发,
I want **正式实装 POST /api/v1/rooms 创建房间接口**：service 层 `RoomService.CreateRoom` 用 `txMgr.WithTx` 包"预检 users.current_room_id → 插入 rooms → 插入 room_members（撞 UNIQUE(user_id) 兜底回 6003）→ 更新 users.current_room_id"四步事务 + repo 层补 `RoomRepo.Create(ctx, *Room)` 写方法 + `RoomMemberRepo.Create(ctx, *RoomMember)` 写方法 + `UserRepo.UpdateCurrentRoomID(ctx, userID, *uint64)` 写方法 + handler 层 `RoomHandler.CreateRoom` 处理空请求体 + 业务码三层映射 + bootstrap 路由组 `authedGroup.POST("/rooms", roomHandler.CreateRoom)` 挂载 + ≥4 case 单测覆盖（happy / 6003 预检 / 6003 兜底 / DB 异常）+ dockertest 集成测试覆盖（用户首次创建 → 验证 rooms / room_members / users.current_room_id 三表行 + 同 user 再次调用 → 6003）,
so that **下游 Story 11.4（POST /rooms/{roomId}/join）/ Story 11.5（POST /rooms/{roomId}/leave）/ Story 11.6（GET /rooms/current + GET /rooms/{roomId}）/ Story 11.7（房间快照真实实装）/ Story 11.8（member.joined / member.left WS 广播）/ Story 11.9（Layer 2 集成测试 - 房间生命周期全流程）/ Story 11.10（GET /home 扩展 room.currentRoomId 真实数据）/ Epic 12 房间页面 SwiftUI 集成**全部能基于一个**已实装、已测试、已挂载到 router 的 POST /rooms 接口**展开 —— Story 11.4 join 路径需要先有 room 存在，Story 11.5 leave 路径需要先有 room 可加入再退出，Story 11.6 GET /rooms/current 需要 users.current_room_id 已被本 story 写入，Story 11.7 RoomSnapshotBuilder 需要从本 story 创建的 rooms / room_members 行查 roster，Story 11.8 member.joined 广播需要先有房间，Story 11.9 集成测试场景"创建 → join → leave → close" 第一步就是本 story，Epic 12 客户端 RoomViewModel 第一个 use case `CreateRoomUseCase` 调的就是本接口。本 story **不**实装 11.4 ~ 11.10 中的任何东西，仅做"创建房间"这一个动作 + 11.4 ~ 11.10 全部依赖的 server 端起点。

**AC2 — `RoomMemberRepo` 扩展 `Create` 方法 + 双路径 UNIQUE 哨兵**

  ```go
  // ErrRoomMembersUserIDDuplicate: RoomMemberRepo.Create 时 UNIQUE(user_id) `uk_user_id`
  // 冲突。语义："用户已在某个房间中"（V1接口设计.md §10.1 / §10.4 / 数据库设计.md §7.1）。

  // 但分两个独立哨兵让 service 层日志能区分哪个约束被打破，便于审计 / debug）。
  // service 层同样翻译为 6003 ErrUserAlreadyInRoom（Story 11.3 / 11.4 共同消费）。

  ```

- `roomMemberRepo.Create` impl（紧接既有 `ListMembers` 之后）：

  ```go
  // Create 插入一行 room_members。GORM 在成功后回填 m.ID (AUTO_INCREMENT)。
  //
  // **ER_DUP_ENTRY 1062 双路径翻译**：room_members 表有两个唯一约束（0008 schema）：
  //   - uk_user_id (user_id)         → ErrRoomMembersUserIDDuplicate
  //   - uk_room_user (room_id, user_id) → ErrRoomMembersRoomUserDuplicate
  //
  // 不解析 mysql.MySQLError.Message 字符串（"Duplicate entry ... for key
  // 'uk_user_id'" / "'uk_room_user'"）—— 字符串国际化 / 版本不可靠。
  // 用 Message contains "uk_user_id" / "uk_room_user" substring 是稳定 contract
  // （MySQL 错误消息的 key 名 part 在所有版本 + 语言下都是英文 ASCII，与
  // ER_DUP_ENTRY 1062 错误号配套；与 ws_integration_test.go::startMySQL...
  // helper 同模式 —— 用 substring 而非 exact match）。
  //
  // **fallback**（review r? 钦定 —— 如未来有第三个 UNIQUE 约束加入）：1062 但
  // 字符串不含两个 key 名 → raw error 透传给 service（service 包成 1009）。
  // 节点 4 阶段两个 UNIQUE 约束已穷举，本 fallback 路径理论不触发。
  func (r *roomMemberRepo) Create(ctx context.Context, m *RoomMember) error {
      db := tx.FromContext(ctx, r.db)
      err := db.WithContext(ctx).Create(m).Error
      if err != nil {
          var mysqlErr *mysql.MySQLError
          if stderrors.As(err, &mysqlErr) && mysqlErr.Number == mysqlErrCodeDupEntry {
              msg := mysqlErr.Message
              if strings.Contains(msg, "uk_user_id") {
                  return ErrRoomMembersUserIDDuplicate
              }
              if strings.Contains(msg, "uk_room_user") {
                  return ErrRoomMembersRoomUserDuplicate
              }
              // 极罕见：1062 但既不含 uk_user_id 也不含 uk_room_user → raw 透传
          }
          return err
      }
      return nil
  }
  ```

- `userRepo.UpdateCurrentRoomID` impl：

  ```go
  // UpdateCurrentRoomID 更新 users.current_room_id 字段（Story 11.3 引入）。
  //
  // roomID 可为 nil（语义：set to NULL，Story 11.5 退出房间路径用）；非 nil
  // 时设为指定 room id（Story 11.3 创建房间 / Story 11.4 加入房间路径用）。
  //
  // **关键 GORM 陷阱**：用 Updates(map[string]interface{}) 显式传 nil 而非
  // Update("current_room_id", v) —— 后者在 v 为 nil 时被 GORM 视为 zero value
  // 而跳过更新（不会执行 UPDATE SQL）；前者用 map 路径强制 SQL 执行 SET col=NULL。
  // 与 users.UpdateNickname 用 Update("nickname", v) 路径不同：nickname 是 string
  // 永远非 nil，不存在 nil-skip 陷阱；current_room_id 是 *uint64，必须用 map 路径。
  //
  // GORM autoUpdateTime tag 让 updated_at 自动写当前时间（与数据库 §3.2 ON UPDATE
  // CURRENT_TIMESTAMP(3) 语义一致）。
  func (r *userRepo) UpdateCurrentRoomID(ctx context.Context, userID uint64, roomID *uint64) error {
      db := tx.FromContext(ctx, r.db)
      return db.WithContext(ctx).
          Model(&User{}).
          Where("id = ?", userID).
          Updates(map[string]interface{}{
              "current_room_id": roomID,
          }).Error
  }
  ```

- `CreateRoom` impl（核心事务编排；详见 AC5 ~ AC8）：

  ```go
  func (s *roomServiceImpl) CreateRoom(ctx context.Context, in CreateRoomInput) (*CreateRoomOutput, error) {
      // (1) 预检 user.current_room_id（事务外，普通连接池查询）
      user, err := s.userRepo.FindByID(ctx, in.UserID)
      if err != nil {
          // ErrUserNotFound 理论不应发生（auth middleware 已校验 token 对应有效 user）；
          // 任何 err 都包成 1009
          return nil, apperror.Wrap(err, apperror.ErrServiceBusy, apperror.DefaultMessages[apperror.ErrServiceBusy])
      }
      if user.CurrentRoomID != nil {
          // 用户已在房间中（V1 §10.1 步骤 1 钦定的预检路径）→ 6003，不开事务
          return nil, apperror.New(apperror.ErrUserAlreadyInRoom, apperror.DefaultMessages[apperror.ErrUserAlreadyInRoom])
      }

      // (2) 开事务（数据库设计 §8.6 / 时序图 §11.1 钦定）
      var roomID uint64
      err = s.txMgr.WithTx(ctx, func(txCtx context.Context) error {
          // (2a) 插入 rooms（status=1, max_members=4；GORM 回填 r.ID）
          r := &mysql.Room{
              CreatorUserID: in.UserID,
              Status:        roomStatusActive,
              MaxMembers:    roomMaxMembersDef,
          }
          if err := s.roomRepo.Create(txCtx, r); err != nil {
              return err
          }
          roomID = r.ID

          // (2b) 插入 room_members；撞 UNIQUE(user_id) → service 层捕获后翻译 6003
          m := &mysql.RoomMember{
              RoomID: roomID,
              UserID: in.UserID,
          }
          if err := s.roomMemberRepo.Create(txCtx, m); err != nil {
              return err
          }

          // (2c) 更新 users.current_room_id = room_id
          if err := s.userRepo.UpdateCurrentRoomID(txCtx, in.UserID, &roomID); err != nil {
              return err
          }

          return nil
      })

      if err != nil {
          // (3) UNIQUE(user_id) / UNIQUE(room_id, user_id) 兜底 → 6003
          if stderrors.Is(err, mysql.ErrRoomMembersUserIDDuplicate) ||
              stderrors.Is(err, mysql.ErrRoomMembersRoomUserDuplicate) {
              return nil, apperror.New(apperror.ErrUserAlreadyInRoom, apperror.DefaultMessages[apperror.ErrUserAlreadyInRoom])
          }
          // (3') 其他失败 → 1009（事务已 rollback）
          return nil, apperror.Wrap(err, apperror.ErrServiceBusy, apperror.DefaultMessages[apperror.ErrServiceBusy])
      }

      // (4) 事务 commit 成功 → 返回
      return &CreateRoomOutput{
          RoomID:        roomID,
          CreatorUserID: in.UserID,
          MaxMembers:    roomMaxMembersDef,
          MemberCount:   1,
          Status:        roomStatusActive,
      }, nil
  }
  ```

**AC5 — 事务边界严格 4 步（含 6003 双路径）**

`CreateRoom` 的事务边界严格按 V1 §10.1 + 时序图 §11.1 + 数据库设计 §8.6 钦定：

- **步骤 1（事务外）**：`s.userRepo.FindByID(ctx, in.UserID)` → 检查 `user.CurrentRoomID != nil` → **预检 6003 路径**（不开事务）
- **步骤 2（事务内，4 步）**：
  - **2a**：`s.roomRepo.Create(txCtx, r)` 插入 rooms，GORM 回填 `r.ID`
  - **2b**：`s.roomMemberRepo.Create(txCtx, m)` 插入 room_members，撞 UNIQUE → 兜底 6003 路径
  - **2c**：`s.userRepo.UpdateCurrentRoomID(txCtx, userID, &roomID)` 更新 users.current_room_id
- **步骤 3（事务后）**：commit / rollback；commit 成功 → 返回 CreateRoomOutput；rollback 后按 err 类型分流 6003 / 1009

**6003 双路径必须等价**：预检路径（步骤 1）+ DB 兜底路径（步骤 2b）都返 `apperror.New(ErrUserAlreadyInRoom, ...)`，handler 端响应 envelope 完全一致 —— client 不应区分（V1 §10.1 行 1146 钦定 "client **不**应区分这两种场景"）。

**ctx 用法（ADR-0007 §2.4）**：fn 内全部 repo 调用必须用 `txCtx` 而非外层 `ctx` —— 用错 ctx 会绕过 tx 走 db pool 新连接，该调用脱离事务，业务语义错乱。本 story 实装严格遵守。

### 与 Story 4.6 / 7.3 业务事务模式的对照

本 story 实装路径直接套用 Epic 4 / 7 已建立的"业务事务 + 单元测试 + 集成测试 + handler 端到端"四件套模式，与下表对应：

| Epic 11.3 (本 story) | Epic 4.6 (auth.firstTimeLogin) | Epic 7.3 (steps.PostSync) | 共同点 |
|---|---|---|---|
| `roomServiceImpl.CreateRoom` | `authServiceImpl.firstTimeLogin` | `stepServiceImpl.PostSync` | 全部用 `txMgr.WithTx(ctx, fn)` 包多 repo 写 |
| 4 步事务（rooms → room_members → users update） | 6 步事务（users → users.UpdateNickname → bindings → pets → step_account → user_chests） | 2 步事务（step_sync_log → step_account UPDATE balance + version） | 全部按设计文档 §8.x 钦定顺序 |
| 6003 双路径（预检 + DB UNIQUE 兜底）| ER_DUP_ENTRY 双路径（uk_guest_uid + uk_auth_type_identifier 都退到 reuseLogin） | 乐观锁版本失败包 1009 | 全部用 `errors.Is + apperror.Wrap` |
| `tx.FromContext + WithContext(ctx)` repo 路径 | 同 | 同 | ADR-0007 §2.3 / §2.4 |
| 单测 ≥4 case + dockertest 集成测试 | 同 | 同 | 业务事务必须双层覆盖 |

**反模式（已避免）**：
- ❌ handler 直接调多 repo 拼事务（违反 §5.1 "service 是事务边界归属"）
- ❌ service 用 `*gorm.DB` 直接调 `db.Transaction(...)`（绕过 tx.Manager interface，handler / service 单测 mock 困难）
- ❌ fn 内 repo 用外层 `ctx` 而非 `txCtx`（绕过 tx 走 db pool 新连接）
- ❌ 6003 预检 + 兜底两条路径返不同 message（client 难以一致处理）
- ❌ `userRepo.UpdateCurrentRoomID` 用 `Update("current_room_id", v)` 而非 `Updates(map[...])`（GORM nil-skip 陷阱让 SET col=NULL 不生效）
