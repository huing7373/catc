## 2. 主题综合（深度 story + 6 篇 lesson 分析）

### 主题 1 — 契约先冻结，bug 在散文里抓掉（✅ 大胜，团队 + Project Lead 一致认同）

26-1 是**零代码**纯文档契约定稿 story，3 轮 codex review 抓出 3 个真实契约 bug：

1. **§8.3 step 4 的 5001/5002 所有权映射歧义**（r1）：原措辞给实装"二选一"自由度，会让 epics.md §26.3 钦定的单测 case「实例不属于当前用户 → 5002」永不可达。修复：强制 step 4 先按 id 查（不带 user_id 过滤），存在但属他人恒 5002，仅"完全无 row"才 5001，实装无自由度。
2. **§8.4 unequip step 5-6 并发竞态**（r2）：SELECT-then-DELETE 下两个并发 unequip 同 petId+slot 都过 step 5，loser DELETE 0 行仍 commit → 空槽误返成功而非冻结的 5004。修复：step 5 `SELECT ... FOR UPDATE` 串行化 + step 6 `DELETE RowsAffected==0 → 回滚+5004` 冗余兜底。
3. **§8.3 step 7 missing-no-row 实例 equip 未定义**（r2）：§8.2 声明 missing-no-row 道具仍可见可 status=1，但 step 7 裸 `SELECT slot FROM cosmetic_items` 无 missing-row 分支 → 落进未定义/内部错误。修复：复用既有 5003，不扩张节点 9 冻结错误码集合。

**实证收益**：26-3 equip 事务、26-4 unequip 事务照加固后的冻结契约实装，codex r1 **一次过**，零返工。26-4 直接把 26-1 r2 钦定的 FOR UPDATE + RowsAffected 兜底实装进去。**"契约冻结 → 散文里找 bug → 实装干净"范式跑通的硬实证，建议固化继续用。**

### 主题 2 — 26-5 over-correction chain，根源是编排 agent 注入未经实证的错误根因（🚨 核心系统性教训）

26-5 集成测试"100 并发同槽 equip"断言跑满 4 轮 fix-review 的 assertion 强弱 ping-pong：

| 轮 | codex 指导 | fix-review 动作 | 评价 |
|---|---|---|---|
| r1 | `==1` 错（Equip 是 swap 可串行多成功）→ 放松 | 放松成 `>=1` + 加终态一致性矩阵 | 方向本来正确 |
| r2 | 放松丢了 uk_pet_slot 回滚覆盖 → 恢复强断言 | **编排 agent 用未经实证的"空槽+屏障⟹swap 结构不可达⟹确定性恰 1"理论模型，强制 fix-review 恢复 `==1` + 加事后被证伪的守门注释** | ❌ 编排 agent 注入错误根因 |
| r3 | **codex 实跑 `-count=3` 复现 91/100 swap 成功**，硬实证证伪 r2 理论 | correct-course：承认 `==1` 是伪不变量；终态一致性矩阵才是真不变量；epics.md §26.5 AC「99 失败」措辞被 26-1 冻结契约 swap 语义 supersede（登记 wontfix-with-rationale）；删被证伪守门注释 | ✅ 实证打破循环 |
| r4 | 放松后 uk_pet_slot 安全网缺失 + helper INNER JOIN 误绿（2×P2，**非** chain 反转） | 新增独立确定性测试补回 uk_pet_slot 覆盖（13 case）+ helper INNER→LEFT JOIN | ✅ 干净收尾 |

**系统性根因**：epic-loop 的 override #3 专门防 over-correction chain，结果**编排 agent 自身成为 chain 的源头**——注入了一个自信但错误的理论根因，作为强制指令压给忠实执行的 fix-review sub-agent → 引入 r2 回归。**打破循环的是 codex 真去跑了测试（实证），不是又一轮推理。** 4 篇 26-5 lesson 已把完整链条 + r2 编排 agent 错误指令作为反例归档。

### 主题 3 — dev-story 自报 BUILD SUCCESS 与 IDE diagnostics 反复打架（⚠️ 持续假警源）

26-3 / 26-4 / 26-6 三次：dev-story sub-agent 报 BUILD SUCCESS，IDE diagnostics 同时报大量 `undefined`/`redeclared` 编译错。每次编排 agent 亲自跑 `bash scripts/build.sh --test` 均证明 **build 实际干净**——diagnostics 是 sub-agent 改文件中途的 stale 快照。epic-loop"不信 sub-agent 自报、独立验证"硬约束每次兜住（且 sub-agent 均诚实），但每次多烧一轮验证，是持续的假警成本。

## 7. 关键 takeaway

1. **契约冻结范式有效，固化继续用**：纯文档 26-1 在写代码前抓掉 3 个真实契约 bug，下游实装 r1 一次过、零返工。
2. **编排 agent 自己会成为 over-correction chain 的源头**：未经实证的理论根因当强制指令下发 = 注入回归。可一次实证证伪的命题必须先取证再下指令（AP1）。
3. **实证 > 推理**：26-5 chain 是 codex 真去跑测试（91/100）打破的，不是第 N 轮推理。
4. **规划文档会被冻结契约 supersede**：epics.md §26.5 AC 措辞已与 26-1 冻结契约 swap 语义脱节，需主动校正避免 demo 验收再踩（AP3）。
5. **独立验证硬约束兜住了 sub-agent 的 stale diagnostics 假警 3 次**——约束有效，但可让 sub-agent 附 build 原文降低复核成本（AP2）。
