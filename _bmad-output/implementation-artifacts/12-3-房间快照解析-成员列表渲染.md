## 核心契约依据（必读）

> 本 story 严格遵循 V1 §12.3 三条核心契约，所有实装 / 测试不允许偏离。

### 契约 1：roster 全量 + LEFT JOIN 不变量

`payload.members[]` 是 server 端"仍在房间"的**全部** `room_members` 行（**不**做"WS 此刻是否连接"层过滤），LEFT JOIN `pets` 后下发：pet-less 账号 `pet: null`、否则 `pet ≠ null + petId: <真实值>`。`memberCount === members[].length` 不变量。Source: V1 §12.3 字段表 `payload.members` 行 + "不变量（snapshot 内部一致性）" 段。

### 契约 2：room.snapshot 是 enrich/correct，**不是** wipe-out（client merge contract）

snapshot 是握手成功后**必发**的第一条 authoritative 消息（V1 §12.1.3），但其权威性是 enrich/correct：

- **roster 集合层**：以 snapshot 的 `userId` 集合为权威集合 —— snapshot 没有的 `userId` → client 应**移除**；snapshot 有但 client 之前没有的 `userId` → client 应**新增** entry（用 snapshot 的字段值填充）。
- **字段级**：
  - **非空值**（`nickname: "Alice"` / `pet.petId: "2002"` / `pet.currentState: 2`）：用 snapshot 的值**覆盖** client 已有值（这是真实 authoritative 数据）。
  - **空字符串**（`""`）：**保留** client 已有值（"server 不知道这个值"的 placeholder 信号，**不是**"请清空"指令）。
  - **`null` 值**（`pet = null`）：**直接覆盖** client 已有值（authoritative pet-less 信号，渲染降级为"无宠物"占位）。
  - **未出现的字段**（如 placeholder 阶段不下发 `avatarUrl` / `pet.equips`）：**保留** client 已有值。

Source: V1 §12.3 "Client merge contract" 段。

### 契约 3：room.id 校验 + stale 丢弃

`payload.room.id` 与 `lastObservedRoomId` 不匹配时**丢弃**该 snapshot（防 leave / room A→B 切换瞬间前一个 stream 上排队的 late snapshot for room A 错误 repopulate 新 room 的 members）。Source: Story 12.1 fix-review r3 lesson `2026-05-09-stale-snapshot-discard-by-room-id-12-1-r3.md`。

## Story

As an iPhone 用户,
I want 进入房间后能立即看到房间内当前成员列表（含 nickname 与头像 / 猫位 placeholder），
So that 我知道这个房间里有谁，并为后续 12.4（成员动态进出）/ 12.5（重连后 roster 对齐）/ 12.7（创建 / 加入 / 退出闭环）提供视觉真实态.

### AC1 — 端到端 wire 联通（room.snapshot wire JSON → vm.members 真实更新）

**给定**：

- Story 12.1 已就位 `RealRoomViewModel.applySnapshot(_:)` 方法（含 room.id stale-discard guard + nickname empty-string preserve + isHost 严格 false 占位）
- Story 12.2 已就位 `WSMessageCodec.decode(_:)` + `WebSocketClientImpl` + `AppContainer.webSocketClient`

**预期行为**：

1. **WS 消息流入路径**：`WebSocketClientImpl` 收到 server text frame `{"type":"room.snapshot","requestId":"","payload":{"room":{...},"members":[...]},"ts":...}` → `WSMessageCodec.decode` → 返回 `WSMessage.roomSnapshot(RoomSnapshotPayload)` → 经 `messages` AsyncStream yield → `RealRoomViewModel.startConsumingMessages()` 内 for-await 循环消费 → `MainActor.run { handle(message:) }` → `applySnapshot(payload)`；
2. **applySnapshot 算法**（已由 Story 12.1 落地，本 story **不**改算法主干）：
   - 校验 `payload.room.id == lastObservedRoomId`（不匹配丢弃 + log debug）
   - 按 snapshot `userId` 集合裁剪 + 增量构造 `[RoomMember]`
   - 字段级 merge：`nickname` 非空覆盖 / 空字符串保留 existing / 首次空字符串降级 "成员" placeholder
   - `level / status` 保留 existing 或默认（`8 / "在玩耍"` 占位）
   - `isHost` 严格 `false`（snapshot 不带 host 字段时不做位置启发式 —— Story 12.1 fix-review r4）
   - **覆盖** `self.members = newMembers`（@Published 触发 SwiftUI 刷新）
3. **`memberPetStates` 节点 4 阶段策略**（Story 12.1 已锁定）：snapshot `pet.currentState` 固定 `1` → **不写入** `memberPetStates`，保持空 map（待 Epic 14 真实驱动）；
4. **UI 自动刷新**：`RoomScaffoldView` 通过 `@ObservedObject state: RoomViewModel` 订阅 `members` @Published 字段；`applySnapshot` 写入新 `members` 后 SwiftUI 自动 invalidate body → `membersList` 重新 render → `roomMember_<index>` rows 数量 / 文字与 snapshot 一致。

**红线**（不允许偏离）：

- **不**改 `WSMessageCodec.decode` 算法（Story 12.2 已稳定）；如发现 codec bug 走单独 hotfix
- **不**改 `RealRoomViewModel.applySnapshot` 算法主干（仅文档 / 注释精修允许；保留 Story 12.1 fix-review r1-r6 lessons）
- **不**让 client 做"member 顺序敏感"的 host 推断（fix-review r4 lesson `snapshot-host-must-not-infer-from-position-12-1-r4.md`）
- **不**写 raw 字符串 nickname `"成员"` 散播到多处；保留 Story 12.1 已锚定的 `applySnapshot` 内 placeholder 行为
- **不**让 `applySnapshot` 写入 `memberPetStates` —— 节点 4 阶段必须保持空 map
- **不**做 wipe-out 全量替换 + 丢失 client 已加载真实 nickname / avatarUrl / petId（必须遵守 client merge contract）
- **不**实装 `member.joined` / `member.left` 增量 mutate（**Story 12.4 范围**）

**对应 Tasks**: Task 1.1（端到端联通验证 e2e fixture / mock server）
