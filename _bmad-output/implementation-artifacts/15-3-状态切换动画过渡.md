## Story

As an iPhone 用户,
I want 猫的状态切换是平滑动画过渡（淡入 + 轻微缩放，200-300ms）而不是硬切换,
So that 房间内成员状态变化 + 主界面自己猫状态变化都有视觉连贯感，UI 不闪烁、不堆叠队列、不残影.

## 故事定位（Epic 15 第 3 条 story；节点 5 iOS 端"状态切换平滑过渡"收口；上承 15.1 渲染基础 + 15.2 WS 实时更新；下启 15.4 self state-sync + 15.5 跨房间状态恢复）

- **Epic 15 进度**：15.1（房间页内多成员猫位渲染 + snapshot pet.currentState 解析，**done**）→ 15.2（pet.state.changed WS 消息处理，**done**）→ **15.3（本 story，状态切换动画过渡）** → 15.4（自己状态变化时上报 state-sync）→ 15.5（跨房间状态恢复）
- **本 story 是 Epic 15 第三条 story**：承接 15.1 / 15.2 已落地的"任意路径触发 `memberPetStates[userId]` mutate"数据通路，把 PetSpriteView 在 RoomScaffoldView 成员行的视觉切换从 "默认瞬时"提升为 epics.md §15.3 行 2418 钦定的 "平滑过渡（200-300ms + `.opacity.combined(with: .scale)`）"
- **关键发现：PetSpriteView 本体已具备 `.id + .transition(.opacity) + .animation(.easeInOut(duration: 0.2), value: state)` 三件套**（`iphone/PetApp/Features/Home/Views/PetSpriteView.swift:60-62`，Story 8.4 review round 2 落地，详见 `docs/lessons/2026-05-04-swiftui-content-swap-needs-id-and-transition.md`）—— 该三件套在 HomeView catStage 主界面猫渲染路径**已验证生效**；本 story 要做的是：

- **本 story 是 Epic 15 后续 stories 的视觉契约前置**：

### AC1 — `PetSpriteView` 过渡升级：`.transition(.opacity)` → `.opacity.combined(with: .scale)`；duration 锁定 200-300ms 区间；`.id + .transition + .animation(value:)` 三件套结构不变

**给定**：

- `iphone/PetApp/Features/Home/Views/PetSpriteView.swift:60-62` 既有三件套（Story 8.4 review round 2 落地）：

  ```swift
  .id(state)
  .transition(.opacity)
  .animation(.easeInOut(duration: 0.2), value: state)
  ```

- epics.md §15.3 行 2418 钦定："`.transition(.opacity.combined(with: .scale))` 或类似 SwiftUI animation 平滑过渡（200-300ms）"
- `docs/lessons/2026-05-04-swiftui-content-swap-needs-id-and-transition.md` 钦定 "三件套缺一即静默失败"
- PetSpriteView 同时用于 (i) `HomeView` catStage 主界面 180pt 大尺寸渲染 + (ii) `RoomScaffoldView` 成员行 40pt 缩略渲染（Story 15.1 落地 + 15-1-r1 lesson 修复）—— 两个 caller 共用同一过渡契约

**预期行为**：

1. **`.transition(.opacity)` 升级为 `.transition(.opacity.combined(with: .scale))`**（epics.md §15.3 行 2418 钦定）：

   ```swift
   .id(state)
   .transition(.opacity.combined(with: .scale))
   .animation(.easeInOut(duration: 0.25), value: state)
   ```

   - `.opacity.combined(with: .scale)` = `AnyTransition` 组合：插入新 view 时同时做 opacity（0→1）+ scale（默认 .scale = 从 .zero 到 .identity）；移除旧 view 反之
   - duration 选 **0.25**（250ms，处于 epics §15.3 行 2418 钦定 200-300ms 区间中点；保留与 Story 8.4 spec 200ms 的视觉相似度，同时给 scale 动效留出更长可感知时间；epics.md §8.4 行 1539 钦定 "淡入淡出 200ms" 是 8.4 原始 spec，本 story §15.3 在该 spec 上加 scale 维度，duration 上调至 250ms 是合理范围内增量）
2. **三件套结构不变**（`.id(state)` + `.transition(_:)` + `.animation(_:value:)`）—— `.id(state)` 是 view replacement 触发点，移除会让 `.transition` 失效（`2026-05-04-swiftui-content-swap-needs-id-and-transition.md` 钦定）
3. **a11y identifier / label / `.accessibilityElement(children: .ignore)` 不动**（既有 `.accessibilityIdentifier(currentIdentifier)` + `.accessibilityLabel(...)` 模式不动；Story 8.4 + 15.1 review fix 路径保留）
4. **doc comment 同步更新**：把 `iphone/PetApp/Features/Home/Views/PetSpriteView.swift:48-55` 既有"设计要点（review round 2 P2 fix）"段落更新：
   - 把 `.transition(.opacity)` 改为 `.transition(.opacity.combined(with: .scale))` 并加一行说明 Story 15.3 把过渡从纯 opacity 升级为 opacity+scale 组合
   - 把 `.animation(.easeInOut(duration: 0.2)` 改为 `0.25` 并加一行说明 Story 15.3 把 duration 上调到 250ms（仍在 §15.3 钦定 200-300ms 区间）

**红线**：

- **不**删 `.id(state)` —— 是 view replacement 触发点；移除则 `.transition()` 静默失效（lesson 2026-05-04 钦定）
- **不**用 `withAnimation { mutate }` 替代 `.animation(_:value:)` —— `withAnimation` 包裹 ViewModel mutate 只能动画化 modifier 数值变化，**不能**让 view body switch / `.id()` view replacement 走 transition；既有三件套是 SwiftUI 文档推荐 idiom（lesson 2026-05-04 钦定）
- **不**改 `spriteImage(...)` helper / `spriteSymbolName(for:)` / `spriteTintColor(for:)` / `currentIdentifier` / `accessibilityLabel` —— 这些是 Story 8.4 / 37.5 / 15.1 落地的稳定 helper，无需触碰
- **不**让 RoomScaffoldView 成员行的 PetSpriteView 走与 HomeView 不同的过渡契约（spec 一致；改 PetSpriteView 本体即同时覆盖两个 caller）
- **不**让 duration 超出 §15.3 行 2418 钦定 200-300ms 区间（即 0.20 - 0.30 闭区间）
- **不**用 `.matchedGeometryEffect(_:in:)` —— 同一 PetSpriteView 跨状态不是 view migration 场景，`.transition` 已足够
- **不**用 `Animation.spring` 替代 `.easeInOut` —— spring 与既有 200ms easeInOut 视觉差异显著，破坏 8.4 主界面已建立的视觉基线（review round 2 已锁定 easeInOut）

**对应 Tasks**: Task 1.1, 1.2

### AC3 — "连续多次状态切换后覆前不堆叠队列" + "同一 state 重复 set 不触发动画"两条 SwiftUI implicit 行为通过单测/UI 测试断言锁定

**给定**：

- epics.md §15.3 行 2419 钦定 "连续多次状态切换（如 rest → walk → rest 在 1 秒内）→ 后一个动画覆盖前一个，不堆叠队列"
- epics.md §15.3 行 2423 钦定 "同一 state 重复 set（state 没变）→ 不触发动画"
- SwiftUI implicit animation 行为（与 lesson `2026-05-04-swiftui-content-swap-needs-id-and-transition.md` + `2026-04-30-swiftui-onchange-equatable-and-stale-task-cancel.md` 一致）：
  - `.animation(_:value:)` 用 Equatable diff 判定是否触发；同值 mutate 走 SwiftUI 内部 short-circuit 不触发 transition（**天然保证 AC3 第二条**）
  - `.id(state)` 在连续 mutate 时只看"当前帧 id 与上一帧 id 是否不同"，连续 mutate 不会堆叠队列 —— 后一个 mutate 直接覆盖前一个 view replacement 状态（**天然保证 AC3 第一条**）
- Story 15.1 / 15.2 已落地 `memberPetStates: [String: HomePetState]` `@Published` dict —— mutate 同一 key 会触发 SwiftUI dependency invalidate；该路径由 `RealRoomViewModel.applyPetStateChanged` (Story 15.2) + `applySnapshot` (Story 15.1) 调用

**预期行为**：

1. **单测层验证**（`PetSpriteViewTests.swift` 新建或 `RealRoomViewModelTests.swift` 扩展；选前者，本 story 是 view-level 视觉契约测试，单独文件更聚焦）：
   - **case#A**：连续 mutate `vm.memberPetStates["u_alice"] = .walk` → `.rest` → `.walk`（间隔 < 100ms）→ 断言每次 mutate 后 `vm.memberPetStates["u_alice"]` 反映最新值 + view body 重新评估次数 ≤ 3 + 无 task 队列堆叠（通过 `await Task.yield()` × N 后断言无 pending side-effect）
   - **case#B**：mutate `vm.memberPetStates["u_alice"] = .walk` 后再次同值 mutate `vm.memberPetStates["u_alice"] = .walk` → 断言两次 mutate 间 ViewModel state 不变 + SwiftUI Equatable diff short-circuit（断言 `@Published memberPetStates` 在第二次同值 mutate 后**不**重发 publish —— 用 `combine` sink + count 验证）
   - **case#C**：PetSpriteView struct 直接构造 `PetSpriteView(state: .rest)` → 改为 `PetSpriteView(state: .walk)` → 断言两个 instance 的 `state` 字段 / `currentIdentifier` / `accessibilityLabel` 视觉契约字段正确（**不**直接断言 SwiftUI animation —— animation 是 render-tree 行为，view-level 单测断不到；交给 AC5 ios-simulator MCP 录屏视觉验证）
2. **UITest 层验证**（`RoomUITests.swift` 扩展 1 case）：
   - **case#D**：UITEST_ROOM_THREE_MEMBERS 路径启动 → 录屏 `record_video` → 通过测试 helper / launch flag 触发 vm.memberPetStates mutate（连续 .rest → .walk → .rest）→ stop_recording → 人工 / AI（ui_view 抓帧）观察录屏：(i) 每次状态切换有可见 fade + 微缩放；(ii) 连续切换无残影 / 无堆叠；(iii) 同一 state 重复 set 视觉无变化
   - 注：UITest 录屏验证不强求自动化断言（SwiftUI render-tree 视觉断言超出 XCUITest 能力）—— 录屏文件作为 dev 验收 artifact，纳入 commit + PR description（与 lesson `2026-05-12-swiftui-frame-clipped-does-not-scale-15-1-r1.md` 钦定 MCP 实跑模板一致）

**红线**：

- **不**在单测层试图断言 SwiftUI implicit animation 是否真的触发 / 视觉是否真的 fade —— SwiftUI render-tree 行为不在 ViewInspector / XCTest 可达范围（lesson 2026-05-04 钦定 "验证方法：模拟器实跑触发 state 变化，**眼睛**看 fade 是否真发生"）
- **不**用 ViewInspector 库（ADR-0002 §3.1 钦定 XCTest only / 零外部依赖）
- **不**为"动画不堆叠"实装自定义 Combine pipeline / `DispatchQueue.debounce` —— SwiftUI implicit animation 已天然保证（Equatable diff + `.id()` view replacement 是覆盖语义）；本 story 的任务是通过测试**锁定**这个性质，**不**是**实装**这个性质
- **不**对"同一 state 重复 set 不触发动画"做 `@State private var lastEmitted` 防抖 —— SwiftUI Equatable diff 已 short-circuit；额外防抖会引入双层防抖 bug（lesson `2026-04-30-swiftui-onchange-equatable-and-stale-task-cancel.md` 钦定 "Equatable diff 已是 SwiftUI 内置防抖"）

**对应 Tasks**: Task 3.1, 3.2, 3.3, 3.4

### Latest tech information

- **SwiftUI `.transition(.opacity.combined(with: .scale))` 语义**：`.opacity` 与 `.scale` 是 `AnyTransition` 的两个内置过渡；`.combined(with:)` 把多个 transition 串联，view insert 时同时执行（opacity 0→1 + scale 0→1）—— `.scale` 默认从 `.zero` 起步过于剧烈，可考虑 `.scale(scale: 0.9, anchor: .center)` 微缩放；本 story 用默认 `.scale` 先实跑，若视觉过激再调整（fallback 预案：`.transition(.opacity.combined(with: .scale(scale: 0.85)))`）
- **SwiftUI implicit animation Equatable diff 行为**：`.animation(_:value:)` 通过 `value` 的 `==` 判断是否触发；`HomePetState: Equatable` + `MotionState: Equatable` 已在 Story 8.4 / 15.1 落地 —— 本 story 直接复用
- **Swift Concurrency 不变**：本 story 不动 `@MainActor` / `ObservableObject` / `@Published` 模式
- **AnyTransition.combined(with:) 性能**：SwiftUI 内置实现，无 perf 风险 —— PetSpriteView 单 view + 单 transition，无 view tree 深度 / 数量爆炸
