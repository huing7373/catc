# Story 24.5: 穿戴入口按钮位置预留（锁定既有 equipButton 契约为节点 9 接入点）

## Story

As an iOS 开发,
I want 仓库页"穿戴入口"按钮的位置 / 大小 / 间距 / a11y 锚 / enable 规则被**钉成机器可校验的守护契约**（按钮本体 Story 37.9 已交付，本 story 只锁契约不重排版），
so that 节点 9 Story 27.1 激活真实 EquipUseCase 时**只换 `onEquipTap` 内部实装**，无需改 `WardrobeScaffoldView` 布局 / a11y / 按钮结构（零返工）.

## 范围红线（必读 —— epics.md 字面 spec 与已交付 UI 形态的 reconciliation）

> **本 story 是「在 Story 37.9 + 24.1/24.2/24.3 已交付的真实仓库页上，把既存 `equipButton` 的位置 / a11y / enable 规则 / placeholder 行为契约钉成守护测试」的零视觉重排 story；不是按 epics.md 字面在 instance 列表每行右侧新建一个 disabled "穿戴（节点 9）" 按钮、不改既有 `equipButton` 视觉与交互、不改 `onEquipTap` 实装、不发任何 API。**

### epics.md §Story 24.5 字面 spec vs 当前已交付形态的偏离（必须 reconcile，不可照抄 epics）

epics.md §Story 24.5（line 3417-3435）字面要求：在 InventoryView 的 **instance 列表每行右侧** 显示一个**预留 / disabled** 的"穿戴"按钮，节点 8 阶段「disabled + 灰色 + 文案 "穿戴（节点 9）"」，节点 9 Story 27.1 实装时变 enabled + "穿戴" + 触发 EquipUseCase；`instance.status=equipped` 显示"已装备"标签替代按钮。

但该 spec 写于 **ADR-0009 推翻 Sheet 主入口之前**，假设的是「Sheet 弹 `InventoryView`，flat instance 列表，每行一个穿戴按钮」形态。与 Story 24.1/24.3 完全同源的偏离根因。**当前已交付的真实形态**（Story 37.9 `WardrobeScaffoldView` + 24.1 `RealWardrobeViewModel` + 24.2 `LoadInventoryUseCase` + 24.3 品质筛选，全 done 且 762 测试不可回归）是：

- **没有** `InventoryView`、**没有** instance flat 列表、**没有** per-instance 行按钮。
- `WardrobeScaffoldView` 是 4 区块：`topCard` / `previewCard` / `categoryTabs`+`rarityFilterBar` / `grid`（`WardrobeScaffoldView.swift` line 36-44）。
- **"穿戴入口按钮"已存在且已交付**：`previewCard`（line 124-185）内的 `equipButton`（line 187-210）—— 单个 `PrimaryButton(fullWidth:true)`，作用于"当前预览的 active item"（不是 per-instance 行）。它已经：
  - 有稳定 a11y 锚 `AccessibilityID.Wardrobe.equipButton`（= `"wardrobeEquipButton"`，line 209；`AccessibilityIDTests.swift` line 78 已断言）。
  - 有 enable 规则 `isEnabled = active?.owned ?? false`（line 196 —— 未拥有道具 disabled）。
  - 有 label 状态机 `equippedNow ? "✓ 已装备 (点击卸下)" : "装备"`（line 194）+ variant 切换（line 195）。
  - 已 wire 到 `state.onEquipTap(item:)`（line 205）；Story 37.9 round1 P1 fix 后 **Mock 与 Real 子类 `onEquipTap` 都是「本地 toggle equipped」的可用 placeholder**（非 no-op；`MockWardrobeViewModel.swift` line 51-60，`RealWardrobeViewModel.swift` line 198-207），即按钮在节点 8 production 路径下**视觉已工作**（点了立即切 label / grid 对勾），仅未调 server。
- gridCell（line 342-374）已有 `isEquippedNow` 右上对勾 badge（line 359-367）—— epics.md 字面「`instance.status=equipped` 显示"已装备"标签替代按钮」的等价已交付物（已装备态由 grid 对勾 + 预览区按钮文案 "✓ 已装备 (点击卸下)" 双重表达，37.9 钦定）。

**结论：epics.md §24.5「预留一个 disabled 占位按钮位置」的工程意图（= 节点 9 接入点已就位、不返工布局），在 Story 37.9 落地 `equipButton` 时已结构性满足。** 本 story **不**应按字面新建 per-instance disabled 按钮（会与已 done 的 37.9 `previewCard` 单按钮架构 + 24.1/24.2/24.3 渲染契约冲突，且与 ADR-0009「Sheet→Tab」同源越界）。

### reconciliation 决策（本 story 钦定，dev 不得自行另选）

1. **「预留按钮位置」⇒ 锁定既有 `previewCard.equipButton` 契约为节点 9 接入点，不新建、不重排**。epics.md「每行 instance 右侧穿戴按钮」与现存「`previewCard` 单 active-item 按钮」语义重叠（都是"对某 owned 道具发起穿戴"的入口），且后者是 37.9 钦定架构。新建 per-row 按钮 = 重复同语义控件 + 重写 37.9 `gridCell`/`previewCard` 视觉契约（越界 + 与 30.x sprite 演进冲突）。
2. **本 story 真正交付物 = 守护契约测试**：把"节点 9 Story 27.1 不需返工布局"这件事，从口头注释**钉成机器可校验的测试** —— 验证 `equipButton` 的 a11y 锚 / enable 规则 / label 状态机 / placeholder 行为契约**全部就位且稳定**。这是 epics.md「按钮位置 / 大小 / 间距按 UX 设计稿固定，节点 9 不需要重排版」(line 3431) 意图的、与已交付架构兼容的等价实现（用守护测试钉死契约 > 口头注释）。
3. **节点 8 vs 节点 9 行为差异 reconcile**：epics.md 字面要求节点 8 阶段按钮 `disabled + "穿戴（节点 9）"`。但 37.9 已交付的是 **enabled + "装备"/"✓ 已装备 (点击卸下)" + 本地 placeholder 可用**（owned 道具可点、点了视觉立即反馈、未拥有道具自动 disabled）。**取「37.9 既有 enabled placeholder」语义，不退回 disabled**。理由：(a) 37.9 round1 codex review P1 已钉死「Real 子类 override 必须 mutate state 让 production 可用，禁止 no-op/disabled 占位」（lesson `2026-04-30-real-viewmodel-override-placeholder-must-mutate-state.md`，severity 1，已复犯一次）—— 退回 disabled 按钮 = 直接违反该 P1 lesson 的预防规则；(b) Story 27.1 的实质改动是「`onEquipTap` 内部从『本地 toggle』换成『调 EquipUseCase → 写 appState → sink 派生』」，**与按钮 enabled/disabled 无关**（按钮一直 enabled，换的是点击后干什么）；(c) Story 27.1 §Given 写「Story 24.5 已预留按钮位置（disabled + 文案"穿戴（节点 9）"）」是 epics.md 字面假设的延续，本 story 在 Dev Notes「epics.md 偏离登记」+「给 Story 27.1 的接入说明」明确 reconcile：27.1 的真实接入点是「替换 `onEquipTap` 实装」而非「把 disabled 按钮变 enabled」。
4. **不改 `onEquipTap` 任何实装**（Mock/Real 既有本地 toggle placeholder 是 37.9 round1 P1 钦定，本 story 零 edit；节点 9 真实 EquipUseCase 是 Story 27.1/27.2 范围）。本 story 是**纯守护契约 + 文档 reconcile** story，**不**产生新 production 行为代码。

> **一句话**：本 story = 不写一行 production 视觉/行为代码，**只新增守护测试**把 37.9 已交付的 `equipButton`「位置/a11y/enable规则/label状态机/placeholder可用」契约钉死 + 在 Dev Notes 完成 epics.md 字面（per-row disabled 按钮）→ 已交付架构（previewCard 单按钮 enabled placeholder）的 reconcile 登记，供 Story 27.1 直接接入与 review 不误判「漏实现/没退回 disabled」。
