- **epics.md §Story 11.5 钦定**（行 1903-1927）：
  - **Given** Story 11.4 已能加入房间
  - **When** 调用 `POST /rooms/{roomId}/leave`
  - **Then** service 在事务中执行：
    1. 查 `users.current_room_id`，非该 roomId → 6004 用户不在房间中
    2. 删除 `room_members WHERE room_id=? AND user_id=?`
    3. 更新 `users.current_room_id = NULL`
    4. 查 `room_members WHERE room_id=?` 剩余数，==0 → 更新 `rooms.status = 2 closed`
    5. 返回 `{roomId, left: true}`
  - **And** 单元测试覆盖（≥5 case，mocked repo）：
    - happy: 用户在房间且非最后一人 → 删除成员，房间仍 active
    - happy: 用户是房间最后一人 → 删除后房间 closed
    - edge: 用户不在该房间 → 6004
    - edge: 房间已 closed（理论不会，因为用户必然不在 closed 房间）→ 6004
    - edge: 删除 room_members 后查剩余出错（mock）→ 事务回滚，user 仍在房间
  - **And** 集成测试覆盖（dockertest）：
    - A + B 在房间 → A leave → DB room_members 剩 B 一行 + rooms.status 仍 = 1
    - B leave → DB room_members 0 行 + rooms.status = 2

- **V1 §10.5 服务端逻辑钦定**（V1接口设计.md 行 1506-1607）—— 注：与 epics.md 钦定的 4 步事务**不完全等价**，V1 §10.5 把"开启事务 / 提交事务"显式列为独立步骤，且**步骤 2（SELECT rooms FOR UPDATE）就是事务内的第一动作**，让"rooms 行锁定 + remaining-count + status update"三件事用同一把行锁覆盖（与 §10.4 join 跨事务串行化），这是 r9 P1#2 钦定的关键 race fix（与 §8.6 / §8.7 钦定一致）。本 story 实装路径**严格按 V1 §10.5 服务端逻辑 + 数据库设计 §8.7**：

  1. 查 `users.current_room_id`，**与请求的 `roomId` 不一致**（含 `current_room_id` 为 null）→ 立即返回 **6004 用户不在房间中**（不开事务）
  2. 开事务，对 `rooms` 行加排他锁：`SELECT ... FROM rooms WHERE id = ? FOR UPDATE`（**与并发 join 串行化**，确保后续步骤 4 的 remaining-count + 步骤 5 的 status update 与并发 join 不交错）；**找不到 rooms 行**：理论上不会（步骤 1 已通过意味着 caller 在该房间，rooms 行必存在），但 race 兜底：回滚 + 返回 **1009 服务繁忙**（数据不一致，按 DB 异常处理；**不**对外暴露 6001，与 V1 §10.5 行 1597 钦定一致）
  3. 删除 `room_members WHERE room_id = ? AND user_id = ?`（删除当前 user 的成员行）；**检查 `RowsAffected`**：若 `== 0`（同一用户两次并发 leave 都通过步骤 1，赢家已先删该行，输家此处 0 行受影响）→ 回滚 + 返回 **6004 用户不在房间中**（与步骤 1 同语义统一）；该兜底是关键并发护栏，否则输家会继续走完步骤 4 ~ 8 产生重复广播 + 重复 close 4007
  4. 更新 `users.current_room_id = NULL`（**首次启用 UpdateCurrentRoomID 的 nil 入参路径** —— 11.3 review r3 P1 已做 GORM `*uint64` nil-skip 防御 + 测试用 `Updates(map[string]interface{}{"current_room_id": nil})` 路径；本 story 是首次端到端走 nil 路径的事务）
  5. 查 `room_members WHERE room_id = ?` 剩余数量；若 `== 0`（最后一人离开）→ 更新 `rooms.status = 2 closed`
  6. 提交事务
  7. **关闭 leaver 自己的 WS Session**（V1 §10.5 步骤 7 钦定）——**由 Story 11.8 实装**；本 story 在 commit 后 return 之前留 `// TODO: Story 11.8 — close 4007 + unregister leaver Session（V1 §10.5 步骤 7）` 注释占位
  8. **事务成功提交后**触发 WS 广播 `member.left`（V1 §10.5 步骤 8 钦定）——**由 Story 11.8 实装**；本 story 在 commit 后 return 之前留 `// TODO: Story 11.8 — broadcast member.left（V1 §10.5 步骤 8）` 注释占位
  9. 返回 `data.{roomId, left: true}`（**这是 leave 成功的 authoritative 信号**：client 收到 HTTP 200 + `data.left: true` 即应立即清本地房间状态并退出 RoomView，**不**等待 close 4007；4007 是 server 端 fire-and-forget WS cleanup，到达不可保证 —— V1 §10.5 行 1546 + §12.1 4007 行 + 时序图 §12.2 r10 锁定）

  **关键约束**（V1 §10.5 行 1601-1608 钦定）：
  - 6004 触发条件包含三种场景（client UX 一致处理）：(a) `users.current_room_id == NULL`（步骤 1 预检，用户当前不在任何房间）+ (b) `users.current_room_id != path roomId`（步骤 1 预检，用户在其他房间）+ (c) 同一用户并发两次 leave 同一房间，赢家事务内已删除 `room_members` 行，输家步骤 3 `DELETE` 0 行受影响（事务内兜底回滚）—— 都返回 6004，client 收到 6004 就清本地房间状态、不重试
  - 步骤 3 的 `RowsAffected == 0` 兜底是**必须**项（不是优化）：缺失则两次并发 leave 中输家会继续走完步骤 4 ~ 8 —— 即使步骤 4 把已 NULL 的 `current_room_id` 再次写 NULL（idempotent）、步骤 5 房间剩余数已经被赢家算过、步骤 7 会试图关闭已不属于该房间的 leaver Session、步骤 8 广播会在房间内产生**重复** `member.left` 事件 —— 全部是错误副作用
  - 步骤 2 的 `SELECT rooms FOR UPDATE` 与步骤 3 的 `RowsAffected == 0` 兜底**职责正交，不可互替**：步骤 2 锁解决"跨事务 leave-vs-join 状态 drift"（详见 V1 §10.5 行 1560 r9 race 说明）；步骤 3 兜底解决"同一 user 并发两次 leave 输家走完后续步骤"。两者都是**必须**项，且不冲突 —— 步骤 2 锁的是 `rooms` 行（与 join 互斥），步骤 3 删的是 `room_members` 行（与 leave 自身重入互斥），锁对象不同
  - "最后一人离开 → 房间 closed"是 server 端**单向**状态变化：closed 房间无法被重新激活（节点 4 阶段无"重启房间"接口）；这条规则确保 `rooms.status` 严格单调（1 → 2，无回退路径），简化 server 状态机
  - `data.left` 固定 `true`：与 §10.4 `data.joined` 设计对称，避免引入"left: false" 模糊语义
  - 错误码触发顺序：步骤 1 → 6004（预检）；步骤 2 找不到 rooms → 1009（DB 异常）；步骤 3 `RowsAffected == 0` → 6004（兜底）；任何 DB 异常 → 1009。**不**对外暴露 6001 / 6002 / 6003 / 6005（V1 §10.5 行 1599 钦定）

  ```go
  // DeleteByRoomAndUser 按 (room_id, user_id) 双键定位删除单行 room_members（Story 11.5
  // 引入；用于 V1 §10.5 步骤 3 leave 事务删除 leaver 自己的成员行）。
  //
  // **必须在事务内调用**（与同事务 SELECT rooms ... FOR UPDATE 步骤一起执行）；事务外
  // 调用违反 V1 §10.5 步骤 2-3 钦定的串行化路径 —— FOR UPDATE 锁立即释放，
  // 跨事务 leave-vs-join race 重新出现。
  //
  // 返回 RowsAffected 让 service 层做 6004 兜底分流：
  //   - == 1：删除成功（happy path）
  //   - == 0：同一 user 并发两次 leave 输家场景（赢家已先删该行）→ service 层翻译为 6004
  //   - >  1：理论不可能（room_members 有 UNIQUE(room_id, user_id)，最多 1 行匹配）；
  //          若发生属严重数据脏 —— **service 层兜底视为成功路径**（rowsAffected != 0 即 commit
  //          继续走步骤 4 ~ 6；多删的副作用由 schema UNIQUE 约束兜底，不会发生）
  //
  // 用 GORM Where(...).Delete(&RoomMember{}) 路径生成 DELETE FROM room_members WHERE
  // room_id = ? AND user_id = ?；走 idx_room_id 索引 + uk_room_user 唯一约束高效定位。
  // 不支持软删除（room_members 无 deleted_at 字段）。
  //
  // query 失败 → 返 (0, raw error) 透传给 service（service 包成 1009）。
  func (r *roomMemberRepo) DeleteByRoomAndUser(ctx context.Context, roomID, userID uint64) (int64, error) {
      db := tx.FromContext(ctx, r.db)
      result := db.WithContext(ctx).
          Where("room_id = ? AND user_id = ?", roomID, userID).
          Delete(&RoomMember{})
      if result.Error != nil {
          return 0, result.Error
      }
      return result.RowsAffected, nil
  }
  ```

  - **不**新增哨兵 error（与 11.4 `CountByRoomID` 同模式；rowsAffected == 0 路径不是异常路径，是合法业务场景，由 service 层判定）

  ```go
  //
  //	步骤 1（事务外）：FindByID + 检查 user.CurrentRoomID == &input.RoomID → 否则预检 6004
  //	步骤 2（事务内 5 步）：
  //	  2a. roomRepo.FindByIDForUpdate（SELECT ... FOR UPDATE）→ 找不到 → 1009
  //	  2b. roomMemberRepo.DeleteByRoomAndUser → rowsAffected == 0 → 回滚 + 6004 兜底
  //	  2c. userRepo.UpdateCurrentRoomID(userID, nil)（set NULL）
  //	  2d. roomMemberRepo.CountByRoomID → remaining
  //	  2e. if remaining == 0 → roomRepo.UpdateStatus(roomID, 2 closed)
  //	步骤 3（事务后）：commit / rollback；commit 成功 → 返回 LeaveRoomOutput；
  //	  rollback 后按 err 类型分流 6004 / 1009
  //
  // **错误码触发顺序锁定**（V1 §10.5 钦定）：步骤 1 → 6004（预检）；2a → 1009；
  // 2b → 6004（DELETE RowsAffected == 0 兜底）；任何 DB 异常 → 1009。**不**对外
  // 暴露 6001 / 6002 / 6003 / 6005（V1 §10.5 行 1599 钦定）。
  //
  // **6004 双路径必须等价**（V1 §10.5 行 1601 钦定）：预检路径（步骤 1）+ DB 兜底路径
  // （步骤 2b）都返 apperror.New(ErrUserNotInRoom, ...)，handler 端响应 envelope
  // 完全一致 —— client **不**应区分这两种场景（含"用户不在任何房间" + "用户在其他
  // 房间" + "并发两次 leave 输家"三个子场景，client 不区分）。
  //
  // **ctx 用法**（ADR-0007 §2.4）：fn 内全部 5 个 repo 调用必须用 txCtx 而非外层
  // ctx —— 用错 ctx 会绕过 tx 走 db pool 新连接，FOR UPDATE 锁立即释放，并发保护
  // 失效，r9 cross-tx race 重新出现。
  //
  // **broadcast member.left + close 4007 unregister leaver Session 由 Story 11.8 实装**：
  // 本 story 实装层 commit 成功后**直接 return**，**不**调任何 WS broadcast；
  // Story 11.8 review 时在 return 前按 V1 §10.5 步骤 7-8 顺序（r13 锁定）插入两个
  // post-commit 钩子调用。
  func (s *roomServiceImpl) LeaveRoom(ctx context.Context, in LeaveRoomInput) (*LeaveRoomOutput, error) {
      // (1) 预检 user.current_room_id（事务外，普通连接池查询）
      user, err := s.userRepo.FindByID(ctx, in.UserID)
      if err != nil {
          return nil, apperror.Wrap(err, apperror.ErrServiceBusy, apperror.DefaultMessages[apperror.ErrServiceBusy])
      }
      // 与 input.RoomID 不一致（含 user.CurrentRoomID == nil）→ 6004 预检路径
      if user.CurrentRoomID == nil || *user.CurrentRoomID != in.RoomID {
          return nil, apperror.New(apperror.ErrUserNotInRoom, apperror.DefaultMessages[apperror.ErrUserNotInRoom])
      }

      // (2) 开事务（数据库设计 §8.7 + V1 §10.5 钦定）
      err = s.txMgr.WithTx(ctx, func(txCtx context.Context) error {
          // (2a) SELECT rooms WHERE id = ? FOR UPDATE → 找不到 → 1009 数据不一致兜底
          if _, err := s.roomRepo.FindByIDForUpdate(txCtx, in.RoomID); err != nil {
              return err // 含 ErrRoomNotFound 哨兵 / DB raw error；service 层兜底翻译为 1009
          }

          // (2b) DELETE room_members WHERE room_id = ? AND user_id = ?；rowsAffected == 0 → 6004 兜底
          rowsAffected, err := s.roomMemberRepo.DeleteByRoomAndUser(txCtx, in.RoomID, in.UserID)
          if err != nil {
              return err
          }
          if rowsAffected == 0 {
              return errLeaverNotInRoomInternal
          }

          // (2c) UPDATE users.current_room_id = NULL（**首次启用 nil 入参路径**）
          if err := s.userRepo.UpdateCurrentRoomID(txCtx, in.UserID, nil); err != nil {
              return err
          }

          // (2d) SELECT COUNT(*) FROM room_members WHERE room_id = ? → remaining
          remaining, err := s.roomMemberRepo.CountByRoomID(txCtx, in.RoomID)
          if err != nil {
              return err
          }

          // (2e) 最后一人离开 → UPDATE rooms.status = 2 closed
          if remaining == 0 {
              if err := s.roomRepo.UpdateStatus(txCtx, in.RoomID, roomStatusClosed); err != nil {
                  return err
              }
          }

          return nil
      })

      if err != nil {
          // (3) 业务码分流（**errors.Is 顺序关键** —— review r10 同源风险：errors.Is
          // 顺序写错会让具体业务码被 generic 1009 兜底覆盖）
          if stderrors.Is(err, errLeaverNotInRoomInternal) {
              return nil, apperror.New(apperror.ErrUserNotInRoom, apperror.DefaultMessages[apperror.ErrUserNotInRoom])
          }
          // **关键**：mysql.ErrRoomNotFound（步骤 2a SELECT FOR UPDATE 找不到 rooms 行）
          // 走 **1009** 路径而**不是** 6001 —— V1 §10.5 行 1597 / 行 1599 钦定 leave
          // 接口**不**对外暴露 6001（理论不会发生，因为步骤 1 已通过意味着 caller 在该
          // 房间，rooms 行必存在；race 兜底按 DB 异常处理走 1009）。
          // 直接走 generic ErrServiceBusy 兜底分支（不需要单独识别 mysql.ErrRoomNotFound）
          return nil, apperror.Wrap(err, apperror.ErrServiceBusy, apperror.DefaultMessages[apperror.ErrServiceBusy])
      }

      // (4) 事务 commit 成功 → 返回（broadcast + close 4007 由 Story 11.8 在此处插入调用）
      // TODO: Story 11.8 — 在此插入 close 4007 + unregister leaver Session（V1 §10.5 步骤 7）
      // TODO: Story 11.8 — 在此插入 BroadcastToRoom(roomID, member.left)（V1 §10.5 步骤 8）
      // 顺序由 V1 §10.5 r13 钦定：先 close + unregister 后 broadcast，让 fanout 列表自然
      // 不含 leaver Session（无需 BroadcastToRoom primitive 加 excludeUserID 参数）。
      return &LeaveRoomOutput{
          RoomID: in.RoomID,
          Left:   true,
      }, nil
  }
  ```

  ```go
  // V1 §10.5 钦定"最后一人离开 → status=2 closed"路径写入；rooms.status 严格单调

  const roomStatusClosed = 2
  ```

**AC4 — 事务边界严格 5 步（含错误码触发顺序锁定）**

`LeaveRoom` 的事务边界严格按 V1 §10.5 + 数据库设计 §8.7 + 时序图 §12.2 钦定：

- **步骤 1（事务外）**：`s.userRepo.FindByID(ctx, in.UserID)` → 检查 `user.CurrentRoomID != nil && *user.CurrentRoomID == in.RoomID` → **预检 6004 路径**（不开事务）
- **步骤 2（事务内，5 步）**：
  - **2a**：`s.roomRepo.FindByIDForUpdate(txCtx, in.RoomID)` → 找不到 → 1009（数据不一致兜底；**不**对外暴露 6001）
  - **2b**：`s.roomMemberRepo.DeleteByRoomAndUser(txCtx, in.RoomID, in.UserID)` → `rowsAffected == 0` → 6004（DELETE 兜底）
  - **2c**：`s.userRepo.UpdateCurrentRoomID(txCtx, in.UserID, nil)` → set NULL
  - **2d**：`s.roomMemberRepo.CountByRoomID(txCtx, in.RoomID)` → remaining
  - **2e**：`if remaining == 0` → `s.roomRepo.UpdateStatus(txCtx, in.RoomID, roomStatusClosed)` → 房间 closed
- **步骤 3（事务后）**：commit / rollback；commit 成功 → 返回 LeaveRoomOutput{RoomID, Left: true}；rollback 后按 err 类型分流 6004 / 1009

**错误码触发顺序锁定**（V1 §10.5 行 1597 / 1599 钦定）：步骤 1 → 6004；2a → 1009；2b → 6004；任何 DB 异常 → 1009。**不**对外暴露 6001 / 6002 / 6003 / 6005 —— leave 接口仅校验"用户与房间归属"，其他错误码归 join 接口。

**ctx 用法（ADR-0007 §2.4）**：fn 内全部 5 个 repo 调用必须用 `txCtx` 而非外层 `ctx` —— 用错 ctx 会绕过 tx 走 db pool 新连接，FOR UPDATE 锁立即释放，并发保护失效，r9 cross-tx race 重新出现。本 story 实装严格遵守。

**broadcast + close 触发点**：本 story 实装层 commit 成功后**直接 return**，**不**调任何 WS primitive；Story 11.8 review 时在 `return &LeaveRoomOutput{...}, nil` 之前按 V1 §10.5 步骤 7-8 顺序（r13 锁定）插入：(1) `s.closeLeaverSession(ctx, in.RoomID, in.UserID)` 调用先执行；(2) `s.broadcastMemberLeft(ctx, in.RoomID, in.UserID)` 调用后执行。顺序锁定让 BroadcastToRoom primitive 不需要 `excludeUserID` 参数（先 close + unregister leaver Session 后 broadcast 让 fanout 列表自然不含 leaver）。

**AC5 — 6004 双路径必须等价**

预检路径（步骤 1）+ DELETE 兜底路径（步骤 2b）都返 `apperror.New(ErrUserNotInRoom, ...)`，handler 端响应 envelope 完全一致 —— client 不应区分这两种场景（V1 §10.5 行 1601 钦定 "client 收到 6004 就清本地房间状态、不重试"）。**注**：与 11.3 / 11.4 创建 / 加入房间事务的 6003 双路径同模式（预检 + DB 兜底）。

**含三个子场景**：
- (a) `users.current_room_id == NULL`（步骤 1 预检；用户当前不在任何房间）
- (b) `users.current_room_id != path roomId`（步骤 1 预检；用户在其他房间）
- (c) 同一用户并发两次 leave 同一房间，赢家事务内已删除 `room_members` 行，输家步骤 3 `DELETE` 0 行受影响（步骤 2b 兜底；事务内回滚）

三种场景**全部**返 6004 ErrUserNotInRoom；message 与 errorCode 完全一致。

- **r1 同源风险（事务 ctx 传播）**：review 必须 grep `room_service.go` 确认 `WithTx(ctx, fn)` 内 fn 全部 5 个 repo 调用都用 **`txCtx`** 而非外层 `ctx` —— 任何漏传 / 用错都视为 P1 修复点（ADR-0007 §2.4 钦定；与 11.3 / 11.4 / auth_service.firstTimeLogin 同模式）。**FOR UPDATE 锁的 ctx 传播是关键**：用错 ctx 会让 GORM 走 db pool 新连接，FOR UPDATE 锁立即释放（autocommit 模式），并发保护失效，r9 cross-tx race 重新出现
- **r2 同源风险（错误码触发顺序锁定）**：review 必须 diff service `LeaveRoom` 实装的事务内步骤顺序，**严格按 V1 §10.5 钦定**：步骤 1 → 6004（预检）；2a → 1009（FOR UPDATE 找不到 rooms）；2b → 6004（DELETE RowsAffected==0 兜底）；任何 DB 异常 → 1009。**不**对外暴露 6001 / 6002 / 6003 / 6005（V1 §10.5 行 1599 钦定 leave 仅校验"用户与房间归属"）
