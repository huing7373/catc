## 故事定位（Epic 12 第 6 条 story）

这是 Epic 12「iOS - 房间页面 + WebSocket 客户端」的**第 6 条 story** —— 上游 Story 12.1（WSState + WebSocketClient protocol + WSMessage enum + WSMessageCodec + RealRoomViewModel.applySnapshot 端到端联通）+ Story 12.2（WebSocketClientImpl 真实 URLSessionWebSocketTask 拨号 + connect await handshake + JWT URL-encode 注入 + WSMessageCodec ping outgoing）+ Story 12.3（端到端 wire 联通 + UITEST_ROOM_THREE_MEMBERS）+ Story 12.4（WSMessage 扩 memberJoined / memberLeft + RealRoomViewModel.applyMemberJoined / applyMemberLeft + streamRoomId 启动捕获守护 cross-room race）+ Story 12.5（WebSocketClientImpl 自动重连状态机：close code 分类 + 指数退避 1/2/4/8/30s + 最大 5 次 + sessionGeneration / streamGeneration 双 counter 隔离 stale task + connectGate generation-scoped + precondition 在 gen 翻新前 + supersede 旧 gate） 已全部 done。

**本 story 的"实装"动作**（一句话概括）：

在 Story 12.5 已落地的 reconnect 状态机基础上，**在 `WebSocketClientImpl` 内部加一个 heartbeat task**：connect 成功后启动定时器 → 每 30s 发一次 `ping`（V1 §12.2 钦定 outgoing 消息）→ 等待 server 在 5s 内回 `pong`（V1 §12.3 钦定 server 收 ping 立即回 pong）→ pong 超时则**通过 cancel underlying URLSessionWebSocketTask with `.goingAway`** 触发 receive-loop catch error → close code = 1001 → 走 12.5 transient 路径自动重连（**不**调 public `disconnect()`，那是 close 1000 = terminal 不重连）。

**心跳的本质**：节点 4 阶段 server 端钦定 60s 静默 → close 4005，client 30s ping 是为了让 server 不达 60s 静默阈值；client 端 5s pong 超时是为了让 client 比 server 更早识别"连接已死"，不依赖系统 TCP keepalive / OS timeout 的不确定时长。

**与 12.5 的边界**：

- 12.5 已落地：reconnect 状态机内部"close code → transient/terminal 分类 → 指数退避 + 最大 5 次"；`disconnect()` cancel reconnectTask；`prepareForReconnect()` cancel reconnectTask；double generation counter 隔离 stale task；
- 12.5 已经处理 close 4005（server-emit heartbeat timeout）→ transient → 重连；
- 本 story **新增**：client 自身的 ping 主动驱动 + pong-arrived 信号 + pong-timeout proactive 失活路径；
- 本 story **不**改 reconnect 状态机（不改 `scheduleReconnect` / `attemptReconnect` / `classifyCloseCode` / 退避序列 / 5 次上限）；
- 本 story **不**改 vm 主干（`subscribeRoomIdConnect` / `startConsumingMessages` / `applySnapshot` / `applyMemberJoined` / `applyMemberLeft` / `handle(message:streamRoomId:)`）—— heartbeat 在 client 内部，vm 不感知；
- 本 story **不**消费 `.connectionStateChanged` 接缝 —— pong 超时通过 task.cancel(with: .goingAway) 注入到 12.5 receive-loop catch error 路径，复用 12.5 已有的 transient → schedule reconnect 链路。

**与 12.7 的边界**：12.7（CreateRoom / JoinRoom / LeaveRoom UseCase）**不在本 story 范围**；本 story 的 heartbeat 在 client 内部启动 / 停止，UseCase 层不感知；UseCase 落地后 connect / disconnect 间隔的 heartbeat 由本 story 透明处理。

**本 story 落地后立即解锁**：

- Epic 13 节点 4 demo 验收（13.2）—— heartbeat 是节点 4 验收的"客户端在线感知"必备；
- Story 12.7 UseCase 落地后弱网 / 后台场景的连接稳定性验证。

## 核心契约依据（必读）

> 本 story 严格遵循 V1 §12.2 ping outgoing 协议 + §12.3 pong incoming 协议 + §12.1 close 4005 transient 语义 + Story 12.5 reconnect 状态机锚定，所有实装 / 测试不允许偏离。

### 契约 1：V1 §12.2 ping outgoing 协议（行 1805-1827）

V1 §12.2 钦定 client → server `ping` 消息：
- 信封：`{ "type": "ping", "requestId": "<optional>", "payload": {} }`（**无** `ts` 字段；client→server 方向不需要 client 时间戳）
- 心跳间隔：**客户端应每 30 秒发送一次 ping**（默认值；config key `ws.heartbeat_interval_sec` 在 client 侧）
- `requestId`：客户端可生成（推荐 `"ping_<seq>"` 或 `"ping_<ts_ms>"`），server `pong` 必须回带；省略 → server `pong.requestId = ""`
- `payload`：固定空对象 `{}`，**不**省略 key

**已有实装锚定**（不要重新发明）：
- `WSOutgoingMessage.ping(requestId: String)` enum case 已就位（`iphone/PetApp/Core/Networking/WSOutgoingMessage.swift:20`）
- `WSMessageCodec.encode(_:)` 的 ping 编码已就位（`iphone/PetApp/Core/Networking/WSMessageCodec.swift:108-115`），输出 `{"payload":{},"requestId":"<rid>","type":"ping"}`（sortedKeys）
- `WebSocketClientImpl.send(_:)` 已就位（`iphone/PetApp/Core/Networking/WebSocketClientImpl.swift:371-382`）；未连接 → throw WSError.notConnected

本 story 在心跳 task 里调 `try await self.send(.ping(requestId: <rid>))` 即可，**不**新增任何 outgoing 编码路径。

Source: V1 §12.2 行 1805-1827 + 既有 12.2 实装锚定。

### 契约 2：V1 §12.3 pong incoming 协议（行 2117-2147）

V1 §12.3 钦定 server → client `pong` 消息：
- 信封：`{ "type": "pong", "requestId": "<回带 ping.requestId>", "payload": {}, "ts": <ms> }`
- 触发：server 收到 client ping 后**立即**回 pong（同步路径，不走 broadcast）
- `pong.requestId` 必须等于 `ping.requestId`（含 `""` 情况）

**已有实装锚定**（不要重新发明）：
- `WSMessage.pong(requestId: String)` enum case 已就位（`WebSocketClient.swift:82`）
- `WSMessageCodec.decode(_:)` 的 pong 路由已就位（`WSMessageCodec.swift:50-51`），返回 `.pong(requestId: envelope.requestId)`
- `RealRoomViewModel.handle(message:streamRoomId:)` 的 `case .pong` 当前是 no-op `break`（`RealRoomViewModel.swift:417-419`）—— 本 story **不**让 vm 消费 pong（pong 由 client 内部消费）

**本 story 关键决策**：
- pong 由 `WebSocketClientImpl` **内部 receive-loop** 消费 —— 在 receive-loop decode 出 `.pong(requestId:)` 后**先**通知 heartbeat task（清 pong-pending），**再**正常 yield 到 stream 让 vm 也看到（保持 vm 路径不变；vm `handle` 仍 break 即可）
- **不**让 vm `handle` 调 `webSocketClient.notifyPongReceived(...)` 之类的 callback —— 那会让 vm 反向命令 client，违反"vm 与 client 唯一通信通道是 messages stream"原则（12.1 锁定）
- pong-arrived 信号必须**在 receive-loop 里就被 client 内部识别**，不靠 vm 中转

Source: V1 §12.3 行 2117-2147 + 既有 12.1/12.3 实装锚定。

### 契约 3：心跳间隔 / pong 超时参数

epic line 2164-2183（Story 12.6 acceptance）钦定：
- **心跳间隔**：30 秒（与 V1 §12.2 行 1807 钦定一致）
- **pong 超时**：5 秒（acceptance 行 2176）—— "5 秒未收到 pong → 视为连接失效，主动 disconnect 触发 Story 12.5 自动重连"
- **disconnect 触发自动重连**：注意此处 acceptance 的"主动 disconnect"是行为意图描述（让连接断开 → 12.5 状态机接管重连），**不是字面调 public `disconnect()`**（那是 close 1000 = terminal 不重连，违反 acceptance 意图）

**关键决策**（pong 超时的注入路径）：

不调 `public func disconnect()`（close 1000 terminal）；改为 cancel underlying URLSessionWebSocketTask with `.goingAway`（close 1001）：
- 1001 在 12.5 `classifyCloseCode` 中是 transient（`WebSocketClientImpl.swift:695`：`case 1001, 1011: return .transient(code: raw)`）
- receive-loop catch error 路径会读 `task.closeCode = .goingAway` (rawValue=1001) → 走 transient → `scheduleReconnectIfCurrent` → 12.5 状态机接管
- **不**手工调 `scheduleReconnect` —— 复用 receive-loop 既有路径让分类逻辑只有一个真相源（避免心跳超时 path 与 server-emit close path 走两条 codepath，给未来 dev 埋坑）

**关键决策**（参数硬编码 vs 实例字段）：
- production 默认值硬编码 **30s ping interval / 5s pong timeout**（与 acceptance 一致）
- 与 12.5 `backoffSequence` / `maxReconnectAttempts` 同模式：**`internal` 实例字段**让单测可注入短值（如 0.05s / 0.02s）跑 ms 级 fake-clock 路径，避免单测真实跑 30s+5s（违反 ADR-0002 §3.1「unit test 必须秒级完成」）
- 字段名建议：`heartbeatInterval: TimeInterval = 30.0` / `pongTimeout: TimeInterval = 5.0`

Source: epic line 2164-2183 + V1 §12.2 行 1807 + Story 12.5 实装锚定（`classifyCloseCode` 1001=transient）。

### 契约 4：heartbeat task 生命周期 + sessionGeneration 守护

heartbeat task **必须**：
- (a) connect 成功（receive-loop 收到第一帧 / `firstFrameReceived = true`）后启动；
- (b) `disconnect()` / `prepareForReconnect()` 路径必须 cancel heartbeat task（否则用户主动断开后 client 仍在循环发 ping → send 抛 notConnected → noise log，最坏情况泄漏 task）
- (c) reconnect 成功（attemptReconnect 走 connectInternal 成功 firstFrameReceived）后必须**重启** heartbeat task（旧 task 在 transient close 时已被识别，但本 story 选择"在 firstFrameReceived 路径统一启动 heartbeat"，让 reconnect 成功路径自动启）；reconnect 期间（backoff sleep 中）不发 ping
- (d) heartbeat task 必须 sessionGeneration 守护（与 12.5 receive-loop / reconnect-attempt 同精神 —— stale heartbeat task 不能 send / 不能 cancel underlying task / 不能 schedule reconnect）

**关键决策**（启动点）：
- heartbeat task 启动 = receive-loop 内 `firstFrameReceived = true` 的同步路径，与 `emitConnectionStateIfCurrent(.connected, ...)` 同位置；
- 启动时捕获 `mySession`（与 receive-loop 内的 mySession 同值即可，因为是同一 session 内的兄弟 task）；
- 字段：`private var heartbeatTask: Task<Void, Never>?`（与 `receiveTask` / `reconnectTask` 平级）；
- 启动前 cancel 既有 `heartbeatTask`（防御性兜底；正常路径下 disconnect/prepareForReconnect 已 cancel，此处仅防 receive-loop firstFrame 路径被 fresh connect 重入）

**关键决策**（cancel 点）：
- `disconnect()`：cancel `heartbeatTask`（与 receiveTask / reconnectTask 同位置一并 cancel）+ 清 nil；
- `prepareForReconnect()`：cancel `heartbeatTask` + 清 nil（caller-driven 全 reset）；
- receive-loop 退出（terminal close / 重试上限切 disconnected）：heartbeatTask 被外部 cancel 时已 cancel；本 story **不**在 receive-loop catch path 显式 cancel heartbeatTask（让 disconnect/prepareForReconnect/重连失败终态走 "更新 currentRoomId=nil 时一起 cancel" 的统一收口；细节见下方 Task 1.5）

**关键决策**（pong 超时如何识别 "5s 未收到 pong"）：
- 选项 A（推荐）：**用 `withTaskGroup` 并发 sleep + pong-event**：`heartbeatTask` 内每轮调 send(ping) → 用 `AsyncStream<Void>`（per-ping 一次性 latch；存为 `pendingPongLatch: AsyncStream<Void>.Continuation?`）等 pong-arrived 信号，`Task.sleep` 计 5s 超时；先到者赢；超时 → cancel underlying task with `.goingAway` + return（receive-loop 接管 transient → reconnect）
- 选项 B（备选）：用绝对时间戳 `pongDeadline: ContinuousClock.Instant?` + 每个 ping 设 deadline，下一轮 ping 前检查 deadline 是否 expired（轮询模型；与 acceptance "5 秒未收到 pong" 字面意思有偏差 —— 实际触发时间可能拖到 30s + ε 而非 5s）
- **选 A**：精确 5s 超时，与 acceptance 一致

**关键决策**（pong notify 接缝）：
- 在 `WebSocketClientImpl` 加私有字段：`private var pendingPongContinuation: AsyncStream<Void>.Continuation?`（每个 ping 一次性 latch；下一轮 ping 创建新 stream + continuation 替换；超时 / cancel / disconnect 时 finish）
- receive-loop decode 出 `.pong(...)` 时调内部 helper `notifyPongReceived(mySession:)`：lock 内 yield + finish 当前 latch（generation-gated）
- 本路径**不**改 vm 行为：vm 仍然收到 `.pong` event 走 `case .pong: break`（既有路径）

**关键决策**（per-ping requestId 设计）：
- 用单调递增 sequence：`private var heartbeatSeq: Int = 0`，每次 ping requestId = `"ping_\(seq)"`；
- **不**用 timestamp（时区 / 重启后 collision；不可控 fixture）
- `internal` 可见让单测可读（验证序列）

Source: 12.5 实装锚定（receiveTask / reconnectTask cancel 模式 + sessionGeneration 守护）+ Swift Concurrency `withTaskGroup` API + 本 story 设计。

### 契约 5：pong-arrived 在 stream 上同时下游传播

V1 §12.3 钦定 pong 是合法 server-active 消息；既有 `WSMessage.pong(requestId:)` enum case + decode 路径就位。本 story 关键约束：

- receive-loop decode 出 `.pong(requestId:)` 时**仍然要 yield 到 stream**（保持 vm 端兼容性；不破坏既有 `case .pong: break` 行为）
- **同时**让 client 内部识别 pong-arrived（清 pendingPongContinuation latch）
- yield 顺序：先 internal notify（清 latch），再 yield 到 stream（让 heartbeat 优先于 vm 看到 pong；避免 vm 路径的 main-actor hop 拖慢 latch 释放）

**关键决策**（不让 vm 消费 pong）：
- vm `case .pong: break` 既有路径不动 —— pong 是 client-internal 心跳信号，vm 业务层不需要感知 RTT
- **不**给 vm 加 "RTT 显示" UI（acceptance 不要求；Epic 13 / Epic 30 才考虑）

Source: 既有 12.1 implementation `WebSocketClient.swift` enum + 本 story 设计。

### 契约 6：与 12.5 close 4005 路径的协调（关键不要双触发）

12.5 已经处理 server-emit close 4005（heartbeat timeout）→ transient → reconnect。本 story 新增的 client-side pong-timeout 路径（cancel underlying task with .goingAway = 1001 → receive-loop catch → transient → reconnect）会在以下时序下与 server-side 4005 路径**潜在交叉**：

- 时序 1：server 60s 静默判定 → emit close 4005 → client receive-loop catch error 拿到 .invalid（如果 server 还没真正发 close frame 就被 TCP RST）或 4005（正常 close frame）→ 走 12.5 transient → reconnect
- 时序 2：client 30s ping → server 收到立即回 pong → client 5s timeout 没事 → 下一轮 ping
- 时序 3：网络断了 → client 30s ping → send 可能成功（写入 socket buffer 但对端永远收不到）或 throw（TCP RST 已传播）→ 5s 后 pong 超时 → client cancel task with .goingAway → receive-loop catch → 1001 transient → reconnect
- 时序 4（关键 race）：client send ping → 网络死了 → client 5s timeout 触发 cancel(.goingAway)；与此同时 server 60s 端达到 emit close 4005；两个 close code 谁先到 receive-loop catch path 难说

**关键决策**：不必担心时序 4 的"双触发"—— 12.5 receive-loop catch path 是单一入口，无论是 1001（client cancel goingAway）还是 4005（server close）还是 .invalid（TCP RST）都走同一 `classifyCloseCode` → 都是 transient → 都进 `scheduleReconnectIfCurrent`；reconnect 状态机内部用 sessionGeneration 守护去重，不会重复 schedule。本 story **不**为时序 4 加额外 guard。

**关键决策**（client ping 把 server 60s 静默推迟）：
- 正常路径下 client 每 30s 发 ping → server reset 60s 静默 timer → 永远不达 60s → server 不主动 close 4005
- 因此本 story 落地后 4005 close 几乎只在"client 已死但 server 还没察觉"场景出现（client app 被 kill / 后台无网长时间）
- 反过来说：client side pong-timeout（5s）也几乎只在网络真死场景触发；正常网络抖动 server 来回时延 < 5s 不会误触

Source: V1 §12.1 close code 表（4005 心跳超时 by server）+ §12.2 ping interval 30s + 12.5 classifyCloseCode 实装。

## Story

As an iPhone 用户,
I want App 在 WS 连接活跃期间定时发心跳给 server，连接死了能比 server 60s 超时更快感知并触发重连,
So that 弱网 / TCP RST 静默断网场景下用户体感的"连接已死"延迟可控（≤35s 而非依赖 server 60s 超时），并为 Epic 13 节点 4 demo 验收 / 12.7 UseCase 端到端体验保驾.

### AC1 — WebSocketClientImpl 内嵌 heartbeat task

**修改文件**：`iphone/PetApp/Core/Networking/WebSocketClientImpl.swift`

**新增 internal 实例字段**（与既有 `reconnectTask` / `backoffSequence` / `maxReconnectAttempts` 平级）：

```swift
// MARK: - Story 12.6 heartbeat 字段

/// Story 12.6：当前 in-flight heartbeat task（connect 成功 firstFrame 后启动；disconnect /
/// prepareForReconnect / 终态切 disconnected 时 cancel + 清 nil）.
/// **关键**：与 reconnectTask / receiveTask 同精神 —— 任何"终止活跃 session"路径必须 cancel
/// 此 task，否则用户主动断开后 client 仍在循环 send(.ping) → send 抛 notConnected → noise log,
/// 最坏情况 task 泄漏.
private var heartbeatTask: Task<Void, Never>?

/// Story 12.6：心跳间隔（秒）—— V1 §12.2 行 1807 钦定 30s 默认值.
/// `internal` 实例字段（非 static）让单测可注入短值（如 0.05s）跑 ms 级 fake-clock 路径,
/// 避免单测真实跑 30s（违反 ADR-0002 §3.1「unit test 必须秒级完成」）.
internal var heartbeatInterval: TimeInterval = 30.0

/// Story 12.6：pong 超时（秒）—— epic line 2176 钦定 5s.
/// 5s 未收到 pong → 视连接失效 → cancel underlying task with .goingAway（1001 transient）→
/// receive-loop catch → 12.5 状态机自动重连.
/// `internal` 实例字段同上 —— 单测可注入短值跑 fake-clock.
internal var pongTimeout: TimeInterval = 5.0

/// Story 12.6：心跳序号（每发一次 ping 自增；ping requestId = "ping_<seq>"）.
/// 单调递增 + 进程内复用即可（V1 §12.2 行 1784 钦定 requestId 仅用于 session 内 request-response 配对，
/// 不进 server 持久化日志）.
/// `internal` 可见让单测可读断言序列（"第 N 次 ping requestId == 'ping_N'"）.
internal var heartbeatSeq: Int = 0

/// Story 12.6：当前 in-flight ping 的 pong-arrived latch.
/// 每发一次 ping 在 lock 内创建新 AsyncStream<Void>.makeStream() 替换；receive-loop 收到
/// `.pong(...)` 调 `notifyPongReceivedIfCurrent(mySession:)` → yield + finish；
/// heartbeat task 用 `withTaskGroup` 并发等"latch yield"或"Task.sleep(pongTimeout)"先到者赢.
/// finish 后 nil；下一轮 ping 重新创建.
/// `private`（非 internal）—— 仅 client 内部 receive-loop / heartbeat task 写读，单测不直接访问.
private var pendingPongContinuation: AsyncStream<Void>.Continuation?
```

**新增内部方法 `startHeartbeatTask(mySession:)`**（在 receive-loop firstFrameReceived 同步路径调用，与 `emitConnectionStateIfCurrent(.connected, mySession: mySession)` 同位置）：

```swift
/// Story 12.6：启动 heartbeat task. 在 firstFrameReceived 同步路径调，与 emitConnectionState(.connected) 同位置.
/// `mySession` 由 receive-loop 内捕获的 mySession 透传 —— heartbeat task 内任何 mutation / 调
/// `cancelUnderlyingTaskWithGoingAway` 之前必须 generation-gated.
///
/// **lifecycle**：
///   - 启动前 cancel 既有 heartbeatTask（防御性兜底；正常路径下 disconnect/prepareForReconnect 已 cancel）
///   - 用 Task { ... } 起 detached-style task（捕获 self weak）
///   - task 内 while !Task.isCancelled 循环：
///     1. await Task.sleep(heartbeatInterval) —— 30s 心跳间隔
///     2. 如果 cancelled / stale session → return
///     3. lock 内创建新 pendingPongContinuation + 写入 + 自增 heartbeatSeq + 取出 underlyingTask
///     4. 通过 self.send(.ping(requestId: "ping_\(seq)")) 发心跳；send 抛错（notConnected / underlying network error）→ return
///        （send 已经间接通过 receive-loop catch 路径触发 reconnect，本 task 退出即可）
///     5. 用 withTaskGroup 并发等：(a) AsyncStream 收到 yield 即 pong-arrived；(b) Task.sleep(pongTimeout) 即超时
///     6. (a) 先到 → 清 pendingPongContinuation = nil → continue 下一轮
///     7. (b) 先到 → cancel underlying task with .goingAway → return（receive-loop catch 接管走 12.5 transient → reconnect）
private func startHeartbeatTask(mySession: Int) {
    lock.lock()
    let oldTask = heartbeatTask
    heartbeatTask = nil
    lock.unlock()
    oldTask?.cancel()

    let task = Task { [weak self] in
        while !Task.isCancelled {
            // 1. 心跳间隔 sleep
            do {
                try await Task.sleep(nanoseconds: UInt64((self?.heartbeatInterval ?? 30.0) * 1_000_000_000))
            } catch {
                return  // cancellation
            }
            if Task.isCancelled { return }
            guard let strongSelf = self else { return }
            if !strongSelf.isCurrentSession(mySession) {
                return  // stale session
            }

            // 2. 创建 pong latch + 取 underlying task + 自增 seq
            let (pongStream, pongCont) = AsyncStream<Void>.makeStream()
            let task: WebSocketTaskHandle?
            let seq: Int
            strongSelf.lock.lock()
            // 再次 generation 校验（持锁内最终一致）
            guard strongSelf.sessionGeneration == mySession else {
                strongSelf.lock.unlock()
                pongCont.finish()
                return
            }
            // 关 finish 既有 pendingPongContinuation（防泄漏；上一轮 ping 若没回 pong 也已被 timeout 处理过）
            strongSelf.pendingPongContinuation?.finish()
            strongSelf.pendingPongContinuation = pongCont
            strongSelf.heartbeatSeq += 1
            seq = strongSelf.heartbeatSeq
            task = strongSelf.underlyingTask
            strongSelf.lock.unlock()

            guard let activeTask = task, activeTask.isRunning else {
                // underlying task 不在了（被 cancel / disconnect）→ 直接 return
                strongSelf.lock.lock()
                strongSelf.pendingPongContinuation?.finish()
                strongSelf.pendingPongContinuation = nil
                strongSelf.lock.unlock()
                return
            }

            // 3. send ping —— 走既有 send(_:) 路径（注意：不能让 send 失败也吞掉）
            do {
                try await strongSelf.send(.ping(requestId: "ping_\(seq)"))
            } catch {
                // send 失败：underlying task 已死 / notConnected —— receive-loop 会接管 catch 走 reconnect.
                // 本 heartbeat task 退出即可；不主动 cancel underlying（避免双触发，让 receive-loop 自然 catch）
                strongSelf.lock.lock()
                strongSelf.pendingPongContinuation?.finish()
                strongSelf.pendingPongContinuation = nil
                strongSelf.lock.unlock()
                return
            }

            // 4. 并发等 pong-arrived OR pong-timeout
            let timedOut = await Self.awaitPongOrTimeout(
                pongStream: pongStream,
                timeout: strongSelf.pongTimeout
            )

            // 5. 清 latch（无论 timedOut 与否；防泄漏）
            strongSelf.lock.lock()
            // 再次 generation 校验
            guard strongSelf.sessionGeneration == mySession else {
                strongSelf.pendingPongContinuation?.finish()
                strongSelf.pendingPongContinuation = nil
                strongSelf.lock.unlock()
                return
            }
            strongSelf.pendingPongContinuation?.finish()
            strongSelf.pendingPongContinuation = nil
            strongSelf.lock.unlock()

            if Task.isCancelled { return }
            if timedOut {
                // 6. pong 超时 → cancel underlying task with .goingAway（1001 = transient）
                //    → receive-loop catch 走 12.5 状态机 schedule reconnect.
                //    **不**调 public disconnect()（那是 close 1000 = terminal 不重连）.
                strongSelf.cancelUnderlyingTaskWithGoingAwayIfCurrent(mySession: mySession)
                return
            }
            // 7. pong 正常到达 → continue 下一轮 sleep
        }
    }
    lock.lock()
    heartbeatTask = task
    lock.unlock()
}

/// Story 12.6：内部 helper —— 并发等 pong-arrived OR Task.sleep timeout，先到者赢.
/// 返回 true = 超时；false = pong arrived.
private static func awaitPongOrTimeout(
    pongStream: AsyncStream<Void>,
    timeout: TimeInterval
) async -> Bool {
    return await withTaskGroup(of: Bool.self) { group in
        // task A: 等 pong yield；stream finish 后 for-await 自动退出 = false（pong 已到）
        group.addTask {
            for await _ in pongStream {
                return false  // pong arrived
            }
            // stream 被 finish 但没 yield —— 视为 cancellation（caller 路径），返回 false 让 caller 决策
            return false
        }
        // task B: timer
        group.addTask {
            try? await Task.sleep(nanoseconds: UInt64(timeout * 1_000_000_000))
            return true  // timeout
        }
        // 第一个返回的赢
        let result = await group.next() ?? false
        group.cancelAll()
        return result
    }
}

/// Story 12.6：在 underlying URLSessionWebSocketTask 上 cancel(with: .goingAway) ——
/// rawValue=1001 在 12.5 classifyCloseCode 是 transient → receive-loop catch → schedule reconnect.
/// generation-gated：stale heartbeat task 不能 cancel 新 session 的 underlying task.
private func cancelUnderlyingTaskWithGoingAwayIfCurrent(mySession: Int) {
    lock.lock()
    guard sessionGeneration == mySession else {
        lock.unlock()
        os_log(.debug,
               log: WebSocketClientImpl.logger,
               "heartbeat pong timeout drop: stale mySession=%{public}d (current=%{public}d)",
               mySession, sessionGeneration)
        return
    }
    let task = underlyingTask
    lock.unlock()
    os_log(.info,
           log: WebSocketClientImpl.logger,
           "heartbeat pong timeout (5s) → cancel underlying task with .goingAway (1001) → receive-loop catch transient")
    task?.cancel(with: .goingAway, reason: "heartbeat pong timeout".data(using: .utf8))
}
```

### AC3 — heartbeat task lifecycle（reconnect 路径）

reconnect 状态机由 12.5 已落地；本 story heartbeat task 必须在 reconnect 成功后**自动重启**。

**关键路径**：
- reconnect attempt 由 12.5 `attemptReconnect(attempt:mySession:)` 调 `connectInternal(roomId:isReconnect:true)`
- connectInternal 走 receive-loop firstFrameReceived 同步路径 → 调 `startHeartbeatTask(mySession:)`（AC1 已落地）
- 因此 reconnect 成功后 heartbeatTask **自动重启**（不需要本 story 在 attemptReconnect 加额外代码）

**关键约束**：
- reconnect 期间（backoff sleep 中）**没有** heartbeatTask 在跑（前一个 task 已被 12.5 receive-loop catch path 直接进入 transient → 状态机；本 story 不在 catch path 显式 cancel heartbeatTask 因为它要么自然退出（send 抛错路径）要么由下次 firstFrame 启动前的"防御性 cancel 既有 task" 兜底）
- reconnect 失败超 5 次切 disconnected 时，receive-loop 已 finish stream → underlyingTask 已 cancel；heartbeatTask 在下一轮 sleep 醒来时会因 send 抛错（notConnected）退出，或在更早路径上因 sessionGeneration 翻新被 stale-drop（12.5 attemptReconnect 终态路径已 generation-gated 写状态）

**关键决策**（本 story 不动 12.5 catch path）：
- **不**在 12.5 `startReceiveLoop` 的 catch path（terminal / transient 分支）显式 cancel heartbeatTask
- 理由：
  - terminal 路径：连同 emit `.disconnected` 一并写 currentRoomId=nil + reconnectAttempt=0；下次 connect 入口已 cancel heartbeatTask（AC1 落地）
  - transient 路径：scheduleReconnect → backoff sleep；旧 heartbeatTask 仍可能在跑，但 send 会抛错（receive task 已被 cancel + underlying task 已被 cancel = notConnected 状态），task 自然退出
  - "自然退出"虽然不优雅但安全（不主动 cancel = 少一个 race surface）
- 如 dev 落地觉得"显式 cancel 更稳"，可加；但要在 disconnect/prepareForReconnect/connect 入口 + receive-loop catch 的 terminal/transient 两路径**所有**地方一致 cancel，不能漏（漏一处就有 task 泄漏）

**对应 Tasks**: Task 3.1-3.2

### 开放问题（dev 落地时如有疑问可决策或 raise）

1. **pong-timeout 触发 reconnect 的注入路径**：本 story 钦定走 `task.cancel(with: .goingAway, reason: ...)` 让 receive-loop catch path 接管。dev 落地时若觉得"直接调 12.5 `scheduleReconnectIfCurrent(mySession:)` 更直白"，需要权衡：
   - 直接调好处：少一次 receive-loop catch 的 unwind；
   - 直接调坏处：classifyCloseCode 路径有两个真相源（receive-loop catch 用 closeCode 分类；heartbeat 直接 schedule 不走 closeCode）；未来 dev 改 classifyCloseCode 时容易漏改 heartbeat 路径；
   - **本 story 主张走 cancel .goingAway 路径** —— 单一真相源更稳。
2. **pong-arrived 是否严格匹配 requestId**：本 story 钦定**不严格匹配**（任何 pong 进 latch 即认 ping 已确认）。dev 落地时若觉得"严格匹配 ping_seq → pong.requestId 更鲁棒"，可加（在 `notifyPongReceivedIfCurrent` 内传入 received requestId 与 lock 内 expectedSeq 比较），但要：
   - 单测加 case 覆盖"server 推不带 requestId 的 pong（V1 §12.3 钦定 requestId 必填，但允许 ""）→ 不让心跳进入误超时"
   - 处理 race：如果上一轮 ping 的 pong 在下一轮 ping 发出后才到达，严格匹配会丢弃合理 pong → 提前 trigger 5s 超时（误判）；本 story 主张不严格匹配避此 race。
3. **heartbeatInterval / pongTimeout 是否走 config**：本 story 钦定**硬编码默认值 30s/5s + internal 实例字段**（与 12.5 backoffSequence 同模式）。如未来需要 remote config 调参，加配置接缝再迁移；本 story 不预留。
4. **heartbeat task 是否在 reconnect backoff sleep 中也跑**：本 story 钦定**不跑** —— receive-loop catch path 后 underlying task 已 cancel，heartbeat task 的 send 会抛 notConnected → 自然退出（task 1.5 实装）；下次 firstFrameReceived 路径起新 heartbeat task。
5. **deinit 路径是否需要更激进的清理**：本 story 在 deinit 加 cancel heartbeatTask + finish pendingPongContinuation。如果 dev 觉得 deinit 期 dangling task 已经被 ARC 释放路径处理（task 持 self weak ref），可以省略；但显式 cancel 防御性更稳。
6. **case#HB2 测试 cancel(.goingAway) 的 fake 接缝**：12.5 既有 FakeWebSocketTaskHandle 应该支持 cancelCalls 记录 + cancel 后 receive() 抛错 + closeCode 被设置；如缺失需要补齐。补齐时注意：cancelCalls 字段必须线程安全（lock 保护或用 actor），因为 cancel(with:) 可能从 heartbeat task / disconnect 路径并发调；既有 12.5 fake 落地若线程不安全本 story 顺手补上.
