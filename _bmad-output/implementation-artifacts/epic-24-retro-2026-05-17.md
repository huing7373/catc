## 2. 关键洞察与 Epic 24 的"主旋律"

### 2.1 主旋律 1: scaffold-then-fill 设计模式回报兑现 —— "占位 sink 闭环"零 edit 接力

Story 37.9（scaffold epic）**有意**把 `RealWardrobeViewModel.subscribeInventory` 留成订阅了 `appState.$currentInventory` 但 closure 内只占位的半成品 sink。Epic 24 是这笔设计债的**正向兑现**：

| Story | 接的 seam | 上游留的口子 | 对上游的 edit |
|---|---|---|---|
| 24-1 | sink closure 真实 mapping | 37.9 占位 sink | 改 closure 本体 |
| 24-2 | 数据源（UseCase→applyInventory→publisher） | 24-1 留的 `appState.currentInventory` 写入点 | **24-1 sink 零 edit** |
| 24-3 | 品质过滤叠加 | 基类 `currentCategoryItems` | Mock/Real **零 edit** |

24-1→24-2 的零 edit 闭环是本 epic 最强证据：每条 story 缩窄到一个"seam"，上一条留的 seam 让下一条变成零 edit 接通。这是 Epic 23 主旋律 1（契约定稿前置吸收痛苦）在 **iOS 端结构层**的同构镜像 —— 区别是 Epic 23 前置吸收的是"接口契约歧义"，Epic 24 兑现的是"架构骨架预留 seam"。

### 2.2 主旋律 2: 锚定度而非功能复杂度才是难度预测变量（Epic 21/23 主旋律 N 次重演）

唯一 review 失败（24-2 r1 P2）落在**低锚定**区：`WardrobeView` 的 cancellation 处理 —— `.task(id:)` 取消被 `APIClient` 包成 `APIError.network(URLError.cancelled)` 而非裸 `CancellationError`，落进通用 catch 误弹 RetryView。这个 APIClient 包裹行为是**没有任何上游契约钉死过**的涌现集成细节。反观全部一次过的：

- 24-2 DTO（严格贴 Epic 23 §8.2 **冻结契约** + fail-fast）→ r1 过
- 24-1 类型决策（`HomeEquip` 不演进，锚定 ADR-0010 §4.4 缓解策略）→ r1 过
- 24-3 transient 状态（锚定 ADR-0010 §3.2 view-specific 不进 AppState）→ r1 过
- 24-5 守护契约（锚定 severity-1 lesson + 节点 9 接入意图）→ r1 过

meta-lesson 与 Epic 21/23 完全一致：**复杂度的"锚定度"才是难度预测变量**。Epic 23 §8.2 那 4 轮死磕的契约，正是 Epic 24 DTO 一次过的原因（跨 epic 回报二次确认）。

### 2.3 主旋律 3: epics.md 三连作废 reconcile —— ADR-0009/0010 下游收敛代价的持续偿付

24-1 / 24-3 / 24-5 **全部**要 reconcile epics.md 原文（写于 ADR-0009/0010 之前，假设 Sheet 主入口 / InventoryView / segment control / per-instance disabled 按钮）vs 已交付架构（Wardrobe Tab + 5 分类 Tab + 单 grid + previewCard 单 equipButton）；24-4 直接 deleted。每条 create-story 都做了显式"epics.md 偏离登记"写进 story 文件防 codex 误判。

**关键正面信号**：这个 reconcile 做对了 —— create-story sub-agent 锚定权威源（sprint-change-proposal-2026-04-29-v2 §5.4 + ADR-0009/0010）+ 偏离逐条登记 → **没有任何一轮 codex review 把偏离当成 bug flag**。机制（story 文件内嵌偏离登记表）有效。但根 planning 文档仍 stale，每条下游 story 持续付 reconcile 噪音成本（非阻塞，详见 §4.2）。

### 2.4 主旋律 4: over-correction 防御单轮收敛 —— AP1(override #3) 第二次兑现

24-2 r1→r2：codex r1 flag P2（wrapped cancellation 误弹 RetryView）→ fix-review 按 override #3 做**窄判**（`isSilentCancellation` helper 识别 3 形态但**不**宽吞所有 `.network` 错误）+ 加守门测试（case#12，4 反例断言 timeout/离线/business/decoding 仍弹 retry）→ r2 **一轮锁定收敛**，零膨胀。对比 Epic 20 race chain 5 轮 / testing chain 7 轮、Epic 21 21-1 4 轮 —— 这是 Epic 23 主旋律 3（AP1 固化进 epic-loop.md override #3 的回报）的**第二次连续兑现**。机制持续有效。

### 2.5 主旋律 5: 纯守护契约 story 模式（24-5）—— 零 production 改动 + RED-proof 的正确姿势

24-5 epics.md 原文要求 per-instance disabled 按钮（pre-ADR），但已交付架构（37.9/24.1/24.2/24.3）**已结构性满足**"节点 9 接入点就位"意图。处理做对了：**不臆造工作量**，reconcile 成零 production 改动纯守护 story —— 只新建 `EquipButtonReservationTests.swift` 把 equipButton 节点 9 接入契约钉成机器可校验守护，配 RED-proof（临时改 no-op 复刻 severity-1 lesson 反例 → 守护正确 fail → 字节级恢复）证明守护真能抓复犯。这是"story 意图已被上游交付"时的**典范处理**：锁契约、不造假活。防的是 `real-viewmodel-override-placeholder-must-mutate-state`（severity 1，已复犯一次）。

### 2.6 固有噪音: SourceKit 陈旧孤立分析快照（Epic 23 gopls 噪音的 iOS 同构）

4 条 iOS story **每条**：sub-agent 编辑后 harness 抛 `<new-diagnostics>` SourceKit 错误（`Cannot find type X in scope` / `No such module 'XCTest'`），但每次主 agent 独立 `bash iphone/scripts/build.sh --test` 均确认 BUILD SUCCESS + 测试全绿。根因：SourceKit 脱离 build module/index 上下文做**孤立单文件分析**，跨文件符号全报找不到。若盲信会误杀 **4/4** story。**机制**：每条 iOS story 主 agent 必须独立 `build.sh --test` 二次核验，**不盲信 harness 抛的 SourceKit diagnostics**。这是 Epic 23 gopls 陈旧中间态快照发现的**同类异构问题**（不同 toolchain，同一类"harness 静态分析快照 ≠ 真实编译结果"陷阱）—— 应合并成一条跨语言通则。

## 5. 关键 Takeaways（Top 5）

1. **scaffold-then-fill 设计债是正资产**：37.9 有意留的占位 sink，让 24-1→24-2 零 edit 接通闭环。架构骨架预留 seam = iOS 端的"契约前置吸收痛苦"。
2. **锚定度是难度预测变量（第 N 次实证）**：唯一 review 失败在低锚定的 APIClient cancellation 包裹涌现细节；全部高锚定区（冻结 DTO / ADR 类型决策 / severity-1 lesson）一次过。
3. **AP1(override #3) 第二次兑现**：24-2 over-correction 单轮收敛零膨胀（vs Epic 20/21 的 5/7/4 轮）；机制持续有效。
4. **纯守护契约 story 是"意图已交付"的典范处理**：24-5 零 production 改动 + RED-proof，锁契约不造假活。
5. **harness 静态分析快照 ≠ 真实编译**：SourceKit 孤立分析（iOS）= Epic 23 gopls 陈旧快照（Go）同构；每条有码 story 主 agent 必须独立真实 build/test 二次核验。
