## 2. 关键洞察与 Epic 21 的"主旋律"

### 2.1 主旋律 1: 21-1 over-correction chain — Epic 20 元模式在 SwiftUI 时序场景的精确重演

21-1 跑满 4 轮 fix-review，是 Epic 21 唯一的 over-correction chain。链条：
- r0: `isUnlockable = remainingSeconds <= 0 || status == .unlockable`
- r1: 未初始化默认 `0` 误判为"超时" → counting chest 首帧闪 unlockable → fix: status-only（表层）
- r2: status-only 矫枉过正，本地 timer tick 到 0 不再切视觉态 → fix: driver 同步初始化 + status-aware 双轴（表层）
- r3: `.receive(on: DispatchQueue.main)` 异步 hop 破坏 happens-before，subsequent hydration 又闪一帧 → fix: 删异步 hop（表层）
- **r4: 根因翻转** —— `unlockAt - Date()` 本地时钟派生违反"状态以 server 为准"原则，device clock skew 时误判 → fix: server `remainingSeconds` 锚定 + 注入式 `clock: () -> Date` + 守门测试
- r5: APPROVED

**关键观察**：r1→r3 都是同一抽象层（view-state 派生时机）的表层微调，每轮 finding 都来自上轮修复；**r4 才上移到"source of truth 锚定"根因层**。这与 Epic 20 三个 chain（20.7 RowsAffected→事务约束 / 20.8 错误码→metrics 路径 / 20.9 断言强弱→责任分离）是**同一元模式**：over-correction chain 第 3+ 轮必须上移一层抽象。

**Epic 20 retro 主旋律 2 已标定此模式，但未固化进 epic-loop 流程**——21-1 本可在 r2 就跳出表层（如果 fix-review prompt r2 起就内置 over-correction warning），实际跑到 r5。→ 转化为本 retro **AP1**。

对应 lesson：`2026-05-15-default-value-vs-meaningful-zero-21-1-r1.md` / `-driver-sync-init-on-sink-21-1-r2.md` / `-driver-sync-sink-on-subsequent-change-21-1-r3.md` / `-driver-server-anchored-time-21-1-r4.md`（4 条，r2 含 `supersedes_partial` 指向 r1）。

### 2.2 主旋律 2: 难度预测的系统性偏差 — "锚定度"而非"功能复杂度"才是预测变量

Epic 20 retro line 194-201 对 Epic 21 的预测：

| 预测 | 实际 | 判定 |
|---|---|---|
| 总轮次 11-14 | **11** | ✅ 总量精准 |
| 21.1 r1-r2（最简单） | **r5（最难，chain）** | ❌ 完全反转 |
| 21.2 r1-r3（async race 风险） | **r1 一次过** | ⚠️ 风险类对，归属错（async race 实际爆在 21-1） |
| 21.3 r2-r4（idempotency 陷阱） | **r1 一次过** | ⚠️ 严重低估契约成熟度 |
| 21.4 r1-r2 | **r1-r2** | ✅ 准 |
| 21.5 r2-r3 | **r1-r2** | ✅ 准 |

**meta-lesson**：story 难度不取决于"功能复杂度"，而取决于"有多少未被上游契约/lesson 覆盖的隐性时序/状态决策"。21-1 看似简单（一个 SwiftUI view），但要新发明 ChestTimerDriver 的时序语义（Combine sink / 初始化时机 / clock 源）——**无上游契约兜底**；21-3 看似复杂（幂等+错误码+双写），但每处都有 Epic 20 V1 §7.2 + lesson 钦定——**高锚定**。**复杂度的"锚定度"才是预测变量**。这进一步强化了 Epic 20 主旋律 3（实装类一次过 = 契约层已迭代到无 ambiguity）。

### 2.3 主旋律 3: wontfix-with-rationale 的成熟运用（21-5 r1）

21-5 r1 codex 把"开箱前双 sync"判为可消除的 duplicate work，要求改成"只共享 in-flight gate"。sub-agent 没盲从——读 Story 8.5 的 triggerManual race-fix 历史后识别出 codex 建议正是 8.5 两轮 race-fix 钦定要避免的 caller-gets-stale-state bug；双 sync 是开箱事务校验步数余额的 safe choice。正式记 **wontfix + 归档决策推理 lesson + 引用前次 race-fix lesson**，r2 codex 自己也认无 actionable bug。

这是 Epic 20 主旋律 2 的进化：不止"上移抽象找根因"，还学会"拒绝会破坏既有 race-fix 的修复并留证"。对应 lesson：`2026-05-16-double-sync-before-chest-open-is-transactional-safe-21-5-r1.md`（含 `related_lessons` 指向 `2026-05-04-manual-trigger-must-await-in-flight.md`）。

### 2.4 主旋律 4: 中断恢复机制再次验证（rate limit + SendMessage resume）

21-4 dev_story 撞 Anthropic rate limit 中断在 in-progress 状态（生产文件 + 测试已大部分落地，未推 review）。恢复方式：**SendMessage 给原 sub-agent**（保留完整 context）让它续跑剩余 workflow（build + test + MCP 实跑 + 推 review）。未丢工作、未重做、未数据破坏。

与 Epic 20（2 次中断走"新 sub-agent audit + resume"）的差异：本次用 **SendMessage 续原 agent** 比派新 agent 更省——原 agent 还记得做到哪了。对未来 epic-loop 长链路的指导：rate limit / quota 类中断**优先 SendMessage 续原 sub-agent**（context 仍在）；只有 socket crash / agent 不可达时才派新 agent audit+resume。

## 5. 关键 Takeaways（Top 5）

1. **over-correction chain 的根因层翻转通常在 r4**：21.1（本地时钟派生 → server-anchored）、Epic 20 的 20.7（RowsAffected → 事务约束）/ 20.1（Redis → MySQL）都在 r4 附近翻转。第 3 轮起每轮 finding 来自上轮修复 = 强信号，应主动上移抽象而非等 r4 撞上。
2. **story 难度的预测变量是"锚定度"不是"功能复杂度"**：21-1（低锚定/新发明时序语义）跑 4 轮，21-3（高锚定/贴 Epic 20 契约）一次过。预测下个 epic 时按"有多少隐性决策无上游契约/lesson 兜底"评估，不按功能表面复杂度。
3. **wontfix-with-rationale 是成熟 review 应对**：codex 要求改一个历经多轮 race-fix 的稳定组件时，正确动作是拒绝盲改 + 归档决策推理 + 引用前次 race-fix lesson（21.5 r1），而非顺从 review 制造新 race。
4. **rate limit 中断优先 SendMessage 续原 sub-agent**：原 agent 保留完整 context，比派新 agent audit+resume 更省（21-4 范例）。socket crash / 不可达时才派新 agent。
5. **流程改进必须固化进模板，不能靠主 agent 临场记得**：Epic 20 AP1/AP3 未固化导致 21-1 仍跑 4 轮。lesson + retro 只是知识，固化进 epic-loop.md prompt 模板才是机制。
