## 2. 关键洞察与 Epic 11 的"主旋律"

### 2.1 主旋律 1: 协议契约 Story 11.1 跑到 **r14**，**超过** Epic 10 末调整的 r10 上限

Story 11.1（接口契约最终化）从 r1 一路演进到 r14 才收敛 —— 比 Story 10.1 的 r10 多出 4 轮，比 epic-loop 子循环上限多出 4 轮。

**这不是"上限失配"，而是"协议契约 Story 的复杂度天花板还没摸到"**：每一轮 r1~r14 都命中真实问题（不是 cosmetic / nit）：

| 轮次 | 问题维度 |
|---|---|
| r1 | 房间 roster 契约自洽 / `member.joined` 必须自包含丰富字段 |
| r2 | HTTP leave 必须关闭 WS Session / 跨文档触发点声明对齐 |
| r3 | WS 断开仅清 ephemeral / 房间归属只能由 HTTP leave 改 |
| r4 | create / leave 并发 race 落到正确业务码（6003 而非 1009）+ 冻结契约引用稳定锚 |
| r5 | `pet.currentState` enum 不能与 `motion_state` 别名 |
| r6 | snapshot 示例 / mermaid 图必须镜像 prose |
| r7 | 接口默认 deny + 显式 allow ACL；prose 与 mermaid zip 对齐 |
| r8 | ACL 校验与受 ACL 保护的数据返回必须共享同一事务 snapshot |
| r9 | snapshot 隔离 + FOR SHARE / 跨事务 FOR UPDATE 串行化 |
| r10 | pet 字段全协议 nullable / WS close 4007 best-effort cleanup |
| r11 | 给已有错误码新增接口用法时反向扫描"该错误码仅用于 X"陈述 |
| r12 | ACL FOR SHARE 锁精确边界 + mermaid pet nullable 同步 |
| r13 | leave step 顺序对调（close 4007 先于 broadcast member.left）匹配 BroadcastToRoom primitive 能力 |
| r14 | §10.3 / §12.3 跨章节同一 placeholder 阶段 pet shape 收敛到单一 going-forward 契约 |

**根因**：房间业务契约的"审计面"比 WS 协议本身更大：5 个 REST 接口 + 2 个 WS 消息 + 错误码 6001-6005 + 节点 4/5/9/10 五阶段过渡表 + 跨 4 份 docs（V1 / DB / 时序 / Go 项目结构）+ ACL 锁 + 并发 race 业务码归属 + 跨章节 placeholder shape 一致性。每一类都是一个独立审计维度。

**结果**：r10 上限对 11.1 不够。如果硬卡 r10，r11~r14 这 4 条不变量（错误码反向扫描 / FOR SHARE 锁精确边界 / leave step 顺序与 primitive 能力匹配 / 跨章节 placeholder shape 收敛）会被静默放过。

### 2.2 主旋律 2: 异步化路径 11.8 跑了 **10 轮折返**（r2~r11），中间出现 **回头路**

Story 11.8 把 `member.joined` / `member.left` 广播 + leaver session close 4007 切到 post-commit 异步路径。10 轮 review 累计了 11.8 r1~r11 lessons，揭露异步化是个"看似简单实则牵动 5 个维度"的陷阱：

| 轮 | 修复方向 |
|---|---|
| r1 | `closeLeaverSession` 缺 nil sessionMgr guard |
| r2 | post-commit fire-and-forget 必须 detached ctx + 独立 goroutine + timeout 兜底 |
| r3 | 异步必须保留 sync observable invariants（joiner self-fanout / leaver stale subscription） |
| r4 | 异步路径必须按业务 key（roomID）做 serialization 才能保留 caller commit 顺序 causal ordering |
| r5 | lock 必须在 caller 同步段获取；long-running side effect 不应持 serialization lock |
| r6 | commit + enqueue 包成原子段；unregister 进 worker 闭包 |
| r7 | **三象限分类（instant / slow / ordered）—— LeaveRoom unregister 移回 lock 段同步执行**（颠覆 r6 的 worker 闭包决策） |
| r8 | per-room worker lifecycle defer + tech-debt 显著标记 |
| r9 | fire-and-forget queue 满阻塞背压替代 silent drop |
| r10 | 昂贵资源分配在 cheap validation 之后；JoinRoom lock 段前补 room-exists check 关闭 attack vector |
| r11 | snapshot + act 模式必须 atomic 持锁或 act 时 re-check 状态 |

**回头路**：r6 把 unregister 进 worker 闭包，r7 又把它移回 lock 段同步执行。这是 **r6 颠覆 r3 的 sync invariant，r7 又推翻 r6**。

**Epic 10 retro §4.3 的"Story 10.6 r3~r10 折返"在 Epic 11 的 11.8 r2~r11 几乎完全复演**。fire-and-forget 异步化的复杂度在 dev-story 阶段无法穷举枚举（join / leave / cross-tx / shutdown × instant / slow / ordered × backpressure / drop / block × snapshot consistency × attack vector），review 阶段一轮一轮挖才能挖到底。

### 2.3 主旋律 3: lesson 文化复利继续兑现 —— **6 个 Story 一次过**

11.2 / 11.4 / 11.6 / 11.7 / 11.9 / 11.10 全部 0 review/fix 子循环 done。这是 Epic 11 最值得高兴的事：

## 3. What Went Well（一定要保留的好做法）

### 3.1 lesson reference 块作为 anti-pattern checklist 的 ROI 翻倍

Story 11.6 / 11.7 在 dev-story 阶段就 hardcode 引用 5+ 条来自 Epic 10 / 11.1 的 lesson 路径（如 `[Source: docs/lessons/2026-05-08-acl-and-protected-data-must-share-snapshot-tx-11-1-r8.md]`），让 dev 阶段的"该不该这么做"决策被前置规约。证据：11.7 dev 阶段 4 单测 + 1 集成测试一次过，0 review fix。

### 3.2 范围红线"本 story 不做"清单 + ws stub 同步纪律

每个 Story dev notes 都列具体到接口 / 字段 / 方法的"不做"清单（例如 11.5 dev notes 第 994 行明确"未引入 11.6 GET 接口；未实装 WS broadcast / close 4007；未消费 idempotencyKey..."）。配合 ws 包 `nilRoomMemberRepo` / `stubRoomMemberRepo` 随 interface 演进每个 story 都补兜底，编译期防止越界。这是 11.3 → 11.4 → 11.5 → 11.6 共用的纪律。

### 3.3 协议契约 Story 11.1 的 14 轮 review 没有"放低标准提前关闸"

Reviewer 在 r10 之后没有因为"已经 10 轮了，差不多够了"而放过 r11~r14 这 4 个独立维度。这是 Epic 10 retro §6 A5 行为级落实。

### 3.4 11.8 r2~r11 的 10 轮异步化折返**没有跨过 epic-loop 上限抽象违反**

虽然 11.8 跑到 r11 超过子循环上限，但每一轮都是"上一轮的修复带来新维度的暴露"，不是同一维度的反复打磨。三象限分类（r7）+ commit-time per-room serialization（r6）是真正落地的 architectural insight，不是补丁。

### 3.5 集成测试 fault-injection wrapper 模式（11.9）落地

`faultRoomMemberRepo` / `faultUserRepoForJoin` / `faultUserRepoForLeave` 三个 wrapper struct 让 11.9 的 3 个回滚 case（fault-inject 第 N 步抛 error → 验证前 N-1 步全部回滚）走真 dockertest 路径而非 mock。这与 Epic 4 Story 4.7 同源模式，复用了已有的"fault wrapper + buildXxxWithCustomRepos helper"模板。

### 3.6 Build 脚本契约稳定性继续保持

整个 Epic 11 期间没有触碰 `scripts/build.sh` 行为；所有 Story 的"完成验证"指令是同一行 `bash scripts/build.sh --test` + `bash scripts/build.sh --integration`。

## 4. What Didn't Go Well（要修正的）

### 4.1 epic-loop r10 上限对协议契约 + 异步化 Story 不够

- **现象**：11.1 跑到 r14（+4），11.8 跑到 r11（+1）。Epic 10 末把上限从 5 → 10 后，本以为覆盖了"WS / 并发 / 资产事务"三类高复杂度 Story；Epic 11 证明协议契约类 Story 的复杂度天花板比此前认知更高。
- **根因**：r10 是经验值上限，但**协议契约 Story 的审计维度**（跨 4 份 docs + ACL 锁 + 并发 race 业务码归属 + 跨章节 placeholder shape 一致性）和 **fire-and-forget 异步化**（5 维笛卡尔积）的复杂度都比 Epic 10 时认知的更深。
- **影响**：这两个 Story 都靠手动 override 突破上限完成。如果硬卡 r10，11.1 r11~r14 + 11.8 r11 都会被静默放过，会埋下 P1 隐患。
- **建议**（非紧急）：考虑把 epic-loop 上限按 Story 类型分级 ——
  - 协议契约 Story（X.1）：去掉硬上限或调到 20，由 reviewer 自然收敛
  - 异步化 / 并发 Story：保持 10
  - 普通业务 Story：保持 10（或 5 也够）

### 4.2 Story 11.8 r6 → r7 出现回头路（worker 闭包 → lock 段同步）

- **现象**：r6 把 unregister 进 worker 闭包（与 broadcast 同 fn），r7 review 揭露这违反"HTTP 200 前 leaver 已 detached"钦定（V1 §10.5 步骤 7），把 unregister 移回 lock 段同步段。
- **根因**：dev-story 阶段把 post-commit hook 的"哪些操作必须 sync / 哪些必须 ordered / 哪些可 fire-and-forget"分类没穷举枚举。r2~r6 反复在 hybrid 切分边界打补丁，直到 r7 引入"三象限分类"才彻底定位。
- **后果**：1 次回头路（r7 推翻 r6）。属正常异步化探索成本，但若 dev-story 阶段提前画三象限矩阵可减少 1~2 轮。
- **状态**：已落地 r7 lesson `2026-05-09-post-commit-three-quadrants-instant-vs-slow-vs-ordered-11-8-r7.md`，未来异步化 Story（如 Epic 14 / 17 / 20 / 32）应在 dev-story 阶段直接 reference。

### 4.3 Race detector 在 Windows 仍然缺位 —— 11.8 进一步放大风险

- **现象**：Epic 10 retro §4.5 已 flag。11.8 引入 per-room mutex + worker queue + fire-and-forget hook + commit-time serialization，并发原语数量比 Epic 10 还多。
- **代价**：Windows dev 环境 race report 完全缺位；测试用真实 wire + 集成路径间接覆盖大部分 race，但 race detector 系统性 risk 没消除。
- **状态**：⏳ 仍属 tech debt（Epic 10 retro A2 钦定 Epic 13 落地）；本 epic 没回归这条。

### 4.4 service 整包集成测试 600s timeout 偶尔触发

- **现象**：11.7 / 11.8 dev/test 阶段都报告 service 包整跑 dockertest 在 Windows 单进程默认 timeout 偶尔超时（auth_service.NewPool 重试 + cross-tx leave 测试 dockertest spin-up 多容器导致总耗时 > 600s）。
- **缓解**：按包分跑（`go test -tags=integration -timeout=600s ./internal/service/...` 隔离跑 24s 通过）稳定；不阻塞功能验证。
- **影响**：CI 还没有对应缓解；如果 Epic 13 demo 验收阶段加 CI 跑全量集成测试需要先调整 timeout 或拆包。
- **状态**：⏳ 已知 infra 限制，与本 epic 业务无关。Epic 13 文档同步阶段处理。

## 5. Lessons & Patterns 提炼（送给 Epic 12 / 14 / 17 / 20 / 32 等下游）

### 5.1 Epic 11 沉淀的 26 条 lesson 中**送给下游的 7 条核心**

| # | Lesson | 来源 |
|---|---|---|
| 1 | 协议契约 Story 不能卡 review 上限 —— 文档审计维度可能客观超过 r10 | 11-1 r1 ~ r14 |
| 2 | post-commit hook 三象限分类（instant / slow / ordered）—— sync 段必须 instant；slow 进 fire-and-forget；ordering-sensitive 进 worker queue | 11-8 r7 |
| 3 | 异步化必须按业务 key（roomID / userID）做 serialization 才能保留 caller commit 顺序 causal ordering | 11-8 r4 / r6 |
| 4 | 异步必须保留 sync observable invariants（joiner 不应自己 fanout / leaver subscription 必须先 detach） | 11-8 r3 |
| 5 | ACL 校验 + 受 ACL 保护的数据必须共享同一事务 snapshot；FOR SHARE 锁精确边界 | 11-1 r8 / r12 |
| 6 | snapshot + act 模式必须 atomic 持锁或 act 时 re-check 状态 | 11-8 r11 |
| 7 | 给已有错误码新增接口用法时必须反向扫描 docs 中"该错误码仅用于 X"陈述 | 11-1 r11 |

下游 epic 改任何 post-commit hook / WS 业务广播 / 跨事务 ACL 路径前**必须先比对这 7 条**。

### 5.2 流程层 lesson（Epic 10 retro 已钦定的）继续验证

1. **协议 contract Story 与实装 Story 同等严格** —— 11.1 r14 是 10.1 r10 的加强证据。
2. **review/fix 子循环上限要匹配 Story 复杂度** —— **协议契约 Story 应去掉硬上限**；其他 Story 维持 10 仍合理。

4. **dev-story 阶段写 anti-pattern 清单 ROI 极高** —— 11.5 / 11.7 / 11.10 各列具体到接口 / 字段 / 方法的"不做"清单，让 dev 阶段范围扩张被主动遏制。

## 10. Closing Note

Epic 11 是 Epic 10 的"相变后半场"：节点 4 第一个 epic（10）建好 WS 框架 + 中间件，第二个 epic（11）在框架上填房间业务。

数字画像：
- **Story 11.1 跑到 r14** —— 协议契约 Story 的复杂度天花板继续被推高
- **Story 11.8 跑到 r11 + 出现 r6→r7 回头路** —— 异步化路径是 5 维笛卡尔积陷阱，dev-story 阶段难以穷举
- **6 个 Story 一次过** —— 比 Epic 10 的 1 个翻 6 倍；lesson reference 块作为 anti-pattern checklist 的 ROI 翻倍兑现
- **27 轮总 review/fix vs 41 轮（Epic 10）** —— 总轮次下降 34%，但 Epic 11 多 3 个 Story（10 vs 7）；**单 story 平均轮次 5.9 → 2.7 减半**

最值得高兴的两件事：
1. **6 个 Story 一次过 = lessons 沉淀复利兑现** —— Epic 10 retro §3.1 的"强制 review/fix 子循环 + lesson 沉淀文化"在 Epic 11 半 epic 范围内已经 ROI 翻倍。
2. **11.8 r7 三象限分类是 architectural insight 不是补丁** —— 经过 r2~r6 五轮反复演进 + r7 回头反思，后续 Epic 14 / 17 / 20 / 32 的异步化 Story 都可以直接 fork 这套分类。

最值得警觉的两件事：
1. **协议契约 Story 的 r10 上限不够** —— 11.1 跑到 r14 比 10.1 r10 还多 4 轮；Epic 14.1 / 17.1 / 20.1 / 32.1 等下游契约 Story 的 review 严格度不能因为"已经做过 4 个 contract Story 了，应该熟练了"而降低。
2. **异步化 Story 的复杂度不可低估** —— 11.8 r2~r11 的 10 轮反复 + r6 → r7 回头路是 Epic 10 §4.3 Story 10.6 的 lesson 在 Epic 11 的复演；Epic 14 / 17 等下游异步化 epic 必须前置画三象限矩阵。
