## Story

As an iPhone 用户,
I want App 在合适时机（启动 / 回前台 / 定时 / 开箱前）自动把我的累计步数同步到服务端，
so that 我不必手动点同步，可用步数总是接近真实值，开箱时也能用上最新累计.

### AC1 — V1 §6.1 wire DTO（`StepsSyncRequest` / `StepsSyncResponse`）

**新建** `iphone/PetApp/Features/Home/Models/StepsSyncRequest.swift`：

```swift
// StepsSyncRequest.swift
// Story 8.5 AC1: V1 §6.1 POST /api/v1/steps/sync 请求 wire DTO.
//
// 契约源（V1 §6.1 已冻结，**禁止**改任一字段）:
// - syncDate: string YYYY-MM-DD（严格 10 字符；client 按本机时区算"今天"；server [today-2d, today+2d] 容忍）
// - clientTotalSteps: int ≥ 0（HealthKit 当日累计；不是增量）
// - motionState: int ∈ {1,2,3}（1=stationary_or_unknown / 2=walking / 3=running；与 V1 §6.1.3 枚举一致）
// - clientTimestamp: int64 ms > 0（仅审计；不参与 server 差值计算）
//
// 字段类型选择:
// - syncDate: String（Codable encode 直接 JSON 字符串；不用 Date 避免跨时区 ISO 化失误）
// - clientTotalSteps: Int（HealthKit 当日步数 64-bit 安全；服务端 INT32 自然范围内）
// - motionState: Int（与 server INT8 字段对齐；MotionState String enum 通过 wireValue extension 桥接，详见 AC4）
// - clientTimestamp: Int64（毫秒；与 V1 §6.1.4 一致；服务端 BIGINT UNSIGNED）

import Foundation

public struct StepsSyncRequest: Encodable, Sendable, Equatable {
    public let syncDate: String
    public let clientTotalSteps: Int
    public let motionState: Int
    public let clientTimestamp: Int64

    public init(
        syncDate: String,
        clientTotalSteps: Int,
        motionState: Int,
        clientTimestamp: Int64
    ) {
        self.syncDate = syncDate
        self.clientTotalSteps = clientTotalSteps
        self.motionState = motionState
        self.clientTimestamp = clientTimestamp
    }
}
```

**新建** `iphone/PetApp/Features/Home/Models/StepsSyncResponse.swift`：

```swift
// StepsSyncResponse.swift
// Story 8.5 AC1: V1 §6.1 POST /api/v1/steps/sync 响应 wire DTO.
//
// 契约源（V1 §6.1 已冻结）:
// - 外层走 APIClient 既有 envelope 解包（`code/message/data/requestId`，已在 Story 2.4 / 5.5 落地）
// - data 字段嵌套：data.acceptedDeltaSteps + data.stepAccount.{totalSteps, availableSteps, consumedSteps}
//   （注意：与 §6.2 `GET /steps/account` 直接平铺三档值不同；§6.1 是"动作型"接口，附带本次入账后的最新账户态）
//
// **关键 schema 选择**（与 Story 5.5 HomeResponse / StepAccountDTO 同模式）:
// - Decodable（不需要 Encodable；client 只解析）+ Equatable + Sendable
// - 嵌套 StepAccountInResponse 结构体单独命名（不复用 HomeResponse.StepAccountDTO，避免改 5.5 落地的 HomeData
//   解析路径——HomeResponse.StepAccountDTO 是 GET /home 解析专用；本 story 是 POST /steps/sync 专用）

import Foundation

public struct StepsSyncResponse: Decodable, Sendable, Equatable {
    public let acceptedDeltaSteps: Int
    public let stepAccount: StepAccountInSyncResponse

    public init(
        acceptedDeltaSteps: Int,
        stepAccount: StepAccountInSyncResponse
    ) {
        self.acceptedDeltaSteps = acceptedDeltaSteps
        self.stepAccount = stepAccount
    }
}

public struct StepAccountInSyncResponse: Decodable, Sendable, Equatable {
    public let totalSteps: Int
    public let availableSteps: Int
    public let consumedSteps: Int

    public init(totalSteps: Int, availableSteps: Int, consumedSteps: Int) {
        self.totalSteps = totalSteps
        self.availableSteps = availableSteps
        self.consumedSteps = consumedSteps
    }
}
```

**关键设计决策**：

1. **`syncDate: String` 而非 `Date`**：避免 Codable 默认 ISO 8601 化（带 T + 时区）；V1 钦定严格 10 字符 `YYYY-MM-DD`。AC6 `DateProvider.todayString(timeZone:)` 负责生成。
2. **`motionState: Int` 而非 enum**：wire 类型与 server INT8 对齐；`MotionState` 通过 AC4 `wireValue` extension 派生。**禁止**直接 encode `MotionState.rawValue`（"rest"/"walk"/"run" 是 string，server 不接受）。
3. **`clientTimestamp: Int64`**：毫秒；用 `Int64(Date().timeIntervalSince1970 * 1000)` 派生（AC6 `DateProvider.nowMillis()`）。**不**用 `Int` —— 32-bit 平台 timestamp ms 会溢出（虽然 iOS 早不存在 32-bit，但 V1 钦定 int64 → 严格对齐契约）。
4. **`StepAccountInSyncResponse` 独立命名**（不复用 `HomeResponse.StepAccountDTO`）：本 story 解析路径独立；避免因为修改 5.5 落地的 DTO 间接破坏 GET /home 解析。
5. **不引 `errorPresenter` / 不持锁**：DTO 是 pure value type；不接 UI / async 行为。

### AC2 — `StepsEndpoints` + `StepRepository`

**新建** `iphone/PetApp/Features/Home/UseCases/StepsEndpoints.swift`：

```swift
// StepsEndpoints.swift
// Story 8.5 AC2: V1 §6.1 POST /api/v1/steps/sync endpoint 工厂；与 AuthEndpoints / HomeEndpoints 同模式.
//
// 提为独立 enum：避免 StepRepository 内 inline `Endpoint(...)` 字面量散落；
// 当 V1 §6.1 path / requiresAuth 改时（理论上不会，已冻结），仅改本文件一处.
//
// path 必须**含** `/api/v1` 前缀（与 AuthEndpoints 同模式 —— APIClient 用 host-only baseURL，
// 拼出的 URL 是 baseURL + endpoint.path）.

import Foundation

public enum StepsEndpoints {
    /// POST /api/v1/steps/sync —— 步数同步（V1 §6.1）.
    /// requiresAuth=true：自动经过 APIClient 的 token 注入（Story 5.3）+ AuthBoundaryAPIClient 装饰器.
    public static func sync(_ request: StepsSyncRequest) -> Endpoint {
        Endpoint(
            path: "/api/v1/steps/sync",
            method: .post,
            body: AnyEncodable(request),
            requiresAuth: true
        )
    }
}
```

**新建** `iphone/PetApp/Features/Home/Repositories/StepRepository.swift`：

```swift
// StepRepository.swift
// Story 8.5 AC2: 封装 POST /api/v1/steps/sync 调用；与 AuthRepository / HomeRepository 同模式.
//
// 与 HomeRepository（Story 5.5）同模式：协议方法返回原始 wire DTO；APIError 原样透传.
//
// 注入的 APIClient 是 container.apiClient —— 已被 ADR-0008 v2 AuthBoundaryAPIClient 包装；
// 业务请求 401 会自动触发全局 cold-start (清 SessionStore + 重跑 bootstrap)，StepRepository 完全无感.
//
// `DefaultStepRepository` 是 `struct`：value type，无内部状态，构造廉价；与 DefaultHomeRepository 同模式.

import Foundation

public protocol StepRepositoryProtocol: Sendable {
    /// 调 POST /api/v1/steps/sync 同步步数.
    /// - Parameter request: V1 §6.1 钦定的请求 wire DTO（syncDate / clientTotalSteps / motionState / clientTimestamp）
    /// - Returns: StepsSyncResponse（含 acceptedDeltaSteps + 最新 stepAccount 三字段）
    /// - Throws: APIError.business(1001 / 1002 / 1005 / 3001 / 1009) / APIError.network / APIError.unauthorized
    ///           / APIError.missingCredentials / APIError.decoding
    func syncSteps(_ request: StepsSyncRequest) async throws -> StepsSyncResponse
}

public struct DefaultStepRepository: StepRepositoryProtocol {
    private let apiClient: APIClientProtocol

    public init(apiClient: APIClientProtocol) {
        self.apiClient = apiClient
    }

    public func syncSteps(_ request: StepsSyncRequest) async throws -> StepsSyncResponse {
        try await apiClient.request(StepsEndpoints.sync(request))
    }
}
```

**关键设计决策**：

1. **`StepsEndpoints.sync(_:)` 接 `StepsSyncRequest` 参数**：与 `AuthEndpoints` 路径处理 body 风格一致；body 通过 `AnyEncodable(request)` 类型擦除装入 Endpoint。**禁止**让 caller 自己构造 `Endpoint(...)`（散落 path 字面量是 5.5 / 4.x 已避免的反模式）。
2. **`requiresAuth: true`**：必须；server `/steps/sync` 走 auth 中间件（V1 §6.1 元信息表）。**禁止** false（会让 APIClient 不注入 Bearer token → 1001 errcode 立即返回）。
3. **`StepRepositoryProtocol: Sendable`**：与 `HomeRepositoryProtocol` 同精神；Swift 6 strict concurrency 跨 actor 调 protocol 必须。
4. **错误透传，**不**包装**：repo 不 catch APIError；UseCase 层（AC3）也透传给 TriggerService（AC5），TriggerService 决定"失败 → 不更新 stepAccount，下次再试"（AC5 钦定）。
5. **独立 `DefaultStepRepository` 而非和 `DefaultHomeRepository` 合并**：边界清晰——HomeRepository 走 GET /home（domain：首屏聚合），StepRepository 走 POST /steps/sync（domain：步数同步动作）；与 iOS 架构 §5.4 钦定的 Repository 分类（StepRepository 单独列）一致。

### AC3 — `SyncStepsUseCase`（healthProvider → repository → appState）

**新建** `iphone/PetApp/Features/Home/UseCases/SyncStepsUseCase.swift`：

```swift
// SyncStepsUseCase.swift
// Story 8.5 AC3: 步数同步 UseCase（业务编排：healthProvider → repository → appState）.
//
// 职责（epics.md AC 行 1572-1576）:
//   1. 读 HealthProvider 当日累计步数（healthProvider.readDailyTotalSteps(date:)）
//   2. 拼请求体（StepsSyncRequest：syncDate / clientTotalSteps / motionState / clientTimestamp）
//   3. 调 repository.syncSteps(_:) 拿响应
//   4. 成功 → 调 appState.applySyncedStepAccount(_:) 写入 AppState.currentStepAccount（addendum 钦定，**不**写 ViewModel）
//   5. 失败 → throw 透传给上层 TriggerService（TriggerService 决定不阻塞 UI / 下次再试）
//
// **不**做的事:
//   - 不接 ErrorPresenter（背景同步失败不弹 toast；与 epics.md AC 行 1576 "不阻塞 UI" 钦定一致）
//   - 不做 retry / 指数退避（TriggerService 层用"下次定时器到达"模式自然兜底；YAGNI）
//   - 不申请 HealthKit 权限（caller 决定是否走 healthProvider.requestPermission()；本 UseCase 仅 readDailyTotalSteps，
//     未授权时 HealthProviderImpl 抛 .permissionDenied → 透传 → TriggerService log warning + 不更新 stepAccount）
//   - 不读 ViewModel.petState（与 8.4 视觉层解耦；motionState 由调用方注入，详见 AC4 motionStateLatch 路径）

import Foundation

public protocol SyncStepsUseCaseProtocol: Sendable {
    /// 执行一次步数同步（4 步流程：read → build → sync → apply）.
    /// - Parameter motionState: 当前活动状态（由 caller TriggerService 持的 motionStateLatch 提供；详见 AC4）
    /// - Throws: HealthProviderError / APIError（原样透传给 caller）
    func execute(motionState: MotionState) async throws
}

public struct DefaultSyncStepsUseCase: SyncStepsUseCaseProtocol {
    private let healthProvider: HealthProvider
    private let repository: StepRepositoryProtocol
    private let appState: AppState
    private let dateProvider: DateProvider

    public init(
        healthProvider: HealthProvider,
        repository: StepRepositoryProtocol,
        appState: AppState,
        dateProvider: DateProvider
    ) {
        self.healthProvider = healthProvider
        self.repository = repository
        self.appState = appState
        self.dateProvider = dateProvider
    }

    public func execute(motionState: MotionState) async throws {
        // Step 1: 读 HealthKit 当日累计步数（按 dateProvider.now() 取本机时区当日）.
        // healthProvider 内部用 HKStatisticsQuery sumQuantity；与 8.1 协议契约一致.
        let now = dateProvider.now()
        let totalSteps = try await healthProvider.readDailyTotalSteps(date: now)

        // Step 2: 拼请求体.
        // - syncDate: 本机时区当日 YYYY-MM-DD（DateProvider 钦定；不用 ISO8601DateFormatter 避免跨时区漂移）
        // - motionState: MotionState.wireValue（AC4 extension）→ 1 / 2 / 3
        // - clientTimestamp: Int64 毫秒（与 V1 §6.1.4 一致）
        let request = StepsSyncRequest(
            syncDate: dateProvider.todayString(),
            clientTotalSteps: totalSteps,
            motionState: motionState.wireValue,
            clientTimestamp: dateProvider.nowMillis()
        )

        // Step 3: 调 server.
        // - 网络成功 + 业务 code=0 → 解 StepsSyncResponse；
        // - 网络失败 / 业务 code != 0 → 抛 APIError（透传给 TriggerService）
        let response = try await repository.syncSteps(request)

        // Step 4: 写 AppState（与 addendum 钦定一致；**不**写 ViewModel）.
        // 用 AppState 提供的 mutation 入口（AC7：applySyncedStepAccount(_:)）;
        // 转 wire DTO → domain HomeStepAccount（与 5.5 HomeData(from:) 同精神，但单字段；Story 21.x ChestOpen 后续也按此模式）.
        let domainAccount = HomeStepAccount(
            totalSteps: response.stepAccount.totalSteps,
            availableSteps: response.stepAccount.availableSteps,
            consumedSteps: response.stepAccount.consumedSteps
        )
        await MainActor.run {
            appState.applySyncedStepAccount(domainAccount)
        }
    }
}
```

**关键设计决策（必须严格遵守）**：

1. **`execute(motionState: MotionState)` 接 motionState 参数**：UseCase 不订阅 MotionProvider —— motionState 由 caller `StepSyncTriggerService` 通过 `motionStateLatch` 提供（AC4 / AC5）。**禁止**给 UseCase 传 MotionProvider 引用（UseCase 单次执行；订阅是 TriggerService 长生命周期职责）。
2. **`appState: AppState`（concrete final class）而非协议**：AppState 是 final class（不可继承；ADR-0010 §3.1 钦定）；测试场景通过 `AppState()` 直接 new 实例（与 5.5 / 37.4 测试模式一致）。**禁止**抽 `AppStateProtocol`（破坏 ADR-0010 §3.1 类型契约）。
3. **`await MainActor.run { appState.applySyncedStepAccount(_:) }`**：AppState 是 `@MainActor final class`；从 UseCase（不在 main actor）写 @Published 必须跨 actor hop。**禁止**直接 `appState.applySyncedStepAccount(_:)`（编译错：non-isolated context 调 main-actor method）。
4. **错误**全部透传**，不 catch / 不 wrap**：`HealthProviderError.permissionDenied` / `APIError.business(3001 ..)` / `APIError.network` 任一种都让 TriggerService 决策（TriggerService 钦定 silent 失败 + 下次再试）。**禁止**在 UseCase 里 catch + 转 NSError（破坏 8.1 / 4.x 错误三态契约）。
5. **`DateProvider` 抽象**：测试场景需要锁固定时间（验 syncDate / clientTimestamp）；不能直接用 `Date()`。AC6 钦定 `DateProvider` 协议。
6. **`StepsSyncResponse.stepAccount → HomeStepAccount` 转换**：用 `HomeStepAccount(totalSteps:availableSteps:consumedSteps:)` 直接构造；与 Story 5.5 `HomeData(from:)` 内 `HomeStepAccount(totalSteps: response.stepAccount.totalSteps, ...)` 同模式（行 68-72）。
7. **不写 `acceptedDeltaSteps` 到 AppState**：epics.md / addendum 都没钦定要持久化"本次入账增量"；仅写 `stepAccount`（与 ADR-0010 §3.2 白名单 7 字段对齐）。`acceptedDeltaSteps` 仅作为响应字段被丢弃（如未来需要"开箱前同步成功提示"，由 caller 接 useCase 返回值；本 story `execute(...)` 返 Void，YAGNI）。
8. **`SyncStepsUseCaseProtocol: Sendable`**：与 `LoadHomeUseCaseProtocol` 同精神；Swift 6 strict concurrency 跨 actor 调 protocol 必须。

#### AC4.1 — `MotionState.wireValue` extension

`StepSyncTriggerService`（AC5）订阅 `MotionProvider.startUpdates(handler:)` 持最近一次 mapper 结果作为 latch；每次同步触发时读 latch 拼请求 motionState 字段：

// StepSyncTriggerService.swift（仅展示 motionStateLatch 字段 + 订阅链路；完整代码见 AC5）

**关键设计决策**：

### AC5 — `StepSyncTriggerService`（4 触发器 + in-flight gate）

**新建** `iphone/PetApp/Features/Home/Services/StepSyncTriggerService.swift`：

```swift
// StepSyncTriggerService.swift
// Story 8.5 AC5: 步数同步触发器服务（4 触发器 + in-flight gate + motionState latch）.
//
// 4 触发时机（epics.md AC 行 1567-1571）:
//   1. App 启动后进入主界面（RootView .task / .onAppear 同步路径调 service.start() 触发首次）
//   2. App 从后台回到前台（RootView .onChange(of: scenePhase) 监听 .active 触发）
//   3. 主界面停留期间每 5 分钟定时同步一次（service.start() 内启动 Timer / Task 循环）
//   4. 手动触发接口（service.triggerManual() 公开入口；Story 21.x ChestOpenUseCase 节点 7 用）
//
// in-flight gate（epics.md AC 行 1577 钦定 "同步不重叠"）:
//   - isSyncing flag 防重叠：当前 sync in-flight 时新触发被忽略（不排队）
//   - 失败不阻塞 UI（背景同步；下次定时器到达再试）
//
// motionStateLatch（详 AC4.2）:
//   - service 持 MotionProvider strong 引用 + 订阅 startUpdates；handler 内更新 latch
//   - syncSteps 触发时读 latch 拼请求 motionState 字段
//
// 生命周期:
//   - 由 AppContainer 单例化；与 App 同生命周期
//   - start() 由 RootView .task 内调（启动 + 定时器循环）
//   - stop() 由 RootView 离开 / scenePhase .background 调（停定时器；scenePhase .active 重新 start）
//   - deinit 时 stopUpdates motionProvider 防泄漏（与 8.4 HomeViewModel.deinit 同模式）
//
// 性能 / 资源约束:
//   - Timer 周期 5 分钟（300 秒）—— 不可配置（YAGNI；prod 默认值锚定 epics.md AC 钦定）
//   - Timer 用 `Task.sleep(nanoseconds:)` 循环，不用 Foundation `Timer`（@MainActor 友好 + 可被 cancel）

import Foundation

@MainActor
public final class StepSyncTriggerService {

    // MARK: - Dependencies

    private let syncStepsUseCase: SyncStepsUseCaseProtocol
    private let motionProvider: MotionProvider

    // MARK: - State

    /// in-flight gate（epics.md AC 行 1577）.
    /// 当前 sync 进行中时新触发被忽略（不排队）.
    private var isSyncing = false

    /// motionState latch（AC4.2）：MotionProvider 推送 activity 后通过 mapper 转换缓存的最新值.
    /// 默认 `.rest`：handler 未触发 / 未授权时合法兜底（V1 §6.1.3 motionState=1 合法）.
    private var motionStateLatch: MotionState = .rest

    /// 定时器循环 task；start() 启动；stop() / deinit 取消.
    private var timerTask: Task<Void, Never>?

    /// 是否已订阅 motionProvider（防重复订阅；与 8.4 bind(motionProvider:) 同模式）.
    private var hasStartedMotionUpdates = false

    /// 是否已启动定时循环（防 .scenePhase .active 多次触发重复启动 timer）.
    private var hasStartedTimer = false

    // MARK: - Init

    public init(
        syncStepsUseCase: SyncStepsUseCaseProtocol,
        motionProvider: MotionProvider
    ) {
        self.syncStepsUseCase = syncStepsUseCase
        self.motionProvider = motionProvider
    }

    // MARK: - Public API

    /// 启动触发器：订阅 motionProvider + 启动 5 分钟定时循环 + 触发首次同步（epics.md AC 行 1568）.
    /// 由 RootView .task 在主界面就绪后调；幂等（多次调安全）.
    public func start() {
        startMotionLatchIfNeeded()
        startTimerIfNeeded()
        Task { [weak self] in
            await self?.performSync(reason: .launch)
        }
    }

    /// 回前台触发（epics.md AC 行 1569）.
    /// 由 RootView .onChange(of: scenePhase) `.active` 边沿调.
    public func triggerForeground() {
        Task { [weak self] in
            await self?.performSync(reason: .foreground)
        }
    }

    /// 手动触发（epics.md AC 行 1571；节点 7 ChestOpenUseCase 调用入口）.
    /// 等待同步完成（与 launch / foreground / timer 不等待不同 —— caller 需要在同步完成后再继续开箱）.
    public func triggerManual() async {
        await performSync(reason: .manual)
    }

    /// 停止触发器：cancel 定时器 + stopUpdates motionProvider.
    /// 由 RootView .onChange(of: scenePhase) `.background` 边沿调（节省电量；下次 .active 重新 start）.
    /// **不**清 isSyncing：让正在 in-flight 的 sync 自然完成；scenePhase .active 时再触发新 sync 时
    /// isSyncing 自然 false（前一个已完成）.
    public func stop() {
        timerTask?.cancel()
        timerTask = nil
        hasStartedTimer = false
        // 注：不调 motionProvider.stopUpdates() —— motionStateLatch 在前后台切换间保留有意义
        // （前台时立即用上次后台的 latch 值；不需要重新等 motion event 到达）.
    }

    // MARK: - Private

    private enum SyncReason: String {
        case launch
        case foreground
        case timer
        case manual
    }

    private func performSync(reason: SyncReason) async {
        guard !isSyncing else {
            // 重叠忽略（epics.md AC 行 1577 + 1583）.
            return
        }
        isSyncing = true
        defer { isSyncing = false }

        do {
            try await syncStepsUseCase.execute(motionState: motionStateLatch)
        } catch {
            // 失败不阻塞 UI（epics.md AC 行 1576）；下次触发再试.
            // 节点 3 阶段不做 logger framework；失败被 silently 吞掉是 by design.
            // future: 接 logger 后此处 log warning（与 server 7.3 防作弊 log warning 同精神）.
        }
    }

    private func startMotionLatchIfNeeded() {
        guard !hasStartedMotionUpdates else { return }
        hasStartedMotionUpdates = true
        motionProvider.startUpdates { [weak self] activity in
            // closure 在 OperationQueue.main 触发（8.2 MotionProviderImpl 钦定）；
            // mapper.map 是 pure function 同步调用；写 actor-isolated 字段必须 main actor.
            let mapped = MotionStateMapper.map(activity)
            Task { @MainActor [weak self] in
                self?.motionStateLatch = mapped
            }
        }
    }

    private func startTimerIfNeeded() {
        guard !hasStartedTimer else { return }
        hasStartedTimer = true
        timerTask = Task { [weak self] in
            // 定时循环：每 5 分钟一次（epics.md AC 行 1570）.
            // 用 Task.sleep + while !Task.isCancelled，不用 Foundation Timer
            //   （Timer 跨 actor / cancel 复杂；Task 模型在 Swift 6 strict concurrency 下更干净）.
            while !Task.isCancelled {
                do {
                    try await Task.sleep(nanoseconds: 5 * 60 * 1_000_000_000)
                } catch {
                    // CancellationError → 退出循环；其它 error 不应发生（sleep 仅抛 CancellationError）.
                    return
                }
                guard !Task.isCancelled else { return }
                await self?.performSync(reason: .timer)
            }
        }
    }

    deinit {
        timerTask?.cancel()
        // motionProvider.stopUpdates() 不能在 deinit 内调（@MainActor isolated；deinit nonisolated）.
        // service 单例与 App 同生命周期；deinit 几乎不发生.
        // 测试场景由 do-block ARC + stop() 显式释放（详见 AC10 单测）.
    }
}
```

**关键设计决策（必须严格遵守）**：

1. **`@MainActor final class`**：与 AppState / SessionStore 同模式；保 `isSyncing` / `motionStateLatch` / `hasStartedMotionUpdates` / `hasStartedTimer` 等可变字段在 main actor 上的串行访问安全。**禁止**抽 actor 类型（actor 跨 hop 成本高 + 与 RootView UI thread 调用语义不一致）。
2. **`isSyncing` flag + `defer` 释放**：不用 lock / NSLock；@MainActor 已保证串行——同时间只有一个 task 能进 critical section。`defer { isSyncing = false }` 保证抛错路径也能释放（与 RootView .task / .onAppear cancel 路径同精神）。
3. **`Task.sleep(nanoseconds:)` 而非 Foundation Timer**：Foundation Timer 跨 actor + cancel 在 Swift 6 strict concurrency 下脏（必须 RunLoop.current 配合）；`Task` + `try await Task.sleep` 是 Swift Concurrency 推荐路径。每次 sleep 后 `Task.isCancelled` 双重保护防 stop 后多走一圈。
4. **`5 * 60 * 1_000_000_000`** 常量 inline 而非常量化：YAGNI；epics.md AC 钦定的 5 分钟是产品契约（不可调）。**禁止**抽 `static let timerInterval` 让外部可配置——一抽就需要测试覆盖配置变更场景，徒增复杂度。
5. **`triggerManual()` async 等待**（与 `triggerForeground()` `start()` 不等待不同）：Story 21.x 开箱前调用方需要"同步完成后开始开箱事务"；返 Void 但 await 自然给 caller 等待入口。**禁止**让 manual 也 fire-and-forget（破坏开箱前同步语义）。
6. **`stop()` 不调 `stopUpdates`**：motion 订阅长期持有 latch；前后台切换之间 `motionStateLatch` 不需要"重置 .rest"——上次后台时的最后 motion 状态在 .active 时立即可用，避免重启订阅 race。**仅** cancel timer + reset hasStartedTimer flag。
7. **`SyncReason` enum 私有 + 仅作为代码语义注释用**：未来如接 logger 可作为 log key；当前仅 internal 区分（log warning 时区分 reason 是 future 扩展点；本 story 内部 only）。
8. **`deinit` 仅 cancel timer**：service 是单例；测试场景靠 do-block ARC 自然释放；不调 `motionProvider.stopUpdates()` —— deinit nonisolated 调 @MainActor isolated provider 方法编译错；motionProvider 是 AppContainer 单例（全 App 同生命周期），不调 stopUpdates 也不会泄漏（与 8.4 ViewModel.deinit 钦定 stopUpdates 不同——8.4 ViewModel 是 short-lived 测试场景）。
9. **`hasStartedMotionUpdates` + `hasStartedTimer` 双 flag**：避免 `start()` / `triggerForeground()` 多次调用重复订阅 / 重复启 timer（与 8.4 bind(motionProvider:) 同精神）。
10. **`performSync` 内部 catch 全部错误并 silent 吞掉**：节点 3 阶段不接 logger；epics.md AC 行 1576 钦定 "失败不阻塞 UI"。**禁止**接 ErrorPresenter 弹 toast——背景同步失败弹 toast 会严重打扰用户。`triggerManual` 也吞错——caller（Story 21.x ChestOpenUseCase）会自己 catch 开箱本身的错误，同步失败时 stepAccount 没更新但开箱仍能继续（开箱失败时再处理）。

### AC6 — `DateProvider` 协议 + impl + mock

**新建** `iphone/PetApp/Core/DateProvider.swift`：

```swift
// DateProvider.swift
// Story 8.5 AC6: 时间抽象（让单测可锁固定时间，验 syncDate / clientTimestamp）.
//
// 用途：SyncStepsUseCase 拼请求体时需要:
//   - syncDate: 本机时区当日 YYYY-MM-DD（不能直接用 Date() 派生 — 单测无法验确定值）
//   - clientTimestamp: 当前毫秒时间戳（同上）
//   - now(): healthProvider.readDailyTotalSteps(date:) 入参
//
// 与 8.1 / 8.2 / 8.3 不同（那几个 story 的 system adapter 不需要时间抽象 —— 单测注入步数 / activity 序列即可）.
// 本 story 单测必须覆盖"跨自然日"等时间敏感场景（epics.md AC 行 1582；定时器单测）；DateProvider 是必备.

import Foundation

public protocol DateProvider: Sendable {
    /// 当前时间（healthProvider.readDailyTotalSteps 入参）.
    func now() -> Date
    /// 当前毫秒时间戳（V1 §6.1.4 clientTimestamp 字段；Int64 = Unix ms）.
    func nowMillis() -> Int64
    /// 本机时区当日 "YYYY-MM-DD"（V1 §6.1.1 syncDate 字段；严格 10 字符）.
    func todayString() -> String
}

public struct DefaultDateProvider: DateProvider {
    public init() {}

    public func now() -> Date { Date() }

    public func nowMillis() -> Int64 {
        Int64(Date().timeIntervalSince1970 * 1000)
    }

    public func todayString() -> String {
        // 本机时区当日 YYYY-MM-DD（与 server 钦定 [today-2d, today+2d] 容忍窗口对齐；
        //   不二次转 UTC，避免跨时区漂移；详见 V1 §6.1.1 + GAP E）.
        let formatter = DateFormatter()
        formatter.calendar = Calendar(identifier: .gregorian)
        formatter.dateFormat = "yyyy-MM-dd"
        formatter.locale = Locale(identifier: "en_US_POSIX")  // 防系统区域影响（i18n locale 安全）
        formatter.timeZone = TimeZone.current  // 本机时区（与 V1 钦定一致）
        return formatter.string(from: Date())
    }
}
```

**关键设计决策**：

1. **`DateProvider: Sendable` + struct 实装**：value type；不接 mock 子类（`DefaultDateProvider` 是 struct 不可继承）；测试场景用 `MockDateProvider` 实装协议（单测文件内）。
2. **`todayString()` 用 `en_US_POSIX` locale**：防设备语言（如阿拉伯语 / 泰语）影响数字字符（i18n 安全）；与 Apple HIG 推荐路径对齐。
3. **`nowMillis()` 直接 `Int64(Date().timeIntervalSince1970 * 1000)`**：iOS 自 2001-01-01 起 timestamp seconds 是 Double（`Date.timeIntervalSinceReferenceDate`），转 Unix epoch（1970）走 `timeIntervalSince1970`；乘 1000 得毫秒；Int64 cast 安全。
4. **不抽 `Calendar` / `TimeZone` 入参**：本期 V1 钦定按本机时区算；测试场景通过 `MockDateProvider` 注入固定 todayString 即可（不需要在 production 接口暴露 timezone 控制）。
5. **不引第三方时间库**（如 swift-clocks）：YAGNI；ADR-0002 锁定零第三方 lib；自实装 ~15 行足够。

/// 不可能为 nil，schema 已冻结）.
public func applySyncedStepAccount(_ stepAccount: HomeStepAccount) {
    self.currentStepAccount = stepAccount
}
```

**关键设计决策**：

1. **`applySyncedStepAccount` 而非直接公开 `currentStepAccount` setter**：与 ADR-0010 §3.3 钦定的"统一 hydrate / mutation 入口"一致。`@Published var currentStepAccount` 已是 public（Story 37.4 落地），技术上 caller 可以直接 `appState.currentStepAccount = ...`，但本入口让代码意图清晰 + 未来若需要在 mutation 时附带副作用（如 log / analytics）有单点位。
2. **不在方法内做 diff 检查**（如 `if currentStepAccount != stepAccount { ... }`）：YAGNI；@Published 自身在新 value == old value 时仍会 emit（`Equatable` 比较成本低；订阅方 SwiftUI 也会 dedup）。
3. **不接 `oldStepAccount` 回调 / 不接 `acceptedDeltaSteps` 参数**：本入口仅做 stepAccount 替换；UI 展示的是 `availableSteps` 数值（HomeView 已订阅 `appState.currentStepAccount?.availableSteps`，自然刷新）。
4. **方法位置在 `setCurrentRoomId` 附近**（与 ADR-0010 §3.5 钦定"显式 setter"段落同精神）：保 mutation 入口聚合便于后续 audit。
5. **不动 `applyHomeData` 内的 `currentStepAccount = data.stepAccount` 路径**：GET /home 启动期 hydrate 仍走 applyHomeData（全 7 字段）；本入口仅做"同步动作"专用单字段更新。

### AC9 边界澄清 — 多订阅 motion 行为（与 Story 8.2 协议契约对齐）

**关键预期**：`MotionProviderImpl.startUpdates(handler:)` 在第二次调用时**覆盖**第一次的 handler（与 8.2 协议契约 `motion-handler-invoke-must-be-in-lock.md` 钦定的 generation token 同精神 — 后启动者赢）。

**本 story 影响**：8.4 `homeViewModel.bind(motionProvider:)` 与 8.5 `stepSyncTriggerService.start() → startMotionLatchIfNeeded()` 都调 `motionProvider.startUpdates`；后调用者覆盖前者意味着**只有最后一个 caller 收到 activity event**。

**红线分析**：
- 视觉层（8.4 PetSpriteView）需要每次 motion event 立即派生 petState；
- 同步层（8.5 motionStateLatch）需要每次 motion event 更新 latch；
- 两路同时存活时只有一个能收到 event → **冲突！**

**缓解**（本 story 落地选择）：

> **option A（推荐）**：让 `StepSyncTriggerService` 复用 `HomeViewModel.petState` 作为 motion 来源：
>   - service 不自己 startUpdates；构造时注入 `homeViewModel: HomeViewModel`（而非 motionProvider）；
>   - service 内 `motionStateLatch` 字段改为 computed property `var motionStateLatch: MotionState { homeViewModel.petState }`；
>   - 仅 8.4 一处订阅 motionProvider；8.5 借用 ViewModel @Published petState（已落地）。
>
> **option A 反对意见**：与 epics.md / 8.4 Dev Notes 行 803 钦定 "ViewModel 与同步触发器解耦" 冲突 — 本 story 应让两层独立。
>
> **option B（备选）**：扩展 `MotionProvider` 协议成 `addObserver(_ handler:) -> ObserverToken` + `removeObserver(token:)` —— 多订阅模型。
>
> **option B 反对意见**：违反本 story 红线第 14 条 "不给 MotionProvider 加新方法"；需新开 story 改 8.2 协议。
>
> **option C（实施期再 spike）**：让 StepSyncTriggerService **不订阅** MotionProvider —— 接受"motionState 字段固定 .rest（=1）"作为节点 3 阶段简化方案；epics.md 明确钦定 motionState 字段 must 上传，但 server 端实际仅作为日志审计用，不影响差值计算。
>
> **option C 反对意见**：上传一律 .rest 不准确；产品语义"用户跑步时同步带 motionState=3" 失准；epics.md AC 行 1574 钦定字段 = "motionState"（隐含字段值正确）。

**实施期 dev 决策（请在 dev-story 阶段第一时间决定）**：

- **首选 option A**：dev-story 落地时把 service 字段从 `motionProvider: MotionProvider` 改为 `homeViewModel: HomeViewModel`（构造注入），服务 init 不接 motionProvider；motionStateLatch 改为 computed property 读 viewModel.petState. 这与 8.4 视觉层共用 motion 订阅，避免重复订阅冲突。**违反** epics.md 钦定的"解耦"指示，但**符合**协议 startUpdates 单订阅契约。
- **若坚持 option A 的"解耦"反对**：在 dev-story 落地前必须先扩 MotionProvider 协议（开新 story 8-6 或合并到 epic-8 retrospective 的 follow-up；本 story HALT）。
- **不建议 option C**（产品语义失准）+ **不建议** "service 自己 startUpdates 同 8.4 共存"（**实质上后调者覆盖前者** —— 本 story 测试通过但运行期 8.4 PetSpriteView 不再随活动切换，破坏 8.4 已 done feature）。

**dev 落地方案锁定（option A）**：

- `StepSyncTriggerService.init(syncStepsUseCase:homeViewModel:)`（不接 motionProvider）；
- 移除 `motionStateLatch` 字段、`startMotionLatchIfNeeded()` 方法、`hasStartedMotionUpdates` flag；
- `performSync(reason:)` 内读 `homeViewModel.petState` 拼请求 motionState；
- AppContainer `makeStepSyncTriggerService(appState:homeViewModel:)` 增 homeViewModel 入参；
- RootView `ensureStepSyncWired()` 调时传 `homeViewModel`（已是 @StateObject）.

> **本节是 dev 必读**：dev-story 落地时**直接走 option A**；不要复制本 AC5 / AC8 内 `motionStateLatch` / `startMotionLatchIfNeeded` 相关代码。AC5 / AC8 内保留这些代码段是**历史推导上下文**（让 dev 理解为何不能"service 自己 startUpdates"）；最终落地代码以 option A 锁定为准。

final class StepSyncTriggerServiceTests: XCTestCase {

    // happy: start() 触发首次同步（epics.md AC 行 1579）
    func testStart_triggersInitialSync() async {
        let useCaseMock = MockSyncStepsUseCase()
        let viewModel = HomeViewModel()
        viewModel.petState = .walk
        let service = StepSyncTriggerService(
            syncStepsUseCase: useCaseMock,
            homeViewModel: viewModel
        )

        service.start()
        // 等 Task spawn 完成
        await Task.yield()
        await Task.yield()

        XCTAssertEqual(useCaseMock.invocations.count, 1, "start() 应触发一次同步")
        XCTAssertEqual(useCaseMock.invocations.first, .walk, "应使用 viewModel.petState")
    }

    // happy: triggerForeground() 触发同步（epics.md AC 行 1580）
    func testTriggerForeground_triggersSync() async {
        let useCaseMock = MockSyncStepsUseCase()
        let viewModel = HomeViewModel()
        let service = StepSyncTriggerService(syncStepsUseCase: useCaseMock, homeViewModel: viewModel)

        service.triggerForeground()
        await Task.yield()
        await Task.yield()

        XCTAssertEqual(useCaseMock.invocations.count, 1)
    }

    // happy: triggerManual() 触发同步并等待完成（epics.md AC 行 1582）
    func testTriggerManual_awaitsSyncCompletion() async {
        let useCaseMock = MockSyncStepsUseCase()
        let viewModel = HomeViewModel()
        let service = StepSyncTriggerService(syncStepsUseCase: useCaseMock, homeViewModel: viewModel)

        await service.triggerManual()

        XCTAssertEqual(useCaseMock.invocations.count, 1, "triggerManual 应等待同步完成")
    }

    // edge: in-flight 时新触发被忽略（epics.md AC 行 1577 + 1583）
    func testTriggerWhileInFlight_isIgnored() async {
        let useCaseMock = MockSyncStepsUseCase()
        useCaseMock.executeDelayMs = 50  // 让第一次 sync 有 50ms 窗口
        let viewModel = HomeViewModel()
        let service = StepSyncTriggerService(syncStepsUseCase: useCaseMock, homeViewModel: viewModel)

        async let first: Void = service.triggerManual()
        // 在第一次 in-flight 时立即触发第二次
        service.triggerForeground()
        service.triggerForeground()
        await first
        await Task.yield()

        XCTAssertEqual(useCaseMock.invocations.count, 1,
            "in-flight 时新触发应被忽略；总同步次数应为 1")
    }

    // edge: useCase 抛错 → 不阻塞 UI 不传染下次触发（epics.md AC 行 1584）
    func testSyncFails_doesNotBlockNextTrigger() async {
        let useCaseMock = MockSyncStepsUseCase()
        useCaseMock.stubError = APIError.network(underlying: NSError(domain: "", code: -1))
        let viewModel = HomeViewModel()
        let service = StepSyncTriggerService(syncStepsUseCase: useCaseMock, homeViewModel: viewModel)

        await service.triggerManual()  // 第一次失败（不抛出，silent 吞掉）
        XCTAssertEqual(useCaseMock.invocations.count, 1)

        useCaseMock.stubError = nil  // 第二次成功
        await service.triggerManual()
        XCTAssertEqual(useCaseMock.invocations.count, 2,
            "第一次失败不应阻塞第二次触发")
    }

    // happy: stop() 后 timer 不再触发（验 cancel 路径）
    func testStopCancelsTimer() async {
        let useCaseMock = MockSyncStepsUseCase()
        let viewModel = HomeViewModel()
        let service = StepSyncTriggerService(syncStepsUseCase: useCaseMock, homeViewModel: viewModel)

        service.start()
        await Task.yield()
        await Task.yield()
        let invocationsAfterStart = useCaseMock.invocations.count

        service.stop()
        // sleep 略多于 timer 触发时机的局部窗口（不能等 5min；仅验证 cancel 路径不抛错）
        try? await Task.sleep(nanoseconds: 50_000_000)  // 50ms

        XCTAssertEqual(useCaseMock.invocations.count, invocationsAfterStart,
            "stop() 后无新增 invocations（cancel 生效）")
    }
}

// MARK: - Test Doubles

final class MockSyncStepsUseCase: SyncStepsUseCaseProtocol, @unchecked Sendable {
    var invocations: [MotionState] = []
    var stubError: Error?
    var executeDelayMs: UInt64 = 0

    func execute(motionState: MotionState) async throws {
        invocations.append(motionState)
        if executeDelayMs > 0 {
            try? await Task.sleep(nanoseconds: executeDelayMs * 1_000_000)
        }
        if let stubError {
            throw stubError
        }
    }
}
```

**关键设计决策**：

1. **`MockSyncStepsUseCase` 内嵌测试文件而非抽到 helper**：本 service 测试独有；与 `MockStepRepository` / `MockDateProvider` 在 SyncStepsUseCaseTests 内嵌同模式（避免 cross-test 引用 + scope 清晰）.
2. **`@MainActor` 测试 class**：service 是 @MainActor；测试用例必须在 main actor.
3. **`await Task.yield()` 两次**：service.start() 内 `Task { ... performSync }` 派发；let task spawn → schedule → 执行需要至少 1-2 个 yield（与 8.4 单测同精神）.
4. **不测 `Task.sleep(5min)`**：本测试不验证 5min 定时（实跑会超时）；验证 cancel 路径 + start/foreground/manual 三入口语义即可。"5min 触发"由 epic-9 跨端 e2e 集成阶段验证（实跑 5min wait + log 检查）.
5. **`testTriggerWhileInFlight_isIgnored`** 用 50ms `executeDelayMs`：让第一次 sync 有 in-flight 窗口；其它触发立即调时刚好命中 isSyncing flag.
6. **断言 `useCaseMock.invocations.count == 1`**：核心契约 "不重叠"；其它入参（reason / motionState）作为辅助断言.
7. **`testStopCancelsTimer`** 不验证 5min 后 timer 触发（不可行）；仅验证 stop 后短窗口内无新 invocations + 不抛错（cancel 路径正确性）.
8. **测试文件路径 `Services/StepSyncTriggerServiceTests.swift`**：与 production 路径 `Features/Home/Services/StepSyncTriggerService.swift` 对齐.

**关键设计决策（必须严格遵守）**：

1. **launch arg 命名前缀 `UITEST_MOCK_*`**：与 8.4 / 5.5 / 37.x 既有 `UITEST_SKIP_GUEST_LOGIN` / `UITEST_FORCE_IN_ROOM` 同前缀；ProcessInfo.processInfo.environment 读路径一致.

3. **`UITestMockStepRepository`** 是 `final class` + `@unchecked Sendable`：与 `MockAPIClient` 同模式；不需 actor 同步（仅一次性同步触发，无并发写）.

5. **NSPredicate `CONTAINS[c]` "5678"**：HomeView stepBalance Label 文本格式可能是 `"5,678"` / `"5678 步"` / `"5678"`；用 contains 而非精确匹配避免 i18n 形态偏差.

7. **不模拟 server 错误路径**（如 3001）：节点 3 阶段集成测试覆盖 happy path 即可；错误路径由单测覆盖.

## Latest Tech Information（Swift 6.3 + iOS 17+ + Xcode 26）

- **`Task.sleep(nanoseconds:)`** vs **`Foundation.Timer`**：在 Swift 6 strict concurrency 下，Task.sleep 是推荐的 async-friendly 定时方案；Timer 跨 actor + cancel 模型与 RunLoop 绑定，在 @MainActor class 内使用复杂；本 story 钦定 Task.sleep 5min 循环 + while !Task.isCancelled 双重保护.
- **`await MainActor.run { ... }` vs **`Task { @MainActor in ... }`**：前者是 await/return；后者是 fire-and-forget；本 story SyncStepsUseCase.execute 在末尾 await MainActor.run（同步等 AppState 写完再返回；让 caller TriggerService.performSync 内 isSyncing flag 释放与写 AppState 在同一原子边界）.
- **`@MainActor final class` 与 `@unchecked Sendable`**：service 是 @MainActor class；测试 doubles（`MockSyncStepsUseCase` / `MockStepRepository` / `MockDateProvider`）需 `@unchecked Sendable`（与 8.1 / 8.2 / 8.4 既有 mock 同模式）.
- **`ScenePhase` API**：iOS 14+ SwiftUI；`@Environment(\.scenePhase) private var scenePhase` + `.onChange(of: scenePhase) { oldPhase, newPhase in ... }`（iOS 17+ 推荐写法）；本 story 复用 8.4 落地的 .onChange 调用块.
- **`DateFormatter` 性能**：DateFormatter 是重对象（创建成本高；缓存推荐）；本 story `DefaultDateProvider.todayString()` 内每次 new DateFormatter — 节点 3 阶段每 5min 调一次 + start/foreground/manual 偶尔调 — 性能不敏感；不抽 static cache（YAGNI）；如未来高频场景可换 ISO8601DateFormatter（thread-safe + 性能更好；但 ISO8601 fixed format 无法表达 "YYYY-MM-DD" 简短形态，需另设 outputFormat = .withFullDate；本期不引入）.
- **HealthKit 与 CoreMotion 在 iPad / 模拟器行为**：8.1 / 8.2 lesson 已揭露 — 模拟器 idle 时 motion event 30s+ 不发；HealthKit 模拟器需手动注入 sample；本 story 集成测试通过 launch arg 注入 mock 路径，不依赖系统真实 health/motion 行为.
- **`Locale(identifier: "en_US_POSIX")`**：Apple HIG 推荐路径；防系统语言（如阿拉伯、泰语）影响数字字符；本 story DateFormatter 钦定使用.
