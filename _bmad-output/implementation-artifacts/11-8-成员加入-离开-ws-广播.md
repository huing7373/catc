  - 广播范围：仅该房间内当前在线的其他 Session（不含加入者自己）—— 由 BroadcastToRoom primitive 的 fanout 路径决定（**本 story 红线偏移**：BroadcastToRoom 既有实装会 fanout 给 ListSessionsByRoomID 全量结果，**含**加入者自己的 Session（如果加入者已建立 WS 连接）；V1 §12.3 行 2063 钦定加入者自己**不应**收到自己的 `member.joined` —— 但**节点 4 阶段时序事实**：HTTP join 接口与 WS 握手是两条独立路径，加入者通常**先**建立 WS 连接（在握手期通过 §12.1 校验顺序步骤 5 `room_members` 表查询，但此刻 user 还不在该 roomID 的 `room_members` 中 → 步骤 5 校验失败 → close 4003 → 加入者 WS 连接被拒），加入者**只能**先 HTTP join → 再 WS 握手 → 此时加入者 WS 已经能通过 §12.1 校验顺序步骤 5（user 已在 room_members）→ 但**此时 broadcast 已发送结束**（HTTP join 路径事务 commit + broadcast 是同步顺序，broadcast 入队时加入者 WS 还没握手），所以**事实上加入者收不到自己的 member.joined**（race 上自然达成 V1 §12.3 行 2063 的"加入者不收自己 member.joined"语义；本 story **不**给 BroadcastToRoom 加 excludeUserID 参数 —— V1 §10.5 行 1544 r13 同模式钦定不加 excludeUserID 参数）—— 见下方"r1 关键决策 - excludeUserID 不加"段
  - `payload` 字段**完整自包含**成员展示所需的全部字段（client 收到后**可直接** append 一条完整 roster entry，不需要走 `GET /rooms/{roomId}` 二次拉取 —— 这是因为 `room.snapshot` 仅在握手时下发一次，server **不**会在 `member.joined` 之后追发 snapshot）

- **V1 §12.3 `### 成员离开` 字段表**（行 2075-2080 + 2095-2101）：

  | 字段 | 类型 | 必填 | 说明 |
  |---|---|---|---|
  | `type` | string | 必填 | 固定值 `"member.left"` |
  | `requestId` | string | 必填 | 固定 `""`（主动推送类消息） |
  | `payload.userId` | string | 必填 | 离开的成员 user 主键（BIGINT 字符串化） |
  | `ts` | number (int64) | 必填 | 服务端发送时间戳（ms） |

  **关键约束**：
  - `payload` 字段**精简**为仅 `userId`（与 `member.joined` 含 `nickname` / `avatarUrl` / `pet` 不同）—— 离开事件 client UX 不需要显示昵称（"X 离开了房间"中的 X 由 client 从已有 roster 查到 nickname；UX 文案降级为"有人离开"也可接受），减少 server 加载压力
  - 广播范围：仅该房间内当前在线的其他 Session（不含离开者自己）；离开者以 HTTP 200 响应作为 leave 完成的 authoritative signal；步骤 7 close 4007 是 server 端 best-effort cleanup
  - **触发不重复**：server 实装层应保证 `member.left` 广播严格 1:1 对应 `room_members` 行被删除事件 —— 由于唯一删行路径是 HTTP leave 事务，同一 user 在同一 leave 事件中**只触发一次** `member.left` 广播；任何 WS 断开（含心跳超时）**禁止**走"被动 leave"路径删 row + 广播 `member.left`

- **V1 §12.1 close code 4007 行钦定**（行 1720）：4007 是业务级拒绝（client **不**自动重连），reason = `"left room via HTTP"`；4007 是 best-effort cleanup signal，**不是** leave 完成的 authoritative confirmation —— HTTP 200 响应才是 authoritative。

**AC5 — `closeLeaverSession` unexported method 实装**

修改 `server/internal/service/room_service.go`（紧接 `broadcastMemberJoined` 之后；与 `roomServiceImpl` 其他方法平级）：

```go
// closeLeaverSession 实装 V1 §10.5 步骤 7（Story 11.8 引入）：
// 从 SessionManager 撤销 leaver 在该 roomID 的 Session + close 4007 reason
// "left room via HTTP"。
//
// 流程：
//  1. s.sessionMgr.ListSessionsByRoomID(ctx, roomID) 拿该 roomID 全部 active Session
//  2. 线性扫描找到 Session.UserID() == leaverUserID 的 Session（节点 4 阶段单 room
//     最多 4 user，O(N) 可接受 —— ws/session_manager.go 行 198-204 钦定单 user
//     单 session 假设由 Register 替换路径兜底）
//  3. 命中 → session.CloseWithCode(4007, "left room via HTTP") + sessionMgr.Unregister
//     —— **两步顺序**：先 CloseWithCode（写 close frame）后 Unregister（清索引）；
//     CloseWithCode 内部走 notifyClosed → Unregister 自动闭环（10.3 落地），但
//     显式 Unregister 兜底保证即使 Close 内部 race / panic 索引仍被清理（Unregister
//     idempotent，二次调用 no-op 无副作用）
//  4. 未命中（leaver 未持该 roomID 的 WS / 已断开）→ no-op + log info（合法场景）
//
// **fire-and-forget 严格语义**（V1 §10.5 步骤 7 钦定）：本方法**永远不返 error** ——
// CloseWithCode / Unregister 的 error 一律 log warn / log info 不返；caller (LeaveRoom)
// 不需要走错误分流。Session 撤销失败（leaver 未持 WS / 已断开）→ no-op，不影响
// HTTP 200 响应。
//
// **事务外严格性**：本方法在 LeaveRoom 事务 commit 成功**之后**调用，**不**包入
// txMgr.WithTx fn 内（与数据库设计 §8.7 退出房间事务边界 + V1 §10.5 步骤 7 钦定一致）。
//
// **r13 顺序约束**：本方法**必须**在 broadcastMemberLeft **之前**调用，让 broadcast
// fanout 时 ListSessionsByRoomID 返回列表自然不含 leaver Session（无需 BroadcastToRoom
// primitive 加 excludeUserID 参数）。
func (s *roomServiceImpl) closeLeaverSession(ctx context.Context, roomID, leaverUserID uint64) {
    logger := slog.Default().With(
        slog.String("component", "room-service-broadcast"),
        slog.String("event", "close.4007"),
        slog.Uint64("roomId", roomID),
        slog.Uint64("leaverUserId", leaverUserID),
    )

    sessions := s.sessionMgr.ListSessionsByRoomID(ctx, roomID)
    var target *ws.Session
    for _, sess := range sessions {
        if sess.UserID() == leaverUserID {
            target = sess
            break
        }
    }
    if target == nil {
        logger.Info("ws close 4007: leaver session not registered; skip (合法场景)")
        return
    }

    sessionID := target.SessionID()
    if err := target.CloseWithCode(4007, "left room via HTTP"); err != nil {
        // ErrSessionClosed (target 已被并发 close) → 走 Unregister 兜底（idempotent）；
        // 其他 raw error → log warn 不返
        logger.Warn("ws close 4007: CloseWithCode returned error",
            slog.String("sessionId", sessionID),
            slog.Any("error", err))
    }

    // **关键**：显式 Unregister 兜底（CloseWithCode → notifyClosed → Unregister 自动
    // 闭环已就绪，但显式调用兜底；Unregister idempotent 双调安全）
    if err := s.sessionMgr.Unregister(ctx, sessionID); err != nil {
        logger.Warn("ws close 4007: Unregister returned error",
            slog.String("sessionId", sessionID),
            slog.Any("error", err))
        return
    }
    logger.Info("ws close 4007: leaver session closed and unregistered",
        slog.String("sessionId", sessionID))
}
```

- **关键 dependency**：本方法依赖 `ws.Session.UserID()` / `ws.Session.SessionID()` / `ws.Session.CloseWithCode(int, string)` 三个 exported 方法（10.3 / 10.4 落地）；review grep `Session.UserID` / `Session.SessionID` / `Session.CloseWithCode` 在 ws 包内必为 exported method 命中（既有 ws/session.go 行 ~150 + ~407 已 ship 这三个方法，本 story 仅消费）。

## Dev Notes

### 关键架构决策

1. **service → ws 单向 import**：service 层 import ws 包是合法的单向依赖（与 ws/broadcast.go 行 40-69 钦定的 BroadcastFn 函数值注入模式一致）。**绝不**让 ws 包反向 import service —— 这会引入循环 import 编译失败。本 story `roomServiceImpl.broadcastFn` 字段类型用 `ws.BroadcastFn` 函数值（type alias），让 service 层只持有函数值不直接持有 SessionManager 实例的内部细节。

2. **`MemberJoinedPayload` 与 `SnapshotMember` 字段集合不同**：MemberJoinedPayload 含 avatarUrl 字段（V1 §12.3 行 2009 钦定），SnapshotMember **不**含 avatarUrl 字段（11.7 r1 P1 已锁定不回工）。本 story **不**给 SnapshotMember 加 avatarUrl —— 走"两个独立 struct，字段集合刻意 drift" 路径，与 V1 §12.3 Story 10.7 placeholder vs Story 11.7 真实 vs Story 11.8 member.joined 三阶段独立锚定的设计哲学一致。

3. **Pet 字段类型一致**：MemberJoinedPayload.Pet 用 `*SnapshotPet` pointer 类型 —— 与 SnapshotMember.Pet 同模式（11.7 r1 P1 钦定），nil → JSON null（pet-less 路径）+ 非 nil → JSON object（含 petId / currentState）。**不**新增 MemberJoinedPet struct 重复 SnapshotPet 字段集合（YAGNI；两个 payload struct 共用 SnapshotPet 节省维护成本）。

4. **fire-and-forget 严格语义**：service 层的 broadcastMemberJoined / closeLeaverSession / broadcastMemberLeft 三个 unexported method **绝不返 error** —— 任何 internal failure 一律 log warn 不返。原因：(a) V1 §10.4 步骤 8 / §10.5 步骤 7 / 8 全部钦定 fire-and-forget；(b) caller (JoinRoom / LeaveRoom) 不需要走错误分流，let HTTP 200 响应严格 1:1 对应事务 commit 成功 —— broadcast 失败不应让 client 看到非 200 状态。

5. **r13 顺序约束的实装层落地**：closeLeaverSession 必须在 broadcastMemberLeft 之**前**调用 —— review grep `s.closeLeaverSession` 调用点必在 `s.broadcastMemberLeft` **之前**；**禁止**调换顺序。原因详见 V1 §10.5 r13 锁定段：让 BroadcastToRoom 调 ListSessionsByRoomID 时 leaver Session 已被 Unregister 不在返回列表中，自然跳过 leaver。

6. **加入者收不到自己 member.joined 由时序事实自然达成**：节点 4 阶段 HTTP join 与 WS 握手是两条独立路径，加入者通常**先**HTTP join → 再 WS 握手 → 但 broadcast 入队时加入者 WS 还没握手，所以加入者 Session 不在 ListSessionsByRoomID 列表中 → 自然收不到自己的 member.joined。**不**需要在 service 层做 excludeUserID 过滤；不需要扩展 BroadcastFn / BroadcastToRoom 签名。

7. **`closeLeaverSession` 对未持 WS 的 leaver 走 no-op + log info**：合法场景（leaver client 在调 HTTP leave 之前已主动 close 了 WS / 心跳超时已被动断开），不视为异常路径。

8. **PetRepo wire 复用兼容性**：`bootstrap/router.go` 内如已 wire `petRepo := mysql.NewPetRepo(deps.GormDB)` 实例（其他 service 复用），本 story 直接复用同实例；如未 wire 则新增（与 roomMemberRepo / roomRepo 平级）。**禁止**重复构造同一 db handle 的多个 NewPetRepo 实例。
