- **epics.md §Story 11.4 钦定**（行 1875-1901）：
  - **Given** Story 11.3 已能创建房间
  - **When** 调用 `POST /rooms/{roomId}/join`
  - **Then** service 在事务中执行：
    1. 查 `users.current_room_id`，非 null → 6003
    2. 查目标 `rooms`，不存在 → 6001；status != 1 → 6005
    3. 查 `room_members WHERE room_id=?`，count >= 4 → 6002
    4. 插入 `room_members` (room_id, user_id)
    5. 更新 `users.current_room_id = room_id`
    6. 返回 `{roomId, joined: true}`
  - **And** 单元测试覆盖（≥6 case，mocked repo）：
    - happy: 房间存在 + 未满 + 用户不在其他房间 → 事务成功
    - edge: 用户已在其他房间 → 6003
    - edge: 房间不存在 → 6001
    - edge: 房间已满 4 人 → 6002
    - edge: 房间 status=closed → 6005
    - edge: 并发竞态（用户同时被加入两个房间）→ DB UNIQUE(user_id) 兜底，第二个事务报 dup error，service 转 6003
  - **And** 集成测试覆盖（dockertest）：
    - 用户 A 创建房间 → 用户 B join → DB room_members 2 行 + B.current_room_id 更新
    - 4 个用户都 join → 第 5 个 user join 返回 6002

- **V1 §10.4 服务端逻辑钦定**（V1接口设计.md 行 1437-1456）—— 注：与 epics.md 钦定的 6 步事务**不完全等价**，V1 §10.4 把"开启事务 / 提交事务"显式列为独立步骤，且**步骤 2（SELECT rooms FOR UPDATE）就是事务内的第一动作**，让"房间存在性 / status / 成员计数"三件事用同一把行锁覆盖，这是 r9 P1#2 钦定的关键 race fix（与 §8.6 / §8.7 钦定一致）。本 story 实装路径**严格按 V1 §10.4 服务端逻辑 + 数据库设计 §8.6**：
  1. 查 `users.current_room_id`，**非 null** → 6003（**特例**：若 `current_room_id == 当前请求的 roomId` 也返回 6003 —— 用同一码统一处理"用户已在某房间"语义；client 仅需识别 6003 即可，不需要区分"已在目标房间" vs "已在其他房间"）（V1 行 1441 钦定）
  2. 开事务，对 `rooms` 行加排他锁：`SELECT ... FROM rooms WHERE id = ? FOR UPDATE`（**同时锁住"房间存在性 / status 字段 / 成员计数判断"三件事，与并发 leave 串行化**，详见 V1 §10.4 "并发保护" 段）；**找不到** → 回滚 + 6001
  3. 查到的 `rooms.status != 1` → 回滚 + 6005（`status = 2 closed` 房间禁止加入）
  4. 查 `room_members WHERE room_id = ?` 数量，`>= 4` → 回滚 + 6002
  5. 插入 `room_members`（`room_id`, `user_id = 当前 user`）；如果遇到 DB UNIQUE(`user_id`) 兜底冲突（理论不会，因为步骤 1 已查过 `users.current_room_id`，但并发竞态下可能发生）→ 回滚 + 6003
  6. 更新 `users.current_room_id = roomId`
  7. 提交事务
  8. **事务成功提交后**触发 WS 广播 `member.joined`（payload 见 §12.3 `### 成员加入`，**由 Story 11.8 实装；本 story 不触发广播 / 不调用 BroadcastToRoom**；广播失败 fire-and-forget 仅 log，不影响 HTTP 200 响应）
  9. 返回 `data.{roomId, joined: true}`

  **关键约束**（V1 §10.4 行 1500-1502 钦定）：
  - 错误码触发顺序严格按服务端逻辑步骤：步骤 1 → 6003（预检）；步骤 2 → 6001；步骤 3 → 6005；步骤 4 → 6002；步骤 5 → 6003（兜底）。**不**允许实装层重排顺序（如先查容量再查 status，会让"closed 房间满员"场景错误返回 6002 而非 6005）
  - 6003 在两条路径下触发（步骤 1 预检 + 步骤 5 DB UNIQUE 兜底），但 client **不**应区分这两种场景（都是"用户已在某房间"，UX 处理一致）—— message / errorCode 完全一致
  - `data.joined` 固定 `true`：成功路径必返；失败路径返回错误码 + 不返回 `joined` 字段

## Acceptance Criteria

**AC1 — `RoomRepo.FindByIDForUpdate` 方法新增 + `ErrRoomNotFound` 哨兵 error**

修改 `server/internal/repo/mysql/room_repo.go`：

- `RoomRepo` interface 新增 `FindByIDForUpdate(ctx context.Context, roomID uint64) (*Room, error)` 方法签名（紧接既有 `Create` 方法之后）：
  - doc comment 说明：**必须在事务内调用**（外层 ctx 必须是 `txMgr.WithTx` 注入的 `txCtx`，否则 FOR UPDATE 锁立即释放无效）；语义"取 rooms 行并对该行加排他锁，与并发 §10.5 leave 跨事务串行化"；找不到 → 返 `ErrRoomNotFound` 哨兵 error；query 失败 → 返 raw error 透传
  - 用 GORM `Clauses(clause.Locking{Strength: "UPDATE"}).Where(...).First(&room)` 路径：用 `gorm.io/gorm/clause.Locking` 显式生成 `... FOR UPDATE` —— 这是 GORM 最稳的"加锁查询"姿势（与 ADR-0003 §3.5 钦定一致；**不**用 raw SQL "SELECT ... FOR UPDATE"）
- `roomRepo.FindByIDForUpdate` impl（紧接既有 `Create` 方法之后）：

  ```go
  // FindByIDForUpdate 取 rooms 行并对该行加排他锁（FOR UPDATE）。
  //
  // **必须在事务内调用**（caller 传入的 ctx 必须是 txMgr.WithTx 注入的 txCtx）；
  // 在事务外调用时 FOR UPDATE 锁会被 driver 立即释放（autocommit 模式下任何
  // SELECT 完成即 commit），等同于普通 SELECT，违反 V1 §10.4 "并发保护" 钦定。
  // ADR-0007 §2.4 钦定 fn 内全部 repo 调用用 txCtx；本方法是 join / leave 事务的
  // 第一步，是 ADR-0007 §2.4 的关键 hot path。
  //
  // 找不到 → 返 ErrRoomNotFound（service 层 errors.Is 后翻译为 6001 ErrRoomNotFound）。
  // **注**：mysql.ErrRoomNotFound 与 apperror.ErrRoomNotFound 是**不同包**的同名常量
  // —— 前者是 repo 层哨兵 error，后者是业务码 6001；service 层用 errors.Is 翻译。
  // 命名一致是**故意**的（让阅读对照容易；review 时确认 import path 不冲突）。
  //
  // GORM clause.Locking{Strength: "UPDATE"} 路径生成 SELECT ... FOR UPDATE
  // SQL；与 raw SQL "SELECT ... FOR UPDATE" 等价但更安全（GORM 处理参数绑定 +
  // SQL injection；与 ADR-0003 §3.5 钦定一致）。
  func (r *roomRepo) FindByIDForUpdate(ctx context.Context, roomID uint64) (*Room, error) {
      db := tx.FromContext(ctx, r.db)
      var room Room
      err := db.WithContext(ctx).
          Clauses(clause.Locking{Strength: "UPDATE"}).
          Where("id = ?", roomID).
          First(&room).Error
      if err != nil {
          if stderrors.Is(err, gorm.ErrRecordNotFound) {
              return nil, ErrRoomNotFound
          }
          return nil, err
      }
      return &room, nil
  }
  ```

  - 加入 `import` `"gorm.io/gorm/clause"`（如尚未导入）

  //
  // **必须在事务内调用**（与同事务 SELECT rooms ... FOR UPDATE 步骤一起执行）；事务外
  // 调用 race 窗口仍存在 —— 步骤序列必须是：
  //   1. SELECT rooms FOR UPDATE（锁住 rooms 行）
  //   2. CountByRoomID（受锁保护，并发 join / leave wait）
  //   3. INSERT room_members（容量已校验，安全）
  // 这是 V1 §10.4 服务端逻辑步骤 2-4-5 钦定的串行化路径；缺步骤 1 锁则 step 2 / 3
  // 之间存在并发 race（5 用户同时 join 满员房间 → 5 个都看到 count == 3 → 5 个
  // 同时 INSERT，超员 → 4 个被 UNIQUE(user_id) 兜底，但仍多 1 行）。
  //
  // 用 SELECT COUNT(*) FROM room_members WHERE room_id = ?，走 idx_room_id 索引。
  func (r *roomMemberRepo) CountByRoomID(ctx context.Context, roomID uint64) (int, error) {
      db := tx.FromContext(ctx, r.db)
      var count int64
      err := db.WithContext(ctx).
          Model(&RoomMember{}).
          Where("room_id = ?", roomID).
          Count(&count).Error
      if err != nil {
          return 0, err
      }
      return int(count), nil
  }

  ```go
  func (s *roomServiceImpl) JoinRoom(ctx context.Context, in JoinRoomInput) (*JoinRoomOutput, error) {
      // (1) 预检 user.current_room_id（事务外，普通连接池查询）
      user, err := s.userRepo.FindByID(ctx, in.UserID)
      if err != nil {
          return nil, apperror.Wrap(err, apperror.ErrServiceBusy, apperror.DefaultMessages[apperror.ErrServiceBusy])
      }
      if user.CurrentRoomID != nil {
          // 用户已在房间中（V1 §10.4 步骤 1 钦定预检路径；含"已在目标房间" + "已在其他房间"两个子场景）→ 6003
          return nil, apperror.New(apperror.ErrUserAlreadyInRoom, apperror.DefaultMessages[apperror.ErrUserAlreadyInRoom])
      }

      // (2) 开事务（数据库设计 §8.6 + V1 §10.4 钦定）
      err = s.txMgr.WithTx(ctx, func(txCtx context.Context) error {
          // (2a) SELECT rooms WHERE id = ? FOR UPDATE → 找不到 → 6001
          room, err := s.roomRepo.FindByIDForUpdate(txCtx, in.RoomID)
          if err != nil {
              return err  // 含 ErrRoomNotFound 哨兵 / DB raw error
          }

          // (2b) status != 1 → 6005
          if room.Status != roomStatusActive {
              return errRoomInvalidStateInternal  // 哨兵让 service 层翻译 6005
          }

          // (2c) SELECT COUNT(*) FROM room_members WHERE room_id = ? → >= 4 → 6002
          count, err := s.roomMemberRepo.CountByRoomID(txCtx, in.RoomID)
          if err != nil {
              return err
          }
          if count >= roomMaxMembersDef {
              return errRoomFullInternal  // 哨兵让 service 层翻译 6002
          }

          // (2d) INSERT room_members；撞 UNIQUE(user_id) / UNIQUE(room_id, user_id) → 6003 兜底
          m := &mysql.RoomMember{
              RoomID: in.RoomID,
              UserID: in.UserID,
          }
          if err := s.roomMemberRepo.Create(txCtx, m); err != nil {
              return err
          }

          // (2e) UPDATE users.current_room_id = roomID
          if err := s.userRepo.UpdateCurrentRoomID(txCtx, in.UserID, &in.RoomID); err != nil {
              return err
          }

          return nil
      })

      if err != nil {
          // (3) 业务码分流（顺序关键 —— review r10 同源风险：errors.Is 顺序写错会让具体业务码被 generic 1009 兜底覆盖）
          if stderrors.Is(err, mysql.ErrRoomNotFound) {
              return nil, apperror.New(apperror.ErrRoomNotFound, apperror.DefaultMessages[apperror.ErrRoomNotFound])
          }
          if stderrors.Is(err, errRoomInvalidStateInternal) {
              return nil, apperror.New(apperror.ErrRoomInvalidState, apperror.DefaultMessages[apperror.ErrRoomInvalidState])
          }
          if stderrors.Is(err, errRoomFullInternal) {
              return nil, apperror.New(apperror.ErrRoomFull, apperror.DefaultMessages[apperror.ErrRoomFull])
          }
          if stderrors.Is(err, mysql.ErrRoomMembersUserIDDuplicate) ||
              stderrors.Is(err, mysql.ErrRoomMembersRoomUserDuplicate) {
              return nil, apperror.New(apperror.ErrUserAlreadyInRoom, apperror.DefaultMessages[apperror.ErrUserAlreadyInRoom])
          }
          // (3') 其他失败 → 1009（事务已 rollback）
          return nil, apperror.Wrap(err, apperror.ErrServiceBusy, apperror.DefaultMessages[apperror.ErrServiceBusy])
      }

      // (4) 事务 commit 成功 → 返回（broadcast 由 Story 11.8 在此处插入调用）
      return &JoinRoomOutput{
          RoomID: in.RoomID,
          Joined: true,
      }, nil
  }
  ```

**AC4 — 事务边界严格 6 步（含错误码触发顺序锁定）**

`JoinRoom` 的事务边界严格按 V1 §10.4 + 数据库设计 §8.6 + 时序图 §11.2 钦定：

- **步骤 1（事务外）**：`s.userRepo.FindByID(ctx, in.UserID)` → 检查 `user.CurrentRoomID != nil` → **预检 6003 路径**（不开事务）
- **步骤 2（事务内，5 步）**：
  - **2a**：`s.roomRepo.FindByIDForUpdate(txCtx, in.RoomID)` → 找不到 → 6001
  - **2b**：`if room.Status != roomStatusActive` → 6005
  - **2c**：`s.roomMemberRepo.CountByRoomID(txCtx, in.RoomID)` → `>= 4` → 6002
  - **2d**：`s.roomMemberRepo.Create(txCtx, m)` → 撞 UNIQUE 兜底 → 6003
  - **2e**：`s.userRepo.UpdateCurrentRoomID(txCtx, in.UserID, &in.RoomID)` → 更新 users.current_room_id
- **步骤 3（事务后）**：commit / rollback；commit 成功 → 返回 JoinRoomOutput{RoomID, Joined: true}；rollback 后按 err 类型分流 6001 / 6005 / 6002 / 6003 / 1009

**错误码触发顺序锁定**（V1 §10.4 行 1500 钦定，**不允许实装层重排**）：步骤 1 → 6003；2a → 6001；2b → 6005；2c → 6002；2d → 6003；任何 DB 异常 → 1009。重排会让"closed 房间满员"场景错误返回 6002 而非 6005（违反 client UX 期望）。

**ctx 用法（ADR-0007 §2.4）**：fn 内全部 repo 调用必须用 `txCtx` 而非外层 `ctx` —— 用错 ctx 会绕过 tx 走 db pool 新连接，FOR UPDATE 锁立即释放，并发保护失效。本 story 实装严格遵守。

**broadcast 触发点**：本 story 实装层 commit 成功后**直接 return**，**不**调任何 WS broadcast；Story 11.8 review 时在 `return &JoinRoomOutput{...}, nil` 之前插入 `s.broadcastMemberJoined(ctx, in.RoomID, in.UserID)` 调用。

**AC5 — 6003 双路径必须等价**

预检路径（步骤 1）+ DB 兜底路径（步骤 2d）都返 `apperror.New(ErrUserAlreadyInRoom, ...)`，handler 端响应 envelope 完全一致 —— client 不应区分这两种场景（V1 §10.4 行 1501 钦定 "client **不**应区分"）。**注**：与 11.3 创建房间事务的 6003 双路径同模式。

**特例**：若 caller `current_room_id == 当前请求的 roomId`（用户尝试加入自己已在的房间）也返回 6003 —— V1 §10.4 行 1441 钦定 "用同一码统一处理"用户已在某房间"语义，client 仅需识别 6003 即可，不需要区分"已在目标房间" vs "已在其他房间""。本 story 实装步骤 1 通过 `if user.CurrentRoomID != nil` 单一判断兜下两个子场景。

review 阶段必须命中的检查（参考 Story 11.3 r1 ~ rN review 教训 + Story 4.6 / 7.3 / 11.2 业务事务 review 模式）：

- **r1 同源风险（事务 ctx 传播）**：review 必须 grep `room_service.go` 确认 `WithTx(ctx, fn)` 内 fn 全部 5 个 repo 调用都用 **`txCtx`** 而非外层 `ctx` —— 任何漏传 / 用错都视为 P1 修复点（ADR-0007 §2.4 钦定；与 11.3 / auth_service.firstTimeLogin 同模式）。**FOR UPDATE 锁的 ctx 传播是关键**：用错 ctx 会让 GORM 走 db pool 新连接，FOR UPDATE 锁立即释放（autocommit 模式），并发保护失效，r9 cross-tx race 重新出现
- **r2 同源风险（错误码触发顺序锁定）**：review 必须 diff service `JoinRoom` 实装的事务内步骤顺序，**严格按 V1 §10.4 行 1500 钦定**：步骤 2a → 6001；2b → 6005；2c → 6002；2d → 6003。重排（如先查容量再查 status）会让"closed 房间满员"场景错误返回 6002 而非 6005，违反 client UX 期望
- **r3 同源风险（6003 双路径等价性）**：review 必须 diff 预检路径（步骤 1）+ DB 兜底路径（步骤 2d）的 `apperror.New / Wrap` 调用，**确认 code / message 完全一致**（V1 §10.4 行 1501 钦定 client 不区分；与 11.3 r2 同模式）
- **r4 同源风险（FOR UPDATE 锁路径）**：review 必须验证 `roomRepo.FindByIDForUpdate` 用 GORM `clause.Locking{Strength: "UPDATE"}` 而非 raw SQL `"SELECT ... FOR UPDATE"` —— GORM 处理参数绑定 + SQL injection 安全（与 ADR-0003 §3.5 钦定一致）。raw SQL 路径在 review 阶段视为 P2 修复点（仍能跑但不符合内部约定）
- **r5 同源风险（FOR UPDATE 必须在事务内）**：review 必须 grep service.JoinRoom 确认 `roomRepo.FindByIDForUpdate(txCtx, ...)` 调用必须在 `s.txMgr.WithTx(ctx, func(txCtx ...) error { ... })` 内；任何 fn 外的调用（即用外层 ctx）属严重 bug —— FOR UPDATE 锁立即释放（autocommit 模式），并发保护失效
- **r6 同源风险（GORM AutoMigrate 反模式）**：review 必须 grep 确认本 story 引入的 `RoomRepo.FindByIDForUpdate` / `RoomMemberRepo.CountByRoomID` impl **没有**触发 AutoMigrate（搜 `AutoMigrate(` 关键词），与 ADR-0003 §3.2 钦定一致

### 角色定位与 Epic 11 的实装路径

本 story 是 Epic 11 第二个**真实业务事务 story**，在 Epic 11 的角色对应 Story 11.5（退出房间事务）+ Story 11.3（创建房间事务）的"事务三件套"中间一环：上承 11.3（创建房间，已 ship）+ 11.2（持久化基础）+ 11.1（契约），下启 11.5（退出房间，需要 join 事务先存在让 leave 路径有数据可删）+ 11.6（房间详情查询，复用 join 事务的 FOR UPDATE 锁知识）+ 11.7（房间快照真实实装，复用 join 写入的 room_members 数据）+ 11.8（member.joined 广播触发，本 story 实装事务的"广播触发点"逻辑但**不**调 broadcast）+ 11.9（Layer 2 集成测试，需要 join 事务作为生命周期第二步）+ 11.10（GET /home 扩展，需要本 story 写入的 users.current_room_id 数据）。**任何超出"POST /rooms/{roomId}/join 单接口加入房间事务"范围的实装都属范围越界**，review 必拒。

### 与 Story 4.6 / 7.3 / 11.3 业务事务模式的对照

本 story 实装路径直接套用 Epic 4 / 7 / 11.3 已建立的"业务事务 + 单元测试 + 集成测试 + handler 端到端"四件套模式，与下表对应：

**反模式（已避免）**：
- ❌ handler 直接调多 repo 拼事务（违反 §5.1 "service 是事务边界归属"）
- ❌ service 用 `*gorm.DB` 直接调 `db.Transaction(...)`（绕过 tx.Manager interface）
- ❌ fn 内 repo 用外层 `ctx` 而非 `txCtx`（绕过 tx 走 db pool 新连接）—— 对 FOR UPDATE 路径**致命**，锁立即释放
- ❌ 6003 预检 + 兜底两条路径返不同 message（client 难以一致处理）
- ❌ 错误码触发顺序重排（如先查容量再查 status，会让"closed 房间满员"场景错误返 6002 而非 6005）
- ❌ FindByIDForUpdate 用 raw SQL "SELECT ... FOR UPDATE"（应用 GORM clause.Locking）
- ❌ FindByIDForUpdate 在事务外调用（FOR UPDATE 锁立即释放）
