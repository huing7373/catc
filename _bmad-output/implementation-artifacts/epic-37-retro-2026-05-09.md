## 2. 关键洞察与 Epic 37 的"主旋律"

### 2.1 主旋律 1: "壳先行 + 接缝设计"模式让 9 个下游 epic 的 acceptance 提前 single source of truth 化

Epic 37 不仅交付 14 个 Story 的代码，更重要的输出是**改写了 Epic 6 / 12 / 21 / 24 / 27 / 30 / 33 / 35 的 9 处 Story acceptance**（详见 sprint-change-proposal-2026-04-29-v2.md §5）：

- **Story 12.1** 从"房间页 SwiftUI 骨架"缩窄为"在 37.8 RoomView Scaffold 上把 Mock 替换为 Real"
- **Story 12.7** 从"主界面入口完善（Sheet 模式）"作废 + 改写为"3 个 UseCase 实装 + 写 AppState"
- **Story 21.1 / 24.1 / 27.x / 33.1** 同模式 —— 都缩窄为"在 Scaffold 上换 ViewModel + 接 server"
- **Story 24.4 / 33.6** 整条作废（Tab 直接路由不需要"主界面入口按钮"）
- **Epic 6 description** 加前置依赖声明（"本 epic 启动时主入口已是 4 Tab + AppState 单 source of truth"）

**根因**：v1 sprint-change-proposal 想用 Moderate scope 包装架构级矛盾被 codex Reject；v2 升 Major + 落 2 ADR + 集中 14 story 一次清债，让下游 epic 的 acceptance 在 Epic 37 done 前就锁死，避免 epic-12 实装时撞旧 acceptance vs. 新架构的二义性。

### 2.2 主旋律 2: 单 Story review 峰值 r5 —— 与 Epic 11 11.1 r14 / 11.8 r11 形成鲜明对比

| Epic | 单 Story 峰值 | 一次过比例 | 总 review 轮 |

**根因**：iOS UI Scaffold 的 review 维度（视觉精度 + ViewModel 注入契约 + a11y identifier）虽多，但**单维度可枚举 + dev 阶段有 ui_design 像素源参考**；不像协议契约 Story（11.1）的审计维度跨 4 份 docs + ACL 锁 + 并发 race + 跨章节 placeholder shape，无法在 dev 阶段穷举。

### 2.3 主旋律 3: ViewModel class 层次 + Mock/Real 接缝设计在 6 屏验证中收敛

每屏 Scaffold story（37.7-37.12）走的是同一接缝模式：

```
基类 ObservableObject (final 去除让子类可继承)
  ├── @Published var <field>... (4 字段)
  ├── abstract method() { fatalError("must override") }
  └── init() {}
    ↓ 继承
Mock<Screen>ViewModel (硬编码 mock + override 仅 print)
Real<Screen>ViewModel (构造注入 AppState + override 实装；本 epic 仅骨架)
```

经过 6 屏验证后，**关键 lessons 集中沉淀**（共 12 条 ViewModel 层 lessons）：

- `real-viewmodel-init-must-seed-scaffold-defaults`（37.8 r1）—— Real init 必须 seed scaffold default 否则首帧渲染空
- `real-viewmodel-injection-must-not-leave-base-fatalerror`（37.7 r1）—— RootView 注入必须切换到子类，不能让基类调用走 fatalError
- `real-viewmodel-override-placeholder-must-mutate-state`（37.7）—— override 不能只 print，必须实际写 @Published
- `real-viewmodel-must-clear-transient-state-on-reset`（37.7 r3）—— reset 时必须清 transient
- `real-viewmodel-transient-must-clear-on-any-identity-change`（37.11 r2-r4 之一）—— 任意 identity 变更都要清 transient
- `published-derived-state-needs-publisher-subscription`（37.7 r4）—— @Published 派生字段需订阅 publisher
- `strong-vs-weak-for-constructor-injected-state`（37.6/37.7）—— 构造注入 AppState 用 strong 引用
- `onappear-vs-task-sync-bind-before-first-paint`（37.8 r2）—— bind 必须在第一次 paint 之前同步完成
- `room-host-name-must-not-derive-from-local-current-pet`（37.8 r3）—— host name 不能派生自 local 猫
- `coordinator-must-mirror-loaded-home-room-state`（37.3）—— coordinator 必须 mirror 已加载状态
- `scaffold-bypass-viewmodel-seam`（37.11 之一）—— Scaffold 不能 bypass ViewModel 接缝直读 AppState
- `realhomeviewmodel-greeting-and-empty-text-overlay`（37.7）—— 空 Text overlay VoiceOver 陷阱

**对 Epic 12+ 的实质意义**：12.1 / 12.7 实装 RealRoomViewModel 时，应在 dev-story 阶段直接 reference 上述 12 条作为 anti-pattern checklist —— 与 Epic 11 11.6 / 11.7 reference Epic 10 / 11.1 lessons 同模式。

### 2.4 主旋律 4: 静态校验脚本本身是个 Story —— 37.13 / 37.14 各跑 r4

37.13（a11y identifier 总表 + CI 脚本）和 37.14（design-package 白名单）是 Epic 37 仅有的两个**纯治理类 Story**，但都跑了 r4：

- **37.13 r1** SCAN_DIRS 漏 App/ + regex 不覆盖 custom wrapper —— a11y coverage CI 脚本盲区
- **37.13 r2** ADR layering guard 必须 token-match 而非 `TypeName(` 构造调用 match
- **37.13 r3** a11y coverage 算法 sound 性 + 暴露 2 处真违规
- **37.13 r4** Tab.identifier(for:) 必须 source 自声明常量（identifier-helper-must-source-from-declared-constants）
- **37.14 r1** 白名单条目必须 cite 完整流程文档 / 真实渲染路径
- **37.14 r2** deferred artifact 位置必须落到真实承载源（app.jsx）+ 区分 ui_design / SwiftUI render path
- **37.14 r3** 白名单未来 routing 必须 cite 真实 Story scope（3D 模型 != Story 30.x；prototype 不上线）
- **37.14 r4** 白名单 deferred theme set 必须 mirror 真实 code token（candy/matcha/sky/dark 四套，不是 candy/dark/mono 三套）

**根因**：静态校验脚本和文档治理类工作的 review 难度被低估。脚本本身需要"测试 the test"（脚本能否扫出真违规、能否避免误报），文档需要"cite 真实承载源"（不能引用过时 Story scope 或不存在的 render path）。

**lesson**：未来类似治理类 Story（如 Epic 6 节点 2 demo 验收 / Epic 13-1 跨端集成测试场景定义 / Epic 36 MVP 整体收官）应预算 r4 review 量，不要按"治理类 = 简单 = 一次过"判断 effort。

### 2.5 主旋律 5: SwiftUI 视觉细节 review 在 dev-story 阶段无法穷举

37.6（primitives）和 37.7（HomeView Scaffold）跑满 r5，r1-r5 都在挖 SwiftUI 视觉/动画/布局精度问题：

- 37.6 r1: codex os_log 误报 + FadeIn 方向反转 + Avatar inset shadow
- 37.6 r2: PrimaryButton fullWidth modifier order（先 padding 再 frame）
- 37.6 r3: strokeBorder vs stroke 内外绘语义 / ButtonStyle vs DragGesture 取消语义
- 37.6 r4: SwiftUI `.id(nil)` 共享 explicit identity 陷阱（ViewModifier 默认参容易踩）
- 37.6 r5: ViewModifier @State 跨 .id 重建幸存 + `.shadow` 投影到 children（破例 round 5）
- 37.7 r1-r5: 浮动 emoji 状态驱动 / onChange Equatable / 派生字段 publisher / VoiceOver overlay / 浮动动画 .id() 重建子视图

**根因**：SwiftUI 的 declarative diffing + state-driven animation + a11y / VoiceOver 行为是高度上下文相关的，dev-story 阶段无法仅凭 ui_design 像素源穷举所有交互边界。这是 SwiftUI UI Scaffold 类 Story 的本质复杂度天花板。

**好消息**：37.8-37.12 因为 37.6 / 37.7 已经替整条产品线把上述坑踩平，全部 r1-r3 就 done。这与 Epic 11 11.6 / 11.7 一次过依赖 11.1 r1-r14 / 10.7 lessons 的复利模式同源。

### 3.2 "重新实装 + caller 漏改靠编译器报错驱动"模式

## 4. What Didn't Go Well（要修正的）

### 4.1 32 个 lessons 集中产出在 4 月 30 日单天 —— 节奏密度过载

`docs/lessons/` 中 2026-04-30 单天产生 29 条 lesson + 2026-05-01 产生 3 条。同一天处理 6+ 屏 Scaffold story 的 fix-review 反馈，**lesson 分类密度高 + 命名重叠 risk**：

- `whitelist-entry-must-cite-full-flow-doc-and-real-render-path.md` 和 `whitelist-entry-must-cite-full-flow-doc-and-real-render-path-2.md` 出现 -2 后缀重名 —— 同主题 2 lesson 没合并
- 12 条 ViewModel 层 lesson 命名前缀都是 `real-viewmodel-` —— 用户事后检索时容易混淆

**建议**（非紧急）：未来 lesson 命名可加更细的二级 scope（如 `real-viewmodel-init-`、`real-viewmodel-bind-`、`real-viewmodel-transient-`），lesson backfill 阶段做一次同主题 dedup pass。

### 4.2 r6 reclassify 例外破坏了 r5 cap 隐含承诺（37.6）

37.6 出现 round 6 reclassify 决议（commit `4909cfc docs(lessons): backfill Story 37.6 lesson commit hashes + 追加 round 6 reclassify 决议`）—— 即 r5 收敛后 reviewer 又补了 r6 的 reclassify 决议。

虽然这是合理的（r6 是事后 lesson 分类的修正，不是新增 review finding），但**破坏了 epic-loop "r10 cap" 的隐含承诺**：用户看到 git log 时会以为 37.6 跑了 r6，需要点开 commit message 才知道是 reclassify 而非真 review。

**建议**（非紧急）：未来类似情形用 `chore(story-X): reclassify lesson rounds` 或 `docs(lessons): reclassify ...` 类型 commit，不要混入 fix(review) 命名空间。本 epic-37 retro 已经在统计中明确把 37.6 r6 单独标注为 reclassify。

### 4.3 ViewModel 注入模式在 6 屏内仍反复 r1-r3 —— 模板未在 37.7 一次锁死

37.8 r1+r2+r3 / 37.11 r2-r4 持续在挖"基类 fatalError 未 override / hydrate 入口 / strong-vs-weak / transient state clear"。这些维度本可以在 37.7 的 r1-r5 阶段就一次穷举枚举，但实际上每个新屏的 RealViewModel 实装都还能挖出新维度。

- 37.8 r2 暴露："RootView 持有的 ViewModel 必须在第一次 paint 之前同步 bind AppState" —— 这是 onAppear vs task 时序问题，不是 37.7 的范围
- 37.11 r2-r4 暴露："Real ViewModel transient state 必须 on any identity change clear" —— 这是 ProfileView 因为 currentUser 可能跨身份变化的特有维度

**根因**：ViewModel 接缝设计的复杂度在不同屏（HomeView 持 currentUser+currentPet+currentChest / RoomView 持 currentRoomId+members / ProfileView 持 currentUser+settings）的 identity 边界不同；37.7 的模板只覆盖了 HomeView 的 identity 边界。

**lesson**（已落地为 `real-viewmodel-transient-must-clear-on-any-identity-change.md` / `scaffold-bypass-viewmodel-seam.md`）：未来 Epic 12 实装 RealRoomViewModel 时应在 dev-story 阶段直接 reference 这两条 + onappear-vs-task-sync-bind-before-first-paint，让 r1-r3 的折返成本降低。

### 4.4 静态校验脚本类 Story 的 effort 估算偏低

37.13 / 37.14 都跑了 r4 review，每条 r4 都不是 cosmetic / nit 而是真问题（脚本盲区 / 算法 sound 性 / 真违规曝光 / 文档 cite 完整性）。

**根因**：dev-story 阶段把"静态校验脚本"当成"写 ESLint 配置"那种 mechanical 工作；实际上 a11y coverage 脚本和 ADR layering guard 是**业务规则的形式化**，需要"测试 the test"。

**建议**（非紧急）：未来类似 Story（如 Epic 6 / 13 / 16 / 19 / 22 / 25 / 28 / 31 / 34 / 36 各 demo 验收 epic 的 "跨端集成测试场景定义"）应预算 r4 review 量。

### 4.5 Race detector 在 Windows 仍然缺位 —— Epic 11 retro §4.3 标记的 tech debt 本 epic 没回归

虽然 Epic 37 是 iOS 端工作 + Race detector 主要影响 Go server，但**Epic 10 retro A2 钦定 Epic 13 落地 race detector**这条 action item 在 Epic 11 / Epic 37 期间都没回归。

**状态**：⏳ 仍属 tech debt（依然钦定 Epic 13 落地）；Epic 12 / 13 期间需要监控不要因为 iOS 工作堆积而再次推迟。
