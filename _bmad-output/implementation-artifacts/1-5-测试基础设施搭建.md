## Story

As a 服务端开发,
I want server 端测试栈一次配齐（mock 库 + fixture + 示范测试 + CI 命令对齐）,
so that 后续 Epic（4 / 7 / 10 / 20 / 32 …）写第一条 service 测试时**直接能跑**，不被工具栈装配卡住，也不用临时 println 风格漂移.

## 故事定位（节点 1 第五条实装 story）

- Story 1.1 已输出 ADR `_bmad-output/implementation-artifacts/decisions/0001-test-stack.md`，**一次性锁定** 6 类工具栈 + 结构化日志字段 + NFR16 指标位 + 版本清单；本 story 是该 ADR 的**落地 story**
- Story 1.2 / 1.3 / 1.4 的测试**仍用 stdlib `testing`**（`t.Errorf` / `t.Fatalf`；`json.Unmarshal` 手写断言），**没有**引入任何 mock / assertion 库
- 本 story 完成后，Epic 4 Story 4-7（游客登录初始化事务）写**第一条真业务 service 测试**时可以直接 import testify/mock + 复制 sample 结构
- 本 story 的 sample 是**后续业务 Layer 1 测试 AC 的模板**（epics.md Story 1.5 AC 明示）—— 后续 Epic 写测试时**直接复制这个文件结构**，避免风格漂移

**严格按需引入原则**（`docs/宠物互动App_MVP节点规划与里程碑.md` §2 原则 7 / 8 + CLAUDE.md §工作纪律）：本 story **只**装工具 + 写 sample，**不**接真 MySQL / 真 Redis / GORM / auth 中间件。

## 范围红线

本 story **只做**以下五件事：

1. `server/go.mod` 新增 3 个 require（testify / sqlmock / miniredis），**pin 版本**（不用 `latest`）
2. 新建 `internal/service/sample/` 包：一个 placeholder service + mocked repo interface + table-driven test（≥ 4 case，使用 `testify/mock` + `testify/require` + `testify/assert`）
3. 新建 `internal/pkg/testing/` 包：提供 `NewSQLMock(t)` / `NewMiniRedis(t)` 两个测试 helper（供后续 Epic 用，本 story 先写 smoke 测试证明可用）
4. 新建 `internal/pkg/testing/slogtest/` 子包：提供 in-memory `slog.Handler` fixture，用于 Story 1.8 / Epic 4+ 断言日志字段
5. 验证 `go test ./...` / `go test -race -cover ./...` 全绿

本 story **不做**：

- ❌ **不**装 `dockertest`：ADR §6 说 "Story 1.5 可先装"，但 `go mod tidy` 会剔除未被 import 的包 → 必须同时 import；本 story 没有 layer-2 集成测试场景需要真容器，现在装会强迫写一个假用途的 import，污染 go.mod。**决策改为**：dockertest 延迟到 Epic 4 Story 4-7（首次 layer-2 集成测试）安装。在 ADR 的 Change Log 追加一行说明该改动
- ❌ **不**装 GORM：GORM 由 Epic 4 Story 4-2（MySQL 接入）引入；本 story 的 sqlmock smoke 测试用**裸 `database/sql`**（`sql.Open("mysql", ...)` 无需真驱动，配合 sqlmock 的 `sqlmock.New` 返回 `*sql.DB`）
- ❌ **不**写 ADR §3.1 承诺的"sqlmock + GORM 正则写法 cheatsheet"：GORM 未落地时谈 GORM SQL pattern 属于**空中楼阁**，Epic 4 Story 4-2 落地后再补（届时 ADR Change Log 更新；以 `_bmad-output/implementation-artifacts/decisions/0002-sqlmock-gorm-cheatsheet.md` 新建更合适）
- ❌ **不**在 sample service 里做 ctx cancellation 测试：Story 1.9 AC 明示它会**追加**一条 ctx cancel 测试到 sample 模板；本 story 保留"同步路径"骨架，Story 1.9 自己扩展
- ❌ **不**动 Story 1.2 / 1.3 / 1.4 的既有测试：已有 `t.Errorf / t.Fatalf` 风格保留，**不**改写成 testify；重写既有测试会污染 Story 1.5 的 file list（违反节点 1 原则 7 按需引入）
- ❌ **不**做 `scripts/build.sh` 重做：Story 1.7 的 scope；本 story 仅要求 `cd server && go test ./...` 命令行跑通，**不**动 `build.sh`
- ❌ **不**引入 `go generate` / `mockgen` / `mockery`：ADR §3.4 明示否决 codegen 方案，手写 testify/mock

**AC3 — SampleService 实装（`internal/service/sample/service.go`）**

**包命名**：`package sample`（**不**是 `package service`）—— 与架构文档 §4 "按 domain 切子包" 对齐；后续真 service 也走同样的 `internal/service/<domain>/` 布局。

**最小结构**：

```go
package sample

import (
    "context"
    "errors"
)

// ErrSampleNotFound 演示 service 层业务错误（Story 1.8 会用 *AppError 替换，
// 本 story 先用 stdlib errors.New 占位，不提前引入 AppError 依赖）。
var ErrSampleNotFound = errors.New("sample not found")

// SampleDTO 演示返回对象（任意字段，本 story 用 ID + Value 两字段）。
type SampleDTO struct {
    ID    string
    Value int
}

// SampleRepo 是 service 依赖的 repo 接口。
//
// 本 story 不落地任何真实现 —— 仅作为 testify/mock 的 mock 目标。
// 后续 Epic 4 Story 4-2 落地 repo 层时，会看到类似的 interface 在
// `internal/domain/<domain>/repo.go` 或 `internal/repo/mysql/<domain>.go` 中定义。
type SampleRepo interface {
    FindByID(ctx context.Context, id string) (*SampleDTO, error)
}

// SampleService 演示一个最小 service：依赖 SampleRepo，封装一次查询调用。
type SampleService struct {
    repo SampleRepo
}

// NewSampleService 构造函数。依赖通过参数注入（便于测试替换 mock）。
func NewSampleService(repo SampleRepo) *SampleService {
    return &SampleService{repo: repo}
}

// GetValue 演示：查询一个 sample 并返回其 Value。
//   - id 为空串 → 返回 ErrSampleNotFound（预校验，不落 repo）
//   - repo 返回 nil + err → service 原样返回 err（Story 1.8 会替换为 Wrap(err, ...)）
//   - repo 返回 dto → 返回 dto.Value
func (s *SampleService) GetValue(ctx context.Context, id string) (int, error) {
    if id == "" {
        return 0, ErrSampleNotFound
    }
    dto, err := s.repo.FindByID(ctx, id)
    if err != nil {
        return 0, err
    }
    return dto.Value, nil
}
```

**契约**：
- **所有**导出函数第一个参数必须是 `ctx context.Context`（Story 1.9 ADR §0007 即将约定；本 story 先按此约定示范）
- `SampleService.repo` 小写不导出（强制依赖通过 `NewSampleService` 注入）
- `ErrSampleNotFound` 使用 `errors.New` 而非 `*AppError`（Story 1.8 未完成；sample 主题是"mock 注入链路"而非"错误映射"）
- **不**在 `SampleService` 上加额外方法（`Create` / `Delete` 等），保持 template 最小

**AC4 — SampleService table-driven test（`internal/service/sample/service_test.go`）**

**包命名**：`package sample_test`（外部测试包）—— 强制通过导出 API 测试，杜绝访问私有字段；与未来真 service 测试风格一致。

**必须 ≥ 4 case**（table-driven，**使用 testify/mock + testify/require + testify/assert 三件套**）：

| # | case | 构造 | 断言 |
|---|---|---|---|
| 1 | happy: 正常查询 | mock `FindByID("abc", ctx)` 返回 `&SampleDTO{ID:"abc", Value:42}` | `value == 42`；`err == nil`；mock expectations 全部满足 |
| 2 | edge: id 为空 → 预校验失败，repo 不被调用 | mock **不**设置 FindByID 期望 | `errors.Is(err, ErrSampleNotFound)`；`value == 0`；mock.AssertNotCalled FindByID |
| 3 | edge: repo 返回底层错误 → service 原样向上传 | mock `FindByID("x", ctx)` 返回 `nil, errors.New("db down")` | `err != nil`；`err.Error() == "db down"`；`value == 0` |
| 4 | happy: ctx 正常传递 | 自定义 ctx with value key；mock `FindByID` 用 `mock.MatchedBy` 断言收到的 ctx 含该 value | 断言 mock 实际收到 "ctx propagated" 信号（见实装模板） |

**实装模板**：

```go
package sample_test

import (
    "context"
    "errors"
    "testing"

    "github.com/stretchr/testify/assert"
    "github.com/stretchr/testify/mock"
    "github.com/stretchr/testify/require"

    "github.com/huing/cat/server/internal/service/sample"
)

// MockSampleRepo 是手写的 testify/mock 实现（不走 codegen，ADR §3.4）。
// 同样的手写模式会在 Epic 4 Story 4-2 之后的 service 层测试反复出现；
// 本文件是它们的**模板范本**。
type MockSampleRepo struct {
    mock.Mock
}

func (m *MockSampleRepo) FindByID(ctx context.Context, id string) (*sample.SampleDTO, error) {
    args := m.Called(ctx, id)
    // 第一个返回值可能为 nil *SampleDTO；用 Get + 类型断言而非 .Get().(*SampleDTO)
    // 直接转换（会 panic on nil）。
    dto, _ := args.Get(0).(*sample.SampleDTO)
    return dto, args.Error(1)
}

type testCase struct {
    name      string
    id        string
    setup     func(*MockSampleRepo)
    wantValue int
    wantErr   error   // 可 nil；断言时用 errors.Is
    wantErrMsg string // 可空；底层 error 判字面
    notCalled bool    // 如果 true，断言 mock FindByID 未被调用
}

func TestSampleService_GetValue(t *testing.T) {
    cases := []testCase{
        {
            name: "happy: 正常查询返回 value",
            id:   "abc",
            setup: func(m *MockSampleRepo) {
                m.On("FindByID", mock.Anything, "abc").
                    Return(&sample.SampleDTO{ID: "abc", Value: 42}, nil).Once()
            },
            wantValue: 42,
        },
        {
            name: "edge: id 空串预校验失败，repo 不被调用",
            id:   "",
            setup: func(m *MockSampleRepo) {
                // 不设置任何期望 —— 如果 repo 被调用，mock.AssertExpectations 会失败
            },
            wantValue: 0,
            wantErr:   sample.ErrSampleNotFound,
            notCalled: true,
        },
        {
            name: "edge: repo 返回错误原样向上",
            id:   "x",
            setup: func(m *MockSampleRepo) {
                m.On("FindByID", mock.Anything, "x").
                    Return(nil, errors.New("db down")).Once()
            },
            wantValue:  0,
            wantErrMsg: "db down",
        },
        {
            name: "happy: ctx 正确传递到 repo",
            id:   "abc",
            setup: func(m *MockSampleRepo) {
                m.On("FindByID",
                    mock.MatchedBy(func(ctx context.Context) bool {
                        return ctx.Value(testCtxKey{}) == "propagated"
                    }),
                    "abc",
                ).Return(&sample.SampleDTO{ID: "abc", Value: 7}, nil).Once()
            },
            wantValue: 7,
        },
    }

    for _, tc := range cases {
        t.Run(tc.name, func(t *testing.T) {
            repo := &MockSampleRepo{}
            tc.setup(repo)

            svc := sample.NewSampleService(repo)

            ctx := context.Background()
            if tc.name == "happy: ctx 正确传递到 repo" {
                ctx = context.WithValue(ctx, testCtxKey{}, "propagated")
            }

            got, err := svc.GetValue(ctx, tc.id)

            assert.Equal(t, tc.wantValue, got)
            switch {
            case tc.wantErr != nil:
                require.Error(t, err)
                assert.ErrorIs(t, err, tc.wantErr)
            case tc.wantErrMsg != "":
                require.Error(t, err)
                assert.EqualError(t, err, tc.wantErrMsg)
            default:
                require.NoError(t, err)
            }

            if tc.notCalled {
                repo.AssertNotCalled(t, "FindByID", mock.Anything, mock.Anything)
            }
            repo.AssertExpectations(t)
        })
    }
}

type testCtxKey struct{}
```

**约束**：
- `_test.go` **所在包必须是** `sample_test`（外部测试包）
- 前置条件（setup 失败）用 `require.NoError` fail-fast；断言失败结果用 `assert.Equal` continue-on-fail —— ADR §3.3 规则
- **不**加 `t.Parallel()`（本 story 无并行需求；future ctx cancel 测试 Story 1.9 添加时再评估）
- `MockSampleRepo` 手写 —— **不**引入 `mockgen` / `mockery`（ADR §3.4）
- 测试文件顶部 import 顺序：stdlib → 第三方 → 本项目（与 `version_handler_test.go` 风格一致）

**AC5 — Testing helpers（`internal/pkg/testing/helpers.go`）**

提供两个 test helper，供后续 Epic 直接 import：

```go
package testing

import (
    "database/sql"
    "testing"

    sqlmock "github.com/DATA-DOG/go-sqlmock"
    "github.com/alicebob/miniredis/v2"
)

// NewSQLMock 返回 (db, mock, cleanup)。
// 使用方式：
//
//	db, mock, cleanup := testhelper.NewSQLMock(t)
//	defer cleanup()
//	mock.ExpectQuery(...).WillReturnRows(...)
//	// 传 db 给 repo 或 GORM Open("sqlmock", ...)
//
// 内部调 sqlmock.New() + 在 t.Cleanup 注册 db.Close() + mock.ExpectationsWereMet。
// mock 未满足期望会在 test cleanup 阶段 t.Errorf，避免 dev 忘记手动断言期望。
func NewSQLMock(t *testing.T) (*sql.DB, sqlmock.Sqlmock, func()) {
    t.Helper()
    db, mock, err := sqlmock.New()
    if err != nil {
        t.Fatalf("sqlmock.New: %v", err)
    }
    cleanup := func() {
        _ = db.Close()
        if err := mock.ExpectationsWereMet(); err != nil {
            t.Errorf("sqlmock expectations not met: %v", err)
        }
    }
    t.Cleanup(cleanup)
    return db, mock, cleanup
}

// NewMiniRedis 启动一个 in-process miniredis server，返回 (server, addr)。
// 使用方式：
//
//	mr, addr := testhelper.NewMiniRedis(t)
//	client := redis.NewClient(&redis.Options{Addr: addr})
//	// 正常跑 Redis 命令；mr.FastForward / mr.FlushAll 可控时间 / 清空
//
// 自动在 t.Cleanup 关停 server，不需要手动 Close。
func NewMiniRedis(t *testing.T) (*miniredis.Miniredis, string) {
    t.Helper()
    mr, err := miniredis.Run()
    if err != nil {
        t.Fatalf("miniredis.Run: %v", err)
    }
    t.Cleanup(mr.Close)
    return mr, mr.Addr()
}
```

**说明**：
- `NewSQLMock` 返回的 `cleanup` 既在 `t.Cleanup` 注册也返回给调用方 —— 允许早期 `defer cleanup()` 风格；两种调用方式都安全（`t.Cleanup` 幂等：`db.Close` 二次调用返回 `sql.ErrConnDone`，`ExpectationsWereMet` 可能 double-report，所以**推荐只用 `t.Cleanup`**，返回 `cleanup` 是为了让调用方能显式早关闭）—— 在注释里**明示**推荐做法（不要 `defer cleanup()`）
- `NewMiniRedis` 不返回 cleanup —— miniredis 无"期望未满足"概念，`t.Cleanup` 一次关停足够
- `testing.T` 参数必须是 `*testing.T`（不是 `testing.TB`）—— sqlmock / miniredis 没有 B benchmark 用例；后续需要时再抽 `TB`
- **包名避坑**：`package testing` 会与 stdlib `testing` 在调用方产生命名冲突。import 时**必须重命名**：
  ```go
  import testhelper "github.com/huing/cat/server/internal/pkg/testing"
  ```
  在 `helpers.go` 顶部包注释里**明示**该 import 规范，防止后续 dev 按默认包名 `testing` import 后自砍脚

**AC6 — slog in-memory handler（`internal/pkg/testing/slogtest/handler.go`）**

提供一个线程安全的 in-memory `slog.Handler`，用于测试断言日志字段。

**背景**：
- ADR §4 列出 6 个结构化日志字段（`request_id` / `user_id` / `api_path` / `latency_ms` / `business_result` / `error_code`）
- Story 1.8 落地 AppError 时，中间件需要把 `*AppError.Code` 写进 `error_code` —— 必须能在单测里**断言**日志含该字段
- Story 1.3 `logging_test.go` 目前用 `bytes.Buffer` + `json.Unmarshal` 手测日志字段（一次性，工具味不浓）；本 story 抽成**公共 fixture**

**实装模板**：

```go
package slogtest

import (
    "context"
    "log/slog"
    "sync"
)

// Handler 是 in-memory slog.Handler，把 Record 存进内部 slice。
// 适用于测试断言"某条日志含 xxx 字段 / xxx 值"。
//
// 使用方式：
//
//	h := slogtest.NewHandler(slog.LevelDebug)
//	logger := slog.New(h)
//	// ... 执行被测代码，让它调用 logger.InfoContext(ctx, ...) ...
//	records := h.Records()
//	require.Len(t, records, 1)
//	got, ok := slogtest.AttrValue(records[0], "request_id")
//	require.True(t, ok)
//	assert.Equal(t, "rid-123", got.String())
//
// 线程安全：内部 sync.Mutex 保护 records slice；可用于并发测试场景。
type Handler struct {
    level slog.Level

    mu      sync.Mutex
    records []slog.Record
    // 继承自 WithAttrs / WithGroup 的 attr chain（按顺序 append 到每条 Record）
    attrs  []slog.Attr
    groups []string
}

// NewHandler 构造 Handler，设定最小日志级别（与 slog.Default 用的 LevelVar 语义一致）。
func NewHandler(level slog.Level) *Handler {
    return &Handler{level: level}
}

// Enabled：按 level 过滤。
func (h *Handler) Enabled(_ context.Context, lvl slog.Level) bool {
    return lvl >= h.level
}

// Handle：克隆 record（深拷贝 attrs），追加到 records slice。
// 必须复制 record —— slog.Record.AddAttrs 可能被 handler 共享修改。
func (h *Handler) Handle(_ context.Context, r slog.Record) error {
    h.mu.Lock()
    defer h.mu.Unlock()
    cloned := r.Clone()
    // WithAttrs 继承的 attr 链前置
    for _, a := range h.attrs {
        cloned.AddAttrs(a)
    }
    h.records = append(h.records, cloned)
    return nil
}

// WithAttrs：返回一个继承 attr 链的新 Handler（共享 records slice + mu）。
func (h *Handler) WithAttrs(as []slog.Attr) slog.Handler {
    newAttrs := append([]slog.Attr{}, h.attrs...)
    newAttrs = append(newAttrs, as...)
    return &Handler{level: h.level, records: h.records, mu: sync.Mutex{}, attrs: newAttrs, groups: h.groups}
    // ^ 注意：records / mu 不能真正"共享"（新 Handler 有自己的 mu）
    // 这是 MVP 实现的简化；真正需要跨 WithAttrs 统计记录数时直接用 parent handler
    // 本 story 的用法是**单个 Handler + logger.With 不跨层**，该简化足够
}

// WithGroup：类似 WithAttrs，记录 group 名。本 story MVP 实现不深入 group 渲染。
func (h *Handler) WithGroup(name string) slog.Handler {
    newGroups := append([]string{}, h.groups...)
    newGroups = append(newGroups, name)
    return &Handler{level: h.level, records: h.records, attrs: h.attrs, groups: newGroups}
}

// Records 返回已捕获的 Record 快照（深拷贝 slice header，元素仍指向原 Record）。
func (h *Handler) Records() []slog.Record {
    h.mu.Lock()
    defer h.mu.Unlock()
    out := make([]slog.Record, len(h.records))
    copy(out, h.records)
    return out
}

// AttrValue 在 Record 的顶层 attrs 里查找指定 key 的 slog.Value。
// 返回 (value, found)；groups / 嵌套 attr 不递归展开（MVP 够用）。
func AttrValue(r slog.Record, key string) (slog.Value, bool) {
    var found bool
    var value slog.Value
    r.Attrs(func(a slog.Attr) bool {
        if a.Key == key {
            value = a.Value
            found = true
            return false // 停止迭代
        }
        return true
    })
    return value, found
}
```

**约束 / 已知局限**（在包注释里明示）：
- MVP 实现**不**深入 group 渲染 —— 如果 Story 1.8 用到 `logger.WithGroup("error").InfoContext(ctx, "...")`，需要扩展 `AttrValue` 递归
- WithAttrs 的 records 共享策略是"新 Handler 复用 parent 的 records slice 指针但各自 mu"—— 简单场景下够用；复杂场景（parent 在并发写，WithAttrs 分支也在并发写）可能丢记录，那时升级实现
- **不**支持反序列化日志为 JSON string（如需断言 JSON 形状，用 `bytes.Buffer` + `slog.NewJSONHandler` 直连，不走本 Handler）

### 项目关键约束（必读，勿绕过）

1. **包名 `testing` 的陷阱**：`internal/pkg/testing/` 包名是 `testing` —— 与 stdlib `testing` 同名。调用方 **必须** `import testhelper "github.com/huing/cat/server/internal/pkg/testing"` 用别名 import，否则会覆盖 stdlib `testing`。本 story 的 `helpers.go` 顶部注释必须明示这点。另一种选择是**改包名**为 `testsupport` / `testhelper`，但 `internal/pkg/testing/` 路径已是 Go 生态常见约定（cobra / kubectl / go-chi 都用 `pkg/testing/`），路径保留，强制别名 import。
2. **sample service 放 `internal/service/sample/` 不放 `internal/domain/sample/`**：template 主题是"service 层 mock 注入"，不是"domain 层业务规则"；domain 层 `CanUpgradeRarity` / `CanOpenChest` 等 pure function 才放 `domain/`。这与架构文档 §4 的 "service 层 ≠ domain 层" 分层一致。
3. **epics.md 原 AC 字面是 `internal/service/sample_service.go`**（扁平文件），本 story 偏离到 `internal/service/sample/service.go`（子包）。理由：真实 service 层会是**每个 domain 一个子包**（Epic 4 会建 `internal/service/auth/`、Epic 7 建 `internal/service/step/`），扁平文件不代表未来实际布局；sample 必须是**未来布局的真实范本**。在 Completion Notes 明示该偏离（与 1.4 的 buildinfo 偏离同样机制）。
4. **`MockSampleRepo` 手写不走 codegen**（ADR §3.4）：未来 Epic 4+ 的 mock 都按本文件模板**手抄**。如果 dev 在 Epic 4 问"能不能用 mockery"，直接引用 ADR §3.4 否决理由。
5. **`testify/mock` 的 `Return(nil, err)` 陷阱**：当 interface 返回 `(*T, error)` 且第一个返回 nil 时，mock args.Get(0) 是 `interface{}(nil)`，**不能**直接 `.(*SampleDTO)` 断言（会 panic）。必须用 comma-ok 写法 `dto, _ := args.Get(0).(*sample.SampleDTO)`。**所有**后续 mock 接口都要注意这点；sample 文件里的 `MockSampleRepo.FindByID` 是标准示范。
6. **`t.Parallel()` 在 sample 测试里不开**：sample service 本身无副作用可并行，但本 story 主题不是"证明测试能并行"；future ctx cancel 测试（Story 1.9）会评估。过度 parallel 会隐藏 testify/mock 的期望竞态（mock.Mock.On 非线程安全）。
7. **sqlmock expectations 必须在 cleanup 检查**：`NewSQLMock` helper 在 `t.Cleanup` 自动调 `mock.ExpectationsWereMet`；dev 写测试时**不需要**自己调这个方法，只需在 test 开头调 `mock.ExpectQuery(...).WillReturnRows(...)` 设置期望。这是 sample 外的**约定**，helpers.go 注释要明示。
8. **miniredis 的 `FastForward` 是时间控制利器**：Epic 20 宝箱倒计时 / Epic 4 游客登录的 Redis TTL 都可以用 `mr.FastForward(time.Minute)` 跳时；sample 阶段不用，但在 helpers.go 顶部注释预告一句。
9. **structured log 字段名用 snake_case**：ADR §4 表格是 snake_case（`request_id` / `error_code` / `business_result`）；slogtest.AttrValue 测试断言也必须 snake_case。**禁止**在 log 调用里写 `slog.String("requestID", ...)` / camelCase —— 与 JSON response envelope（camelCase）**不同** body 的约定（logs snake_case / HTTP camelCase 是 Go 社区常见惯例）。

### 常见陷阱

1. **`go mod tidy` 剔除未 import 的包**：`dockertest` 如果加到 `require` 但没在任何 `.go` 文件 import → `go mod tidy` 会**自动删掉**。这就是本 story 不装 dockertest 的根本原因。**不要**为了"满足 ADR §6 版本清单"强加 `_ "github.com/ory/dockertest/v3"` 这种假 import（会被 code review 拒）。
2. **testify/mock 的 `args.Get(i)` 返回 `interface{}`**：强转必须 comma-ok（见 #5 Dev Notes），否则 nil 会 panic；单元测试 panic 会让 `go test` 输出 stack trace 压过测试结果，排障痛苦。
3. **sqlmock 匹配默认是 `QueryMatcherRegexp`**：`mock.ExpectQuery("SELECT .* FROM users")` 是**正则**不是**字面量**。如果想字面量匹配，构造时传 `sqlmock.QueryMatcherOption(sqlmock.QueryMatcherEqual)`。本 story 的 smoke test 用最简单的 `ExpectQuery("SELECT 1")` 避开这个陷阱 —— 但在 helpers.go 注释里**明示**默认 regex 行为。
4. **miniredis `FlushAll` vs `FlushDB`**：miniredis 默认只有 db 0（无多 db 概念），`FlushAll` 与 `FlushDB` 等价；别在代码里基于"多 db"设计 Redis key，会在 miniredis 下跑通但在真 Redis 下（Epic 4+）表现不同。
5. **slogtest.Handler 的 WithAttrs 局限**：本 story MVP 实现不真正"跨 WithAttrs 共享 records"—— 如果 Epic 4+ 用到 `logger.With(...).InfoContext(ctx, ...)` 跨层统计记录，需要扩展。这是已知局限，在 handler.go 顶部包注释明示。
6. **`t.Cleanup` 的执行顺序是 LIFO**：helpers.go 里 `t.Cleanup(cleanup)` 会在 test 结束时**最后注册最先执行**。如果 test 自己也注册 `t.Cleanup(...)`，注意顺序（测试 level cleanup 先于 helper level cleanup 执行）。本 story 的 helpers 无 ordering 依赖，不会踩坑；但模板被 Epic 4+ 扩展时要警惕。
7. **Windows CRLF**：`go test` 输出在 Windows 下可能带 CRLF；如果 Completion Notes 要贴 test output，用 `git-bash` 执行确保输出干净（与 Story 1.4 Completion Notes 做法一致）。
8. **`go.sum` 必须 commit**：`go mod tidy` 更新 `go.sum` 后**必须** `git add go.sum` —— 否则 CI 环境 `go mod download` 会因为 checksum 不匹配失败。这是 Go module 的 hash 校验机制，绕不过。
9. **`testify` 版本 v1.11.1 的 `require.ErrorIs`**：testify v1.11 才加了 `ErrorIs` / `ErrorAs` 对齐 Go 1.20+ errors.Is 语义；v1.10 及更早需用 `errors.Is(err, target)` + `assert.True`。本 story 锁定 v1.11.1 可以直接用 `assert.ErrorIs(t, err, sample.ErrSampleNotFound)`。
