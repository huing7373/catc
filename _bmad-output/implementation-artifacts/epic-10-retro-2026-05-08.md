## 2. 关键洞察与本 Epic 的"主旋律"

### 2.1 主旋律：复杂度与子循环成本的对撞

Epic 10 是项目第一次对接**双新中间件**（Redis + WS）+ **并发原语**（Session 生命周期、心跳、广播 fanout、presence 钩子），3 个 Story 的 review/fix 子循环跑满 10 轮，远超此前所有 Story 的 review 轮次峰值。

**结果**：commit `b5327df` 在 Epic 末尾把 epic-loop 子循环上限从 5 → 10。这是 Epic 10 期内最重要的一项**流程修订**。

**根因（不是单点）**：
1. **WS / 并发场景天然 review 轮次高** —— 每一轮都暴露上一轮的盲区（TOCTOU、shutdown 顺序、reconnect 顺序、跨 room 自愈、TTL 续期、per-user mutex…）。
2. **设计文档 §12 WS 协议本身在 10.1 里被 review 反复打磨** —— close code 段位、error 双语义、snapshot 全 roster 不变量、frozen 范围收窄都是 review 中暴露的合约盲区。
3. **Lessons 文档生效需要时间** —— 10.3 r1~r10 的不变量到 10.4/10.5/10.6 都还在被引用、被违反、再被加固。
4. **5 轮上限对此类 Story 是设计性失配**，硬卡上限会导致两种坏结果：要么 review 提前关闸放过未发现的不变量违反，要么 dev 回避高风险点降低 AC 严格度。

### 2.2 验证：Story 10.7 一次过

Story 10.7（snapshot 框架）位于 Epic 10 末尾，**0 轮 review/fix 子循环**就直接 done。这反向印证了一个非常正面的现象：**前 6 个 Story 沉淀的 35+ lesson 文件 + r10 系列锁定的不变量，让 10.7 的 dev/review/fix 循环在第一次就命中所有红线**。Lessons → 成本递减是真的，但前提是前面那些昂贵的 r1~r10 子循环不被人为掐死。

### 2.3 设计文档作为 single source of truth 的边界

10.1 跑满 10 轮 review 的核心症结：**V1 §12 WS 协议在 contract 阶段被作为"信封"反复打磨**，每一轮都暴露文档内部不一致或与下游 Story 假设冲突。教训：**协议契约 Story 不是"纯文档活"**，它的 AC 严格度等同于实装 Story；reviewer 必须把"协议文档自洽"当成 acceptance 一等公民来 hunt。

## 3. What Went Well（一定要保留的好做法）

### 3.1 强制 review/fix 子循环 + lesson 沉淀文化

- 每一轮 review 命中的问题都强制写 `docs/lessons/2026-05-XX-...-rN.md`，并在 dev-story 阶段被前置加载到 context；
- 文件名带 `-rN` 后缀（如 `10-6-r10`）让"这是某 Story 第 N 轮 review 才发现的"信息显式化；
- 后续 Story（如 10.5 / 10.7）显式 reference 前序 Story 的 lesson 名做 anti-pattern 列表 —— **Story 文件本身就是 lesson 的 retrieval 层**。

> 证据：10.5 story §"前置 lessons 提炼" 直接引用 10.4 r5 的 fanout drain；10.4 story §"前置 lessons 提炼" 引用 10.3 r2 的 onUnregister 钩子；10.7 不变量章节列出 10.3~10.6 各 r 锁定的 9 条不变量。

### 3.2 集成测试基线（dockertest MySQL + Redis）

10.3 起每个 Story 都以 dockertest 真实 wire 路径作为集成测试基线，而不是 mock conn / mock redis。这暴露了大量 mock 永远抓不到的问题（gorilla close frame 字节、`information_schema` 权限、`go-redis` ctx timeout 默认 false 等）。

### 3.3 严格的范围红线（"本 Story 不做"清单）

每个 Story 的 dev notes 都列出一份具体到接口/字段/方法的"不做"清单（如 10.3 list 16 条 anti-pattern），让 dev 阶段的范围扩张被主动遏制。这与"小步快跑"是同一种纪律的两面：**做的小 + 不做的清晰**。

### 3.4 ADR 紧跟实装

ADR-0011（WS stack）在 10.3 dev 过程中暴露 gorilla v1.5.4 不可获取的事实后立即落地，没有让"AC 钦定的版本号"与"实装的版本号"长期错位。

### 3.5 Build 脚本契约稳定

整个 Epic 期间没有触碰 `scripts/build.sh` 行为（`--test` / `--integration` / `--devtools` 互斥的形态稳定），所有 Story 的"完成验证"指令是同一行命令。

## 4. What Didn't Go Well（要修正的）

### 4.1 子循环上限失配（已修复）

- **现象**：10.1 / 10.3 / 10.6 跑到 r10 才收敛；epic-loop 原硬上限 5 在 10.1 阶段已被 manual override 突破。
- **影响**：epic 早期 epic-loop 行为与人工节奏不匹配，靠 commit 之间手动协调。
- **修复**：commit `b5327df` 把上限调到 10，并在 epic-loop skill 文档中记录。
- **状态**：✅ **已落地**，进入 Epic 11 时直接生效。

### 4.2 协议契约 Story（10.1）的 review 工作量没有提前预期

- **现象**：10.1 是"纯文档"Story，但 review 轮次与实装 Story（10.3 / 10.6）持平。
- **根因**：V1 §12 WS 协议草稿在 review 之前的内部一致性检查不充分（close code 段位、error 双语义、frozen scope 边界、placeholder 全 roster 不变量…）。
- **代价**：Story 10.1 的 r1~r10 本质上是"协议文档评审"挪到了 review 阶段做。

### 4.3 Story 10.6 进入 r5 后的折返（presence reconcile）

- **现象**：10.6 r3~r10 共 8 轮 fix 集中在 presence reconcile（scanner / partial-fail / room-aware guard / per-user mutex / hook 同步性…）。
- **根因**：lifecycle hook 与 Redis presence 的失败模式分析在 dev-story 阶段没有被穷举（reconnect / cross-room / scanner reconcile / shutdown 四种场景的笛卡尔积没拉表）。
- **后果**：r10 锁定的 hook **同步**语义颠覆了 r6 的 fire-and-forget 设计 —— **属于一次回头路**。

### 4.4 Lesson 命名长度

- **现象**：`docs/lessons/2026-05-07-presence-cross-room-reconnect-srem-old-room-and-scanner-isregistered-guard-10-6-r4.md` 这类文件名超过 100 字符。
- **影响**：Glob 列出来时 token 占用大；文件名级别难快速 scan。
- **状态**：未紧急到必须改，但可考虑把"主题 + r 标签"分离（主题做文件名，r 标签放 frontmatter）。

### 4.5 Race detector 在 Windows 缺位

- **现象**：dev 环境 Windows 无 cgo + gcc，`-race` 跑不起来；Epic 10 含大量并发原语（Session、scanner、broadcast fanout、presence hook），race 暴露面高。
- **缓解**：测试用真实 wire + 集成路径间接覆盖大部分 race；但 race detector 缺位是已知系统性 risk。
- **状态**：⏳ 暂留 tech debt；macOS / Linux / CI 上后续补 race 跑一遍。

### 4.6 Lesson 没有索引

- **现象**：`docs/lessons/` 目录已 80+ 文件，按时间命名但无主题索引（WS 协议 / Redis / presence / heartbeat / broadcast 等都混在同一层）。
- **影响**：未来 Story 如果想"找所有与 ws shutdown 相关的 lesson"只能 grep 文件名。
- **状态**：⏳ 入 tech debt；Epic 13（节点 4 demo 验收阶段）可以做一次 lesson 主题索引。

## 5. Lessons & Patterns 提炼（送给 Epic 11 / 14 / 17 等下游 WS 业务 epic）

### 5.1 横跨 7 个 Story 的不变量（review fix 阶段沉淀，必须保留）

| # | 不变量 | 来源 |
|---|---|---|
| 1 | Send / Close 并发安全（atomic.Bool + sync.Once） | 10.3 r1 |
| 2 | onUnregister hook 在所有 close 路径触发 | 10.3 r2 + 10.4 r6 |

| 8 | RemoveOnline 必须带 sessionID guard 走 Lua atomic | 10.6 r1 |
| 9 | AddOnline 必须自动 SREM 旧 room（cross-room reconnect 不留 stale） | 10.6 r10 |
| 10 | Presence hook 必须**同步**（不是 fire-and-forget） + 共享 per-user mutex | 10.6 r9 |
| 11 | Shutdown 必须 wait fire-and-forget hook 跑完才关 Redis | 10.6 r7 |

| 13 | snapshot 必须返全 roster（成员丰富字段允许空字符串降级，**不**允许整体 `members: []`） | 10.1 r8 |

下游 epic 改任何 WS 路径前**必须先比对这 13 条**。

### 5.2 流程层 lesson

1. **协议 contract Story 与实装 Story 同等严格** —— V1 §12 WS 协议的 r1~r10 教训之一就是"纯文档 Story 不能放松 reviewer 严格度"。
2. **review/fix 子循环上限要匹配 Story 复杂度** —— WS / 并发 / 资产事务这三类 Story，子循环上限 10 是经验值。
3. **Story 之间 lesson reference 是契约延续机制** —— Story 文件 §"前置 lessons 提炼" 不是装饰，是真正起作用的 prompt 工程结构。
4. **dev-story 阶段写 anti-pattern 清单 ROI 极高** —— 16 条 anti-pattern（10.3）让 review 阶段的 hunter 有现成 checklist。
