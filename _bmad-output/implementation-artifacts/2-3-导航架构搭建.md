## Story

As an iOS 开发,
I want 主界面有清晰的导航架构（push 走 NavigationStack，全屏页面走 Sheet）,
so that 后续 Epic（房间 / 仓库 / 合成 / 设置）按既定模式接入，不混用导航 pattern.

## 故事定位（Epic 2 第二条实装 story；Story 2.2 主界面骨架的导航接续）

这是 Epic 2 第二条实装 story，紧接 Story 2.2 (`done`) 的主界面骨架。Story 2.2 把三个主按钮的 `onRoomTap` / `onInventoryTap` / `onComposeTap` 闭包**注册占位空函数**，本 story 把这些闭包替换为 `coordinator.present(.room/.inventory/.compose)` 路由调用，并第一次落地 `AppCoordinator` + `SheetType` 枚举 + `RoomPlaceholderView` / `InventoryPlaceholderView` / `ComposePlaceholderView` 三个 Sheet placeholder。

**本 story 是后续 Epic 12 / 24 / 33 的桥头堡**：
- Epic 12 Story 12.1 RoomView 替换 `RoomPlaceholderView` 内容（保持 Sheet pattern + 路由约定）
- Epic 24 Story 24.1 / 24.4 InventoryView 替换 `InventoryPlaceholderView` 内容
- Epic 33 Story 33.1 / 33.6 ComposeView 替换 `ComposePlaceholderView` 内容

**不涉及**：APIClient（→ 2.4）/ ping & version 真实调用（→ 2.5）/ 错误 UI（→ 2.6）/ 测试基础设施完整搭建（→ 2.7）/ Dev 重置 Keychain（→ 2.8）/ LaunchingView（→ 2.9）/ README（→ 2.10）/ NavigationStack 内部 push 流程（节点 4+ 实际有 push 需求时再加）/ Sheet 内嵌套 Sheet（MVP 不做）/ Sheet 转场动画自定义（用 SwiftUI 默认）。

**范围红线**：

- **不动 `ios/`**：本 story **绝对不**修改 `ios/` 任何文件（ADR-0002 §3.3 + Story 2.2 AC9 强约束的延续）。最终 `git status` 须确认 `ios/` 下无任何 modified / staged。
- **不动 `server/`**：本 story 是 iOS 端导航实装，不涉及 server。
- **不写 APIClient / Networking**：`PetApp/Core/Networking/` 仍不创建任何 .swift 文件（避免提前实装）。
- **不动 Story 2.2 已落地的 6 大占位区块的 a11y identifier 字符串值**：`AccessibilityID.Home.userInfo / petArea / stepBalance / chestArea / btnRoom / btnInventory / btnCompose / versionLabel` 8 个常量值**保持不变**（防止 UITest `testHomeViewShowsAllSixPlaceholders` 回归失败；可在 `AccessibilityID.swift` **追加** Sheet / placeholder 相关的新 namespace 与新常量，但**不改老的**）。

## Acceptance Criteria

**AC1 — `AppCoordinator` 落地（对应 ADR-0002 §3.3 / iOS 架构 §4 `App/AppCoordinator.swift`）**

新建 `iphone/PetApp/App/AppCoordinator.swift`，含：

```swift
import Foundation
import Combine
import SwiftUI

/// Sheet 路由枚举：列举主界面可弹出的全屏 Sheet 类型。
/// MVP 阶段仅 3 类（房间 / 仓库 / 合成），后续 epic 扩展时追加。
public enum SheetType: Identifiable, Equatable {
    case room
    case inventory
    case compose

    public var id: String {
        switch self {
        case .room: return "sheet_room"
        case .inventory: return "sheet_inventory"
        case .compose: return "sheet_compose"
        }
    }
}

/// 集中管理 App 内全屏 Sheet 的状态。
/// 主界面三个 CTA 按钮通过 coordinator.present(.room/.inventory/.compose) 触发 Sheet 弹出。
/// Sheet 内部点击关闭按钮调用 coordinator.dismiss() 返回主界面。
@MainActor
public final class AppCoordinator: ObservableObject {
    @Published public var presentedSheet: SheetType?

    public init(presentedSheet: SheetType? = nil) {
        self.presentedSheet = presentedSheet
    }

    public func present(_ sheet: SheetType) {
        // 已有 Sheet 时被新值覆盖（`@Published` 直接赋值即可）。
        // SwiftUI .fullScreenCover(item:) 在 item 从非 nil 切到非 nil 时
        // 会先 dismiss 旧 sheet 再 present 新 sheet（系统行为，本 story 不改）。
        presentedSheet = sheet
    }

    public func dismiss() {
        presentedSheet = nil
    }
}
```

**关键约束**：
- 命名用 `AppCoordinator`（与 iOS 架构 §4 目录约定一致）；epics.md Story 2.3 AC 提到 "AppCoordinator 或 NavigationRouter"，本 story 选 **`AppCoordinator`**（避免引入 "Router" 概念让 dev 困惑 push 与 sheet 的边界）。
- `SheetType` 是 `Identifiable`：SwiftUI `.fullScreenCover(item:)` 要求 item 类型符合 `Identifiable`。
- `SheetType` 是 `Equatable`：测试中 `XCTAssertEqual(coordinator.presentedSheet, .room)` 需要。
- `@MainActor`：与 `HomeViewModel` 一致风格；coordinator 状态仅在 UI 主线程被读写。
- `presentedSheet` 默认值 `nil`（无 sheet）。
- `import Combine`（Story 2.2 review 已 lesson learned：`ObservableObject` / `@Published` 必须显式 import Combine）。
- `init(presentedSheet:)` 暴露 `nil` 默认值便于测试时构造非 default 状态。

**AC2 — `RootView` 接入 AppCoordinator + `.fullScreenCover(item:)`**

修改 `iphone/PetApp/App/RootView.swift`：

```swift
import SwiftUI

struct RootView: View {
    @StateObject private var coordinator = AppCoordinator()
    @StateObject private var homeViewModel = HomeViewModel()

    var body: some View {
        HomeView(viewModel: homeViewModel)
            .onAppear {
                wireHomeViewModelClosures()
            }
            .fullScreenCover(item: $coordinator.presentedSheet) { sheet in
                sheetContent(for: sheet)
            }
    }

    /// 把 HomeViewModel 三个 CTA 闭包接到 coordinator.present(...)。
    /// .onAppear 时机重新 wire 一次（防止 RootView 重新构建后失去引用），不重复注册不会导致 leak —— 闭包都是值类型，每次赋值覆盖前一个。
    private func wireHomeViewModelClosures() {
        homeViewModel.onRoomTap = { [coordinator] in
            coordinator.present(.room)
        }
        homeViewModel.onInventoryTap = { [coordinator] in
            coordinator.present(.inventory)
        }
        homeViewModel.onComposeTap = { [coordinator] in
            coordinator.present(.compose)
        }
    }

    @ViewBuilder
    private func sheetContent(for sheet: SheetType) -> some View {
        switch sheet {
        case .room:
            RoomPlaceholderView(onClose: { coordinator.dismiss() })
        case .inventory:
            InventoryPlaceholderView(onClose: { coordinator.dismiss() })
        case .compose:
            ComposePlaceholderView(onClose: { coordinator.dismiss() })
        }
    }
}
```

**关键约束**：
- 两个 `@StateObject`（`coordinator` + `homeViewModel`），都在 `RootView` 持有生命周期（与 Story 2.2 选定的 "RootView 持有 + HomeView 观察" 模式一致）。
- 三个 closure wire 在 `.onAppear` 中（不是 `init`）：① `init` 时 `coordinator` 还没真正初始化完毕（`@StateObject` 在 view 第一次出现前才建好），在 init 写会捕获到错误的实例；② `.onAppear` 在 view 出现时被调一次，足够稳定；③ 多次 `.onAppear`（如系统从后台回前台）只是覆盖闭包，不影响行为。
- `[coordinator]` capture list 确保闭包不强引用 self（`RootView` 是 struct，但保险起见显式 capture coordinator）。
- `.fullScreenCover(item:)` 用 `$coordinator.presentedSheet` 双向绑定：用户从屏幕下拉 Sheet（系统手势）也能让 coordinator 状态归零（**注意**：`.fullScreenCover` 默认**不允许**用户下拉 dismiss，本 story 仅靠 placeholder 内的"关闭"按钮触发 `dismiss()`；这是按 epics.md AC "Sheet 顶部有'关闭'按钮"的明确要求）。

### 关键技术细节

#### 1. SwiftUI `.fullScreenCover(item:)` 与 `Identifiable`

`.fullScreenCover(item:)` 的标准签名：
```swift
func fullScreenCover<Item, Content>(
    item: Binding<Item?>,
    onDismiss: (() -> Void)? = nil,
    @ViewBuilder content: @escaping (Item) -> Content
) -> some View where Item : Identifiable, Content : View
```

**为什么用 `item:` 而不是 `isPresented:`**：
- `.fullScreenCover(isPresented: Binding<Bool>)` 需要外部维护"哪个 sheet 在 present"；用 enum + `item:` 一步到位，coordinator 只持有一个 `SheetType?`。
- `Identifiable` 的 `id` 让 SwiftUI 能区分"同一类 sheet"vs"切到另一类 sheet"，避免错位。
- `item: $coordinator.presentedSheet` 双向绑定：当 sheet dismiss（系统手势 / 程序主动）时 SwiftUI 把 binding 设为 nil，coordinator 状态自动同步。

#### 2. `@StateObject` 在 RootView 中的生命周期

```swift
struct RootView: View {
    @StateObject private var coordinator = AppCoordinator()      // 实例 A
    @StateObject private var homeViewModel = HomeViewModel()     // 实例 B
    ...
}
```

- `@StateObject` 在 view 第一次出现时初始化一次，view 重建时**不**重新初始化（这是与 `@ObservedObject` 的核心区别）。
- 两个 `@StateObject` 在同一 view 持有，生命周期都跟 RootView 走 —— App 整个生命周期内同一 coordinator + 同一 homeViewModel。
- **不要** 把 coordinator 改成 `@StateObject` 在 `HomeView` 内：① `HomeView` 在 `.fullScreenCover` 弹出时仍是 hostingView，不会被 unmount，但放 HomeView 内意味着主界面以外（如 Story 2.9 LaunchingView）将无法访问 coordinator；② RootView 持有更对齐 iOS 架构 §4 `App/` 目录语义。

#### 3. RootView 的 closure wire 时机：`.onAppear` vs `init`

`init` 内**不能** wire，因为 `@StateObject` 的实际实例在 `init` 阶段未初始化（SwiftUI 内部用 lazy storage）。最稳的时机是 `.onAppear`：
- view 第一次显示时调一次 → wire 三个 closure
- view 从后台回前台再次 onAppear 时再调一次 → 闭包覆盖（值类型，不会 leak）
- view dismiss 时不需要 unwire，闭包随 viewModel 销毁

**替代方案**（不选）：`.task { ... }` async 上下文。本 story 闭包 wire 是同步的，用 `.onAppear` 更直白。

#### 4. SheetType 命名与扩展性

#### 5. Story 2.2 review 的 lessons learned 应用

Story 2.2 在 review 阶段被发现两个问题，本 story 必须避免重复：
- **`@Published` / `ObservableObject` 必须显式 `import Combine`**（commit `6b1f45e`）：`AppCoordinator.swift` 顶部加 `import Combine`，不依赖 SwiftUI 隐式传递。
- **`ZStack` overlay 不能盖在底部 CTA 行上**（commit `4de0140`）：`RootView` 的 `.fullScreenCover(item:)` modifier 加在 view body 之后即可，**不**用 ZStack overlay 实现自定义 sheet（SwiftUI 原生 fullScreenCover 已经全屏覆盖，不会和按钮区域冲突）。

#### 6. SwiftUI `.fullScreenCover` 默认不允许下拉关闭

iOS 上 `.fullScreenCover` 是**模态**全屏，没有下拉手势 —— 必须靠按钮触发 `dismiss()`。这正是 epics.md Story 2.3 AC "Sheet 顶部有'关闭'按钮"的要求。**不要**改用 `.sheet(item:)`（后者是半屏、可下拉，不符合 epics 要求）。

#### 7. UITest 中验证 sheet "消失" 的方式

XCUITest 中验证某 element "消失" 不能简单 `XCTAssertFalse(element.exists)`，因为 SwiftUI sheet dismiss 有动画时长。推荐两种方式之一：

**方式 A**：用 `XCTNSPredicateExpectation`：
```swift
let notExistPredicate = NSPredicate(format: "exists == false")
let expectation = XCTNSPredicateExpectation(predicate: notExistPredicate, object: element)
wait(for: [expectation], timeout: 5)
```

**方式 B**：等待主界面元素重新出现（间接证明 sheet 已 dismiss）：
```swift
XCTAssertTrue(app.descendants(matching: .any)[AccessibilityID.Home.userInfo].waitForExistence(timeout: 5))
```

**推荐方式 B**（更轻量，与 Story 2.2 已用的 `waitForExistence(timeout:)` 风格一致）。

### Debug Log References

#### 1. UITest 第一次跑 3/3 失败：sheetPlaceholder_*Title 找不到

**症状**：
- `xcodebuild test -only-testing:PetAppUITests` 跑下来，3 个新增导航 UITest 全部 fail（room/inventory/compose 各 1）；`waitForExistence(timeout: 5)` 5 秒等不到 `sheetPlaceholder_roomTitle` 等元素出现。Story 2.2 老 UITest 仍然全绿。
- Sheet 实际**已经**弹出（按钮点击不抛错，hierarchy 中能看到 "Room Placeholder" 文字）。

**根因**：
- 通过 `xcrun xcresulttool export object --legacy` 导出 fail 时刻的 UI 树发现，placeholder 内的 `Button` 和 `StaticText` 的 `accessibilityIdentifier` 全部被父 `VStack` 的容器 identifier `sheetPlaceholder_room` 覆盖（SwiftUI 默认行为：父 view 的 a11y identifier 会传播到所有子 view）。
- 导致 `app.descendants(matching: .any)["sheetPlaceholder_roomTitle"]` 在整个树里找不到任何匹配。

**修复**：
- 三个 placeholder view 的最外层 VStack 改成同时加 `.accessibilityElement(children: .contain)` + `.accessibilityIdentifier(...)` 两个 modifier。前者告诉 SwiftUI"我是容器，但子元素保留独立的 accessibility 身份"，后者再给容器自身一个 identifier。
- 修复后 UI 树中容器、关闭按钮、标题文字三个 identifier 都独立存在，UITest 全绿。
- 在三个 placeholder 文件加了详细注释说明这一约束（避免后续维护时踩坑）。

**Lesson Learned 沉淀（供 Story 2.4+ 参考）**：
- ⚠️ **SwiftUI `.accessibilityIdentifier(...)` 在父容器上会传播到所有子元素**，导致子元素自己的 identifier 被覆盖。如果父容器和子元素都需要独立 identifier（如本 story 的 placeholder 容器 + 子标题 + 子按钮），父容器必须配 `.accessibilityElement(children: .contain)`。
- ⚠️ **新增 source 目录后必须跑 `xcodegen generate`**，让 `project.pbxproj` 接收新 file references；否则 xcodebuild 不会编译新文件。
