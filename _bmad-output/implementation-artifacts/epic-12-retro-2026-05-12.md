## 2. 关键洞察与 Epic 12 的"主旋律"

### 2.1 主旋律 1: Story 12.7 跑到 **r14**，与 Epic 11 Story 11.1（r14）持平

Story 12.7（CreateRoom / JoinRoom / LeaveRoom UseCase + 主界面状态机闭环）从 r1 一路演进到 r14 才收敛 —— 与 Epic 11 Story 11.1 协议契约 Story 的 r14 纪录持平。

**复杂度根因不同于 11.1**：11.1 是协议契约文档审计；12.7 是 **多层组件的运行时交互**：HTTP APIClient + WebSocketClientImpl + RoomViewModel + AppState 状态机 + TabView 导航栈 + UITEST 旁路 + ErrorPresenter wiring 共 **7 层组件** + 跨"创建 / 加入 / 退出 / UITEST 启动 / 错误恢复"5 条业务路径笛卡尔积。每一轮都命中真实 P1（不是 cosmetic）：

| 轮次 | 暴露维度 | 关键 lesson |
|---|---|---|
| r1 | WS connect 同步抛错 + URL path roomId 注入风险 | connect() 同步 throws 路径必须区分于 async firstFrame await |
| r2 | UITEST 路径必须保留 UseCase nil-fallback + LeaveRoom guard target==current | stale response 防御要覆盖 leave path |
| r3 | WS connect 失败不弹 errorPresenter alert（transient）+ UITEST 路径必须跳过 real WS / errorPresenter wiring | transient WS 失败语义与业务错误不同语义 |
| r4 | UITEST nil-WS-client 路径 errorPresenter wiring 不能掉 | UITEST 旁路必须为子集兼容而非全屏蔽 |
| r5 | LeaveRoom thrown-error 也要 stale guard + create useCase nil 不能 hard no-op | nil-UseCase 与 nil-Client 是两个独立旁路维度 |
| r6 | CreateRoom / JoinRoom 必须 guard entry==current 防 stale-response 抹掉新房间 | 异步 success path 与 error path 必须共享 stale-guard 框架 |
| r7 | RoomEndpoints percent-encode pre-escaped 输入也要 escape `%` | encoder 必须幂等：double-escape > silent corruption |
| r8 | 业务错误码 fallback 必须 forward 原 error（禁止 rewrap 成空 message/requestId） | error wrapping must be additive, not replacement |
| r9 | 异步 error catch 必须做 stale-guard：roomId + client identity 两层 | 两层 identity（roomId + client）才能根治 reconnect-during-error race |
| r10 | Room navigation 用 monotonic generation token 而非 currentRoomId equality | ABA race 必须 monotonic counter，equality 不够 |
| r11 | Room 进入路径必须先清 scaffold roster 再 connect | scaffold seed 与真实 connect 之间必须显式 reset |
| r12 | applyHomeData 仅在 currentRoomId 实际变更时 bump generation token | generation token 必须严格对应"逻辑切换"而非"消息收到" |
| r13 | UITEST nil-WS-client 路径 nil→A 必须保留 RoomScaffoldDefaults seed roster | r11 与 UITEST 旁路语义冲突的再次精化 |
| r14 | UseCase stale-success 路径必须 throw error 触发 home refresh | success path 也可能是错误：成功了"错误的事" |

**结果**：12.7 是"客户端集成 Story 的复杂度天花板还没摸到"。如果硬卡 r10，r11~r14 这 4 条不变量（roster reset 时机 / generation token 实际语义 / UITEST 子集兼容 / stale-success 也要 throw）会被静默放过。

**与 Epic 11 retro §2.1 主旋律 1 同源结论**：客户端"集成 Story"（X.7 收尾 use-case + 状态机）的审计面与服务端"协议契约 Story"（X.1）的审计面**复杂度相当**，都可能突破 r10。

### 2.2 主旋律 2: Story 12.6 心跳维护跑了 **8 轮折返**（r1~r8），中间出现**回头路**

12.6 心跳维护跑了 8 轮。每一轮揭露"看似简单的 ping/pong" 实则牵动 **5 个并发 / 生命周期维度**：

| 轮 | 维度 |
|---|---|
| r1 | transient reconnect 前 heartbeat 必须 reset（pendingPongContinuation 切换 + cancel + finish） |
| r2 | pong 配对 —— 严格 requestId vs 宽容 any-pong：选 **宽容**（避免 stale ping 的迟到 pong 误清新 ping 的等待） |
| r3 | heartbeat send 抛错必须走 cancel(.goingAway) → receive-loop classifier 单一真源，**禁止直接 scheduleReconnect()** |
| r4 | lock-unlock 后 pre-send 必须重新校验 + send 必须用 captured task 引用，不能 re-read self.underlyingTask（TOCTOU） |
| r5 | task identity 守护必须独立于 sessionGeneration（task 可能跨多个 session lock 段） |
| r6 | terminal post-handshake close（如 4007 LeaveRoom）必须与 transient 分支对齐 cancel heartbeat 子系统 |
| r7 | cancel(.goingAway) **不能** 覆盖 server 真实 close code（heartbeat send catch 路径） |
| r8 | closeCode TOCTOU re-check 必须原子化折进 helper 内部 |

**回头路**：**r3 单一真源（必须走 receive-loop classifier）** → **r7 反例（cancel 注入 .goingAway 会覆盖 server 真实 close code，所以 send catch 路径不能简单 cancel）**。r7 不是推翻 r3，而是 r3 的 boundary case 暴露：r3 的"单一真源"在 send catch 路径里若使用同样 cancel(.goingAway) 会污染 close code 分类。

**这是 Epic 11 retro §2.2 主旋律 2 的客户端版本**：fire-and-forget 异步化 + 共享可变状态 + lock TOCTOU 共同形成的"5 维笛卡尔积陷阱"，dev-story 阶段无法穷举枚举。

### 2.3 主旋律 3: "Generation counter 隔离 stale task" 成为 Epic 12 的核心原语

12.1 r5 `bind() 替换 client instance 必须先 disconnect 旧 client + cancel 旧 task` 第一次显式提出"消费者重启 gated on 实际 client swap"。

12.5 r2 把它升级为 **WS reconnect 状态机用 generation counter 隔离 stale task 共享状态写入**：每次 reconnect 把 sessionGeneration++，stale Task 在每次访问共享状态前必须 `assert isCurrentSession(myGen)`。

12.5 r3 / r4 进一步把 generation 推广：
- r3：**connectGate resolve 也必须 generation-gated**（stale defer 不能污染新 session）
- r4：**stream 复用 vs session 翻新必须双 generation 解耦**（streamGeneration 独立于 sessionGeneration）

12.6 整 Story 都在这个原语上运行（task identity 守护、pre-send 重新校验都是 generation-gate 的派生）。

12.7 r10 把这个原语 **lift 到导航层**：Room navigation 用 monotonic generation token，currentRoomId equality 不够（ABA race：A→B→A 在 equality 视角看"无变化"）。

**结果**：Epic 12 沉淀出"**双层 generation token**"架构原语 —— 数据层（sessionGeneration / streamGeneration）+ 导航层（roomNavigationGenerationToken），两层独立计数、独立 reset 边界。**Epic 14 / 15 / 17 / 18 等下游异步化 Story 直接 fork 此原语**。

## 4. What Didn't Go Well（要修正的）

### 4.1 一次过 Story 仅 1 个 —— lesson 复利**未**兑现

- **现象**：7 Story 仅 12.3 一次过；12.1 r6 / 12.5 r5 / 12.6 r8 / 12.7 r14 都跑满或超 r10。
- **根因**：
  1. Story 之间互相 reference 密度低（除 12.5→12.6→12.7 generation 原语线）
  2. 客户端 anti-pattern 库没建立（vs 服务端的事务 / ACL / post-commit）
  3. Epic 37 lesson 与 Epic 11 lesson 都不直接覆盖客户端并发原语
- **建议**：Epic 13 文档同步阶段建立**客户端 anti-pattern checklist**（async task generation / Combine pipeline / @MainActor hop / SwiftUI Sheet stack 四大主题）。

### 4.2 epic-loop r10 上限对客户端集成 Story（X.7）不够

- **现象**：12.7 跑到 r14，与 Epic 11 Story 11.1（协议契约）持平。
- **根因**：客户端集成 Story 的"组件层数 × 业务路径"笛卡尔积复杂度（7 层组件 × 5 条业务路径）已与协议契约 Story 持平。
- **建议**：Epic 11 retro A1 钦定的"epic-loop 上限按 Story 类型分级"必须**扩展**：协议契约 Story（X.1）+ **客户端收尾集成 Story（X.7 多组件状态机）** 都应去掉硬上限，由 reviewer 自然收敛。

### 4.3 Story 12.6 r3 → r7 出现 boundary case 回头路

- **现象**：r3 钦定"send 抛错必须走 cancel(.goingAway) → receive-loop classifier 单一真源"；r7 揭露"send catch 路径若用 .goingAway cancel 会覆盖 server 真实 close code"，事实上需要在 send catch 时**区分有无 server close code**才能决定是否 cancel(.goingAway)。
- **根因**：r3 的"单一真源" architectural decision 在 send catch 这个 boundary 处与"close code 信号完整性"原则冲突，dev-story 阶段未穷举 boundary。
- **后果**：1 次回头精化（r7 不是推翻 r3，是 r3 的 boundary 覆盖）。属正常并发探索成本。
- **状态**：已落地 r7 lesson `2026-05-XX-ws-heartbeat-send-catch-cannot-inject-going-away-close-code-12-6-r7.md`。

### 4.4 客户端 race condition test 仍偏弱

- **现象**：12.5 / 12.6 / 12.7 的 stale-task / generation-counter race 都靠 review/fix 子循环挖出，单元测试 fake-clock 路径无法穷举枚举（reconnect during error / leave during connect / cross-room rapid switch 等）。
- **代价**：r1~r14 的 stale-guard fix 都靠"reviewer 想象路径"+"dev 手动追测"，自动化覆盖率低。
- **状态**：⏳ tech debt（Epic 13 文档同步阶段登记）；Epic 13 Story 13.1 E2E 场景 7（WS 重连）+ 场景 9（Background 模式）必须显式覆盖 stale-task race。

### 4.5 UITEST 旁路成为独立维度，未来扩展可能挤压主干清晰度

- **现象**：12.7 r2 / r3 / r4 / r13 反复在"UITEST 路径 nil-Client / nil-UseCase / 保留 ErrorPresenter wiring / 保留 RoomScaffoldDefaults seed roster"四个边界来回演进。
- **根因**：UITEST 旁路（XCUITest 启动时通过 launch arguments 注入 mock state）与 production 路径在 ErrorPresenter / WS client / UseCase 三处都有"是否注入"开关，组合数 2^3 = 8 种状态，r2~r4 / r13 都在补全这 8 种状态的兼容矩阵。
- **建议**（非紧急）：Epic 13 / 后续 epic 增加 UITEST 旁路时，dev-story 阶段必须画"UITEST 兼容矩阵"（哪些组件 mock-out / 哪些保留 wiring / 哪些保留 seed），避免在 review 阶段一格一格挖。
