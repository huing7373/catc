# Story 21.5: 开箱前主动同步步数（DI wire StepSyncTriggerService 进 OpenChestUseCase + "同步步数中…" loading 提示）

Status: review

<!-- Validation 可选。建议运行 validate-create-story 在 dev-story 前做一次质检。 -->

## Story

As an iPhone 用户,
I want 我点开箱前 App 自动同步一次步数到 server,
so that server 端用最新可用步数判断我是否够开（避免本地步数已增加但 server 还没收到导致 3002 "步数不足" 误报；同步耗时超过 2 秒时还能看到 "同步步数中…" 提示，知道 App 没卡死）.

## 故事定位（Epic 21 第 5 条 = 收官 story；接 Story 8.5 `StepSyncTriggerService.triggerManual()` + Story 21.3 已预埋的 OpenChestUseCase hook）

这是 Epic 21「iOS - 首页宝箱倒计时 + 奖励弹窗」最后一条 story。Epic 21 进度：21-1（ChestCardView，**done**）→ 21-2（LoadChestUseCase + ChestRefreshTriggerService，**done**）→ 21-3（OpenChestUseCase + onChestOpenTap + isOpening / pendingReward，**done**）→ 21-4（RewardPopupView + sheet 装配，**done**）→ **21-5（本 story；开箱前主动 sync steps）**。本 story done 后 Epic 21 全 done，解锁 Epic 22（节点 7 宝箱开箱链路联调）。

**关键事实：本 story 的核心编排早已在 Story 21.3 预埋落地，本 story 主要是「接通线缆」 + 加 loading 提示。** Story 21.3 在 `DefaultOpenChestUseCase` 已实装：


因此本 story 的实际工作量集中在 **3 件事**：

1. **DI wire（核心）**：把 RootView 已持有的同一个 `stepSyncTriggerService` 实例传进 `makeOpenChestUseCase` → `DefaultOpenChestUseCase(stepSyncTriggerService:)`，替换 21.3 占位的 `nil`。
   - **关键陷阱**：`AppContainer.makeStepSyncTriggerService(...)` 每次调用返回**新实例**（Story 8.5 钦定 + lesson `2026-04-26-stateobject-debug-instance-aliasing.md`）；RootView 通过 `ensureStepSyncWired()` lazy 构造并用 `@State private var stepSyncTriggerService` 持有**唯一实例**（持 5min 定时器 + in-flight gate + timerTask）。**绝对不能**在 `makeOpenChestUseCase` 内部再 `makeStepSyncTriggerService(...)` 造第二个 —— 那会造出第二个独立的 in-flight gate + 第二个 5min timer + 第二个 weak homeViewModel 借用链，违反 Story 8.5「同步不重叠」「单 timer」不变量（lesson `2026-04-26-stateobject-debug-instance-aliasing.md` + `2026-05-04-scenephase-idempotent-start-no-duplicate-trigger.md`）。必须复用 RootView 持有的那一个。
   - **关键时序问题**：RootView `.onAppear` 块内，`realHomeVM.bind(openChestUseCase: container.makeOpenChestUseCase(appState:))` 在**第 388-389 / 404-406 行**（两个 UITEST gate 分支各一处），但 `ensureStepSyncWired()`（构造 `stepSyncTriggerService`）在**第 486 行** —— 即 OpenChestUseCase 被造出来时 `stepSyncTriggerService` 还是 `nil`。本 story 必须**把 `ensureStepSyncWired()` 上移到两处 `bind(openChestUseCase:)` 之前**，并把 `self.stepSyncTriggerService` 实例传进 `makeOpenChestUseCase`。
2. **"同步步数中…" loading 提示（spec AC 行 3140）**：同步耗时 > 2 秒时 ChestCardView 副标题显示 "同步步数中…"（在既有 21.3 `isOpening` → "开箱中…" 之上多一档 "同步步数中…" 中间态）。需要一个新的 ViewModel transient @Published 标志 + ChestCardView 副标题三态派生 + RealHomeViewModel.onChestOpenTap 内用 2 秒延迟 Task 控制显隐。
3. **测试**：≥4 单测 case（mocked SyncStepsUseCase + APIClient）+ ≥1 集成测试（XCUITest + mock server，验证 sync→open 两次请求顺序 + open 用新余额）。

- 现状：`.onAppear` 块内 `realHomeVM.bind(openChestUseCase: container.makeOpenChestUseCase(appState: appState))` 在第 388-389 行（非 UITEST 分支）+ 第 404-406 行（UITEST_SKIP_GUEST_LOGIN 分支）；`ensureStepSyncWired()` 在第 486 行。
- 改动：把 `ensureStepSyncWired()` 调用**上移到两处 `realHomeVM.bind(openChestUseCase:)` 之前**（同时保留它在 `ensureLaunchStateMachineWired()` 之后的相对顺序约束 —— 见 RootView line 484-485 注释「必须在 ensureLaunchStateMachineWired 之后调」；所以 `ensureLaunchStateMachineWired()` 也需一并上移，或验证其上移不破坏既有依赖）。

- **关键不变量**：上移 `ensureStepSyncWired()` 不能破坏它原依赖 `homeViewModel`（`makeStepSyncTriggerService(appState:homeViewModel:)` 需 homeViewModel）+ `appState`。验证 `.onAppear` 块内 `homeViewModel.bind(appState:)`（line 337）+ ensureStepSyncWired 依赖的 homeViewModel / appState 在上移后仍已就绪（homeViewModel 是 @StateObject 一直存在；appState 是 @StateObject 一直存在；ensureStepSyncWired 仅需这两者 + container，不依赖 line 380 的 bind(createRoomUseCase:) 等 → 安全上移）。
- **关键不变量**：保留 `ensureLaunchStateMachineWired()` → `ensureStepSyncWired()` → `ensurePetStateSyncWired()` → `ensureChestRefreshWired()` 的相对顺序（RootView line 483-491 钦定「与未来 audit 期望顺序一致」）；本 story 整体上移这一组 ensure*Wired() 到 bind(openChestUseCase:) 之前即可，组内顺序不变。

**验收口径**：

- 非 UITEST 路径：用户点开箱 → `OpenChestUseCase.execute()` 内 `if let stepSync` 为真（stepSyncTriggerService 非 nil）→ `await stepSync.triggerManual()` 真实执行 → POST /steps/sync → 再 POST /chest/open；

**对应 spec AC 行 3140**：`同步耗时 > 2 秒时显示 loading 提示 "同步步数中…"`。

**改动文件**：

- `iphone/PetApp/Features/Home/ViewModels/RealHomeViewModel.swift`（onChestOpenTap 内编排 2s 延迟 task 控制显隐）

**ViewModel 新增字段契约**（紧跟 `isOpening` / `pendingReward` 之后，HomeViewModel.swift line 163-171 区）：

```swift
/// Story 21.5 AC2: 开箱前步数同步耗时 > 2s 时的 loading 提示标志.
///
/// 单位：Bool；初值 false.
///
/// **派生契约**：
/// - ChestCardView.unlockableView 副标题三态优先级：isSyncingSteps → "同步步数中…" > isOpening → "开箱中…" > "可开启"
///   （isSyncingSteps 优先于 isOpening，因为 sync 发生在 open 之前，时序上先 sync 提示后 open 提示）
/// - 由 RealHomeViewModel.onChestOpenTap 控制：
///     - 起 2s 延迟 Task：sleep 2s 后若 sync 仍未完成 → isSyncingSteps = true
///     - sync（triggerManual）完成后立即 cancel 该延迟 task + isSyncingSteps = false
/// - **不**上 AppState（与 isOpening / pendingReward 同精神：transient view-state；ADR-0010 §3.2 钦定 loading flag 归 ViewModel @Published）.
///
/// 不变量:
/// - isSyncingSteps 仅在 "开箱按钮已点 + sync in-flight 且已超 2s" 窗口为 true
/// - sync 完成（不论成功 / 失败）后必复位 false（与 isOpening defer 复位同精神）
/// - 同步耗时 < 2s 时该标志全程 false（2s 延迟 task 在 sync 完成时被 cancel，没机会 set true）
@Published public var isSyncingSteps: Bool = false
```

**RealHomeViewModel.onChestOpenTap 编排改动契约**（line 394-455 既有方法内，在 `useCase.execute()` 调用前后包裹 2s 延迟控制）：

> **关键设计约束**：`OpenChestUseCase.execute()` 内部 Step 0 才调 `await stepSync.triggerManual()`，ViewModel 层**看不到** sync 与 open 的边界（execute() 是黑盒）。本 story 不改 execute()（红线）。因此 "sync 耗时 > 2s" 的判定无法精确锚定到 "triggerManual 返回那一刻"。**钦定折中方案**：以 "execute() 整体调用" 起点起 2s 延迟 task —— 2s 后若 execute() 仍未返回则 set `isSyncingSteps = true`；execute() 返回（成功 / 失败 / catch）后立即 cancel 延迟 task + set false。
>
> 语义解释：execute() 内 Step 0 sync 是第一步且通常是最慢的网络往返（HealthKit 读 + POST /steps/sync），Step 2 的 POST /chest/open 紧随其后。"execute 整体 > 2s" 在绝大多数场景下即 "sync 阶段慢"（chest/open 单请求通常 < 1s）。spec AC 行 3140 的意图是「让用户在慢同步时看到提示别以为卡死」，本折中方案满足该用户意图（提示文案锚定 "同步步数中…" 是因为 sync 是 execute 内首个且最可能慢的阶段）。**这是经过权衡的实装选择，dev 不要为了"精确"去改 execute() 内部加回调（违反红线 + 过度工程）。**
>
> 单测口径据此调整（AC4）：测「execute mock 耗时 3s → isSyncingSteps 在 2s 后变 true」「execute mock 耗时 1s → isSyncingSteps 全程 false」，而非测 triggerManual 精确边界。

伪代码（在既有 `Task { @MainActor [weak self] in ... }` 内，`useCase.execute()` 前后加 2s 延迟控制）：

```swift
// onChestOpenTap 内既有 Task 块（line 409-454）改动：
self.isOpening = true
Task { @MainActor [weak self] in
    defer { self?.isOpening = false }
    guard let self else { return }

    // Story 21.5 AC2: 起 2s 延迟 task —— 超 2s 仍未完成 → 显 "同步步数中…".
    let syncHintTask = Task { @MainActor [weak self] in
        try? await Task.sleep(nanoseconds: 2_000_000_000)
        guard !Task.isCancelled else { return }
        self?.isSyncingSteps = true
    }
    defer {
        // execute 返回（成功 / 失败 / catch）后必复位（与 isOpening defer 同精神）.
        syncHintTask.cancel()
        self.isSyncingSteps = false
    }

    do {
        let snapshot = try await useCase.execute()
        self.pendingReward = snapshot
    } catch let error as APIError {
        // ... 既有 21.3 错误码 case-by-case mapping 完全不变 ...
    } catch {
        // ... 既有 21.3 非 APIError 兜底完全不变 ...
    }
}
```

> **defer 复位双保险**：`isSyncingSteps = false` 走 defer（与 `isOpening = false` 同 defer 块或紧邻 defer），保证 execute() 抛错 / cancel / 成功三路径都复位。`syncHintTask.cancel()` 防止 execute 在 2s 内返回时延迟 task 仍 set true（cancel 后 `guard !Task.isCancelled` 短路）。

### AC3 — 同步失败不阻塞开箱不变量

**对应 spec AC 行 3139**：`同步失败也继续开箱（不阻塞，让 server 用上一次 sync 后的余额判断）`。

**不变量（无需新增代码，验证既有契约成立）**：

- `StepSyncTriggerService.triggerManual()` 签名是 `func triggerManual() async`（**非** `async throws`）—— 内部 `runSync` 的 catch 块吞掉 `SyncStepsUseCase.execute()` 抛的任何 error（StepSyncTriggerService.swift line 209-217）；triggerManual 永不向 caller 抛错；
- `OpenChestUseCase.execute()` 内 `await stepSync.triggerManual()`（line 65-67）后无条件继续 `repository.openChest(request)` —— sync 失败被 triggerManual 静默吞掉，不中断 openChest；
- 结论：sync 失败时 server 用上一次成功 sync 后的 `user_step_accounts` 余额判定开箱；本 story 不需新增任何 "失败兜底" 代码，验证既有链路语义即可。

**验收口径**：

- 单测：mock SyncStepsUseCase 抛 APIError → triggerManual 不抛 → openChest 仍被调（AC4 case 2）；
- 集成测试：mock server /steps/sync 返 500 → 仍发 /chest/open → 用旧余额判定（AC5 场景）。

### Story 8.5 接缝（triggerManual await 语义 = 本 story 能成立的前提）

- Story 8.5 已把 `StepSyncTriggerService.triggerManual()` 实装为「await 到 sync 真跑完才返回」（不是 fire-and-forget）—— 经过 review round 3（[P2] 不能复用 fire-and-forget 短路路径，caller 会拿 stale state）+ round 4（[P2] await 后必须 re-check single-flight gate，main actor 让出期间 automatic trigger 可能又 spawn 新 task）两轮 race fix。详见 StepSyncTriggerService.swift line 112-162 + lessons `2026-05-04-manual-trigger-must-await-in-flight.md` / `2026-05-04-await-then-recheck-single-flight-gate.md`。
- **本 story 不需要也不允许动 triggerManual** —— 它已是钦定契约。OpenChestUseCase `await stepSync.triggerManual()` 拿到的返回时机即「sync 已完成、appState.currentStepAccount 已更新」，紧接着的 `repository.openChest` server 端就能读到最新余额。
- **单 in-flight gate 共享的价值**：开箱前 sync 与 launch/foreground/5min-timer sync 走**同一个** StepSyncTriggerService 实例的 in-flight gate → 若 timer sync 恰好 in-flight，triggerManual 会 await 它完成而非重复发请求（Story 8.5 AC 行 1577 "同步不重叠"）。这正是 AC1 红线「必须复用 RootView 持有的同一实例，不能新造第二个」的核心理由。

### "同步步数中…" 提示的实装折中（dev 必读，防过度工程）

- spec AC 行 3140 "同步耗时 > 2 秒时显示 loading 提示" —— 精确锚定「sync 阶段 > 2s」需要 OpenChestUseCase.execute() 在 triggerManual 前后回调通知 ViewModel。但**改 execute() 是红线**（21.3 冻结编排）。
- **钦定折中**：以 `execute()` 整体调用为基准起 2s 延迟 task（execute 内 Step 0 sync 是第一步且通常最慢的网络往返；chest/open 单请求通常 < 1s → "execute > 2s" 在绝大多数场景即 "sync 慢"）。提示文案锚 "同步步数中…" 因 sync 是 execute 内首个且最可能慢的阶段，满足 spec 用户意图（慢同步时别让用户以为卡死）。
- dev **不要**为追求"精确 sync 边界"去改 execute() 加回调 / 改 triggerManual 返回值 —— 那违反红线 + 过度工程。本折中已是经权衡的钦定方案，单测口径据此（测 execute 整体耗时 → isSyncingSteps，AC4 case 3/4）。

### 副标题三态优先级（isSyncingSteps > isOpening > idle）

- 时序：用户点开箱 → isOpening = true（execute 开始前）→ execute 内 Step 0 sync（此阶段若 > 2s → isSyncingSteps = true）→ sync 完成 → Step 2 open → execute 返回 → defer 复位 isSyncingSteps + isOpening。
- 因 isOpening 整个 execute 期间为 true，sync 阶段 isSyncingSteps 与 isOpening 同时 true → 副标题需 isSyncingSteps 优先（先显 "同步步数中…"，sync 完进 open 阶段 isSyncingSteps 复位 false → 自动降级显 "开箱中…"）。
- 开箱按钮 `.disabled(isOpening)` + ProgressView（21.3 落地）整个 execute 期间生效，本 story 不改按钮 disable 逻辑，仅副标题文案三态。
