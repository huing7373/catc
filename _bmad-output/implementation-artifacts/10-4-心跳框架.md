## Story

As a 服务端开发,
I want 在 Story 10.3 已就绪的 Session / SessionManager / Gateway 三对象之上，**精确实装** V1 §12.3 `pong` 字段表（`requestId` 必须等于 client `ping.requestId`、`payload` 固定空 object `{}`、`ts` 必须 server 当前 unix ms epoch）+ **新增**进程内单例 HeartbeatScanner（基于 `time.NewTicker(30s)` 周期扫描 SessionManager.ListAllSessions 列表 + 比较 `Session.LastHeartbeatAt()` 与 `time.Now().UnixMilli()` 差值 > `cfg.WS.HeartbeatTimeoutSec * 1000` ms 时强制 `Session.Close()` 并触发 close frame code=`4005` reason=`"heartbeat timeout"`）+ 在 SessionManager 接口上**新增** `ListAllSessions(ctx) []*Session` 方法（让 scanner 一次扫所有 room）+ Session 提供 `CloseWithCode(code int, reason string) error` 接口（让 scanner 调用时 emit 业务自定义 close code 而不是 1006/1011）+ scanner 在 main.go bootstrap wire 启动 + `defer scanner.Stop()` cleanup + 严格按 V1 §12.1 close code 4005 行钦定（reason 字符串字面量 `"heartbeat timeout"`、log level info 不是 warn/error），
so that 客户端断网（手机切后台超过 60s / 网络抖动 / TCP keepalive 失效场景）时服务端能在最多 90s（30s 扫描周期 + 60s 阈值）内主动释放 Session 资源 + 通知 client 走 4005 close → 触发 iOS Story 12.5 自动重连指数退避路径，避免"客户端已死服务端 Session 永远 leak"导致 SessionManager / sessionsByID / sessionsByRoom 三索引 unbounded growth + 节点 9+ prod 上线后 OOM；且**严格不破** V1 §12.1 ~ §12.3 已冻结的 WS 协议骨架契约（自 2026-05-05 起冻结）。

## Acceptance Criteria

**AC1 — `SessionManager` 接口新增 `ListAllSessions(ctx) []*Session` 方法**

`server/internal/app/ws/session_manager.go`：

- 在 `SessionManager` interface 上**新增**方法签名：
  ```go
  // ListAllSessions 返回 manager 内所有 active Session（按 sessionID 字典序，
  // 让 HeartbeatScanner（Story 10.4） / graceful shutdown 遍历顺序确定）。
  //
  // 内部实装：与 ListSessionsByRoomID 同语义 —— 锁内 read-lock copy 切片返回；
  // 调用方拿到的切片在 manager 改 map 后保持不变（避免遍历期 manager 改 map
  // 的并发问题）。
  //
  // 0 个 Session → 返 ([], nil)。
  ListAllSessions(ctx context.Context) []*Session
  ```
- 在 `sessionManager` struct 上实装该方法：
  ```go
  func (m *sessionManager) ListAllSessions(ctx context.Context) []*Session {
      m.mu.RLock()
      defer m.mu.RUnlock()
      out := make([]*Session, 0, len(m.sessionsByID))
      ids := make([]string, 0, len(m.sessionsByID))
      for id := range m.sessionsByID {
          ids = append(ids, id)
      }
      sort.Strings(ids)
      for _, id := range ids {
          out = append(out, m.sessionsByID[id])
      }
      return out
  }
  ```

**关键约束**：

- 与 ListSessionsByRoomID 完全同模式（read-lock copy + sessionID 字典序）—— 让 HeartbeatScanner / 未来 Epic 36 graceful shutdown 复用同套并发不变量
- ctx 参数当前不消费（list 操作纯内存）；保留参数让未来 distributed manager 抽象时可以走 ctx-aware 实装
- **不**导出 sessionsByID（仍是 manager 私有 map）；外部仅通过 ListAllSessions / ListSessionsByRoomID 接口访问

**AC2 — `Session.CloseWithCode(code int, reason string) error` 接口**

`server/internal/app/ws/session.go`：

- 在 Session 上**新增**公开方法 `CloseWithCode`，**不**修改既有 `Close()` 方法（保持幂等语义）：
  ```go
  // CloseWithCode 先写 WS close frame（code + reason）再调 Close()。
  //
  // 用途（Story 10.4 引入）：HeartbeatScanner 在心跳超时时调本方法，让 client
  // 收到 close code 4005 + reason "heartbeat timeout"，触发 iOS Story 12.5 指数
  // 退避自动重连路径（V1 §12.1 钦定 4005 = transient network failure）。
  //
  // 行为：
  //   1. 用 conn.WriteControl(CloseMessage, FormatCloseMessage(code, reason),
  //      deadline) 同步写 close frame（lesson 2026-04-26-startup-blocking-io-needs-deadline
  //      钦定 IO 必带 deadline；用 closeFrameWriteDeadline = 500ms 与
  //      gateway.go closeWriteDeadline 同 const semantics）
  //   2. 调 Close() —— Close 幂等（atomic.Bool + sync.Once 兜底，详见 Close 注释）
  //
  // 错误：
  //   - Session 已 Close → 返 ErrSessionClosed（不 emit close frame；调用方应在
  //     CloseWithCode 之前避免重复触发 —— scanner 拿到的 sessionID 通过 list 路径
  //     可能已经被 readLoop 触发的 Close 清出 manager；scanner 看到 closed=true
  //     就 skip）
  //   - WriteControl 失败 → log warn 但**不**返 error 给 caller（close frame 是
  //     best-effort；wire 已死时 client 收不到也是预期）；继续调 Close 释放本地
  //     资源
  //   - Close 自身**不**返 error（幂等设计已就绪）
  //   - 入口成功 → nil
  //
  // 并发：调用 CloseWithCode 与 Close 安全并发（共用 closed atomic.Bool + sync.Once
  // 兜底）；CloseWithCode 内部 WriteControl 持 conn 写锁（gorilla conn 自身保证
  // control frame 写入并发安全）
  func (s *Session) CloseWithCode(code int, reason string) error {
      if s.closed.Load() {
          return ErrSessionClosed
      }
      deadline := time.Now().Add(closeFrameWriteDeadline)
      if err := s.conn.WriteControl(
          websocket.CloseMessage,
          websocket.FormatCloseMessage(code, reason),
          deadline,
      ); err != nil {
          s.logger.Warn("ws CloseWithCode write frame failed",
              slog.Int("code", code), slog.String("reason", reason),
              slog.Any("error", err))
          // **不** return；继续调 Close 释放本地资源（与 gateway closeWithCode 同模式）
      }
      return s.Close()
  }
  ```
- 在 session.go 包级 const 区**新增** `closeFrameWriteDeadline = 500 * time.Millisecond`（与 gateway.go `closeWriteDeadline` 同 semantics —— 不同包内 const 不能直接复用，但**注释**必须显式标注与 gateway.go 对齐，方便未来重构合并）

**关键约束**：

- **不**修改既有 `Close()`：HeartbeatScanner 走 CloseWithCode 路径；business code（如 SessionManager.Close 批量、readLoop / writeLoop defer 自闭）走 Close 路径，两者各自语义清楚
- close frame 写入失败**不阻断** Close 调用 —— close 是 best-effort，wire 已死也要清本地资源
- **必须**先 WriteControl 再 Close —— 顺序反了会让 conn 在 close frame 写入前被关闭，client 永远收不到 4005
- 用 atomic.Bool 快速预检（`if s.closed.Load() return ErrSessionClosed`）—— 让 scanner 在并发场景下 skip 已 close 的 Session 而不写无意义的 close frame；Close 内部仍有 sync.Once 兜底，多次 CloseWithCode 调用安全

**AC3 — `HeartbeatScanner` 单例（`server/internal/app/ws/heartbeat_scanner.go` 新建）**

新建文件 `server/internal/app/ws/heartbeat_scanner.go`：

```go
package ws

import (
    "context"
    "log/slog"
    "sync"
    "time"
)

// heartbeatScanIntervalSec 是 HeartbeatScanner 的扫描周期（秒）。
//
// 选 30s 的理由：
//   - V1 §12.2 钦定 client 应每 30 秒发一次 ping（默认 ws.heartbeat_interval_sec
//     在 client 侧）+ 60 秒未收到任何消息触发 server-side 4005 close
//   - server-side 扫描周期取 30s = 半个阈值 —— 让"刚好达到 60s 阈值的 Session"
//     在最多 30s 内被检测到，总检测延迟最大 60+30 = 90s（仍 << 节点 4 阶段
//     合理 UX 容忍度）
//   - 不取 5s（频率过高，扫描 goroutine CPU 浪费）；不取 60s（与阈值同周期，
//     最坏检测延迟 60+60 = 120s 接近 2 倍阈值 → 客户端体感"server 反应慢"）
//
// 不走 YAML：30s 是契约一部分（与 60s 阈值组合产生最大 90s 检测延迟），prod
// 不可覆盖；与 cfg.WS.HeartbeatTimeoutSec 不同 —— HeartbeatTimeoutSec 是协议
// 钦定的 client-side 阈值（dev/test 可覆盖让单测加速），ScanInterval 是 server
// 内部 tick 频率（与协议字段无关，但与 60s 阈值组合产生检测延迟 SLO，不开放
// 用户调整避免 SLO 漂移）。
const heartbeatScanIntervalSec = 30

// HeartbeatScanner 是后台心跳超时扫描器。
//
// 用途：周期性遍历 SessionManager 内所有 active Session，对超过 cfg.WS.
// HeartbeatTimeoutSec * 1000 ms 未活跃的 Session 触发 CloseWithCode(4005,
// "heartbeat timeout")，自动释放 leaked Session 资源 + 通知 client 走 iOS
// Story 12.5 自动重连路径。
//
// 生命周期：
//   1. main.go bootstrap wire：scanner := NewHeartbeatScanner(mgr, cfg.WS, slog.Default())
//   2. ctx, cancel := context.WithCancel(context.Background())
//   3. go scanner.Run(ctx)
//   4. defer cancel()（让 scanner.Run 在 ctx.Done 时优雅退出）
//
// 并发安全：
//   - Run 是单 goroutine 主循环（不 fanout）—— SessionManager.ListAllSessions
//     已经做了 read-lock copy 切片，scanner 拿到切片后并发 CloseWithCode 安全
//   - Stop 通过 ctx 取消语义间接触发（main.go cancel ctx）；本身没有 Stop()
//     方法 —— 与 *time.Ticker 风格 Stop 不同，因为 scanner 没有需要外部释放的
//     资源（ticker 在 Run 内 defer Stop）
type HeartbeatScanner struct {
    mgr      SessionManager
    cfg      heartbeatScannerConfig
    logger   *slog.Logger
    interval time.Duration
}

// heartbeatScannerConfig 是 HeartbeatScanner 接受的最小配置面（让单测可注入
// 自定义阈值不依赖完整 config.WSConfig 全字段）。
type heartbeatScannerConfig struct {
    timeoutMs int64 // = cfg.WS.HeartbeatTimeoutSec * 1000
}

// NewHeartbeatScanner 构造 scanner。
//
// 参数：
//   - mgr: 既有 SessionManager 单例（main.go 已构造）
//   - timeoutSec: 心跳超时阈值（秒）；通常传 cfg.WS.HeartbeatTimeoutSec
//   - logger: base logger（传 slog.Default() 即可；scanner 内部 With 加 component=ws-heartbeat）
//
// timeoutSec ≤ 0 → 走默认 60s（防御性兜底；正常路径 cfg.WS.HeartbeatTimeoutSec
// 已经在 loader 内兜底过 60，但本构造函数也兜底一次让 scanner 在 testing 场景
// 不必依赖 loader）。
func NewHeartbeatScanner(mgr SessionManager, timeoutSec int, logger *slog.Logger) *HeartbeatScanner {
    if timeoutSec <= 0 {
        timeoutSec = 60
    }
    return &HeartbeatScanner{
        mgr:    mgr,
        cfg:    heartbeatScannerConfig{timeoutMs: int64(timeoutSec) * 1000},
        logger: logger.With(slog.String("component", "ws-heartbeat")),
        // 使用包级 const heartbeatScanIntervalSec（30s）。**不**接受参数覆盖：
        // 该值是 SLO 契约一部分（详见 const 注释）。但单测路径用 newHeartbeatScannerForTest
        // helper（同包内 unexported）注入小 interval（如 50ms）让单测加速。
        interval: heartbeatScanIntervalSec * time.Second,
    }
}

// Run 启动扫描主循环；ctx.Done 时优雅退出。
//
// 主流程：
//   1. ticker := time.NewTicker(s.interval)  // 30s
//   2. defer ticker.Stop()
//   3. for {
//        select {
//        case <-ctx.Done(): return
//        case now := <-ticker.C:
//            s.scanOnce(ctx, now)
//        }
//      }
//
// **关键约束**：
//   - 用 time.NewTicker 而非 time.AfterFunc：ticker 的 panic-safe 语义更强（即使
//     scanOnce 抛 panic，ticker 自身仍在跑；下一 tick 仍触发）
//   - **不**在 ctx.Done 后再做 cleanup（除 ticker.Stop）：scanner 没有持久状态
//   - **不**在每 tick 启动新 goroutine：scanOnce 内部并发 CloseWithCode 即可，
//     单 ticker 串行触发避免 goroutine 数量随时间无限增长
func (s *HeartbeatScanner) Run(ctx context.Context) {
    ticker := time.NewTicker(s.interval)
    defer ticker.Stop()
    s.logger.Info("ws heartbeat scanner started",
        slog.Int64("timeoutMs", s.cfg.timeoutMs),
        slog.Duration("interval", s.interval),
    )
    for {
        select {
        case <-ctx.Done():
            s.logger.Info("ws heartbeat scanner stopped", slog.Any("ctx", ctx.Err()))
            return
        case now := <-ticker.C:
            s.scanOnce(ctx, now)
        }
    }
}

// scanOnce 是单次扫描逻辑（提取出来方便单测直接调用，绕过 ticker）。
//
// 流程：
//   1. mgr.ListAllSessions(ctx) → 拿当前所有 active Session 切片
//   2. 对每个 Session 计算 idle = now.UnixMilli() - s.LastHeartbeatAt()
//   3. idle > timeoutMs → 并发 go s.CloseWithCode(4005, "heartbeat timeout")
//      （并发避免某一 Session 写 close frame 慢阻塞其他 Session 检测）
//   4. log info（V1 §12.1 钦定 4005 写 log info，因为心跳超时是常态网络抖动 / 切后台）
//
// **关键约束**：
//   - 用 goroutine fanout 触发 CloseWithCode：单 Session 的 close frame 写超时
//     是 500ms（closeFrameWriteDeadline），如果 1000 个 Session 同时超时 + 串行
//     写 = 500s 才能跑完一轮扫描，明显违反 SLO。fanout 后所有 close frame 在
//     500ms 窗口内并发写完
//   - **不**等待所有 goroutine 退出（用 fire-and-forget）：scanOnce 单次扫描
//     不需要"原子完成"语义；Close 自身幂等，下一轮扫描如果同 Session 仍在 list
//     里且仍 idle 就再触发一次（Close 走 sync.Once 兜底，第二次是 no-op）
//   - 用 sync.WaitGroup 仅在测试路径上同步（让单测能 assert "所有 close 已触发"）—— 
//     生产路径不阻塞 ticker（详见 scanOnce 注释）
func (s *HeartbeatScanner) scanOnce(ctx context.Context, now time.Time) {
```

## Dev Notes

### 实装关键决策

#### 1. HeartbeatScanner 是单例后台 goroutine，不是 SessionManager 内置功能

**决策**：HeartbeatScanner 是独立 struct + main.go 单独 wire，**不**塞进 SessionManager。

理由：
- 单一职责原则：SessionManager 是注册中心（Register / Unregister / List / Close 四个方法），HeartbeatScanner 是 lifecycle event 触发器（基于 list + 时间比较）—— 两者职责清晰分离让代码更易维护
- 测试隔离：scanOnce 可以在不构造完整 ticker 的情况下被单测直接调用（mock now 时间戳 + assert close 行为），而塞进 SessionManager 后会与 manager 测试纠缠
- 未来扩展：节点 9+ prod launch 阶段可能需要多种 scanner（如未来的 Session reaper / metrics 推送），各自 wire 不必共享 manager 内部状态
- 与 Story 8.5（步数同步触发器）类似的"背景 goroutine + cancel ctx 优雅退出"模式

**反例**（不要做）：
- 不把 scanner 实装为 SessionManager 上的 method（如 `mgr.StartHeartbeatScan(ctx, timeoutSec)`）—— 让 manager 接口变胖，且 stop 语义模糊
- 不把 scanner 实装为 init() 自动启动 —— init() 路径不可控，main.go 显式 wire 让生命周期清晰

#### 2. ListAllSessions 必须 read-lock copy，与 ListSessionsByRoomID 完全同模式

**决策**：实装时**严格**复用 ListSessionsByRoomID 的并发不变量。

理由：
- review 10-3 r5 P2 / r2 P1 反复修过 SessionManager 的并发陷阱（reconnect 替换 + 钩子触发顺序）；任何新增的 list-style 接口都必须遵守"锁内 copy → 锁外消费"模式，否则 caller 在遍历期间 manager 改 map 会触发 race
- HeartbeatScanner.scanOnce 拿到切片后会并发触发 CloseWithCode（goroutine fanout），如果切片是 manager 内部 map 的引用，goroutine 跨 tick 持有切片时 manager Register / Unregister 改 map 直接 panic

**反例**（不要做）：
- 不直接返 `m.sessionsByID`（map 引用）；必须 copy 到独立切片
- 不在 ListAllSessions 内部启动 goroutine（list 是同步只读操作）

#### 3. scanOnce 的 goroutine fanout 是必须的

**决策**：scanOnce 内对每个超时 Session **并发**调 CloseWithCode（go func {}），不串行。

理由：
- 单 Session 的 CloseWithCode 写 close frame 最多 500ms（closeFrameWriteDeadline）；如果 1000 个 Session 同时超时 + 串行 close = 500s 才能跑完一轮扫描，明显违反"30s 扫描 + 60s 阈值 → 90s 检测延迟" SLO
- fanout 后所有 close frame 在 500ms 窗口内并发写完，scanOnce 总用时不超过 500ms（即便所有 Session 同时超时）
- Close 自身幂等（atomic.Bool + sync.Once）—— 即便下一 tick 再次扫到同 Session（理论不会，因为 onUnregister 钩子会清出 manager），第二次 CloseWithCode 也是 no-op + log warn，不会 panic

**反例**（不要做）：
- 不用 sync.WaitGroup 等所有 goroutine 退出再退出 scanOnce —— 阻塞 ticker 让下一 tick 推迟
- 不用 work pool（如 50 worker channel + queue）—— Session 数量在节点 4 阶段最多几十个 + 节点 9+ 也只是几千，goroutine fanout 不需要复杂 pool

#### 4. log level 必须是 info，不是 warn / error

**决策**：scanner 触发的 close 路径全程 log info。

理由（V1 §12.1 close code 表 4005 行钦定）：
- 4005 = transient network failure（心跳超时多半是手机切后台 / 网络抖动），是**常态**
- 写 warn / error 会让正常网络抖动场景下日志噪声暴涨 —— 节点 4 阶段 4 user / room 看不出来，节点 9+ 几千 user 在线时 server 每天可能产生数千次 warn/error，淹没真正异常
- 类比 Story 10.3 的 4001 也是 info（token 过期是正常业务）

**反例**（不要做）：
- 不写 warn / error；仅在 ListAllSessions 自身失败 / 不可恢复异常时才走 error 路径

#### 5. close 路径必须触发 onUnregister 钩子（review 10-3 r2 P1 不变量保留）

**决策**：HeartbeatScanner 触发的 CloseWithCode → Session.Close → notifier.notifyClosed → manager.Unregister(sessionID) → onUnregister 钩子。

理由：
- review 10-3 r2 P1 已经修过类似陷阱（reconnect 替换路径漏触发钩子）：lessons `2026-05-06-ws-reconnect-unregister-hook-and-prod-contract-gate.md` 钦定"任何 close 路径必须走 onUnregister 钩子"
- 10.6 Redis presence cleanup 挂在 onUnregister 钩子上；如果心跳超时 close 漏触发钩子，Redis presence 永远 leak（key TTL 续期靠每次心跳，但 Session 已 close 后无心跳触发，TTL 到期前 presence 与真实 SessionManager 状态不一致）
- 单测 #10 专门验证：scanner 触发的 close → onUnregister 钩子被调一次

**实装路径**：
- HeartbeatScanner.scanOnce 调 `s.CloseWithCode(4005, "heartbeat timeout")` → 内部先写 close frame → 再调 `s.Close()` → Session.Close 内调 `s.notifier.notifyClosed(s.sessionID)` → manager.notifyClosed → manager.Unregister → 触发 onUnregister 钩子（已存在 lifecycle 路径）

**反例**（不要做）：
- 不绕过 Session.Close 直接调 manager.Unregister —— 会让 conn 泄漏
- 不在 scanner 内自定义 cleanup 钩子 —— 复用 SessionManager 既有 onUnregister 钩子（10.6 落地时统一接通）

#### 6. CloseWithCode 与 Close 的并发安全

**决策**：CloseWithCode 用 atomic.Bool 快速预检 + 调用 Close（其内部 sync.Once 兜底）。

理由：
- 多个 goroutine 可能同时触发 CloseWithCode（如 readLoop 收到错误自闭 + scanner 检测超时同时触发）—— 第二个调用看到 closed=true 应直接返 ErrSessionClosed，不重复写 close frame（写已 close 的 conn 会返 net.ErrClosed，clutter logs）
- Close 内部已用 sync.Once 兜底，所以即便 atomic 预检 race（两个 goroutine 都看到 closed=false 同时进 CloseWithCode），第一个 WriteControl 后 closed.Store(true) 在 Close 内执行；第二个的 WriteControl 走 conn.WriteControl 时 conn 已 close，gorilla 返 net.ErrClosed → log warn → 继续调 Close → sync.Once 已跑过 → no-op
- 这是与 Story 10.3 review r1 相同的设计（Send / Close 并发用 sendMu RWMutex 互斥；CloseWithCode 不需要参与 sendMu，因为它写 control frame 而非 data frame，与 Send 不互斥）

**反例**（不要做）：
- 不在 CloseWithCode 内加新 mutex —— Session 已经有 sendMu / closeOnce 两个并发原语，再加一个会让锁顺序复杂
- 不在 atomic 预检失败时 panic —— 多次 Close 是合法语义（idempotency）

#### 7. interval 30s 是 SLO 契约，不开放 YAML 覆盖

**决策**：`heartbeatScanIntervalSec = 30` 是包级 const，不走 YAML / env override。

理由：
- 30s 与 60s 阈值组合产生最大 90s 检测延迟（V1 §12.2 client-side 30s 心跳间隔的对偶）；如果让用户调整 interval，会破坏 SLO 一致性
- prod 不可覆盖（与 cfg.WS.HeartbeatTimeoutSec 相同语义）
- dev / test 路径用同包 unexported helper `newHeartbeatScannerForTest(mgr, timeoutMs, intervalMs, logger)` 注入小 interval（如 200ms），避免污染 production 配置面

**反例**（不要做）：
- 不加 `cfg.WS.HeartbeatScanIntervalSec` YAML 字段 —— 增加 prod gate 复杂度，且无业务理由开放
- 不通过环境变量 override —— 与 Story 10.3 r2 P2 prod gate 模式一致（最小 env 表面积）

#### 8. main.go defer 顺序：cancelHeartbeat 必须先于 sessionMgr.Close

**决策**：

```go
heartbeatScanner := wsapp.NewHeartbeatScanner(...)
go heartbeatScanner.Run(heartbeatCtx)
defer cancelHeartbeat()       // FIRST registered → LAST executed
// ... 
sessionMgr := wsapp.NewSessionManager()
defer func() { sessionMgr.Close() }()  // SECOND registered → FIRST executed
```

实际上 Go defer 是 LIFO（后注册先执行）；上面注释笔误 —— **正确**顺序是：

```go
sessionMgr := wsapp.NewSessionManager()
defer func() { sessionMgr.Close() }()    // 先注册 → 后执行（main exit 时排在 cancel 后跑）
heartbeatScanner := wsapp.NewHeartbeatScanner(...)
go heartbeatScanner.Run(heartbeatCtx)
defer cancelHeartbeat()                   // 后注册 → 先执行（main exit 时先 cancel scanner）
```

理由：
- main 退出时 LIFO：先 cancelHeartbeat → scanner.Run 退出 ticker / 不再 fanout 新 close → 再 sessionMgr.Close 批量清理 Session
- 反序（先 sessionMgr.Close 再 cancelHeartbeat）会出现：sessionMgr.Close 跑批量 s.Close() 同时 scanner 还在 ticker 上 fanout CloseWithCode → 双方对同 Session 并发 close → 虽然 Close 幂等不 panic，但日志噪声 + 钩子可能多调一次（依 10.6 实装是否真 idempotent）
- 单实例进程 main 退出场景实际上 cleanup 时间窗口很短，理论上不出问题；但**显式正确顺序**让未来重构者不必反复推理

**反例**（不要做）：
- 不省略 `defer cancelHeartbeat()` —— scanner goroutine 会泄漏到 main 退出后（虽然 main 退出时进程整体退出，goroutine 强制结束，但 lint 工具会报 leak warning）
- 不把 cancel 拼到 sessionMgr.Close 同一 defer 内 —— 顺序耦合让两个独立 lifecycle 难以分别测试

#### 9. 单测 ticker 路径的隔离策略

**决策**：scanOnce 是 exported（同包内）+ ticker 路径单独测一个 ctx-cancel goroutine leak 防御 case。

理由：
- 单测**不**等真实 30s ticker 触发（让单测加速）；scanOnce 直接调用 + 注入 mock now 时间戳
- ticker 路径仅测 #9（启动 scanner.Run + cancel ctx → Run 退出 + 无 goroutine leak）—— 用 t.Cleanup + runtime.NumGoroutine 探测在 cancel 后是否 goroutine count 回归基线
- 集成测试用 newHeartbeatScannerForTest 注入小 interval（200ms）走 production-like 路径，但仅 1 个 case 避免集成测试时间膨胀

**反例**（不要做）：
- 不在单测内 sleep 几秒等真实 ticker —— 让 CI 时间膨胀
- 不在 production NewHeartbeatScanner 接受 interval 参数 —— 让 SLO 契约破防（详见 §7）

#### 10. 时间源用 time.Now().UnixMilli()（与 Session.lastHeartbeatAt 写入路径对齐）

**决策**：scanOnce 调 `time.Now().UnixMilli()` 减 `s.LastHeartbeatAt()` 比 `timeoutMs`。

理由：
- Session.readLoop 内更新 lastHeartbeatAt 用 `time.Now().UnixMilli()`（10.3 实装已固定）—— scanner 必须用同一时间源 + 同精度（毫秒）才能正确比较
- ticker 在 select-case 内拿到的 `now := <-ticker.C` 是 time.Time，用 `now.UnixMilli()` 转毫秒后与 lastHeartbeatAt 对齐
- 不引入 clock interface（项目当前没有 clock package，详见 §"Project Structure Notes"）

**反例**（不要做）：
- 不用 time.Now() vs time.Now() 直接 Sub（结果是 time.Duration），再转毫秒 —— 多此一举且引入精度损失风险
- 不在 scanner 内引入新的 clock abstraction —— Story 8.5 步数同步 trigger 也未引入 clock interface（直接用 time.Now），保持一致

### 前置 lessons 提炼（review 10-3 系列必读规则）

1. **close 路径必须触发 onUnregister 钩子**（review 10-3 r2 P1）：HeartbeatScanner.scanOnce → CloseWithCode → Session.Close → notifier.notifyClosed → manager.Unregister → onUnregister；任何绕过 Session.Close 直接调 manager.Unregister 都会让 conn 泄漏 + 钩子链断开

3. **WS 协议字面量字符串严格不变**（review 10-1 系列）：close reason `"heartbeat timeout"` 必须**字面**匹配 V1 §12.1 钦定；任何 typo（如 "Heartbeat Timeout" / "heartbeat-timeout"）都会让 iOS 解析层不匹配（虽然 4005 + Code 数值是 primary signal，reason 是 secondary 但仍是契约一部分）
4. **log level 区分**（review 10-1 r3 + V1 §12.1 关键约束）：4005 写 info（常态），4001 写 info，4002/4003/4004 写 warn，4006/1011 写 error；本 story scanner.scanOnce 内 log "ws session heartbeat timeout" 必须用 slog.Info
5. **ctx-aware 严格规约**（ADR-0007 + lesson 1-9 ctx 传播）：HeartbeatScanner.Run / scanOnce / mgr.ListAllSessions 都接 ctx 参数；scanner.Run 在 ctx.Done 时 return；scanOnce 内对每个 close 用 fire-and-forget goroutine（不需要 wait）但 ctx 需要传给 mgr.ListAllSessions
6. **defer LIFO 顺序**（lesson 2026-04-26-startup-blocking-io-needs-deadline）：cancelHeartbeat 必须**后**注册（先执行）；sessionMgr.Close 必须**先**注册（后执行）；让 main 退出时 scanner 先停 → 再清 manager
7. **唯一 envelope producer**（lesson 2026-04-24-error-envelope-single-producer）：本 story 不写 server-active error envelope（4005 走 close frame 不走 error type 消息）；与 §12.3 error 小节"snapshot 构建失败处理路径"注一致 —— transient 网络断开走 close frame，应用错误走 error type 消息

### Anti-patterns to AVOID

1. **不要**绕过 Session.CloseWithCode 直接调 manager.Unregister：会让 conn 泄漏 + close frame 不写到 client + onUnregister 钩子链可能断开（如果 Unregister 内部不调 Close）
2. **不要**让 HeartbeatScanner 持有 SessionManager 内部 map 引用：必须通过 ListAllSessions 接口 read-lock copy 拿切片
3. **不要**在 scanOnce 内串行 close（必须 goroutine fanout）：500ms × N session 会让单 tick 阻塞过久，违反 90s SLO
4. **不要**让 scanner 写 `cfg.WS.HeartbeatScanIntervalSec` YAML 字段：interval 30s 是 SLO 契约，不开放用户调整
5. **不要**给 HeartbeatScanner 加 Stop() 方法：用 ctx-cancel 优雅退出语义（与 *time.Ticker.Stop 风格不同；scanner 没有持久状态需要外部释放）
6. **不要**在单测里 sleep 30s 等真实 ticker：scanOnce 抽出来直接调 + #9 ticker 路径仅测 ctx-cancel exit
7. **不要**修改 Session struct 字段定义：lastHeartbeatAt / closed / closeOnce / sendChan / sendPriorityChan / sendMu 全部已就绪；CloseWithCode 仅消费这些字段
8. **不要**修改 Gateway.Handle 校验顺序：已在 10.3 r10 P1 锁定；本 story 不动握手路径
9. **不要**在 close frame reason 里包含 stack trace / 具体内部错误：V1 §12.1 close code 表"reason 应包含简短错误提示但不泄漏 stack trace"钦定；本 story 严格用字面量 "heartbeat timeout"
10. **不要**为 heartbeat-timeout 加 metrics counter（节点 13+ 才做）：本 story 仅 slog log
11. **不要**在 production 路径上接受 interval 参数（NewHeartbeatScanner 不接受 interval）：仅 unexported test helper newHeartbeatScannerForTest 注入小 interval；prod 强制 30s
12. **不要**让 scanOnce 在 ListAllSessions 失败时 panic：list 是同步只读，不会失败；防御性返空切片即可（与 ListSessionsByRoomID 同模式）
13. **不要**让 scanner 触发 close 时 emit `error` type WS 消息（V1 §12.3 末尾 snapshot-build-failed 注同模式）：transient 网络断开走 close frame；应用错误才走 error type
14. **不要**在 main.go 把 scanner 构造放在 sessionMgr 之前：scanner 依赖 mgr 接口实例
15. **不要**预先实装 4006 message too large 显式 emit：节点 4 阶段 conn.SetReadLimit 已生效，gorilla 自动 close；4006 frame emit 由后续 epic 业务消息消费阶段配套
