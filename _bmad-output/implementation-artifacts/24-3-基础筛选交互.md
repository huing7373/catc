# Story 24.3: 基础筛选交互（在既有分类 Tab 上叠加品质筛选）

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As an iPhone 用户,
I want 仓库页能在「分类 Tab」基础上再按品质（N/R/SR/SSR）筛选,
so that 我有大量道具时也能在某个分类内快速找到想要品质的装扮.

## 范围红线（必读 —— epics.md 字面 spec 与已交付 UI 形态的 reconciliation）

> **本 story 是「在 Story 37.9 + 24.1 + 24.2 已交付的真实仓库页上，additively 叠加品质筛选」的窄范围 UI 交互 story；不是按 epics.md 字面重建一套 segment control UI、不改 24.1 sink/mapping、不改 24.2 UseCase/触发、不发任何 API。**

### epics.md §Story 24.3 字面 spec vs 当前已交付形态的偏离（必须 reconcile，不可照抄 epics）

epics.md §Story 24.3（line 3374-3394）字面要求：在 `InventoryView` 顶部加 **"全部 / 品质 / 槽位" 三段 segment control**，"全部" 平铺、"品质" 按 common/rare/epic/legendary 分 4 区段、"槽位" 按 slot 分区段。

但该 spec 写于 ADR-0009 推翻 Sheet 主入口 **之前**，假设的是「Sheet 弹 `InventoryView`，flat grid，无分类 Tab」形态。**当前已交付的真实形态**（Story 37.9 WardrobeScaffoldView + 24.1 RealWardrobeViewModel + 24.2 LoadInventoryUseCase，全 done）是：

- `WardrobeScaffoldView` 已有 **5 分类 Tab**（`CosmeticCategory` = hat/bow/scarf/outfit/bg，`categoryTabs` 区块 line 211-257），grid 渲染 `state.currentCategoryItems`（= `inventory.filter { $0.category == selectedCategory }`）。
- `CosmeticCategory.category(forSlot:)`（CosmeticCategory.swift line 56-64）已把 server 8 个 slot 值（`{1,2,3,4,5,6,7,99}`）**归并**进这 5 个 client 桶。**即：epics.md 说的「槽位」筛选，语义上已由现存的分类 Tab 承担**（slot→category 是确定映射，分类 Tab 就是 slot 维度的 client 投影）。
- 没有 `InventoryView`、没有 segment control、没有 flat grid 模式。

**reconciliation 决策（本 story 钦定，dev 不得自行另选）**：

1. **"槽位" 维度 ⇒ 复用既有分类 Tab，不新建**。epics.md「槽位」分段与现存 `CosmeticCategory` 分类 Tab 语义重叠（都是 slot 投影）；重建一套 segment 会与 37.9/24.1 既有渲染契约冲突且越界。**本 story 不动分类 Tab**。
2. **"品质" 维度 ⇒ 本 story 真正新增的能力**：在分类 Tab **之下、grid 之上** 叠加一条 **品质筛选条**（"全部 / N / R / SR / SSR" 5 个 chip），对**当前分类的道具**再按 rarity 过滤。这是 epics.md「品质」分段意图的、与已交付架构兼容的等价实现（client 纯过滤，不重新发 API）。
3. **"全部" ⇒ 品质筛选条的默认/重置态**（`selectedRarity == nil` ⇒ 不按品质过滤，等价 epics.md「全部：显示所有」在「当前分类内」的语义）。
4. **不做按品质「分 4 区段并列展示」的 sectioned 布局**（epics.md 字面的「4 个区段」）。理由：当前 grid 是单一 `LazyVGrid(currentCategoryItems)`，改成多 section 会大幅重写 37.9 grid 视觉契约（越界 + 与 30.x sprite 渲染演进冲突）。**以「单选品质 chip 过滤」实现等价的「快速找到某品质」用户价值**（epics.md user story「我有大量道具时也能快速找到想要的」由此满足）。本偏离在 Dev Notes「epics.md 偏离登记」明确记录，供 review 不误判为漏实现。

> **一句话**：本 story = 在既有分类 Tab 与 grid 之间插入一条**品质单选筛选条**，`WardrobeViewModel` 新增 `selectedRarity: Rarity?` transient state + 派生属性把 rarity 过滤叠加进 `currentCategoryItems`，`WardrobeScaffoldView` 渲染该筛选条 + 平滑动画。**不动** sink/mapping/UseCase/触发/分类 Tab/Mock 数据。

## Acceptance Criteria

**权威来源**：epics.md §Story 24.3（line 3374-3394，user story + 「纯客户端筛选不发 API」+「切换有平滑动画」+ ≥4 单测 case 钦定）+ 上文 reconciliation 决策（epics 字面 segment 形态 → 与已交付 5 分类 Tab 架构兼容的「品质单选筛选条」等价实现）+ Story 24.1/24.2 接续红线（sink/mapping/UseCase 已 final，本 story 零 edit）+ ADR-0010 §3.2（`selectedRarity` 属 view-specific transient → `WardrobeViewModel @Published`，与既有 `selectedCategory`/`selectedCosmeticId` 同归属）。

**Given** Story 37.9 `WardrobeScaffoldView`（5 分类 Tab + `currentCategoryItems` grid）+ Story 24.1 `RealWardrobeViewModel`（`[HomeEquip]→[CosmeticItem]` sink mapping，`CosmeticItem.rarity: Rarity`）+ Story 24.2 `LoadInventoryUseCase`（Wardrobe Tab 出现触发 + 写 `appState.currentInventory`）全部 done 且不可回归

**When** 用户在仓库页（Wardrobe Tab）点品质筛选 chip

**Then**：

1. **AC1 — `WardrobeViewModel` 新增 `selectedRarity: Rarity?` transient state + 单选切换方法**
   - 基类 `WardrobeViewModel`（`WardrobeViewModel.swift`）新增 `@Published public var selectedRarity: Rarity? = nil`（`nil` = "全部"，不按品质过滤；与 `selectedCategory`/`selectedCosmeticId` 同为 view-specific transient @Published，ADR-0010 §3.2）。
   - 新增 concrete 方法 `public func selectRarity(_ rarity: Rarity?)`（**非** abstract —— 与 `selectCategory`/`selectItem` 同精神，纯 view-state 行为无 Mock/Real 分化需求，放基类避免子类重复实装）：set `self.selectedRarity = rarity`；**副作用**：清空 `selectedCosmeticId`（与 `selectCategory` 同精神 —— 过滤变化后老选中 id 可能不在过滤后列表内，让预览区 `activeItem` fallback，防止选中态指向已被过滤掉的 item）。
   - **不**改 `selectCategory` 现有副作用，但 `selectCategory` 切换分类时**额外**重置 `selectedRarity = nil`（切分类回到「全部品质」，符合用户心智：换一个分类重新看；在 `selectCategory` body 内追加一行，**不**改其既有「清空 selectedCosmeticId」副作用）。
   - `selectRarity` / `selectCategory` 的 `selectedRarity` 重置都不触碰 `inventory`/`equipped`/`catName`/`selectedCategory`(在 selectRarity 内)（单字段 transient mutation 隔离）。

2. **AC2 — 派生过滤属性：rarity 叠加进 `currentCategoryItems`（client 纯过滤，零 API）**
   - 新增派生计算属性 `public var filteredCategoryItems: [CosmeticItem]`：在既有 `currentCategoryItems`（= `inventory.filter { $0.category == selectedCategory }`）基础上**再叠加** rarity 过滤 —— `selectedRarity == nil` ⇒ 返回 `currentCategoryItems` 原样（"全部"）；`selectedRarity == r` ⇒ `currentCategoryItems.filter { $0.rarity == r }`。
   - **保持 `currentCategoryItems` 原样不改**（37.9/24.1 既有 `activeItem`/`isEquipped`/badge 派生与 `categoryTabButton` count 仍依赖原 `currentCategoryItems`；`filteredCategoryItems` 是**新增**叠加层，不替换、不破坏既有契约）。
   - `WardrobeScaffoldView.grid` 的 `ForEach` 数据源由 `state.currentCategoryItems` 改为 `state.filteredCategoryItems`（这是 grid **唯一** edit 点；分类 Tab badge count 仍用 `state.inventory.filter{...}.count` 不变 —— badge 反映该分类**全部**件数不受品质筛选影响，符合用户心智「Tab 上数字 = 该类总数」）。
   - **纯客户端**：不调任何 UseCase/Repository/APIClient，不写 `appState`，不触发 24.2 触发点重拉（epics.md line 3388「筛选纯客户端，不重新发 API，由 ViewModel 对 groups 做 filter」）。
   - **`activeItem` 与品质过滤的一致性**：grid 渲染 `filteredCategoryItems` 后，预览区 `activeItem`（line 79-90，派生自 `currentCategoryItems`）可能指向一个已被品质过滤掉的 item。本 story 处理方式：`activeItem` 派生**改为基于 `filteredCategoryItems`**（让预览区与可见 grid 一致 —— 选中只可能落在可见项；fallback `equipped`/`first` 也在过滤后集合内）。这是 reconcile 「过滤后选中态一致」的必需点，**不是** scope creep（不改 `activeItem` 的 fallback 优先级逻辑，仅换其底层 items 源 `currentCategoryItems → filteredCategoryItems`）。`isEquipped(_:)` 不依赖列表不改。

3. **AC3 — `WardrobeScaffoldView` 渲染品质筛选条（分类 Tab 下 / grid 上）+ 平滑动画**
   - 在 `categoryTabs` 区块**之后**、`grid` 区块**之前**插入新区块 `rarityFilterBar`（横向 chip 排：`全部` / `N` / `R` / `SR` / `SSR`，共 5 个 —— "全部" 对应 `selectedRarity = nil`，4 个品质对应 `Rarity.allCases`）。
   - 每个 chip：选中态用既有 `theme.colors.accent` 填充 + 白字、未选中 `theme.colors.surface` + `theme.colors.border` stroke（**复用** `categoryTabButton` 的视觉规则与 shadow 挂载路径 —— shadow 必须挂在 `RoundedRectangle.fill(...).shadow(...)` 那一层，遵守 37.6 round5 lesson `2026-04-30-swiftui-state-survives-id-and-shadow-over-children.md`；**不**在最外层 chain `.shadow` 投影 children）。chip label：4 品质用 `Rarity.rawValue`（"N"/"R"/"SR"/"SSR"，与既有 `RarityTag` a11y `rarityTag_<rawValue>` 命名一致）；"全部" 用字面 `"全部"`。chip 可附 `RarityTag` 小色条做视觉提示（可选，dev 视 37.9 视觉密度决定；不强制）。
   - 切换 chip 时 grid 内容变化要有**平滑动画**（epics.md line 3389「切换 tab 时有平滑动画」）：用 `withAnimation(.easeInOut(duration: ~0.2))` 包裹 `state.selectRarity(_:)` 调用（与 SwiftUI 既有动画风格一致；**不**新造动画系统 / 不引第三方）。同样在切分类 Tab 时（`selectCategory`）grid 切换也应平滑（若 37.9 既有未包动画则**可**补 `withAnimation` 在 `categoryTabButton` 的 action —— 这是允许的最小补强；若已有则不重复）。
   - **a11y identifier**：新增 `AccessibilityID.Wardrobe.rarityFilter(_ rawValue: String) -> String`（模式 `wardrobeRarityFilter_<rawValue>`，与既有 `category(_:)`/`item(_:)` helper 同风格），"全部" chip 用 `rawValue = "all"`（即 `wardrobeRarityFilter_all`），4 品质用各自 `Rarity.rawValue`（`wardrobeRarityFilter_N` 等）。**不**破坏 37.9/24.1/24.2 既有锚（`wardrobeView`/`wardrobeCategory_*`/`wardrobeItem_*`/`wardrobeDiamondCount`/`wardrobeComposeEntry`/`wardrobeEquipButton`/`wardrobe_loading_indicator` 全保留）。a11y 常量进 `AccessibilityID.swift` 的 `Wardrobe` enum（与既有 `category`/`item` helper 并列），并补 `AccessibilityIDTests.swift` 对应断言（与既有模式一致）。
   - **不**改 `categoryTabs`/`topCard`/`previewCard` 视觉与既有 a11y 锚；`grid` 仅改 `ForEach` 数据源（AC2）+ 可选包 `withAnimation`；`gridCell` 零 edit。

4. **AC4 — `MockWardrobeViewModel` 零 edit；`RealWardrobeViewModel` 零 edit；不碰 sink/mapping/UseCase/触发/AppState**
   - `selectedRarity`/`selectRarity`/`filteredCategoryItems` 全在**基类** `WardrobeViewModel`，Mock 与 Real 子类**零 edit**自动继承（Preview / UITest / Mock 单测走 18 件 mock 不回归 37.9 视觉与 UITest 锚；Real 路径 24.1 sink 写 inventory 后过滤叠加在派生属性层，子类无需感知）。
   - **不**改 `RealWardrobeViewModel.subscribeInventory`/`mapToCosmeticItems`/`rarity(forServerValue:)`（24.1 final）。
   - **不**改 `LoadInventoryUseCase`/`InventoryRepository`/`InventoryEndpoints`/`InventoryResponse`/`AppState.applyInventory`/`WardrobeView.task(id:)`触发（24.2 final）。
   - **不**改 `AppState`（`selectedRarity` 是 view transient 归 ViewModel，**不**进 AppState —— ADR-0010 §3.2「current Tab/sheet/loading/筛选选中 → ViewModel transient，不进 AppState」）。
   - **不**改 `CosmeticItem`/`CosmeticCategory`/`HomeEquip` struct（`Rarity` enum 已有 `CaseIterable`，直接用 `Rarity.allCases` 渲染 4 chip，不改 `Rarity`）。
   - **不**反向修改 V1 §8.x 接口文档（schema Epic 23 冻结；本 story 纯 client 交互）。

## Dev Notes

### 本 story 一句话定位

37.9 给了仓库页 5 分类 Tab + grid，24.1 接通真实 inventory sink，24.2 接通 Wardrobe Tab 出现自动拉取。本 story 在「分类 Tab 与 grid 之间」**additively 插入一条品质单选筛选条**：基类 `WardrobeViewModel` 加 `selectedRarity: Rarity?` transient + `filteredCategoryItems` 叠加派生 + `selectRarity`，`WardrobeScaffoldView` 渲染该条 + 平滑动画。**零 edit** sink/mapping/UseCase/触发/AppState/分类 Tab/Mock 数据。

### epics.md 偏离登记（必读 —— 防 review 误判为「漏实现 segment control」）

epics.md §Story 24.3 写于 ADR-0009 推翻 Sheet 主入口之前，假设「Sheet 弹 InventoryView，flat grid」形态，故字面要求「全部/品质/槽位 三段 segment control + 品质分 4 区段并列展示 + 槽位按 slot 分区段」。当前已交付形态（37.9+24.1+24.2 全 done）是「Wardrobe Tab + 5 分类 Tab + 单一 grid」，字面 spec 已不适配。本 story 做的等价 reconcile（每条都有用户价值对等关系，**非缩水**）：

| epics.md 字面 | 本 story 等价实现 | 等价理由 |
|---|---|---|
| "槽位" 分段（按 slot） | **复用既有分类 Tab**（`CosmeticCategory`） | `CosmeticCategory.category(forSlot:)` 已把 server 8 slot 归并进 5 client 桶；分类 Tab 就是 slot 维度的 client 投影。新建 slot segment 会与 37.9/24.1 渲染契约冲突且越界（重复同语义控件）。 |
| "品质" 分段（4 区段并列） | **品质单选 chip 过滤条**（全部/N/R/SR/SSR） | 当前 grid 是单一 `LazyVGrid(currentCategoryItems)`，改 sectioned 多区段会大幅重写 37.9 grid 视觉契约（越界 + 与 30.x sprite 演进冲突）。单选 chip 过滤同样满足 user story「大量道具时快速找到想要品质」核心价值。 |
| "全部"（显示所有） | 品质条 `selectedRarity == nil`（不按品质过滤，显示当前分类全件） | 等价「全部」语义在「当前分类内」的投影（当前形态下「所有道具」总在某分类 Tab 下，无 flat-all 模式）。 |
| "某品质 0 件 → 折叠或显示暂无" | 取「折叠」=空 grid（`filteredCategoryItems.isEmpty` → `ForEach` 空集自然空 grid） | epics 给了「折叠**或**暂无」二选一，本 story 取「折叠」（更简、零新增 placeholder 视觉，不越界改 grid 空态）。 |

> 这些偏离是 **ADR-0009 主入口架构变更的必然下游 reconcile**，与 24.1（Sheet→Tab）/24.4（整条作废）同源逻辑。dev **不得**按 epics.md 字面重建 segment control / sectioned grid（会与已 done 的 37.9/24.1/24.2 冲突 + 越界）。review 应按本 reconcile 表 + AC 判定，不按 epics.md 字面判「漏实现」。

### 关键约束 / 最易踩坑点

- **`filteredCategoryItems` 是叠加层不是替换**：`currentCategoryItems` 必须**保留原样**（37.9/24.1 既有 `categoryTabButton` 的 badge count 用 `state.inventory.filter{$0.category==}.count`、`isEquipped`、24.1 测试都可能依赖原派生）。本 story 只新增 `filteredCategoryItems` 并把 **grid 的 `ForEach` 数据源** + **`activeItem` 底层 items 源** 这两处换成它；其余引用 `currentCategoryItems` 处不动。
- **分类 Tab badge count 不受品质筛选影响**：`categoryTabButton`（line 228）的 `count = state.inventory.filter { $0.category == category }.count` **保持不变**。用户心智：Tab 上的数字 = 该分类**总**件数；品质筛选只影响 grid 可见项，不改 Tab 数字（改了反而混乱「我明明有 5 顶帽子 Tab 怎么显示 2」）。这是有意决策，dev 勿"顺手"把 badge 也接过滤。
- **`activeItem` 必须基于 `filteredCategoryItems`**：否则选中一个 R 道具后切到 "N" 品质，预览区还显示那个已被过滤掉（grid 不可见）的 R 道具 → 视觉撕裂。AC2 钦定 `activeItem` 底层 items 源换 `filteredCategoryItems`（fallback 优先级 selected→equipped→first 逻辑不动，仅换 `let items =` 源那一行）。`selectRarity`/`selectCategory` 都清 `selectedCosmeticId` 进一步保证选中态不悬挂。
- **`selectedRarity` 不进 AppState**：view-specific transient（ADR-0010 §3.2「current Tab/sheet/loading/筛选选中 → ViewModel transient 或 SwiftUI @State，不进 AppState」），与既有 `selectedCategory`/`selectedCosmeticId` 同归属（`WardrobeViewModel @Published`）。**不**改 `AppState`/`applyInventory`/sink。
- **纯客户端，零 API**：epics.md line 3388 钦定。`filteredCategoryItems` 是纯计算属性，`selectRarity` 只写 transient @Published。**不**触发 24.2 的 `.task(id: coordinator.currentTab)` 重拉（那个只在 `currentTab` 变时触发；本 story 不动 `currentTab`/不动 `WardrobeView.task`）。
- **动画用 SwiftUI 原生**：`withAnimation(.easeInOut(duration: ~0.2))` 包 `selectRarity`/（可选）`selectCategory` 调用。`@Published` 变化触发 SwiftUI diff，grid `ForEach` 数据源变 → 自动过渡。**不**新造动画 / 不引第三方。duration 取 0.2 与既有交互手感一致（dev 可微调，记录在 Completion Notes）。
- **基类加能力 = Mock/Real 子类零 edit 自动继承**：`selectedRarity`/`selectRarity`/`filteredCategoryItems` 全在 `WardrobeViewModel` 基类。`MockWardrobeViewModel`（18 件 mock）与 `RealWardrobeViewModel`（24.1 sink）**不写一行**即获能力。这是为何本 story 不碰子类的根因 —— 也是 37.9 把 `selectCategory`/`selectItem` 设计成基类 concrete 方法的同模式延续。
