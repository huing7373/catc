# Story 15.5: 跨房间状态恢复（重连后由 snapshot 对齐）

Status: done

<!-- Validation 可选。建议运行 validate-create-story 在 dev-story 前做一次质检。 -->

## Story

As an iPhone 用户,
I want WS 重连后所有成员的猫状态自动对齐到 server 最新值，且我自己在断线期间发生的本地状态变化在重连成功后能被自动上报给 server,
So that 我和房间内其他成员都不会在断线后看到错的猫状态，server `pets.current_state` 与本地最终一致.

## 故事定位（Epic 15 第 5 条 / 收官 story；节点 5 iOS 端"跨连接边界状态对齐"收口；上承 12.5（自动重连状态机）+ 12.6（心跳）+ 15.1（snapshot 解析）+ 15.2（pet.state.changed WS 处理）+ 15.4（PetStateSyncTriggerService），下启 16.1 / 16.2 节点 5 demo 验收）

  - **他人 pet state 对齐**：靠 server 在 WS handshake 成功后**自动重发** `room.snapshot` 作为第一帧（V1 §12.1.3 + §12.5 + Story 12.5 契约 5）→ 走既有 `RealRoomViewModel.applySnapshot` → 既有 Story 15.1 路径解析 + 写 `memberPetStates` —— **零新代码**，仅靠现有路径在新 connection 上自然对齐。**本 story 的责任是写下断言这条链路在 reconnect 边界确实工作的测试**（见 AC1 案例 (i) / 案例 (iii)）。
  - **自己 pet state 对齐**：本 story 引入**单一新行为** —— 当 WS connection 从非 connected 变为 connected 时（含首次 connect 与 reconnect），主动让 `PetStateSyncTriggerService` 触发一次"当前 motion state → state-sync"，**绕过 Story 15.4 的 5s 同 state 节流**（节流锚点 reset）—— 这样 user 在断线期间 motion 已经从 .rest 变成 .walk 但 5s 节流还没过期 / 或者 reconnect 期间 motion 没变化但 server 端因别的客户端的 state-sync 已经把它老 state 推走需要重新 ack —— 都能在 reconnect 边界回到一致状态。
- **本 story 是 Epic 16 demo 验收强前置**：

  - **Story 16.1 验证场景 1 / 6 端到端时序**：A 与 B 都断网 → 各自 motion 变化 → 各自重连 → 各自看到对方最新 state（依赖本 story 的 reconnect 边界 self push）。
- **节点 5 验收要求（§4.5）**：本 story 落地后，节点 5 iOS 端 §4.5 全部要点齐 —— 房间内 ≥2 用户在线时双向状态实时同步 + 状态变化延迟可接受 + 网络断/恢复后状态自动对齐。

## ⚠️ 实装边界（必读 —— 防止 Story 15.4 attempt 1 重演）

> **本 story 是 Epic 15 收官 story，spec 范围必须严格收敛**。Story 15.4 attempt 1 因为引入 spec 没要求的"per-state 节流字典 / coalesce-to-latest / room edge 主动 sync / publisher replay 区分 / stop in-flight rollback"跑 13 轮 codex review 未收敛（recovery tag `epic-loop/15-4-attempt-1-halt`）。本 story 必须严守 epics.md §15.5 行 2449-2464 钦定的 3 个 case，**不**为未来可能 race 提前抗蚀。

### 禁止复刻的过度设计（红线，违反即 review reject）

1. **❌ 在 RealRoomViewModel 内新增 `reconcilePetStatesAfterReconnect()` 等"专门为 reconnect 写的对齐方法"** —— spec 钦定"接收新 snapshot → 全员 petState 重置为 server 最新值"完全由既有 `applySnapshot` 路径覆盖（Story 15.1 已落地）；**不**要新建专用对齐方法。
2. **❌ 在 RealRoomViewModel 内监听 `wsState` 变化主动触发 `manualResync()`** —— spec 没要求 vm 层做任何 reconnect-edge 主动行为；snapshot 自然走既有路径就已对齐他人 pet state。
3. **❌ 让 `PetStateSyncTriggerService` 订阅 `appState.$currentRoomId`** —— Story 15.4 attempt 1 r6/r7/r12 红线已钦定。`currentRoomId` 仅在 sink 内同步读 snapshot；reconnect 边界识别走 `WSConnectionState` 维度（见下方 §AC2）。
4. **❌ 引入"在 motion state 没变化时也定时 push state-sync"轮询机制** —— spec 行 2460 钦定"断线期间收到的本地 MotionState 变化（如自己走了几步）→ 重连后自动通过 Story 15.4 上报"，**自然语义**是"自下次 motion 变化触发"+"reconnect 边界主动 push 一次"双层保证；**不**要做定时轮询 / 跨 connection 状态对账。
5. **❌ 在 `PetStateSyncTriggerService` 内部维护 connection state 状态机** —— 不要在 service 内 cache `lastKnownConnectionState` 然后做 `.connected → .reconnecting → .connected` 边沿识别；外部 caller 在合适时机调 `service.triggerManualResync()` 即可。
6. **❌ 把 reconnect 后的 self push 路径与"用户主动按按钮触发 sync"混为一谈** —— spec 没引入"用户按按钮触发 sync"概念；本 story 的 manual trigger 仅由 reconnect-edge 触发，**不**暴露公共 manual trigger UI。
7. **❌ 测试中用真实 `Task.sleep(5_000_000_000)` 验证节流绕过** —— 用 internal `nowProvider` test seam（service 已就位）+ `triggerManualResync` 直接调用方式覆盖。

### 允许的最小可行设计（spec literally 实装）

1. **✅ 复用既有 snapshot 路径对齐他人 pet state** —— `applySnapshot` (Story 15.1) 在重连成功后由 server 自动重发的第一帧 snapshot 上自然触发，**零新代码**。本 story 仅写测试断言这条链路在 reconnect 边界工作。
2. **✅ 在 `PetStateSyncTriggerService` 上加 `triggerManualResync()` 公共方法** —— 该方法语义：**绕过 5s 节流锚点**（reset `lastSentState/lastSentAt = nil`）+ 调 `handlePetStateChange(homeViewModel.petState)` 触发一次正常的 throttle / roomId / commit / fire-and-forget 流水线；如果当前不在房间，roomId guard 会自然短路（与 Story 15.4 现有 preflight 一致），所以 `triggerManualResync` 调用本身是 idempotent / safe。
3. **✅ 在 `RealRoomViewModel.handle(message:streamRoomId:streamGeneration:)` 的 `.connectionStateChanged(.connected)` 分支内**，在切完 `wsState = .connected` 之后通知一个**新增的 reconnectAlignmentDelegate**（弱引用）—— delegate 持有方就是 `PetStateSyncTriggerService`，由 RootView 在 wire 时绑定。这是 vm 与 service 之间唯一的反向通知接缝，避免让 vm 直接 import / 持有 service 类型（保持 vm 不向 Home feature 反向耦合）。
4. **✅ delegate 接缝用 `protocol PetStateReconnectAlignmentDelegate: AnyObject` + 单方法 `func didReconnectAlignmentRequested()`** —— 让 vm 依赖一个"语义中立的"协议，由 service 适配；测试可注入 mock delegate 验证调用次数。
5. **✅ 既有 `applyPetStateChanged` 路径不动** —— 断线期间收到的旧 `pet.state.changed` 是不可能事件（断线就收不到 WS 消息），但即使有"重连后旧 stream late message"残留也由 Story 15.2 已落地的 `streamGeneration` gate 拦下；本 story **不**复刻。
6. **✅ 测试用 `WebSocketClientMock.emit(.connectionStateChanged(.connected))` + 验证 mockDelegate.alignmentRequestedCount** 覆盖 reconnect 边界触发；用 `WebSocketClientMock.emit(.roomSnapshot(...))` + 验证 `vm.memberPetStates` 覆盖他人 pet state 对齐。

**预期文件修改**：

#### `iphone/PetApp/Features/Home/Services/PetStateSyncTriggerService.swift`（修改 —— 新增 `triggerManualResync()` 公共方法 + 节流锚点 reset 私有 helper）

实装契约（**目标增量 ≤ 30 行包含注释**；保持 service 总行数 ≤ 200 行预算 —— 当前 167 行 + 30 行 = 197 行 OK）：

```swift
// 在现有 stop() 方法之后、handlePetStateChange(_:) 之前 新增：

/// Story 15.5 AC2: 重连成功后由 RealRoomViewModel 通过 PetStateReconnectAlignmentDelegate 触发的
/// "manual resync" 入口 —— **绕过 5s 同 state 节流**，让 reconnect 边界的状态对齐不被节流锚点错挡.
///
/// 语义：
///   1. reset 节流锚点：`lastSentState = nil; lastSentAt = nil` —— 让本次 sync 必发（不论上次 sync 时间）
///   2. 调 handlePetStateChange(homeViewModel.petState) —— 走完整 throttle (now no-op) / roomId preflight /
///      commit-to-send / fire-and-forget 流水线 —— 与 reactive subscription 路径同流水线，确保 roomId guard
///      与 throttle "重新建立锚点" 等行为一致.
///
/// **idempotent / safe**: 当 user 不在房间（appState.currentRoomId == nil）时，roomId preflight 自然短路 ——
/// 调用本方法是无害的；但 caller 应仅在合适时机调（ws connected + 在房间），避免无谓 syscall.
///
/// **fire-and-forget**: 与 reactive 路径同语义；本方法**同步**返回，UseCase Task 在背景 spawn；失败 silently 吞.
public func triggerManualResync() {
    guard let homeViewModel else { return }  // service 已 deinit / wire 残缺 → no-op
    // Step 1: reset 节流锚点 —— 让 handlePetStateChange 内 throttle preflight 必通过.
    // **不**重置 nowProvider —— 那是 test seam，与节流逻辑解耦.
    // **不**调 stop() —— subscription 不动，避免 reactive 路径被打断.
    lastSentState = nil
    lastSentAt = nil
    // Step 2: 走完整流水线（throttle 已 reset → 必通过；roomId guard / commit / fire-and-forget 走原路径）.
    handlePetStateChange(homeViewModel.petState)
}
```

**预期行为**：

1. **reset 节流锚点必先于调 handlePetStateChange** —— 否则 5s 内同 state 调用本方法仍被节流挡（违反 spec 行 2460 "断线期间收到的本地 MotionState 变化 → 重连后自动通过 Story 15.4 上报"；如果 user 5s 内连续 reconnect，第二次 reconnect 不应被错挡）
2. **不重启 subscription** —— reactive 路径不动；triggerManualResync 是与 reactive 互补的"边界触发器"
3. **roomId guard 自然短路** —— user 不在房间时调本方法应 no-op（实际由 handlePetStateChange Step 2 在 reset 锚点后才 guard，所以**节流锚点会被 reset 但 sync 不发**；这是合理边界 —— 下次 reactive 触发依然会按新锚点工作）
4. **fire-and-forget Task 内行为不变** —— 失败 silently 吞；UseCase 已就位

**红线**：

- **不**在本方法内重新 subscribe / 重启 service —— 仅 reset throttle 锚点 + 调 handlePetStateChange
- **不**在本方法内拷贝 / 重写 4 步流水线 —— 必须复用 handlePetStateChange（DRY；attempt 1 r9-r13 教训：service 内部多路径 = 易出 race）
- **不**在 reset throttle 锚点之前调 handlePetStateChange（否则节流不会被绕过）
- **不**让本方法返回 Task 或 async —— 同步返回保持 caller 调用简单（与 reactive sink 一致：sink 闭包同步返回）
- **不**在本方法内访问 `appState.currentRoomId` 做 preflight —— 让 handlePetStateChange Step 2 的 preflight 统一负责（DRY）
- **不**在 reset 锚点后清 `homeViewModel` weak 引用，**不**调 stop()

#### `iphone/PetApp/Features/Home/Services/PetStateSyncTriggerService.swift`（同文件 —— 新增 `PetStateReconnectAlignmentDelegate` 协议 conformance）

```swift
// 在文件顶部 import 之后、@MainActor public final class PetStateSyncTriggerService 之前 新增 protocol：

/// Story 15.5 AC3: WS 重连成功后由 RealRoomViewModel 通知 PetStateSyncTriggerService 触发对齐的 delegate 协议.
///
/// 设计：vm 不 import service 类型（避免反向耦合 —— Room feature 不应依赖 Home feature 类型）；
/// 通过本"语义中立"协议接缝传递通知；service 适配 conformance；RootView 在 wire 时把 service
/// 注入为 vm.reconnectAlignmentDelegate（弱引用，避免循环）.
///
/// 调用契约：
///   - vm 在 `.connectionStateChanged(.connected)` 分支（含 first connect / reconnect）调用一次本方法
///   - service 在 `didReconnectAlignmentRequested()` 内调 `triggerManualResync()` 触发 self push
public protocol PetStateReconnectAlignmentDelegate: AnyObject {
    @MainActor
    func didReconnectAlignmentRequested()
}

// 在 PetStateSyncTriggerService class 末尾、deinit 之前 新增 conformance：

extension PetStateSyncTriggerService: PetStateReconnectAlignmentDelegate {
    public func didReconnectAlignmentRequested() {
        triggerManualResync()
    }
}
```

### Project Structure Notes

- **service 单一职责扩展** —— Story 15.4 service 已是"reactive 触发器 + 5s 节流"；本 story 加"manual reconnect-edge 触发器"，两者通过共享 `handlePetStateChange` 复用流水线（DRY）

### Latest tech information

- **Swift 6 strict concurrency `@MainActor` weak delegate**：`weak var reconnectAlignmentDelegate: PetStateReconnectAlignmentDelegate?` 在 @MainActor 隔离的 vm 内使用 —— 协议方法用 `@MainActor` 标注让跨边界类型安全（vm 调 delegate 时已在 main actor，无需 await）
- **Swift protocol with @MainActor method**：协议里的 `@MainActor func didReconnectAlignmentRequested()` 让 conformance 类型必须在 main actor 上调用本方法；`PetStateSyncTriggerService` 是 `@MainActor public final class` 自然满足
- **`AnyObject` constraint on protocol**：`PetStateReconnectAlignmentDelegate: AnyObject` 让协议**只能**被 class type conformance（保证 weak ref 语义）；与 NSObjectProtocol 类似但不强制 ObjC runtime
- **WebSocketClientMock.emit(_:)** 既有 API（Story 12.x 落地）—— `mockClient.emit(.connectionStateChanged(.connected))` 模拟 server-side connection 事件
- **WebSocketClientMock.prepareForReconnect()**：测试可手动调来翻新 streamGeneration（Story 15.2 r4 落地路径）—— case (iii) 用此模拟 reconnect 后的 generation 翻新
