## Dev Notes

### ADR §6 第 2 条 vs Story 37.1 顺序矛盾（核心 dev 注意点）

ADR-0009 §6 第 2 条字面要求 "Story 37.3 落地后跑 `bash iphone/scripts/build.sh --test` 通过"，但 Story 37.1 epic AC 要求"§6 3 条全部勾选"。两者按字面读**矛盾**：

- 若严格按 ADR §6 字面：Story 37.1 完成时 37.3 未开工，第 2 条**不能**勾 → 违反 epic AC
- 若严格按 epic AC 字面：Story 37.1 一次性勾完所有 → 第 2 条在 37.3 落地前**预先勾选**

**推荐解读（路径 B，与 epic AC 字面一致）**：

§6 第 2 条的"Story 37.3 落地后跑 build.sh --test 通过"应理解为「ADR-0009 contract validity 的**外部前置依赖**已经在 Sprint Change Proposal v2.5 终审 + ADR-0009 §1.3 + §2 决策表 + §3.1-§3.5 详细落地步骤的 architect/PM 评估中**契约级**确认过」。Story 37.1 改 Accepted 的语义是"契约 freeze、解锁下游开工"，不是"已物理验证"。物理验证（build 通过）由 Story 37.3 实装期 codex review 兜底；若届时发现 ADR-0009 §3 决策有偏差，走"ADR 修订 patch + 改 v2 Accepted"的修订路径（参考 ADR-0008 v2 的先例：`ec5beb3 docs(adr): ADR-0008 v2 Accepted + Story 0008-impl-1 done`）。
