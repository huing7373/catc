```markdown
#### 服务端逻辑

1. **认证 & 限频**：auth 中间件校验 Bearer token；rate_limit 中间件按默认配置限频
2. **参数校验**：`syncDate` 格式 + `clientTotalSteps` ≥ 0 + `motionState` ∈ {1,2,3} + `clientTimestamp` > 0；任何不通过 → 返回 1002 参数错误
3. **差值计算**（Story 7.3 service 层实装）：
   - 查 `user_step_sync_logs WHERE user_id=? AND sync_date=?` 取最近一条
   - 若**无**最近记录（首次同步） → `delta = clientTotalSteps`
   - 若**有**最近记录 → `delta = max(0, clientTotalSteps - lastClientTotalSteps)`
   - 倒退场景（clientTotalSteps < lastClientTotalSteps）→ delta = 0，仍**写日志**
   - 跨自然日（sync_date 不同） → 按新一天起算，**不**读上一天的 lastTotal
4. **防作弊阈值**（Story 7.3 增量 AC GAP K 修补；server 权威实施，**iOS 端不加限制**）：
   - **单次截断**：若 `delta > 5000`（配置 `steps.single_sync_cap`） → delta 截断为 5000 + log warning（**不**返回错误，避免误伤真实跑步用户）；接口仍返回 200 OK + `acceptedDeltaSteps = 5000`
   - **当日封顶**：若同 sync_date 当日 `accepted_delta_steps` 历史累计 + 本次截断后 delta **将超过** 50000（配置 `steps.daily_cap`，即 `prevAccepted + curDelta > 50000`） → 当次 delta 强制 = 0 + log warning + **返回 3001 步数同步数据异常**（注：判断条件是"本次入账后是否越界"，**不**是"已达上限"，否则会放过最后一次跨界 sync，例如 prev=49000 + cur=4000 应被拒）
   - 阈值通过配置文件可调；本 story 文档侧只锚定**默认值** 5000 / 50000；具体配置 key 见 Story 7.3 实装
5. **事务**（一个 MySQL 事务内完成）：
   - UPDATE `user_step_accounts` SET `total_steps += delta, available_steps += delta, version += 1` WHERE user_id=?
   - INSERT `user_step_sync_logs` (user_id, sync_date, client_total_steps, accepted_delta_steps=delta, motion_state, source=1, client_ts=clientTimestamp)
   - 注：`source=1` 代表客户端正常上报；dev grant 走 source=2，见数据库设计 §6.6 + Story 7.5
   - 任一步失败 → 整体回滚，返回 1009 服务繁忙
6. **响应**：返回 `acceptedDeltaSteps`（实际入账增量，可能因截断 / 封顶 ≠ 客户端预期）+ 最新 `stepAccount` 三字段（见 §6.1.5）
```

**§6.1.6 可能的错误码**（**新增** H4 块，紧接响应体之后）：

```markdown
#### 可能的错误码

| code | message | 触发条件 |
|---|---|---|
| 1001 | 未登录 / token 无效 | auth 中间件拦截（无 Bearer token / token 过期 / token 解析失败） |
| 1002 | 参数错误 | `syncDate` 格式不符 `YYYY-MM-DD` / `clientTotalSteps` 缺失或为负数 / `motionState` 不在 {1,2,3} / `clientTimestamp` ≤ 0 / 任一字段缺失 |
| 1005 | 操作过于频繁 | rate_limit 中间件拦截（**已认证路由**按 user_id 限频，每用户每分钟 > 60 次；按 Story 4.5 默认值，配置可调） |
| 3001 | 步数同步数据异常 | 本次同步入账后当日 `accepted_delta_steps` 累计**将超过** 50000 上限（即 `prevAccepted + curDelta > 50000`），本次 delta 被强制 = 0（防作弊封顶；GAP K 修补） |
| 1009 | 服务繁忙 | DB 异常 / 事务回滚 / 内部 panic（见 Story 7.3 service 实装） |
```

**关键约束**：

- 错误码 3001 与 1002 的语义差异**必须**明确：1002 是**参数本身**不合法（客户端错），3001 是**参数合法但触发服务端业务限制**（防作弊封顶，本次 delta 被强制 = 0）；客户端处理策略不同（1002 应停止重试并修正参数；3001 应静默接受本次返回 + UX 提示"今日步数已达上限"）。**注意**：3001**不是粘性错误码** —— 当日只要 `prevAccepted + curDelta > 50000` 触发条件成立才返 3001；若客户端后续 sync 是倒退（`clientTotalSteps < lastClientTotalSteps`）或重复（`clientTotalSteps == lastClientTotalSteps`），按"差值计算"步骤 delta 自然 = 0，**仍返 code = 0**，不会再次触发 3001。客户端**不应**假设当日首次 3001 后所有 sync 都失败而停止上报（仍需上报维持 sync_log 审计）
- 单次截断（`delta > 5000`）**不**返回错误，仅 log warning + 截断；客户端从 `acceptedDeltaSteps` 字段感知实际入账值，**不**通过错误码感知截断（avoid false negative：真实跑步用户单次同步可能合法超 5000，体验上不应被错误码打断）

```markdown
> **schema 嵌套差异**：本接口 `data` 字段直接是三档值对象，**不**包一层 `stepAccount`（区别于 §6.1 `POST /steps/sync` 响应的 `data.stepAccount.totalSteps` 嵌套结构）。两端不同设计的原因：§6.1 是"动作型"接口（响应需附带本次动作影响的"账户最新态"，故聚合在 `stepAccount` 子对象下避免与 `acceptedDeltaSteps` 等动作返回值平铺混淆）；§6.2 是"纯读型"接口（响应**只**含账户三字段，无嵌套必要）。客户端 Codable 解析时**必须**区分 `StepsSyncResponse.data.stepAccount.*` 与 `StepsAccountResponse.data.*` 两种结构。

{

  "message": "ok",

  },
  "requestId": "req_xxx"
}
\`\`\`
```

**§6.2.3 服务端行为**（**新增** H4 块）：

```markdown
#### 服务端行为

1. **认证 & 限频**：auth 中间件校验 Bearer token；rate_limit 中间件按 `user_id` 限频（每用户每分钟 60 次，按 Story 4.5 默认值，配置可调；与 §6.1 `POST /steps/sync` 同 scope —— 已认证业务路由统一按 `user_id` 而非 IP）
```

## Dev Notes

### 关键设计原则

1. **契约先行（contract first）**：本 story 是节点 3 第一个 epic 的开局，**禁止**先写 Go 代码再回头补文档。理由：iOS Epic 8 必须基于稳定契约并行开工，先实装后补文档会导致 iOS 端 Codable struct 与 server response 字段对不上 → 节点 3 demo 时双向返工。
2. **数据库 → API 单向映射**：API 字段语义由 `docs/宠物互动App_数据库设计.md` §5.4 / §5.5 + §6.5 / §6.6（motion_state / source 枚举）+ §8.2（步数同步事务）推导，**不**反向修改数据库。如发现 API 需求与数据库 schema 不符（如某个字段长度不够），优先改 API 设计或后置到下一 epic 处理，**不**直接改 DB。
3. **server 权威 + 客户端无校验**：所有防作弊阈值（5000 / 50000）只在 server 实施；iOS 端**不**加任何客户端校验。理由：客户端代码可被反编译 / 修改，校验放客户端等于无校验；server 是唯一可信源。这条规则文档侧必须**显式**写明，避免 iOS Story 8.5 实装时主动加客户端 5000 校验导致真实跑步用户体验损坏。
4. **截断 vs 错误码差异**：单次 5000 截断**不**返 3001（仅 log warning + acceptedDeltaSteps 截断），当日 50000 封顶**才**返 3001。理由：单次截断是"系统给你打了个折扣"，用户的下次同步还能正常入账（剩余真实步数下次 sync 自动累加）；当日封顶是"今天到此为止"，必须给客户端明确信号停止重试。两者业务语义不同，错误处理路径不同，文档侧必须明确区分。
5. **syncDate 客户端时区权威**：客户端按本机时区算"今天"，server 直接采用不二次转换。理由：用户跨时区旅行（如美国 → 中国 13 小时差）时，server UTC "今天"与客户端本机"今天"可能不同；以 server 为准会导致用户在飞机上跨时区的步数被记到错误日期。客户端时区是**用户体验真相源**，server 只做记录不做时区裁判。
6. **§6.1 vs §6.2 嵌套差异**：§6.1 是"动作型"（响应携带本次动作影响的"账户最新态"，需要嵌套 stepAccount 子对象避免与 acceptedDeltaSteps 等动作返回值平铺混淆）；§6.2 是"纯读型"（响应只含账户三字段，无嵌套必要）。两端不同设计的根因是不同的接口定位，**不**应统一为同一结构（统一会要么破坏动作型语义，要么过度嵌套纯读型）。
7. **dev 端点 schema 不进 V1 文档**：`/dev/grant-steps` 在 Story 7.5 落地，schema 直接在 Story 7.5 内联定义。理由：dev 端点不暴露给客户端（生产 build 返回 404），无跨端契约对齐需求；写进 V1 文档反而会让客户端开发者误以为是公共 API。

### 架构对齐

**领域模型层**（`docs/宠物互动App_总体架构设计.md` §3.X 步数账户）：
- 步数是可消费资产，按"账户模型"记账（账户 = 总步数 + 可用 + 已消耗）
- 同步入账走"客户端总步数 - 上次同步总步数 = delta"差值算法（不接受客户端直接报 delta，避免重放攻击）
- 这两条规则决定 §6.1 必须**accept POST 客户端 totalSteps（非 delta）+ server 算 delta**

**数据库层**（`docs/宠物互动App_数据库设计.md`）：
- §5.4 user_step_accounts → §6.2 三档值字段直接映射（total_steps / available_steps / consumed_steps）
- §5.5 user_step_sync_logs.client_total_steps + accepted_delta_steps + motion_state + source + client_ts → §6.1 请求体字段映射（去掉 user_id / id / created_at 等 server 维护字段）
- §6.5 motion_state 枚举（1/2/3）→ §6.1 motionState 字段值域
- §6.6 source 枚举（1=healthkit, 2=admin_grant）→ §6.1 sync_log 写入用 source=1，dev grant 用 source=2（Story 7.5）
- §8.2 步数同步事务边界：查询最近记录 + 计算增量 + 更新 step_accounts + 写 sync_logs，一个事务完成

**服务端架构层**（`docs/宠物互动App_Go项目结构与模块职责设计.md` §4）：
- handler 层 DTO 严格按本契约：handler 不做业务计算，只做参数校验 + 调 service + 转 response
- service 层负责事务（Story 7.3 走数据库设计 §8.2 事务）+ 错误转换（DB error → 1009 / 防作弊触发 → 3001）
- repo 层不直接对外暴露错误码，错误码不在 repo 层产生

**iOS 客户端层**（`docs/宠物互动App_iOS客户端工程结构与模块职责设计.md` §4 + ADR-0002）：
- DTO 用手写 Codable struct（不上 Swift OpenAPI Generator）
- 字段命名严格 camelCase；syncDate 用 String 解析（**不**用 Swift Date —— Date 解析需 DateFormatter，引入时区歧义；保持 String 让 server 做格式校验）
- StepsSyncRequest / StepsSyncResponse / StepsAccountResponse 三个 struct 在 iOS Story 8.5 落地

**配置层**（`docs/宠物互动App_Go项目结构与模块职责设计.md` §4 + Story 7.3 实装）：
- `steps.single_sync_cap` (default 5000) / `steps.daily_cap` (default 50000) 通过 YAML 配置文件可调
- 默认值是契约一部分（AC8）；运行时覆盖默认值不视为契约变更，但修改默认值视为契约变更走完整冻结流程
