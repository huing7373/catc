**§5.2.4 服务端逻辑**（**新增** H4 块，紧接 state 枚举之后 + 返回示例之前）：

```markdown
#### 服务端逻辑

1. **认证 & 限频**：auth 中间件校验 Bearer token；rate_limit 中间件按默认配置限频（60 次/分按 user_id 计；已认证业务路由统一按 user_id 而非 IP，与 §6.1 / §6.2 / §10.x 同语义）
2. **参数校验**：`state` 必填 + 类型为 number (int) + 值 ∈ {1, 2, 3}；任何不通过 → 返回 1002 参数错误（不开 DB 事务）
3. **查找当前用户的默认 pet**：`SELECT id FROM pets WHERE user_id = ? AND is_default = 1`（数据库设计 §5.3 唯一约束保证最多 1 行）
   - **0 行**（pet-less 账号 —— 与 §5.1 / §10.3 / §12.3 pet-less 合法 edge case 同语义，contract 内显式覆盖；非 invariant 损坏）→ 走 server-acknowledged noop 路径：跳过步骤 4 + 跳过步骤 5 + 直接进入步骤 6 返回 200 OK + code = 0 + `data.state` = 入参 state（回显）；不 log error（可 log info 级 noop 标记）
   - **1 行** → 进入步骤 4
4. **UPDATE**：`UPDATE pets SET current_state = ?, updated_at = NOW() WHERE id = ?`（按上一步查到的 pet.id）
   - **`err == nil`**（DB 调用无错误）→ **一律**返回 200 OK + code = 0，进入步骤 5；service 层**不**读 `RowsAffected`、**不**根据该值分支业务逻辑（与 V1 doc §5.2 line 532-537 实装锁定一致：1009 ⇔ `err != nil`；200 OK + code = 0 ⇔ `err == nil`；两个互斥二分，无第三条路径）
   - **`err != nil`**（driver / 网络 / 约束冲突 / 任何 DB 异常）→ 返回 1009 服务繁忙
   - 理由：(a) `RowsAffected == 0` 在 MySQL/GORM 语义下主要触发条件是"同 user 同 state 重复上报"幂等场景（WHERE 命中 1 行但所有列值与入参完全一致），业务上必须返回 200 OK + code = 0；(b) "步骤 3 到步骤 4 之间 pet 行消失"是节点 5 阶段无业务路径触发 DELETE pets 的理论不可达 race，即便发生仍按 `err == nil` ⇒ 200 OK 路径走，**不**降级为 1009 / 1003、**不**读 `RowsAffected`
5. **广播（fire-and-forget，仅当用户在房间时）**：service 层检查 `users.current_room_id`：
   - **非 null** → 调用 `BroadcastToRoom(currentRoomId, {type: "pet.state.changed", payload: {userId, petId, currentState}})`（详见 §12.3 `### 宠物状态变更（pet.state.changed）`）；广播失败仅 log warning，**不**回滚 UPDATE，**不**影响 HTTP 响应（与 Story 11.8 `member.joined` 广播失败语义一致）
   - **null** → 用户当前不在任何房间，**不**广播（无房间内成员需要被通知）
6. **响应**：返回 `data.state` = 入参 state 值（回显，client 用作 server-acknowledged 确认信号）；详见 §5.2.5 响应体字段表

**事务边界规则**：本接口**不**需要 MySQL 事务（仅 1 个 UPDATE 单语句，DB 引擎默认 autocommit；广播是 fire-and-forget 不在事务内）—— 与 §10.1 / §10.4 / §10.5 房间事务接口的多语句事务边界**不同**（参见数据库设计 §8.x 关键事务设计，本接口**不**入 §8.x 事务列表）。

**WS 广播 vs HTTP 响应的关系（含发起者自己的 self-broadcast 兜底规则）**：HTTP 200 是 server-acknowledged 入账确认信号；`pet.state.changed` WS 广播是事件通知信号。两者通过两条独立连接送达 client（HTTP 经 ApiClient；WS 经 WebSocketClient）；client 同房间内**会同时**收到自己的 HTTP 200 + 自己的 `pet.state.changed`（广播范围包含发起者自己，详见 §12.3 `### 宠物状态变更` 关键约束）。

- **对"别人的状态变化"的权威信号**：以 WS 广播 `pet.state.changed` 为**唯一权威信号**（与 CLAUDE.md §"工作纪律 / 状态以 server 为准"一致）；client **不**通过任何其他渠道驱动别人的 roster pet state 更新
- **对"发起者自己的状态变化"的权威信号（self-broadcast 例外，基于到达顺序的对称无操作）**：考虑到 self-broadcast 是 fire-and-forget 不重试、若该唯一信号丢失会让发起者 UI 永久 stale，契约层**允许**发起者在收到 HTTP 200 OK **或** self-broadcast WS 消息（取**先到的任一信号**）后**立即更新自己**的本地 roster pet state，**不等**另一路信号到达。按到达顺序对称展开：(a) HTTP 200 先到（典型）→ 立即用 `response.data.state` 更新 self entry，后续 self-broadcast 到达走字段级 merge → 值已相同 → no-op；(b) self-broadcast 先到（罕见但合法）→ 立即用 `payload.currentState` 更新 self entry（与"别人的广播"统一走 client merge contract 路径），后续 HTTP 200 到达比对后 → no-op；(c) 对称无操作不变量：任一路径先到的信号都立即驱动 UI，后到的信号是 no-op —— 两路信号 server 端来源同一次 `UPDATE pets ...` 写库行为，值必定相同，merge 幂等。详见 V1 doc §5.2 line 549-555 完整表述。
```

```markdown
> **设计权衡**：本接口 response data 选择**回显入参**（即 `data.state` = request `state`）而非**返回 server 端写入后的最终值**。原因：两者在 happy path 下完全等价（service 层步骤 4 UPDATE 入参值入库，无任何转换 / 截断 / 业务转换），让 client 反推没有信息增量；同时回显语义让 client 实装层（如 iOS Story 15.4）可以用 `response.data.state == request.state` 作为简单的 ack 校验，不需要额外解析 DB 写入结果。**HTTP `data.state` 的 self entry UI 驱动职责**：依据 V1 doc §5.2 line 549-555 的对称兜底规则，HTTP `data.state` 在 (a) 路径（HTTP 200 先到）中**承担 self entry UI 驱动职责**，client 实装层应立即用该值更新本地 self entry roster pet state，不等 self-broadcast 到达；在 (b) 路径（self-broadcast 先到）中 HTTP `data.state` 退化为 server 端入账成功的二次确认信号（值已与本地相同 → no-op）。**该规则仅对 self entry 生效**；对房间内其他成员（他人 entry）仍以 WS 广播为唯一权威信号（client 端没有"别人的 HTTP 200"信号到达）。后续若引入"server 端状态聚合"语义（如多端同时上报取最新），可由对应 epic 的契约 story 重新审视回显 vs 真实态的选择。
```

```markdown
#### 可能的错误码

| code | message | 触发条件 |
|---|---|---|
| 1001 | 未登录 / token 无效 | auth 中间件拦截（无 Authorization 头 / token 非法 / token 过期 / token 解析失败） |

| 1009 | 服务繁忙 | DB 异常（UPDATE 执行返回 `err != nil`，含 driver / 网络 / 约束冲突等）/ 内部 panic（见 Story 14.2 service 实装）。**不**包含 `RowsAffected == 0` —— 该分支在本接口下视为幂等成功（详见 §5.2 服务端逻辑步骤 4） |
```

**关键约束**：

- **1002 语义（唯一保留的请求侧错误码）**：1002 是**请求参数本身**不合法（`state` 字段缺失 / 类型非 int / 值不在 {1, 2, 3}），客户端应停止重试并修正参数。本接口**不**触发 1003 —— pet-less 账号（用户无活跃默认 pet 行）是与 §5.1 / §10.3 / §12.3 同语义的合法 edge case（contract 内显式覆盖，非 invariant 损坏），server 对 pet-less 走 server-acknowledged noop 路径返回 200 OK + code = 0；client 实装层（iOS Story 15.4）**不**需要为 pet-less state-sync 做 special-case suppress
- 用户不在房间（`users.current_room_id == NULL`）**不**视为业务错误 —— 接口仍返回 200 OK + code = 0，**不**广播 WS（与 §10.4 join 失败抛 6004 的语义**不同**：join 是要"操作房间"，无房间归属是前置不满足；state-sync 是要"更新 pet 状态"，无房间归属仅意味着无需广播给他人 —— pet 状态更新本身仍合法 + 仍写库）
- BIGINT 主键 / 外键（`userId` / `petId` 字段在 `pet.state.changed` payload 中）严格按 §2.5 全局约定字符串化下发；**本接口请求 / 响应 body 中无 BIGINT 字段**（仅 `state` 是 int 枚举），无字符串化诉求
- `state` 字段类型为 `number (int)`，**不**字符串化（枚举字段不受 §2.5 BIGINT 字符串化规则约束 —— 类似 §6.1 `motionState`）
- **HTTP 200 vs WS 广播的端到端语义（基于到达顺序对称无操作）**：HTTP 200 = server-acknowledged 入账成功；WS 广播 = 事件通知。两者通过两条独立连接送达，client 实装层不应**假设**两者同时到达 —— 可能 HTTP 200 先到（典型）/ WS 广播先到（罕见但合法）/ WS 广播完全收不到（连接断开 / fire-and-forget 失败）。**对 self entry**：依据 V1 doc §5.2 line 549-555，任一信号（HTTP 200 或 self-broadcast）先到都**立即驱动**本地 self entry 的 roster pet state 更新；后到的信号走字段级 merge → 值已相同 → **no-op**（两路信号 server 端来源同一次 `UPDATE pets ...` 写库行为，值必定相同）。**对他人 entry**：仅以 WS 广播为唯一权威信号驱动（client 端没有"别人的 HTTP 200"信号到达，HTTP 是 caller 自己请求的响应）。client 实装层（iOS Story 15.2 / 15.4）**必须**对两路信号到达顺序做幂等设计，**不**假设固定先后顺序，**不**因 self-broadcast 先到而忽略其 UI 驱动职责，也**不**因 HTTP 200 先到后让 self-broadcast 沦为"仅 ack/活性探测"

## Dev Notes

### 关键设计原则

1. **契约先行（contract first）**：本 story 是节点 5 第一个 Server epic 的开局，**禁止**先写 Go 代码再回头补文档。理由：iOS Epic 15（房间内成员宠物状态展示 + 自己状态上报）必须基于稳定契约并行开工，先实装后补文档会导致 iOS 端 Codable struct 与 server response 字段对不上 → 节点 5 demo 时双向返工。对标 Story 4.1 / 7.1 / 10.1 / 11.1 同模式。
2. **数据库 → API 单向映射**：API 字段语义由 `docs/宠物互动App_数据库设计.md` §5.3 pets 表 + §6.4 `pets.current_state` 枚举（1=rest, 2=walk, 3=run）推导，**不**反向修改数据库。如发现 API 需求与数据库 schema 不符（如某个字段长度不够），优先改 API 设计或后置到下一 epic 处理，**不**直接改 DB。
3. **单一职责 WS 消息**：`pet.state.changed` 仅广播 `currentState` 变化，**不**携带装备 / 头像 / 昵称等其他字段变化（装备变更走 Epic 27 独立路径）；保持每条业务 WS 消息单一职责，避免"全字段事件"的扩展复杂度。
4. **广播范围 = 包含发起者自己**（与 `member.joined` / `member.left` 排除发起者不同）：state 是事件本身，让 client 用统一路径处理（包括自己）；joined/left 是关系变化，发起者已通过 HTTP response 知道结果。这是节点 5 与节点 4 的显式协议差异，必须在 §12.3 关键约束 + §1 冻结声明里**显式**写明，避免 Story 14.4 实装时误抄 11.8 排除逻辑。
5. **server 不限频，client 自律节流**：state-sync 不是资产入账类接口（不像步数 §6.1 的 3001 防作弊），不需要 server 强制阈值；server 只走通用 rate_limit 60 次/分；iOS 端在动作识别状态机切换瞬间上报（self-imposed 节流策略），不每秒重复上报。这条约束在 §5.2 服务端逻辑段 + Dev Notes 里**显式**写明，避免 iOS Story 15.4 实装时主动加客户端 1Hz 节流。
6. **HTTP 200 vs WS 广播基于到达顺序的对称无操作**：HTTP 200 是 server-acknowledged 入账确认；WS 广播是事件通知。两者通过两条独立连接送达，可能乱序甚至 WS 完全收不到。**对 self entry**：任一信号（HTTP 200 或 self-broadcast）先到都**立即驱动**本地 self entry 的 roster pet state 更新；后到的信号走字段级 merge → 值已相同 → **no-op**（两路信号 server 端来源同一次 `UPDATE pets ...` 写库，值必定相同）。**对他人 entry**：仅以 WS 广播为唯一权威信号驱动（client 端没有"别人的 HTTP 200"信号）。client 实装层（iOS Story 15.2 / 15.4）**必须**对两路信号到达顺序做幂等设计，**不**假设固定先后顺序。这条约束 in §5.2 line 549-555（含 (a)/(b)/(c) 完整对称展开）+ §12.3 关键约束**两处**显式标注。
7. **fire-and-forget 广播失败不影响 HTTP 响应**：BroadcastToRoom primitive 失败仅 log warning，HTTP 仍 200 OK + code = 0；不回滚 DB UPDATE。原因：状态同步是 UI 展示，不是资产入账；广播失败的下游影响只是某些 client 暂时看不到状态变化，会在下次状态切换 / WS 重连 / room.snapshot 重新拉取时自动对齐。与 Story 11.8 `member.joined` / `member.left` fire-and-forget 语义一致。
8. **dev 端点 schema 不进 V1 文档**：节点 5 阶段 epics.md 未规划 pet dev 端点（如 `/dev/force-set-pet-state`），无跨端契约对齐需求；写进 V1 文档反而会让客户端开发者误以为是公共 API。
