**AC2 — §8.1 GET /api/v1/cosmetics/catalog schema 锚定**

V1 文档 §8.1 编辑后必须包含以下完整内容（**就地修改**，新增 H4 块紧接 `### GET /api/v1/cosmetics/catalog` 之后、现有 `#### 返回示例` 之前；现有 `#### 返回示例` JSON 示例保留并升级为 `#### 响应体` 下的示例）：

**§8.1 接口元信息**（新增 H4 块）：

| 字段 | 值 |
|---|---|
| HTTP Method | GET |
| Path | /api/v1/cosmetics/catalog |
| 认证 | **需要** Bearer token（auth 中间件） |
| 限频 | 默认（按 Story 4.5 rate_limit 默认值 60 次/分按 user_id 计） |
| 幂等 | 天然幂等（GET 查询，无副作用） |
| 节点 | 节点 8（Epic 23 落地 / Epic 24 客户端集成） |
| 分页 | **无**（约 15-50 条配置，不分页；与 epics.md §Story 23.3 AC 一致） |
| Query 参数 | **无**（不接受任何 query string） |

**§8.1 请求**（新增 H4 块）：无请求体（GET 接口）；仅需 `Authorization: Bearer <token>` Header。

**§8.1 服务端逻辑**（新增 H4 块）：

1. **认证 & 限频**：auth 中间件校验 Bearer token；rate_limit 中间件按默认配置（user_id 60/min）限频。
2. **查询配置目录**：`SELECT id, code, name, slot, rarity, icon_url, asset_url FROM cosmetic_items WHERE is_enabled = 1 ORDER BY rarity ASC, slot ASC, id ASC`。
   - 仅返回 `is_enabled = 1` 的配置（disabled 配置**不**返回）。
   - 排序：`rarity ASC` 优先，再 `slot ASC`，最后 **`id ASC`** 决定性 tie-breaker（与 epics.md §Story 23.3 AC 一致；`id ASC` 是 §1 AR18 约束下同 (rarity, slot) 多行场景保证全序确定的必需补充，client 可依赖整个列表顺序）。
3. **响应**：返回 `{items: [{cosmeticItemId, code, name, slot, rarity, iconUrl, assetUrl}]}`；列表大约 15-50 条，**不分页**。
4. **事务边界规则**：本接口**不**需要 MySQL 事务（仅 1 个 SELECT 查询）。

**§8.2 服务端逻辑**（新增 H4 块）：

1. **认证 & 限频**：auth 中间件校验 Bearer token；rate_limit 中间件按默认配置（user_id 60/min）限频。
2. **查询玩家实例**：`SELECT id, cosmetic_item_id, status FROM user_cosmetic_items WHERE user_id = ? AND status IN (1, 2)`。
   - 仅返回 `status = 1 (in_bag)` 与 `status = 2 (equipped)`；**不含** `status = 3 (consumed)`（合成已消耗）/ `status = 4 (invalid)`（无效，数据库 §6.10 定义但 inventory 不返回）。

3. **按 cosmetic_item_id 聚合**：把步骤 2 结果按 `cosmetic_item_id` 分组；每组关联 `cosmetic_items` 拿配置元信息（`name / slot / rarity / icon_url / asset_url`）。
   - 配置缺失的 `cosmetic_item_id`（§1 允许 admin 删 `cosmetic_items` 行，删后已拥有实例仍存）→ **保留该组（不 skip）+ 降级占位元信息（`name="未知装扮"` / `slot=99` / `rarity=1` / `iconUrl=""` / `assetUrl=""`，均在既有枚举值域内）+ log error**；用户已拥有实例不得因 admin 删配置静默消失（fix-review 23-1 r1 [P2]）。

6. **事务边界规则**：本接口**不**需要 MySQL 事务（只读查询；步骤 2 + 步骤 3 关联可用单 JOIN 或两次 SELECT，实装层 Story 23.4 决定，契约层不约束）。

### 字段语义唯一来源 = 数据库设计文档（DB → API 单向）

- `cosmetic_items`：`docs/宠物互动App_数据库设计.md` §5.8（`id / code / name / slot / rarity / asset_url(DEFAULT '') / icon_url(DEFAULT '') / drop_weight / is_enabled`）

- **关键**：DB `asset_url` / `icon_url` 是 `VARCHAR(255) NOT NULL DEFAULT ''` → API `assetUrl` / `iconUrl` **允许空字符串**（catalog 是全量配置，可能含未配图项）；这与 §7.2 reward.* "不允许空"是**有意差异**（reward 是抽中奖励，必须有图），AC5.3 要求显式标注理由防 review 误判为不一致。

### 易错点（review 高频命中，提前规避）

1. **聚合维度搞错**：groups 按 `cosmetic_item_id` 聚合（"小黄帽 x 3"是一个 group），**不是**按 instance 平铺、**不是**按 slot/rarity 聚合。
2. **status 过滤范围**：inventory 只返回 `status IN (1,2)`；DB §6.10 定义了 4 个枚举值，但 API 只暴露 1/2（3=consumed 已被合成消耗、4=invalid 无效都不返回）。字段表 `instances[].status` 约束必须写"枚举 {1,2}"不是 {1,2,3,4}。
3. **`count` 是否含 equipped**：含。`count` = instances 数组长度（in_bag + equipped 都算）。不要写成"仅 in_bag 计数"。
4. **空背包返回值**：`{groups: []}`，不是 `null` / `{}` / 省略字段。client Swift Codable 按非可选 `[InventoryGroup]` 解析。
5. **不反向改 DB / §7.x / §8.3 / §8.4 / §9.x**：本 story 只锚定 §8.1 / §8.2 + §1 追加；其他章节已锚定，只引用不修改。§8.3 / §8.4 是节点 9 Story 26.1 的（**不要**顺手锚定 equip/unequip）；§9.x 是节点 11 Story 32.1 的。
