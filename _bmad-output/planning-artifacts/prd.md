## Executive Summary

裤衩猫是一款 Apple Watch 原生的触觉社交宠物应用。用户手腕上住着一只极简风格的猫，它实时映射用户的身体状态——用户走路猫就走路，用户静坐猫就打盹，用户睡觉猫就蜷起来。猫不是虚拟宠物，不会饿死、不会生病、不评判用户。它是用户状态的情感化投射——一面可爱的镜子。

核心循环极轻：每挂机 30 分钟掉落一个猫咪礼物（皮肤组件），走 200-300 步即可打开，直接获得新皮肤。每日签到获得额外皮肤。皮肤不显示稀有度标签——只有好看不好看，没有等级焦虑。

杀手级功能是触觉社交：用户与好友配对后，双方的猫同屏显示。好友在走路，你手腕上好友的猫也在走——不需要打字、不需要发消息，一只在动的猫就代表"那个人还在"。用户可以点击好友的猫发送一次无言的触碰——手腕一震，对方就知道"你在想我"。不需要选表情、不需要打字，一次触碰就够了。在一个社交越来越重的时代，裤衩猫提供了最轻形态的人际连接。

目标用户是渴望微小正反馈的 Apple Watch 年轻用户，核心场景是异地情侣和办公室搭子。产品入口在 iPhone 伴侣 App（获客和传播），核心体验在手表（Complication 入口 + App 深度交互）。商业模式为免费下载 + 皮肤内购（iPhone 端支付）。

### What Makes This Special

**活的感觉。** 裤衩猫的猫不是循环播放的动画，是真实世界的因果投射。猫在走是因为你在走，好友的猫在动是因为好友此刻真的在动。这种"活"的体验是裤衩猫和所有其他手表宠物应用的根本区别。

**触觉社交。** 手表是离心脏最近的屏幕，也是最私密的数字空间。裤衩猫将社交压缩到了触觉层——穿越物理空间的手腕震动，比文字更轻、比语音更安静、比任何消息都更私密。这是 2026 年最稀缺的亲密感来源。

**零压力设计。** 永远没有惩罚——不签到猫不会变丑，不走路猫不会生病，盲盒错过了不会累积焦虑。只有正反馈，没有负反馈。用户不是被迫运动，而是获得一个"起身走几步的可爱理由"。

## Success Criteria

### User Success

- **首次尖叫时刻（<1分钟）：** 用户首次打开 App，看到猫跟随自己走路而动
- **社交尖叫时刻（<24小时）：** 用户配对好友后，首次收到手腕震动
- **收集满足感：** 7 天内收集到 10+ 款不同皮肤组件
- **零压力验证：** 用户从未因"错过盲盒"或"没走够步数"而感到焦虑

### Business Success

| 指标 | 3 个月目标 | 12 个月目标 | 预警线（需 pivot） |

| 30 日留存率 | ≥ 30% | ≥ 40% | < 15% |
| 皮肤内购转化率 | N/A（内购 Growth 阶段上线） | ≥ 5% | < 1% |
| ARPU（付费用户月均） | N/A（内购 Growth 阶段上线） | ≥ ¥25 | < ¥8 |
| App Store 评分 | ≥ 4.5 | ≥ 4.5 | < 3.5 |
| 自然获客占比 | ≥ 50% | ≥ 60% | < 20%（获客模型不可持续） |

### Technical Success

- 额外耗电 ≤ 10%（动画 ≤5% + 网络 ≤3% + 传感器 ≤2%）
- 动画帧率 ≥ 24fps（抬腕交互时）
- 好友状态同步延迟 < 30 秒
- App 启动到猫出现 < 2 秒
- 崩溃率 < 0.1%
- 手表端 App 包体 ≤ 30MB

### Measurable Outcomes

- 盲盒解锁率 ≥ 70%（掉落后被走步解锁而非忽略）
- 配对用户日均触觉互动 ≥ 3 次
- 病毒系数 K ≥ 0.5（每个用户邀请 ≥ 0.5 个新用户）

## User Journeys

### Journey 0：陌生人的前 60 秒（Onboarding）

**任何新用户，第一次打开裤衩猫。**

打开 App → 猫立刻出现，做出"镜像时刻"——映射用户当前状态（抬腕时猫也抬起爪子看）。无文字引导，无教程，无弹窗。用户自己发现"它在跟着我"。走几步，猫跟着动。放下手腕，下次抬腕猫还在。

第 3 天签到时，猫举着小牌子自然引导设置 Complication。不在第一天增加摩擦。

**揭示的能力：** 零引导 Onboarding 设计、镜像时刻动画、延迟引导机制

### Journey 1：小林的一天（核心用户 - 单机体验）

**小林，26岁，程序员，杭州。**

上午 10:15 坐了两小时，手腕震了一下——猫旁边有个礼物在闪。显示"230 步解锁"。起身接水绕走廊，回来手腕震动——猫兴奋跳起来，礼物打开：一顶厨师帽。猫举起来展示给小林看，存入库存（不自动穿戴，不打乱当前搭配）。

单盲盒机制：每次只有 1 个待解锁盲盒，解锁后才掉下一个。不堆积，不焦虑。

午饭路上瞟一眼手腕，猫在小跑。下午 Complication 上猫从坐着变成趴着（一小时没动了）。手表上左右滑动，从最近 5 套皮肤中快速换了一套。

**情绪弧线：** 无聊→微小惊喜→会心一笑→回归平静

**关键设计决策：**
- 盲盒不堆积，每次只有 1 个（零压力）
- 皮肤展示后存入库存，不自动穿戴
- 手表端支持最近 5 套皮肤快速切换

**揭示的能力：** 盲盒状态持久化、步数恢复机制（App 重启后 CMPedometer 补查）、单盲盒队列、手表快速换装

### Journey 2：小雨和男友的远距离触碰（社交用户）

**小雨，23岁，设计师，上海。男友小凯在深圳。**

周三晚 22:00，抬手看表——她的猫在打盹，小凯的猫也在打盹（状态约每分钟可感知变化，技术上每 30 秒 HTTP 轮询 + 动画插值平滑过渡）。她点了小凯的猫——手腕震了一下作为发送确认。三秒后手腕震回来——小凯回了一个触碰。

MVP 只有一种社交交互：**点击好友的猫 = 发送触碰。** 不选表情，不选类型。"我碰了你一下"——暧昧比精确更有力量。

第二天早上抬手，小凯的猫在走路——他已经在通勤了。好友状态超过 2 分钟未更新时，猫半透明 + 💤，表示"好友可能不在看手表"。

**关键设计决策：**
- MVP 好友状态 HTTP 轮询每 30 秒（Growth 升级 WebSocket），动画插值平滑过渡
- 社交互动只有"触碰"一种（Growth 阶段再加手势区分）
- 状态过期标记（半透明 + 💤）

**揭示的能力：** 近似实时同步 + 插值动画、单一触碰交互、状态过期标记、APNs 后台触发震动、按好友静音/频率限制/智能免打扰

### Journey 2b：邀请好友配对（全链路）

**小雨想邀请小凯一起用。**

iPhone 伴侣 App → 点击"邀请好友" → 系统分享面板弹出 → 选择微信发送 → 小凯收到一张精美卡片（猫的图 + "你的猫在等一个伴"）带链接 → 点击跳转 App Store → 下载安装 → 打开后自动配对完成。

全程从 iPhone 发起，不从手表发起。不做二维码。

**揭示的能力：** iPhone 端邀请链接生成、分享卡片设计（自传播素材）、深度链接自动配对

### Journey 3：阿毛的皮肤收集（收集用户）

**阿毛，28岁，设计师，北京。**

周六打开 iPhone App 翻皮肤库，37 款组件已收集。今天第 15 天签到，日历标着太空系列主题周——签到获得宇航员护目镜。

在 iPhone App 上搭配：护目镜 + 银色围巾 + 星星项链。猫在预览中会动（不是静态图）。确认后手表同步更新（WatchConnectivity 发送 → 手表恢复前台时检查更新 → 备用通道：服务端推送）。

和闺蜜喝咖啡比了比猫——闺蜜海盗套装，阿毛太空套装。晚上发现限定"樱花和服"¥12，买了。截屏发朋友圈（猫的截屏要好看，利于自传播）。

iPhone App 皮肤体验是核心留存和传播引擎，不是辅助功能。

**关键设计决策：**
- iPhone App 皮肤预览猫会动
- 手表-手机多通道皮肤同步（WCSession + 服务端备用）
- 截屏友好的视觉设计
- MVP 不做穿搭收藏夹（Growth 阶段加）

**揭示的能力：** iPhone App 动态皮肤预览/搭配、多通道同步、StoreKit 内购、签到主题周序列化内容、iPhone App 显示同步状态

### Journey 4：Zhuming 的运维日常（管理员）

**Zhuming，开发者兼运维。**

周一早上收到每日摘要（Go 脚本定时发送）：DAU 涨了 300，崩溃 0 次，盲盒解锁率 73%。设计师发来新皮肤素材——用 CLI 脚本上传到 CDN + 更新数据库，不需要发 App 新版本。

下午 Crashlytics 报警——某型号手表皮肤切换崩溃。查堆栈，修素材，推 CDN。想深挖数据时手动查 SQL。

MVP 不做后台管理界面，不部署 Grafana。CLI + SQL + 每日摘要脚本够用。

**关键设计决策：**
- 每日数据摘要脚本替代 Grafana
- CLI 工具替代后台管理界面
- 皮肤资源 CDN 热更新（不需要 App 更新）

**揭示的能力：** CDN 皮肤热更新、CLI 管理工具、每日摘要脚本、Crashlytics 集成

## Innovation & Novel Patterns

### Detected Innovation Areas

**1. 触觉社交语言（Haptic Social Language）**
将手表震动从"通知工具"重新定义为"人际连接通道"。用户通过手腕触觉传递情感——不是信息传递，是存在感传递。市面上没有任何 Apple Watch App 以触觉作为社交核心。Bond Touch 用专用手环验证了触觉社交的需求，裤衩猫将其带入已有 1 亿用户的 Apple Watch 生态。

与 Apple Digital Touch 的关键区别：Digital Touch 失败是因为入口太深、使用太独立（四步操作只为碰一下）。裤衩猫将触碰嵌入核心场景——你在看猫，好友的猫就在旁边，点一下就发送了。

**2. 活的映射（Living Mirror）**
宠物不是虚拟养成对象，是用户物理状态的实时因果投射。猫走路是因为你在走路，好友的猫在动是因为好友此刻在动。这打破了"虚拟宠物 = 喂养 + 成长 + 死亡"的三十年范式（从 Tamagotchi 到 Finch），创造了"宠物 = 你的镜子"的新品类。

技术上是"感知层面的创新"——每次抬腕映射准确即可，用户不关心中间是否暂停。状态过期标记（半透明+💤）保证诚信。

**3. 零对话社交（Zero-Conversation Social）**
所有社交产品都在让用户"说更多"。裤衩猫反其道而行：你什么都不用说。一只在动的猫 = "那个人还在"。一次手腕震动 = "我在想你"。这是社交产品的减法创新——在信息过载时代，最轻的连接反而最有力量。

裤衩猫不争夺通讯场景，而是占据"微信消息之间的空白时间"——不值得发消息，但又想要连接的时刻。

**设计原则补充：永远不做通讯工具——裤衩猫是陪伴工具，不是对话工具。永远不加文字/语音/图片通讯功能。**

### Market Context & Competitive Landscape

| 竞品 | 做到了什么 | 裤衩猫的差异 |
|------|-----------|------------|
| Bond Touch | 验证了触觉社交需求 | 不需要额外硬件，复用 Apple Watch |
| Tamagotchi / Finch | 验证了虚拟宠物+健康的结合 | 猫不会死，不评判，是镜子不是宠物 |
| Apple Digital Touch | 验证了手表触觉通讯 | 嵌入宠物场景，不是独立功能 |

| 泡泡玛特 | 验证了盲盒收集的社交货币价值 | 数字化，零成本分发，可社交展示 |

裤衩猫的独特定位：**上述所有产品验证过的需求碎片，第一次被组合在一个产品中。**

### Complication 设计规格

表盘上的猫不是动画，是通过 WidgetKit 时间线切换的静态插画（系统约每 15 分钟更新一次）。

**主 Complication：accessoryRectangular（最大可用区域）**

| 表壳尺寸 | 实际像素 |
|---------|---------|
| 41mm | ~300×130 px |
| 45mm | ~320×140 px |
| 49mm (Ultra) | ~340×150 px |

布局：左侧猫状态插画（~100×100 px）+ 右侧状态文字（"在走路"/"在打盹"/"💤"）。根据运动状态切换 3-5 种猫插画。

**辅助 Complication：accessoryCircular**

| 表壳尺寸 | 实际像素 |
|---------|---------|
| 41mm | ~100×100 px |
| 45mm | ~106×106 px |
| 49mm (Ultra) | ~110×110 px |

圆形裁剪，显示猫头像/极简猫图标。

**关键限制：**
- 不支持自定义动画（苹果系统级限制），只能静态图片
- 苹果米奇/玩具总动员表盘能动是系统特权，第三方无法做到
- "伪动画"策略：通过时间线切换不同姿态的猫插画，每次抬腕可能看到不同状态
- **必须为 Complication 专门设计像素级猫插画**，不是缩小 App 内的猫
- 点击 Complication → 跳转 App 全屏体验（猫真正会动的地方）

| 合规项 | 策略 |

### Implementation Considerations

**手表端架构：** MVVM + Repository Pattern，离线优先。SwiftData 本地持久化。SpriteKit 动画层与业务逻辑解耦。

**手表-手机通信：** WatchConnectivity 为主通道，APNs 为备用。手表端启动时始终从本地存储读取最新状态。iPhone App 换肤后显示同步状态（"已发送 ✓" / "手表不在附近，下次自动更新"）。

**电量预算：** 动画 ≤5% + 网络 ≤3% + 传感器 ≤2% = 总额外耗电 ≤10%。

**存储管理：** 皮肤本地最多缓存 10 套（当前+最近9套），LRU 淘汰。iPhone App 设置中提供"清理缓存"。

**兼容性：** iPhone App 检测配对手表版本，不兼容时显示友好提示。

## Functional Requirements

### 猫动画与状态映射

- FR1: 用户可以在手表上看到一只跟随自身运动状态实时变化的猫（走路/跑步/静坐/睡觉）
- FR2: 猫可以随机展现 2 种微行为动画（打哈欠、伸懒腰），Growth 阶段扩展到 5+ 种
- FR3: 用户首次打开 App 时，猫通过"镜像时刻"让用户无需文字引导即可理解猫在跟随自己。镜像时刻必须覆盖所有运动状态：走路→猫走路；静坐→猫抬头看用户（抬腕触发）；手腕转动→猫歪头。**验收标准：** 内测 10 名用户中 ≥8 名在首次打开后无提示自行描述出"猫在跟着我动"
- FR4: 用户可以在手表上通过左右滑动在最近 5 套皮肤之间快速切换
- FR5: 猫在 Always-On Display 模式下显示为静态简化状态图

### 盲盒与奖励系统

- FR6: 系统每 30 分钟自动掉落一个盲盒（单盲盒队列，当前盲盒解锁后才掉落下一个）
- FR7: 用户可以通过累积 200-300 步解锁盲盒。若盲盒持续 2 小时未解锁，提供替代解锁方式（旋转 Digital Crown 完成小游戏），确保久坐用户也能获得奖励
- FR8: 免费盲盒可在离线状态下解锁（客户端开奖，基于服务端下发的序列化礼物列表）；付费盲盒需联网解锁（StoreKit 交易验证）。采用序列化礼物制，非随机概率，规避 loot box 审核
- FR9: 盲盒开启时展示获得的皮肤组件（猫举起展示 + 触觉反馈震动），存入库存不自动穿戴
- FR10: 盲盒状态在 App 被系统终止后可以恢复（持久化存储）
- FR11: App 重启后可以从 CMPedometer 补查丢失的步数数据

### 签到系统

- FR12: 用户每日首次打开 App 即自动完成签到，获得皮肤组件。签到动画作为仪式感展示（猫举牌），但不要求额外点击操作。确保零压力——用户不会因"忘记点击"而错过签到
- FR13: 签到采用月累计制（非连续制），断签不影响进度
- FR14: 用户每月可以补签 2-3 次
- FR15: 签到可在离线状态下完成，联网后自动同步
- FR16: 签到从去重队列按序发放皮肤（不重复），主题周序列化内容放到 Growth 阶段

### 皮肤收集与管理

- FR17: 用户可以在 iPhone App 上浏览自己拥有的全部皮肤库存
- FR18: 用户可以在 iPhone App 上对猫进行多层自由搭配，分层规格如下：

  | 层（z-order 从底到顶） | 说明 | MVP 最小组件数 |
  |----------------------|------|---------------|
  | 1. 身体（Body） | 猫的基础体型+颜色 | 3（白/橘/灰） |
  | 2. 表情（Expression） | 眼睛+嘴巴组合 | 5 |
  | 3. 服装（Outfit） | 衣服/围巾等 | 8 |
  | 4. 头饰（Headwear） | 帽子/发饰等 | 8 |
  | 5. 配件（Accessory） | 项链/眼镜等 | 6 |

  MVP 独立组件总计 ≥30 件，组合数 ≥50 种。每层至少有 1 个"无"选项（裸猫可用）。渲染时按 z-order 从底到顶叠加。

- FR19: 用户在 iPhone App 皮肤预览中可以看到猫的动态效果（猫会动）
- FR20: 用户在 iPhone App 确认搭配后，手表端自动同步更新猫的外观。手表每次进入前台时主动检查是否有待同步的皮肤配置，恢复连接后 ≤10 秒完成同步
- FR21: 皮肤资源支持 CDN 按需下载，无需更新 App 即可获得新皮肤。皮肤资源带版本号，客户端启动时校验缓存版本。服务端可标记某皮肤版本为"已撤回"，客户端检测到后自动清除本地缓存并回退到默认皮肤
- FR22: 手表端本地缓存最多 10 套皮肤资源，超出按 LRU 淘汰

### 社交配对与同屏

- FR23: 用户可以在 iPhone App 生成邀请链接，通过系统分享面板发送给好友
- FR24: 好友点击邀请链接后可以自动完成配对（下载安装后自动关联）
- FR25: 配对后用户可以在手表屏幕上同时看到所有已配对好友的猫（MVP 最多 2 个好友 = 3 只猫同屏，Growth 最多 4 个好友）
- FR26: App 启动时自动拉取好友最新状态，前台期间 HTTP 轮询每 30 秒拉取好友状态（Growth 升级 WebSocket），状态切换动画插值平滑过渡
- FR27: 好友状态超过 2 分钟未更新时，好友的猫显示离线标记（半透明 + 💤）

### 触觉社交

- FR28: 用户可以点击好友的猫发送一次触碰，双方手腕同时震动
- FR29: 用户在 App 后台或未打开时也能通过推送接收好友触碰的震动
- FR30: 用户可以对单个好友设置静音
- FR31: 系统自动限制同一好友 10 分钟内最多推送 3 次触碰
- FR32: 当 iPhone 处于专注模式时，触碰通知自动降级为静默

### Complication（表盘入口）

- FR33: 用户可以在表盘上添加裤衩猫 Complication，显示猫当前状态的专用像素级插画
- FR34: Complication 根据猫的运动状态切换不同的静态插画（3-5 种）
- FR35: 用户点击 Complication 可直接打开裤衩猫 App

### iPhone 伴侣 App

- FR36: 用户可以通过 Sign in with Apple 登录
- FR37: 用户可以管理好友列表（查看、删除、解除配对、屏蔽、举报）。"解除配对"为中性操作（双方猫消失但保留重新配对能力），"删除"为强操作（历史互动数据不再显示）。30 天零互动的配对关系自动变为"非活跃"状态，不再推送好友猫状态更新，用户可手动恢复
- FR38: 用户可以调整通知偏好设置（好友触碰/盲盒掉落独立开关）
- FR39: iPhone App 显示手表端皮肤同步状态（"已发送 ✓" / "手表不在附近，下次自动更新"）
- FR40: iPhone App 检测配对手表的系统版本，不兼容时显示友好提示
- FR41: 用户可以在 iPhone App 查看个人简要统计（今日步数、盲盒数、皮肤收集数、触碰次数）

### 后端与运维

- FR42: 管理员可以通过 CLI 工具上传新皮肤资源到 CDN 并更新皮肤池配置
- FR43: 系统在 App 启动时下发最新序列化礼物列表（预分配的盲盒内容序列），客户端缓存供离线使用。**一致性模型：** 服务端维护序列消费位置为权威。客户端离线开奖时预消费序列，联网后上报；若服务端检测到位置冲突（如另一设备已消费同一位置），以服务端为准，客户端获得的皮肤视为"待确认"，联网后重新校准
- FR44: 系统每日自动生成数据摘要（指标范围见 FR56）
- FR45: 系统对盲盒开奖结果进行事后审计，检测统计异常。响应链：异常检测 → 日摘要标红告警 → 管理员 CLI 确认 → 可选操作（回收非法获得的皮肤 / 标记账号 / 忽略）

### 信任与安全

- FR46: 用户可以屏蔽任意好友（屏蔽后双方猫互不可见，触碰不可达）
- FR47: 用户可以举报好友的不当行为
- FR48: 用户可以设置免打扰时段
- FR49: 用户可以在 iPhone App 中删除自己的账号和所有关联数据。边界情况：(a) 删除后对方好友列表中该用户显示为"已注销用户"，猫消失；(b) 该用户发出的触碰统计从对方数据中匿名化（计入总数但不显示来源）；(c) 删除有 30 天冷却期，期间用户可通过 Sign in with Apple 重新登录撤销删除
- FR50: 皮肤 UI 中不显示稀有度等级文字、图标或颜色分级。所有皮肤卡片使用统一视觉样式，无获取难度区分
- FR51: 盲盒掉落时系统发送本地通知并触发手腕震动提醒用户
- FR52: 用户发送触碰后，自己手腕收到一次确认震动（表示已发送）
- FR53: iPhone App 生成邀请好友时附带精美分享卡片（猫的图+文案），适合社交平台传播
- FR54: 第 3 天签到时猫自然引导用户设置表盘 Complication（P1，延迟引导）
- FR55: 系统追踪邀请链接的生成、点击、下载、配对完成全链路转化数据

### 数据统计

- FR56: 系统统计并汇总以下指标供日摘要和分析使用：DAU、配对率、皮肤收集数分布、Complication 使用率、邀请转化率、盲盒解锁率、触碰频率
- FR57: 皮肤掉率随用户收集进度动态调整——已拥有皮肤的重复掉率降低，未拥有皮肤的掉率提升，目标中度活跃用户 60 天收集 80% 独立组件
- FR58: 用户更换设备后通过 Sign in with Apple 登录即可恢复云端数据（好友、库存、签到）；本地临时数据（当前盲盒进度）从服务端最新状态重建
- FR59: 无好友用户可看到一只"系统猫"（随机行为的 NPC 猫）陪伴在旁，模拟社交氛围，降低冷启动期的孤独感。系统猫不可触碰发送消息，仅提供视觉陪伴
- FR60: 手表电量 ≤20% 时自动进入低电量模式——停止 SpriteKit 动画渲染，Complication 保持更新，猫显示为静态图。用户可在设置中关闭此行为
- FR61: iPhone-only 好友（无 Apple Watch）可通过 iPhone App 查看好友猫状态、接收触碰推送通知（无手腕震动），发送触碰通过 App 内按钮。邀请流程中明确提示"需要 Apple Watch 才能获得完整体验"
- FR62: iPhone App 皮肤库页面显示收集进度条（已收集 X/Y）。凑齐同系列 ≥3 件组件时触发"套装发现"彩蛋动画（猫换上全套做特殊动作），给予成就感但不显示稀有度标签
- FR63: 皮肤资源版本校验——客户端启动时比对本地缓存版本与服务端最新版本，若服务端标记为"已撤回"则自动清除本地缓存并回退默认皮肤，防止有 bug 的资源持续使用

### 错误状态与边缘场景

以下场景需要明确的用户反馈设计（视觉 + 触觉）：

| 场景 | 用户看到什么 | 触觉反馈 |
|------|------------|---------|
| 触碰发送失败（网络中断） | 好友猫上方短暂显示"❌"标记，1 秒后消失 | 不震动（乐观 UI 不适用于触觉社交——错误反馈必须明确） |

| 蓝牙断连（手表-手机） | iPhone App 同步状态显示"手表不在附近"，手表端正常运行（离线模式） | — |

## Design Guide（设计指南，供设计师参考）

### 猫的性格

所有猫统一一种性格——**就是猫**。慵懒但偶尔好奇，傲娇但偶尔粘人，自来熟又忽然高冷。不讨好用户，不卖萌求关注。它只是碰巧住在你手腕上的一只猫。

性格在动画中的体现：
- 被触碰时不是立刻回应，而是先看一眼，再慢慢转过来
- 盲盒掉落时假装不在意，然后"不经意"地走过去
- 打哈欠不是因为无聊，是因为猫就是爱打哈欠
- 好友的猫出现在屏幕上时，两只猫不一定立刻互动，可能各自待着

### 视觉风格

**参考：黄油猫（Butter Cat）**

- 线条简洁、圆润，填充色块为主
- 暖色调，柔和不刺眼
- 表情极简——眼睛和嘴巴的微小变化传达大量情绪
- 身体比例偏 Q 版（头大身子小）
- 皮肤组件风格统一：同样的线条粗细、同样的色彩饱和度

### 设计原则（动画相关）

- **一瞥即懂：** 猫的状态在 0.5 秒内可被识别（走路/坐着/睡觉一眼就能分辨）
- **克制不炫技：** 动画简洁不花哨，手表屏幕上信息量要少
- **截屏友好：** 任何时刻截屏都好看，猫在任何状态下都值得分享

### 多猫同屏布局规则

| 猫数量 | 布局 | 点击区域 |
|--------|------|---------|
| 1 只（仅自己） | 居中偏下 | 点击猫 = 无操作（或弹出状态信息） |
| 2 只（自己 + 1 好友） | 自己的猫左侧，好友猫右侧 | 点击好友猫 = 发送触碰 |
| 3 只（自己 + 2 好友） | 自己的猫居中偏下，两只好友猫分列左上、右上 | 点击对应好友猫 = 发送触碰给该好友 |

- 自己的猫始终最大（约占屏幕 40%），好友猫稍小（约占 25%）
- 猫之间不重叠，各自有独立点击区域
- 快速换装（左右滑动）只在单猫模式或从屏幕边缘滑入时触发，避免与点击好友猫冲突
- 好友猫带名字标签（好友昵称，极小字体显示在猫下方）

## Cat Animation State Machine（猫状态机转换规则）

### 状态定义

| 状态 | 动画 | 触发条件 |
|------|------|---------|
| **idle（静坐）** | 猫坐着，偶尔眨眼 | 默认状态，CMMotionActivity = stationary |
| **walking（走路）** | 猫四脚走路 | CMMotionActivity = walking，持续 ≥ 3 秒（防抖） |
| **running（跑步）** | 猫快速奔跑 | CMMotionActivity = running，持续 ≥ 3 秒 |
| **sleeping（睡觉）** | 猫蜷起来闭眼，有呼吸起伏 | idle 持续 ≥ 30 分钟 且 时间在 22:00-7:00 |
| **micro_yawn（打哈欠）** | 微行为：张嘴打哈欠 | idle 状态下随机触发，平均每 5 分钟一次 |
| **micro_stretch（伸懒腰）** | 微行为：前爪伸展 | idle 状态下随机触发，平均每 8 分钟一次 |

### 状态转换规则

```
idle ──(walking detected 3s)──→ walking
idle ──(running detected 3s)──→ running
idle ──(30min + nighttime)────→ sleeping
idle ──(random ~5min)─────────→ micro_yawn → idle
idle ──(random ~8min)─────────→ micro_stretch → idle

walking ──(stationary 10s)────→ idle
walking ──(running detected)──→ running

running ──(walking detected)──→ walking
running ──(stationary 10s)────→ idle

sleeping ──(wrist raise)──────→ idle（醒来动画过渡）
```

### 好友猫状态机

- 好友猫使用相同的状态定义（idle/walking/running/sleeping）
- 状态来源：客户端每 30 秒 HTTP 轮询好友的 CMMotionActivity 状态（与 FR26 一致）
- 好友猫**不触发微行为**（节省渲染资源，MVP 简化）
- 状态切换使用插值动画平滑过渡（0.5 秒渐变）

## Non-Functional Requirements

### Performance

- 动画帧率：用户抬腕交互时 ≥ 24fps，非交互时可降至 15fps。**测量：** Xcode GPU Report + XCTest Performance Metric，内测阶段 Series 6/8/Ultra 各测 3 次
- 启动速度：App 启动到猫出现 < 2 秒（冷启动）。**测量：** XCTest Performance Metric `measure(metrics: [XCTOSSignpostMetric.applicationLaunch])`
- 触碰延迟：用户发送触碰到对方收到震动 < 5 秒（网络正常时）。**测量：** 服务端日志 timestamp diff（发送→APNs 回执）
- 皮肤切换：手表快速换装响应 < 0.5 秒（本地缓存命中时）。**测量：** SpriteKit 纹理切换到首帧渲染完成的时间戳差
- 电量消耗：全天使用（16 小时唤醒、约 80 次抬腕）额外耗电 ≤ 总电量 10%（动画 ≤5% + 网络 ≤3% + 传感器 ≤2%）。**测量：** Xcode Energy Gauge + 真机 8 小时连续佩戴测试
- App 包体：手表端 ≤ 30MB（含基础皮肤），iPhone 端 ≤ 50MB。**测量：** App Store Connect 包体报告
- 内存占用：手表端 App 运行时内存 ≤ 50MB。**测量：** Xcode Memory Debugger，3 只猫同屏 + 10 套皮肤缓存场景
- 皮肤同步：手表恢复前台/连接后 ≤10 秒完成 iPhone 端待同步的皮肤配置更新。**测量：** WatchConnectivity 日志 timestamp diff（iPhone 发送 → 手表确认接收）

### Security

- 传输加密：所有网络通信使用 HTTPS / WSS（TLS 1.2+）
- 认证：JWT Token，过期时间 ≤ 24 小时，支持自动刷新
- 健康数据：步数和运动状态仅本地处理，不上传原始健康数据到服务端
- 数据存储：服务端仅存储聚合数据（日步数总计），不存储运动轨迹或详细健康记录
- 账号删除：用户请求删除后 30 天内清除所有关联数据（GDPR + PIPL 合规）
- 社交隐私：好友之间仅可见猫的运动状态，不可见具体位置、步数、健康数据
- PIPL 合规（中国《个人信息保护法》）：(a) 服务端部署在中国大陆境内，用户数据不出境；(b) 隐私政策覆盖 PIPL 要求的告知同意条款（收集目的、处理方式、第三方共享范围）；(c) Apple ID、社交关系图谱、行为数据（触碰频率、在线时间）均按个人信息处理；(d) 提供个人信息查询/导出/删除功能（FR49 已覆盖删除）

### Scalability

- MVP 目标：支持 5,000 DAU / 2,000 同时在线
- 爆发目标：架构可在 24 小时内扩容到 50,000 DAU / 20,000 同时在线
- WebSocket 连接：单 Go 实例支持 ≥ 10,000 并发连接
- CDN：皮肤资源全球 CDN 分发，单资源下载 < 3 秒
- 数据库：PostgreSQL 单实例支撑 MVP，Growth 阶段预留读写分离

### Accessibility

- 触觉备选：每种触觉反馈都有对应的视觉动画反馈
- 字体大小：遵循 watchOS Dynamic Type，支持系统级字体大小调整
- 色彩对比：Complication 插画和 App 内文字满足 WCAG AA 标准（4.5:1）
- VoiceOver：关键交互元素支持 VoiceOver 标签

### Reliability

- 服务可用性：后端 ≥ 99.5% 月可用率
- 崩溃率：手表端 + iPhone 端 < 0.1%
- 数据一致性：离线操作联网后最终一致，冲突时以服务端为准
- 降级策略：后端不可达时，单机核心体验（猫动画+盲盒+签到）100% 可用
